## Context

After `git worktree add` returns success, the main process immediately spawns an agent terminal in the new worktree directory. node-pty uses that path as `cwd`, which must exist on disk when the PTY starts.

## Problem

Intermittent ENOENT failures when spawning agents in freshly-created worktrees. The agent PTY would fail to start because its `cwd` did not yet exist at the moment `node-pty` called into the OS.

## Root cause

`git worktree add` returning success does not guarantee the directory is visible to `fs.access` on the same process. Filesystem metadata caches, APFS/NTFS directory visibility latency, and cwd normalization differences between git (which may resolve symlinks differently) and Node's fs layer can produce a small window where the path exists to git but `fs.access` returns ENOENT. Spawning the PTY within that window throws.

## Lesson

After any shell-out that creates a directory you are about to `cwd` into, verify the path with an exponential-backoff `waitForPathExists` helper before proceeding. Normalize to absolute paths before checking to avoid cwd mismatches between the child process and the verifier. Fail fast on EACCES/EPERM/ENOTDIR (these are not transient), but retry on ENOENT. Always `unref()` the retry timers so they do not keep the process alive during shutdown.

## Applies when

- Spawning a process whose `cwd` was just created by another process (git worktree add, mkdir via shell, clone, etc.)
- Seeing intermittent ENOENT from `node-pty` `spawn` on paths the caller believes exist
- Writing any "create then use" sequence that crosses a shell boundary
