## Context

`hardenedGit.ts` creates a git instance with credential helpers, SSH config, and interactive prompts stripped — designed for safe background operations like status polling.

## Problem

The clone dialog would hang permanently on "Starting clone..." when cloning a private repository. The cancel button had no effect during the hang.

## Root cause

`HARDENED_GIT_CONFIG` strips `credential.helper`, `core.askpass`, and `core.sshCommand` from all git operations. For clone, this means git can't authenticate — but without `GIT_TERMINAL_PROMPT=0`, git blocks indefinitely waiting for interactive terminal input that can never arrive in an Electron context. The hardened config was correct for background polling but wrong for user-initiated clone.

## Lesson

The hardened git config is a security feature for passive background operations. For user-initiated git operations (clone, push, pull) that legitimately need credentials, use a separate `gitForClone()` helper that:

1. Sets `GIT_TERMINAL_PROMPT=0` so git fails fast instead of blocking on interactive input
2. Preserves credential helpers and SSH config from the system environment
3. Accepts an `AbortSignal` for cancellation and enforces a hard timeout (5 min)

Never use the hardened git instance for operations that require authentication. The distinction: hardened = background/passive, authenticated = user-initiated/explicit.

See also PR #4948 which formalised this as `createAuthenticatedGit()`.

## Applies when

- Any new git operation in `projectCrud.ts` that a user explicitly triggers (clone, push, pull, fetch)
- Any git handler that could block on auth input in an Electron context
- Wiring cancellation for long-running IPC operations
