## Context

The project switcher list passed `selectedIndex` as a prop to each `ProjectListItem`, alongside an `items.indexOf()` call inside each item to determine if it was selected.

## Problem

Every arrow-key press changed `selectedIndex`, which caused every list item to re-render (even items whose selection state hadn't changed). Each item also instantiated its own `TooltipProvider`, meaning 45–60 `TooltipProvider` instances were recreated on each keystroke.

## Root cause

1. `selectedIndex` as a prop is a numeric value that changes on every keypress, so React's default reference equality check always fires a re-render for every item in the list.
2. `items.indexOf(item)` inside the render was an O(n) scan per item per render, making the full render O(n²).
3. `TooltipProvider` was mounted inside each list item rather than being hoisted to wrap the palette once.

## Lesson

For virtualized or long lists where selection changes frequently:

1. Pass `isSelected: boolean` instead of `selectedIndex: number` — the boolean only changes for two items (the one gaining selection and the one losing it), not all n items.
2. Wrap list items in `React.memo` with a comparator that checks `isSelected` and the item identity.
3. Pre-compute derived display values (`displayPath`, status booleans) in the data model or hook, not inside each item's render function.
4. Hoist context providers (`TooltipProvider`, `ThemeProvider`, etc.) to the list container level, not inside list items. Provider instantiation per item is expensive at scale.

## Applies when

- Any list with keyboard-driven selection (palette, command menu, autocomplete)
- Any component tree where a frequently-changing parent prop flows into many leaf components
- Nested Radix UI components — check whether providers can be hoisted
