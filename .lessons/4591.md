## Context

The project switcher palette was redesigned to show modifier key hints in a footer bar: `↵ Switch`, `⌘↵ New window`, `⌥↵ Background`. The hints update live as the user holds modifier keys.

## Problem / Lesson

The new `useModifierKeys.ts` hook reveals a subtle browser event gotcha: **modifier keys can get "stuck"** if the user switches windows (Cmd+Tab, for example) while holding a modifier key. The `keyup` event never fires for the modifier because focus was transferred to a different window.

## Root cause

`keydown` sets `cmd: true` or `alt: true`, but `keyup` is only received by the document that has focus. If the user presses `Cmd+Tab` while the palette is open, the window loses focus mid-modifier and the `keyup` for `Cmd` never arrives in the Canopy renderer — leaving `cmd: true` stuck in state.

## Fix

The hook adds a `blur` event listener on `window`. On blur, it resets all modifier state to `false`. This is the standard pattern for preventing stuck modifiers in web UIs.

```ts
window.addEventListener("blur", resetModifiers);
```

## Lesson

- Any hook that tracks modifier key state must reset on `window blur`, not just on `keyup`. Without this, Alt/Cmd/Shift can appear permanently held after the user Cmd+Tabs away and back.
- This applies to any UI showing modifier-dependent hints (modifier-key shortcuts, contextual tooltips, drag-and-drop operations).
- Test the blur reset: `window.dispatchEvent(new Event('blur'))` should zero all modifier flags.

## Applies when

- Building keyboard shortcut hint UIs (tooltips, footer bars, overlays)
- Drag-and-drop with modifier-key modes
- Any React hook that tracks `keydown`/`keyup` for modifier keys
