## Context

Canopy applies per-terminal-kind multipliers to the base scrollback line count. Agent terminals used a 2.5× multiplier and regular terminals used 0.5×.

## Problem

The multipliers looked modest on paper, but the actual memory cost is: lines × columns × bytes-per-cell. At 160 columns, each scrollback line costs ~1.5–2 KB. With 12 terminals (mostly agents at 2.5× the 1000-line base = 2500 lines each), scroll buffers alone consumed 50–60 MB across processes.

## Root cause

The multipliers were set based on "agents need more context" reasoning without accounting for the cumulative memory across a typical multi-terminal session. The formula wasn't wrong — it was just never stress-tested against a realistic workload of 10+ concurrent agent terminals.

## Lesson

When sizing buffer limits for multi-instance scenarios, multiply out the full cost: `limit × instances × bytes_per_unit`. For xterm scrollback:

- `lines × columns × ~1.5 KB per line`
- With 12 terminals at 2500 lines × 160 cols ≈ 60 MB just in scroll buffers

The fix: agent multiplier 2.5× → 1.5× (1500 lines), regular 0.5× → 0.3× (300 lines). Still exceeds VS Code's 1000-line default while being 40% cheaper.

Also note: users who need deeper history can increase the base scrollback setting, so the default doesn't need to cover the worst case.

## Applies when

- Setting any per-instance buffer/cache limit that will multiply across many concurrent instances
- Sizing xterm scrollback, shell history depth, or any per-terminal buffer in an app where 10+ terminals can run simultaneously
