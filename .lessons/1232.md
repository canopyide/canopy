## Context

The workspace host utility process sends WorktreeSnapshot objects to the main process via MessagePort.postMessage. Snapshots were assembled ad-hoc from live service state that included class instances (AdaptivePollingStrategy, NoteFileReader), closures, and NodeJS.Timeout handles. Worktrees started failing to load with no obvious error because the IPC silently broke.

## Problem

structuredClone (used internally by postMessage) throws DataCloneError when it encounters functions, class instances with unclonable internals, or timer handles. In a utility process this can bubble up as a crash or a dropped message, and the caller sees "nothing arrived" with no stack pointing at the offending field. Triple-nested fallbacks were needed to keep the host alive when a send still failed.

## Root cause

MonitorState and related service objects held references to live non-plain objects. Adding a new field to any tracked record could at any time drag an unclonable reference into the snapshot path without a type error, because the compile-time types only described the intended shape.

## Lesson

Any object that crosses a MessagePort or utility-process boundary must be deep-cloned to plain JSON before it leaves. Do this centrally at the point-of-send — not at each call site — and wrap the actual postMessage in defensive try/catch so one bad payload can't kill the host. The `shared/utils/serialization.ts` ensureSerializable helper exists for this reason; use it for new IPC surfaces rather than trusting that "it looks like a plain object".

## Applies when

Adding new fields to WorktreeSnapshot / any cross-process state payload, introducing a new MessagePort channel, or refactoring services whose state is later serialized. Also when debugging "IPC payloads arrive empty / don't arrive at all" in the workspace or pty hosts.
