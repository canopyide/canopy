## Context

External editor detection failed when users configured a macOS `.app` bundle path (e.g., `/Applications/IntelliJ IDEA.app`) instead of a CLI binary name.

## Problem

`EditorService.findEditorBinary()` only checked PATH-based binary lookup. macOS app bundles follow the `Contents/MacOS/<executable>` convention — the CLI binary is buried inside the bundle, not on PATH by default. Users naturally configure app paths from `/Applications` rather than digging for CLI binaries.

## Root cause

The detection logic was written for Linux/Windows conventions where editors are binaries on PATH. macOS bundles are directories ending in `.app` that contain their executables at a known sub-path.

## Lesson

On macOS, external application detection must handle two cases: (1) plain binary names looked up via PATH (e.g., `code`, `vim`), and (2) `.app` bundle paths found in `/Applications` or `~/Applications`. For bundles, resolve the CLI via `<bundle>/Contents/MacOS/<appname>`. Add `findInApplicationDirs()` that scans standard macOS app directories before falling back to PATH. Always add the bundle resolution step before PATH lookup.

## Applies when

- `EditorService.findEditorBinary()` and any new editor detection logic
- macOS-only: any feature that needs to find applications by path
- "External editor not found" errors on macOS when the app is clearly installed
