## Context

Browser panels and other panel types would jump to different grid positions when switching projects and returning.

## Problem

During hydration, panels are replaced with fresh instances that receive new IDs. The terminal registry's ordering array still referenced the old IDs. When old IDs were no longer found, the store fell back to default ordering — losing the user's saved arrangement.

## Root cause

Hydration creates a semantic break: old IDs → new IDs with same labels. The ordering array was ID-based but hydration didn't remap it. Any ID-based data structure must be treated as stale after hydration and remapped using stable identifiers (labels, panel kinds, etc.).

## Lesson

After hydration that replaces panel instances (project switch, app startup), use **label matching** to remap saved ordering arrays to new IDs. Save a `savedOrder` snapshot before hydration begins, then run `restoreTerminalOrder(savedOrder, newPanels)` that matches on label. This pattern must be applied in both `useAppHydration` and `useProjectSwitchRehydration` — any path that replaces panels.

## Applies when

- Project switch rehydration (`useProjectSwitchRehydration`)
- App startup hydration (`useAppHydration`)
- Any panel ordering state that must survive ID changes across hydration
