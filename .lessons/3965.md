## Context

When an AI agent asks for tool approval (permission to run a command, write a file, etc.), Canopy should detect this as "waiting for input" within 500ms via the fast-path. Each agent has per-agent `promptHintPatterns` but none covered approval flows — only the standard chat prompt was matched.

## Problem

All five supported agents (Claude Code, Gemini CLI, Codex CLI, OpenCode, Cursor Agent) have tool approval flows that pause execution and display a permission prompt. Despite different TUI frameworks, they share a consistent vocabulary:

- `[y/N]`, `[Y/n]`, `(y/n)` — shell-style confirmation indicators
- `Approve`, `Allow`, `Deny`, `Reject` — permission action words
- `allow once`, `allow always` — Gemini/Claude scope qualifiers

Without matching these, the `PromptDetector` found no match and the `ActivityMonitor` fell through to the 6-second `IDLE_DEBOUNCE_MS` timeout. The agent appeared stuck in "working" state for up to 6 seconds when it was actually waiting.

## Root cause

Agent detection configs were written to recognize working state and generic chat prompts, but approval flows were not considered. The universal vocabulary was only discovered by auditing all five agents' TUI output side-by-side.

## Lesson

Approval prompt vocabulary is highly consistent across agent CLIs regardless of TUI framework. A `UNIVERSAL_PROMPT_HINT_PATTERNS` fallback layer covering `[y/N]`, `Approve`, `Allow`, `Deny` catches approval flows from all current and future agents without per-agent configuration. Universal patterns should be checked after agent-specific patterns so specificity is preserved. Any new agent integration should be audited for both its chat prompt pattern AND its tool approval prompt pattern.

## Applies when

Agents appear stuck in "working" when displaying a tool/command approval prompt. Also relevant when adding a new agent — check approval prompt vocabulary before shipping.
