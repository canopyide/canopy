## Context

`mergeFetchedWorktrees` was called on every git status poll cycle (every few seconds per worktree). It correctly preserved individual worktree object references via `worktreeStatesEqual` — if fetched data matched the existing entry, the existing object was kept. However, the function always returned `new Map(...)`, so the Map container itself was always a new reference.

## Problem

The new Map reference propagated to all 12+ consumers of `useWorktreeDataStore`. `useWorktrees()` ran its `useMemo` sort and produced a new sorted array, causing `SidebarContent`, `ContentGrid`, `AgentButton`, `QuickSwitcher`, `useAgentLauncher`, and 6+ other components to re-render on every poll — even when nothing had changed.

## Root Cause

The function built a fresh Map unconditionally (`new Map(fetchedStates.map(...))`) then merged metadata from the existing Map. Even with individual entries preserved by reference, there was no final check comparing the result Map against the existing Map before returning. A new container object triggers Zustand's default equality check (referential equality), causing all subscribers to re-render.

## Lesson

After building a merged Map, add an identity check before returning:

```typescript
// If size changed, definitely different
if (result.size !== existingMap.size) return result;
// Check every entry is reference-equal
for (const [key, value] of result) {
  if (existingMap.get(key) !== value) return result;
}
// All entries identical — return the original Map to preserve reference
return existingMap;
```

This is O(n) where n is the number of worktrees (typically 5-10), so the overhead is negligible compared to the render cascade it prevents.

The identity check must account for the `issueAssociations` override path — entries modified after the main merge must still trigger a new Map return.

`useWorktreeDataStore` subscribers use referential equality by default (no `useShallow`), so the Map reference is the sole signal for re-renders. Returning the existing Map when contents are unchanged eliminates all no-op re-renders.

## Applies when

- Any Zustand store that stores a `Map` and updates it periodically from polling
- Any `merge*` function that builds a new collection from existing + fetched data
- Performance profiling shows excessive re-renders during polling cycles
