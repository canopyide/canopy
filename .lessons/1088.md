## Context

Canopy users were losing terminal sessions by hitting Cmd+R (muscle memory from browsers). Reload is genuinely destructive here — it tears down xterm instances, IPC channels, and in-flight agent state. We wanted reload accessible from the menu for debugging but not via keyboard.

## Problem

Setting `{ role: "reload" }` on a menu item injects the default Cmd+R accelerator. Adding `accelerator: ""` or `accelerator: null` does not remove it — the role wins. Users still triggered reload accidentally.

## Root cause

Electron roles bundle a label, click handler, AND default accelerator as a single unit. The accelerator is baked in and cannot be overridden while the role is set. The only way to opt out of the shortcut is to drop the role entirely and write an explicit menu item with your own click handler.

## Lesson

For any destructive or sensitive action where you want the menu entry but not the keyboard shortcut, do not use `role`. Write it as a plain menu item with an explicit `click` handler and no `accelerator`. Watch out for the same issue with `reload`, `forceReload`, `quit`, `close`, `toggleDevTools` and similar roles — they all have default accelerators. When writing the click handler, use a helper that resolves the target window safely (the `browserWindow` argument can be undefined when the menu is triggered without a focused window) and type-guards against destroyed windows. Keep reload reachable from the menu for debugging but never from a single accidental keystroke.

## Applies when

Adding or auditing any Electron menu item where the default keyboard shortcut would be dangerous, any time you want to keep menu functionality but drop an accelerator, or investigating "why can't I remove Cmd+R from my reload item".
