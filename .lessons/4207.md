## Context

The PtyHost UtilityProcess called `global.gc()` in three places: after `kill-by-project`, after `graceful-kill-by-project`, and in `ResourceGovernor.engageThrottle()` when heap utilization exceeded 85%. The `--expose-gc` flag was passed to the UtilityProcess to enable this.

## Problem

Forced V8 GC (`global.gc()`) causes 200–400ms full main-thread pauses in the PtyHost. During this pause, no terminal I/O is processed — terminals visibly "freeze". Additionally, forced full GC fragments the V8 heap's old space: memory that would be efficiently compacted by V8's incremental GC becomes scattered across non-contiguous pages that cannot be returned to the OS. This is worse than letting V8 manage collection autonomously.

## Root cause

The calls were added with the intent to reclaim memory after killing terminals and during memory pressure. However, V8's generational GC handles this automatically during the next incremental sweep. Forced `global.gc()` in a UtilityProcess is especially harmful because the PtyHost is single-threaded — any GC pause directly blocks terminal I/O for all open terminals simultaneously. The ResourceGovernor's terminal-pause mechanism already reduces allocation pressure, giving V8 time to collect without forcing a disruptive full sweep.

## Lesson

Do not call `global.gc()` in the PtyHost or any latency-sensitive UtilityProcess. The correct response to memory pressure in PtyHost is pausing terminal I/O (which reduces allocation rate) and letting V8's incremental GC collect freed objects naturally. Forced GC in V8 causes observable latency spikes proportional to heap size and is documented to increase fragmentation. Remove `--expose-gc` from `execArgv` once `global.gc()` calls are removed.

## Applies when

- Responding to memory pressure events in PtyHost or ResourceGovernor
- Reviewing any new `global.gc()` usage anywhere in the codebase
- Investigating terminal freeze/stutter after project-level operations
- Extending ResourceGovernor with new memory mitigation strategies
