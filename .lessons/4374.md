## Context

`closeSharedDb()` was defined in the database module but was never called in the production shutdown sequence. The SQLite connection was left open, relying on the C++ destructor to eventually close it when the process exited.

## Problem

SQLite WAL (Write-Ahead Log) mode leaves `-wal` and `-shm` sidecar files open until the connection is explicitly closed and the WAL is checkpointed. Relying on process exit to close the connection means the WAL may not be fully merged on every clean shutdown, potentially leaving database state inconsistent on the next launch.

## Root cause

`closeSharedDb()` was defined but its call site in the shutdown sequence was never wired up. The function existed as dead code. In Electron, the app lifecycle ends via `app.exit(0)` which terminates the process, causing the C++ SQLite destructor to run — but this bypasses the explicit checkpoint-and-close that flushes the WAL cleanly.

## Lesson

Always wire `closeSharedDb()` (or equivalent connection close) into the shutdown sequence explicitly, not relying on GC or process exit. The correct order: complete all writes (`saveAllProjectStates()`) → close the DB connection (`closeSharedDb()`) → exit (`app.exit(0)`). Wrap the close in try/catch so a close failure never blocks the exit. Verify the shutdown sequence in unit tests by asserting call order.

## Applies when

- Adding a new database or file-backed store to the Electron main process
- Reviewing the shutdown sequence in `electron/lifecycle/shutdown.ts`
- SQLite WAL sidecar files accumulating or database appearing stale on relaunch
