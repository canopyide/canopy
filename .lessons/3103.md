## Context

The terminal wake-restore cycle (triggered on focus/visibility) calls `terminal.reset()` then `terminal.write()` on the xterm instance to replay serialized scrollback.

## Problem

Users scrolled back in an agent terminal would have their viewport snapped to the bottom every time they clicked back into it, because `reset()` inherently clears the scroll position and `write()` auto-scrolls.

## Root cause

xterm's `terminal.reset()` destroys viewport state. There is no built-in "restore scroll" option — callers must capture and reapply it themselves.

## Lesson

Before any `reset()` + `write()` replay sequence on an xterm instance, capture `scrollBackOffset` (distance from bottom), then after the write completes call `scrollToLine` to reapply it. Gate the restore on `isUserScrolledBack` so users already at the bottom still get auto-scroll. This pattern applies to every restore path — sync restore, incremental restore, deferred-output restore — and is easy to miss when adding a new replay path.

## Applies when

Touching xterm restore/replay logic, adding a new serialization path, or anywhere `terminal.reset()` is called on a live xterm instance that the user may have scrolled.
