## Context

xterm.js 6.0 natively supports OSC 52 (clipboard write). Canopy sets `allowProposedApi: true` in shared terminal config, which enables OSC 52 parsing. The renderer also has `clipboard-sanitized-write` permission in the trusted session, allowing writes without a user gesture.

## Problem

AI agent output rendered in terminals could silently overwrite the user's system clipboard via OSC 52 escape sequences embedded in terminal output — a pastejacking attack. The sequence `\x1b]52;c;BASE64_PAYLOAD\x07` causes xterm.js to call `navigator.clipboard.writeText()` with the decoded payload.

If the payload ends with `\n`, it executes immediately upon paste in most shells.

## Root cause

Three converging factors: `allowProposedApi: true` (enables parsing), `clipboard-sanitized-write` permission (allows write), and no `registerOscHandler(52, ...)` override to block the sequence. OSC 11 was already overridden for alt-buffer detection using the same pattern — OSC 52 was overlooked.

## Lesson

Whenever `allowProposedApi: true` is set on an xterm.js instance, audit which proposed APIs become active and which pose security risks. OSC 52 is the critical one for clipboard hijacking.

Block it with a no-op handler on every terminal instance:

```ts
terminal.parser.registerOscHandler(52, () => true);
```

Returning `true` signals to xterm.js that the sequence was handled, suppressing default behavior. This must be registered on every terminal instance including prewarm instances, since the config is shared.

This is especially important in IDEs that orchestrate AI agents — the terminal is the primary interface and agents regularly produce large volumes of output.

## Applies when

Any xterm.js terminal with `allowProposedApi: true`; reviewing terminal security; adding new terminal instance types; any environment where untrusted content may be rendered in a terminal.
