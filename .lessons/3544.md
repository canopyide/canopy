## Context

The directing state (pencil indicator) is meant to appear when the user is actively typing a prompt. It should not appear when the user is navigating the terminal (arrow keys, Home/End, Page Up/Down, function keys) or performing control operations (Escape).

## Problem

Arrow keys, Home/End, Page Up/Down, F1-F24, and Escape triggered the directing indicator even though they are navigation/control actions, not prompt composition. Escape would trigger the indicator and then it would take 2500ms to clear rather than immediately cancelling.

## Root cause

The `isNonKeyboardInput` filter only checked a small set of sequences. ANSI navigation sequences (e.g., `\x1b[A` for up arrow) were not filtered, so they fell through to the directing trigger path.

## Lesson

ANSI navigation sequences to filter in `isNonKeyboardInput`:

- Arrow keys: `\x1b[A`, `\x1b[B`, `\x1b[C`, `\x1b[D`
- Home/End: `\x1b[H`, `\x1b[F` (and `\x1b[1~`, `\x1b[4~`)
- Page Up/Down: `\x1b[5~`, `\x1b[6~`
- Function keys F1-F24: `\x1bOP` through `\x1bOS`, `\x1b[11~` through `\x1b[34~`
- Modifier-bearing variants: `\x1b[1;2A` (Shift+Up), etc. — modifiers are encoded in the `1;N` section before the final byte

Escape (`\x1b` alone, or `\x1b\x1b`) deserves special handling: it should not trigger directing, and if directing is active, Escape should immediately cancel it (revert to waiting) rather than waiting for the debounce to expire.

## Applies when

Any input classifier that needs to distinguish "user is composing text" from "user is navigating." ANSI escape sequences require explicit enumeration — a simple "is printable ASCII" check is insufficient.
