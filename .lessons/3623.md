## Context

`WorktreeMonitor` polled `git status` on a fixed 2s/10s schedule regardless of file activity. With 5+ worktrees, this spawned many git processes unconditionally, even when nothing had changed.

The existing `GitFileWatcher` already watched `.git/index` and `.git/refs/heads/` but missed the most common case: edits to working directory files. `.git/index` only updates on staged operations, not plain file saves.

## Problem

Most git spawns during a quiet session were redundant. The watcher infrastructure existed but watched the wrong paths.

## Root Cause

Working directory files were not watched — only git internals. The primary refresh driver remained the fixed timer for the majority of real-world changes (editing, saving, new files).

## Lesson

For accurate git status freshness, the file watcher must cover the working directory (excluding `.git/`) in addition to git internals. Without this, the "event-driven" watcher only handles staged operations and the timer remains the primary mechanism.

Three implementation considerations:

1. **Max-wait debounce ceiling**: rapid file changes (build tool, linter) flood the debouncer. Add a `MAX_WAIT_MS` ceiling (5s) so sustained rapid changes still produce periodic status updates rather than being indefinitely deferred.
2. **Retry with exponential backoff**: file watchers can fail on macOS due to fd exhaustion. Add `retryWatcher()` with backoff (3 attempts) and a graceful fallback to timer-only mode — never silently lose status freshness.
3. **Timer as fallback**: do not remove the fixed-interval timer entirely; keep it as a safety net for missed events.

## Applies when

- Adding file-system-event-driven git status refresh
- Diagnosing stale status in the worktree panel
- macOS file descriptor exhaustion with many watched worktrees
