## Context

Canopy supports auto-switching the active theme when the macOS system appearance changes (light/dark mode toggle). The implementation listens to `nativeTheme.on('updated')` in the main process and pushes a theme-change event to the renderer via IPC.

## Problem

`nativeTheme` fires its `updated` event on macOS before the system appearance has fully propagated. Reading `nativeTheme.shouldUseDarkColors` immediately inside the handler returns the old value, causing the theme to switch to the wrong mode (e.g., switching to dark when the user just enabled light mode).

## Root cause

This is a known macOS race condition with Electron's `nativeTheme` event. The event fires as part of the appearance change notification, but the underlying value is not yet stable at the moment the handler runs. A short delay (300ms) is required to let the OS finish the transition before sampling `nativeTheme.shouldUseDarkColors`.

## Lesson

When listening to `nativeTheme.on('updated')` on macOS, always debounce the handler by at least 300ms before reading `nativeTheme.shouldUseDarkColors`. Reading it synchronously in the event handler will return stale data.

## Applies when

Implementing any feature that reacts to system light/dark mode changes in Electron on macOS — theme switching, background color updates, `BrowserWindow.setTitleBarOverlay()` palette changes, etc.
