## Context

Agents were showing "failed" state after Ctrl+C, SIGTERM, and other routine user-initiated exits.

## Problem

`isRoutineExit()` in `terminalForensics.ts` had **inverted boolean logic** — it returned `true` for crash scenarios and `false` for routine exits, the opposite of its name. This caused SIGINT (130), SIGTERM (143), and SIGHUP (129) to be classified as non-routine, triggering the failed state.

Additionally, the `AgentStateMachine` had no recovery path from `failed`: once in that state, subsequent `busy` or `prompt` events were ignored.

## Root cause

A logic inversion bug in the forensics classifier. Since the function name was `isRoutineExit`, the caller correctly passed its return value to "is this OK to ignore", but the function body returned the wrong polarity. Tests were written to match the (wrong) behavior rather than catching the inversion.

## Lesson

Boolean classifier functions where the name says "is routine" must return `true` for the routine/expected case. When writing tests for these functions, test against known-good inputs (e.g., SIGINT should definitely be routine) rather than only testing the implementation's current output. Also: state machines should always have recovery paths — a `failed` state with no exit is a UX trap.

## Applies when

- `terminalForensics.isRoutineExit()` or `classifyExit()`
- Agent state machine transitions from `failed`
- Debugging "agent shows failed after Ctrl+C or normal exit"
