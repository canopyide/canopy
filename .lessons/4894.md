## Context

When a terminal is spawned during session restore, it needs the active worktree's path as its cwd. The renderer hook `useActiveWorktreeSync` was emitting the cwd before the worktrees list had finished loading from the store.

## Problem

On cold start the worktrees store is empty initially. `useActiveWorktreeSync` fired immediately, emitting a cwd derived from a stale `homeDir` fallback. The spawned terminal opened in the user's home directory instead of the project worktree.

A secondary gap existed in the main process: `lifecycle.ts` also fell back to `os.homedir()` when given a `worktreeId` it couldn't resolve synchronously at spawn time.

## Root cause

Two independent race windows:

1. **Renderer**: no `isInitialized` guard — the hook emitted a path event as soon as it mounted, before the worktree list had hydrated.
2. **Main process**: the fallback chain in `lifecycle.ts` went straight to `os.homedir()` without attempting an async worktree path lookup when `worktreeId` was present in spawn options.

## Lesson

Hooks that derive paths from async-loaded store data must gate their output behind an `isInitialized` flag. Emitting before the store slice is loaded produces a valid-looking but stale value.

In the main process, when a `worktreeId` is present in spawn options, always attempt `worktreeService.getMonitorAsync(worktreeId)` before falling back to `os.homedir()` — async resolution is safe at spawn time.

## Applies when

- Any hook that reads from a store slice populated by IPC/async hydration and emits an event or side-effect on initial render.
- Any main-process spawn path that accepts a `worktreeId` hint and has a `os.homedir()` fallback.
