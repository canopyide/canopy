## Context

`node-pty` fires individual data events for every chunk of terminal output. These events are forwarded to the renderer via MessagePort. During bulk output (e.g., `cat large-file`, test runs), this produces hundreds of individual `postMessage` calls per second.

## Problem

High-throughput terminal output caused visible UI jank and high IPC overhead because each PTY data chunk was posted as a separate MessagePort message.

## Root cause

No batching on the PTY → MessagePort path. Each `data` event mapped 1:1 to a `port.postMessage()` call, overwhelming the MessagePort with tiny messages.

## Lesson

Add a `PortBatcher` between PTY data events and `port.postMessage()`. Use `setImmediate` for near-immediate (~1-2ms) flushes when the buffer is empty (low-throughput, low-latency), batch to ~16ms intervals when data arrives continuously (high-throughput, amortized cost), and flush immediately when the buffer hits 64KB. Integrate with existing watermark-based backpressure so the batcher pauses when the consumer is behind. This pattern applies to any high-frequency event stream crossing an IPC boundary.

## Applies when

Any high-frequency data producer sending to a MessagePort or IPC channel: PTY output, file watcher events, log streaming, diagnostics data.
