## Context

The non-polling agent terminal path in `ActivityMonitor` uses a rolling 2000-character `PatternBuffer` to detect working state patterns (e.g., agent spinner text). A debounce timer fires when output stops; before transitioning to idle, it checks `lastPatternResult.isWorking`.

## Problem

Agents streaming large responses (code files, verbose logs) would briefly flip to "waiting" mid-response. The working indicator would disappear and re-appear while the agent was still actively generating output.

## Root cause

When a large response arrived, the `PatternBuffer` filled entirely with response text and evicted the spinner/working-indicator pattern that was previously buffered. The `lastPatternResult` was updated to `isWorking: false`. When the debounce timer fired (even before output had stopped), it found no working pattern and transitioned to idle.

The debounce extension logic checked only the pattern buffer state, not whether data had arrived recently.

## Lesson

Pattern buffer state is a useful signal for extending the working-state hold, but active data flow is a stronger signal. The debounce idle transition should be suppressed whenever data arrived recently, regardless of what the rolling buffer currently contains:

```typescript
// Track the timestamp of the most recent data chunk
lastDataReceivedAt = Date.now();

// In resetDebounceTimer callback:
const timeSinceData = Date.now() - lastDataReceivedAt;
if (timeSinceData < 500) {
  // Data is still arriving — suppress idle transition
  return;
}
```

A 500ms recency window is sufficient. This guard is additive to, not a replacement for, the pattern-based working-state extension.

This fix is scoped to the non-polling path (`resetDebounceTimer`). Polling-mode terminals use `runPollingCycle()` and are unaffected.

## Applies when

Any debounced idle transition that relies on pattern matching over a bounded rolling buffer. If the buffer can be overflowed by the very output stream being detected, a recency guard is required.
