## Context

`ActivityMonitor` classifies agent terminals as idle or busy based on PTY output patterns — byte rate, line-rewrite spinner detection, and specific ANSI sequences. Resize events go to node-pty via `SIGWINCH`.

## Problem

Resizing a terminal (panel drag, window resize, layout switch) flipped idle agents to "busy" for several seconds with no real work happening. The UI showed a working indicator and confused agent-state-based workflows.

## Root cause

When the TTY is resized, the running program receives `SIGWINCH` and re-renders its TUI to the new dimensions. That re-render emits a burst of ANSI sequences — cursor moves, clears, redraws — which looks identical to an agent actively printing output. The same byte-rate and line-rewrite heuristics that detect real work fire on reflow bursts.

## Lesson

On resize, suppress ActivityMonitor idle→busy transitions for ~1s via a `notifyResize(suppressionMs)` method that sets a `resizeSuppressUntil` timestamp and resets the high-output tracking window. The suppression must cover every path that can flip state: high-output byte-rate recovery, pattern-based `becomeBusyFromPattern`, spinner/line-rewrite detection in `updateLineRewriteDetection`, and the polling-cycle working-signal and high-output recovery branches. Rapid successive resizes should extend the window. Already-busy terminals must stay busy through the resize (only suppress idle→busy, never busy→idle).

## Applies when

Adding activity heuristics on PTY output, changing `TerminalProcess.resize()`, or debugging phantom "working" indicators that correlate with window/panel resize events.
