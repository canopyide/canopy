## Context

Canopy's integrated browser blocks all cross-origin navigations (e.g., `window.location.href = 'https://example.com'`) via `will-navigate` and `will-redirect` handlers. This was intentional — added in PR #3727 to close a TOCTOU gap where `will-attach-webview` validates `src` at attachment time but the guest page could navigate away afterwards. The silent failure meant OAuth/OIDC login flows (Google, GitHub, SAML) were completely non-functional.

## Problem

Users experienced silent failures: clicking "Sign in with Google" in a localhost app loaded in the integrated browser would do nothing. The app stayed on the current page with no feedback. OAuth is explicitly prohibited in embedded webviews by [RFC 8252] — Google actively enforces this.

## Root cause

The `will-navigate`/`will-redirect` event handlers blocked the navigation but emitted nothing. The UX contract was: "silently fail" rather than "explain and offer an escape hatch."

## Lesson

When intentionally blocking a user action for security reasons, always emit a signal back to the renderer with enough context for recovery. The pattern used here:

1. In the blocking handler, send a `WEBVIEW_NAVIGATION_BLOCKED` IPC event with `{ url, canOpenExternal }` to the relevant renderer
2. In the renderer, show a non-modal notification bar (auto-dismiss after 10s) with the blocked URL and an "Open in Browser" button
3. Only offer external open for `http:` and `https:` — block `javascript:`, `data:`, `file:`, `about:` without offering escape

`canOpenExternal` flag is critical: dangerous protocols must not be offered as external links even if the webview blocks them.

The security invariants (localhost-only restriction, TOCTOU guard, `will-attach-webview` src validation) are all preserved — only the UX changes.

## Applies when

- Any Electron handler that blocks a user-initiated navigation or action
- Debugging blank/non-functional OAuth flows in the integrated browser — the restriction is intentional
- Adding new protocol handlers in `electron/setup/protocols.ts`
