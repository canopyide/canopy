## Context

The NewWorktreeDialog in the renderer proposed a default worktree path by splitting the project path on `/` and appending the branch name. On macOS and Linux this worked; on Windows the project path used backslashes (`C:\Users\...\canopy`), so `split("/")` returned the whole path as one segment and produced nonsensical suggestions.

## Problem

Renderer code performing filesystem path arithmetic via string operations is inherently cross-platform broken because the renderer has no reliable way to know the host separator. Even Node's `path.posix` vs `path.win32` picks are guesses from the renderer side.

## Root cause

Path manipulation happened in the renderer using hardcoded `/` assumptions. The renderer is a browser context — it should never assume a filesystem separator. The main process has `process.platform` and native `path` module.

## Lesson

Any filesystem path arithmetic (basename, dirname, join, normalize) must happen in the main process using Node's `path` module, then be sent to the renderer via IPC. The renderer should treat paths as opaque strings it passes around but never decomposes. This pattern applies to worktree paths, project roots, file picker defaults, recent-files lists, and anywhere the UI wants to display or derive a related path.

## Applies when

Adding any feature where the renderer needs to display or derive a filesystem path from another path — default save locations, "open containing folder", breadcrumbs, suggested names from an existing path. Add an IPC handler, don't inline path math in React components.
