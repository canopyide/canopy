## Context

On startup, terminals restore their scrollback from a `.restore` file by writing the serialised buffer back into xterm before the new PTY spawns.

## Problem

If the terminal was in an interactive full-screen program (vim, htop, less, tmux) when the previous session ended, the restored content was a frozen snapshot of that TUI's alt-screen state — with no running process behind it, so every keystroke went to the wrong place or did nothing. Users saw a ghost vim they couldn't close.

## Root cause

Alt-screen programs write `\x1b[?1049h` (enter alt-screen) on start and `\x1b[?1049l` (exit) on quit. xterm's buffer serialiser captured whatever buffer was active. If the process crashed or the app was killed mid-alt-screen, the serialised bytes included `\x1b[?1049h` but never the matching exit sequence. On restore, xterm dutifully re-entered alt-screen and painted the dead TUI.

Second compounding issue: restore banners (separator marks injected after restore) were serialised back into the next snapshot, so they accumulated across hibernation cycles.

## Lesson

Before writing restored content to xterm, scan for an unmatched `\x1b[?1049h` (or the older `\x1b[?47h` / `\x1b[?1047h`). If found, skip the raw buffer write entirely and inject a plain-text "previous session was in <program>" notice instead. Also strip previously-injected banners from the buffer before each serialisation so they don't compound.

```ts
const hasUnmatchedAltScreen =
  /\x1b\[\?(1049|1047|47)h/.test(buf) && !/\x1b\[\?(1049|1047|47)l/.test(buf);
if (hasUnmatchedAltScreen) {
  writeNotice("Previous session ended in full-screen mode; history not restored.");
  return;
}
```

## Applies when

Any code that persists and replays xterm buffers across restarts. Also applies to any feature that serialises a terminal state for later viewing (session recording, sharing). Look for `\x1b[?1049h`, `\x1b[?1047h`, and `\x1b[?47h` — they all trigger alt-screen.
