## Context

`PtyManager.spawn()` called `new TerminalProcess(id, options, ...)`, which internally spawned a node-pty process before performing additional setup (agent config parsing, headless terminal creation, handler registration). If any step after the PTY spawn threw, the spawned process would be left alive but permanently unreferenced.

## Problem

The PTY process was spawned inside the constructor, but registry registration (`this.registry.add(id, terminalProcess)`) happened after the constructor call. A constructor failure meant the PTY existed but was invisible to all cleanup paths — `kill()`, `dispose()`, graceful shutdown, and orphan recovery all iterate the registry and would never find it.

## Root Cause

No try/catch around `new TerminalProcess(...)` in `PtyManager.spawn()`. The constructor did significant work beyond spawning (handler setup, headless terminal init, process detector config), any of which could throw. Once thrown, the spawned PTY was permanently orphaned.

## Lesson

When a constructor spawns an OS-level resource and then performs fallible setup steps, wrap the constructor call in try/catch and kill the resource on failure:

```typescript
let rawPty: IPty | undefined;
try {
  rawPty = spawnPtyProcess(options);
  const process = new TerminalProcess(id, rawPty, options);
  this.registry.add(id, process);
} catch (err) {
  rawPty?.kill();
  throw err;
}
```

The fix also extracted `spawnPtyProcess()` as a standalone function so the raw PTY handle is available for cleanup before it gets buried inside the constructor. This is the correct pattern: separate resource acquisition from resource initialization.

Pool-acquired PTYs have an additional consideration: if a pool entry is acquired but construction fails, kill it rather than returning it to the pool (it may be in a bad state after partial initialization).

## Applies when

- Any constructor that acquires an OS resource (process, file handle, socket) and then performs fallible setup
- Reviewing PTY spawn paths in `PtyManager.ts`
- Adding new resources to `TerminalProcess` construction
