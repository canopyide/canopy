## Context

With multi-window support added in #4645, IPC event handlers that broadcast to all windows (`win.webContents.send()` looped over all windows) started delivering events to unintended windows. A copy tree progress event for window A would appear in window B's UI.

## Problem

Before multi-window, broadcasting to "all windows" was equivalent to targeting the one window. After multi-window, the same broadcast delivers events to windows that didn't request them.

## Root cause

Request-response patterns implemented as broadcast notifications: the handler broadcasts `COPYTREE_PROGRESS` or `PROJECT_ON_SWITCH` to every window, but only the window that initiated the request should receive it.

## Lesson

- **Capture `event.sender` at handler entry and target it directly**: for request-response IPC patterns, the requesting window is already available via `event.sender` in the IPC handler. Store it and send responses back to that window only.
- **For push events (non-request-triggered), use ownership**: `WEBVIEW_CONSOLE_MESSAGE` should go to the window that owns the webview. Use `BrowserWindow.fromWebContents(webContents)` to find the owner.
- **Keep a broadcast fallback**: if `BrowserWindow.fromWebContents()` returns null (e.g., view is detached), fall back to broadcasting to prevent silent drops during edge cases.
- **The `IpcContext` object should carry the sender window**: threading the sender through the context type makes targeted routing straightforward without accessing global state.
- Affected event types: `COPYTREE_PROGRESS`, `PROJECT_ON_SWITCH`, `PROJECT_INIT_GIT_PROGRESS`, `CLONE_REPO_PROGRESS`, `WEBVIEW_CONSOLE_MESSAGE`, `WEBVIEW_CONSOLE_CLEAR`.

## Applies when

Any Electron IPC handler that was written before multi-window support and sends events back to renderers. Audit all `sendToAll()` / `webContents.send()` loops when adding multi-window support.
