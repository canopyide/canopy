## Context

A modal used `position: fixed; inset: 0` to cover the viewport with a backdrop. It rendered as a child of the sidebar (which had `backdrop-blur-sm`) rather than being portaled to `document.body`.

## Problem

The modal rendered clipped inside the sidebar instead of covering the window. `position: fixed` was resolving relative to the sidebar, not the viewport.

## Root cause

CSS `backdrop-filter` (and `filter`, `transform`, `perspective`, `contain: paint`) creates a new containing block for `position: fixed` descendants. Any ancestor with `backdrop-blur-*` (Tailwind) therefore silently breaks viewport-relative fixed positioning inside its subtree. This is spec-defined but non-obvious and not surfaced by devtools as a warning.

## Lesson

Modals, overlays, and dropdowns that need viewport coverage must be portaled to `document.body` via `createPortal`, not rendered in place — otherwise any ancestor applying `backdrop-filter`, `filter`, `transform`, `perspective`, or `contain: paint` will anchor their "fixed" positioning to that ancestor. Also register with the app's overlay stack (`useOverlayState`) for consistent dismissal behavior. Tailwind's `backdrop-blur-*` utilities are a common culprit because they are applied liberally to surfaces for visual polish.

## Applies when

Building any floating UI (modal, popover, toast) that uses `position: fixed` and might be rendered inside a styled surface. Debugging a "fixed" element that mysteriously respects an ancestor's bounds. Adding `backdrop-blur` to a container and noticing child overlays break.
