## Context

Canopy's tier system already managed WebGL contexts and addon lifecycle for background terminals, but the core xterm.js `Terminal` instance (including its TypedArray-backed scrollback buffer) was never disposed until the panel was explicitly removed. With 20–30 terminals open, cumulative xterm.js memory was 200–400 MB.

## Problem

Each xterm.js `Terminal` instance consumes 2–15 MB depending on scrollback content (baseline 2–5 MB + ~1.2 MB per 1000 lines at 100 columns). The `TerminalInstanceService` stored all instances in a `Map<string, ManagedTerminal>` with no eviction path.

## Root cause / Design decision

The infrastructure for hibernation already existed: `SerializeAddon` was loaded on every terminal, PtyHost could serialize/restore terminal state via the existing `TerminalWakeManager`. The missing piece was calling `terminal.dispose()` on background terminals and recreating them on wake.

The implementation: when a terminal enters BACKGROUND tier, a 30-second hibernation timer starts. On expiry, the xterm.js `Terminal` instance is disposed (DOM element, scrollback buffer, addons all freed), and the terminal is marked `hibernated`. When promoted back to VISIBLE/ACTIVE tier, a new `Terminal` instance is created and restored from PtyHost-serialized state. Terminals with active agent output are skipped — the skip check uses the `AgentStateMachine` state.

Key edge cases handled: rapid tab switching (timer cancellation), already-hibernated terminals, and panel removal cancelling pending timers.

## Lesson

xterm.js `Terminal` instances are expensive and must be treated like heavyweight resources that need lifecycle management beyond just tier-based addon control. The core Terminal object itself must be disposed for background terminals. The pattern — timer-delayed disposal with cancel-on-promotion and wake-from-serialized-state — is reusable and already tested with 17 cases. Never assume that reducing addon count is sufficient memory management for xterm.js.

## Applies when

- Investigating high memory usage with many open terminal panels
- Modifying the terminal tier system (`BACKGROUND`/`VISIBLE`/`ACTIVE`)
- Extending `TerminalInstanceService` with new lifecycle hooks
- Adding new terminal addons that must participate in hibernate/wake cycles
