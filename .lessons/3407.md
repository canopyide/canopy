## Context

`ProcessTreeCache.refreshWindows()` was hardcoding `cpuPercent: 0` for all Windows processes. Agent activity detection uses CPU% as a fallback signal when no command-name patterns match. With CPU always zero on Windows, agent state transitions (idle → working → idle) were blind to actual process activity.

## Problem

Two bugs:

1. `cpuPercent` was hardcoded to `0` for all Windows processes — the calculation was never implemented
2. The `terminalActivityPatterns` fallback logic had a bug: it was supposed to return `cpuPercent > 0` when no patterns matched, but the condition was inverted, always returning `false`

## Root cause

Windows doesn't provide CPU% directly via WMI — it requires computing a delta from two `KernelModeTime + UserModeTime` samples taken between polls. This calculation was deferred and left as a hardcoded zero. The pattern-match fallback bug was a separate logic error in the boolean condition.

## Lesson

**Windows CPU% must be delta-based**, not instantaneous. Store `KernelModeTime + UserModeTime` per PID between polls, then compute:

```
cpuDelta = (newKernel + newUser) - (prevKernel + prevUser)
cpuPercent = cpuDelta / (elapsedMs * logicalCpuCount * 10000) * 100
```

(The `* 10000` converts 100-nanosecond intervals to milliseconds.)

Always return `0` on the first poll for a PID (no prior sample). This approach matches what Windows Task Manager reports.

For the activity fallback: when no pattern matches, the correct check is `cpuPercent > 0` (process is active) — not a negated or inverted form.

## Applies when

Any Windows process monitoring code. Delta-based CPU computation is required on Windows; instantaneous values are always zero from WMI's `PercentProcessorTime` without sampling over an interval.
