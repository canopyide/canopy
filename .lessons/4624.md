## Context

`windowServices.ts` held 17 module-level singleton variables for services and ports. On any window close, ALL services were disposed, which silently killed PortalManager, EventBuffer, MessagePorts, and ProjectSwitchService for every other open window simultaneously.

## Problem

A multi-window architecture where `setupWindowServices()` stored all instances as module globals meant closing Window A would destroy services that Window B was still actively using. The bug was silent — no error was thrown, the other windows simply lost functionality.

## Root cause

`win.on('closed')` cleanup called disposal functions for module-level service references rather than window-scoped ones. With a single window this was fine; with multiple windows the first close poisoned all subsequent windows.

The separation of "truly global" (PtyClient, WorkspaceClient, CliAvailabilityService, AgentVersionService) vs "per-window" services (PortalManager, EventBuffer, MessagePorts, ProjectSwitchService) was not encoded in the code structure, only in mental models.

## Lesson

When writing `setupWindowServices()` for multi-window Electron apps, always separate service lifetimes explicitly:

- **Per-window services** → store in `WindowContext.services`; dispose in that window's `closed` handler
- **Global services** → keep module-level; dispose only when the last window closes (check `WindowRegistry.size === 0`)

Make `setupWindowServices()` handle re-entry: guard global init so it only runs once, while per-window init runs for each call. Add a test that opens two windows, closes one, and asserts the other's services are still alive.

## Applies when

- Adding any new service to `windowServices.ts` — classify it as per-window or global before placing it
- Writing cleanup code in `win.on('closed')` handlers
- Any service that sends IPC to a specific window's renderer (per-window), vs services that are app-wide (global)
