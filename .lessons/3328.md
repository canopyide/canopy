## Context

The terminal wrapper div had CSS padding and a `bg-canopy-bg` class to create a gutter around the xterm canvas. When users picked a custom terminal color scheme, the canvas background followed the scheme but the wrapper frame stayed on the app's default surface color, producing a visible rectangle of wrong-colored padding.

## Problem

Any terminal color scheme with a background that differed from the app surface color showed a mismatched frame around the terminal content.

## Root cause

CSS padding on the wrapper lives outside xterm's rendering surface entirely — xterm only paints the canvas area and has no idea the wrapper exists. The wrapper's background color has to be kept in sync with the scheme manually. Worse, xterm 6.0 exposes an `ITerminalOptions.padding` option that handles this natively: xterm extends its own theme background to fill the entire visible rectangle including the gutter.

## Lesson

For terminal gutters, use `ITerminalOptions.padding` (xterm 6.0+) rather than CSS padding on the wrapper. Pass the padding value at construction time; the internal gutter is owned by xterm and automatically matches `ITheme.background`. If you still need the wrapper to match (for alt-buffer modes that strip the padding, or edge cases), drive its `backgroundColor` from an inline style sourced from the active scheme selector — never from a static Tailwind class.

## Applies when

Any custom terminal color scheme, theme picker, or wrapper chrome change that needs to stay in sync with the xterm canvas background. Also applies when reviewing any `padding-*` Tailwind class on an xterm host element.
