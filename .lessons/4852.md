## Context

App hydration flips `isStateLoaded` to render the UI. Before flipping, it awaited `recipeLoadPromise`. Recipes are loaded from disk via IPC.

## Problem

Initial cold-start had 50-160ms of blank-screen time on every launch caused by the awaited recipe load blocking the hydration completion.

## Root cause

The project-switch path already treated recipe loading as fire-and-forget (non-blocking). The initial hydration path was inconsistent — it awaited the same operation. The recipe store already has concurrent-request protection via `loadRecipesRequestId` tracking that discards stale responses, so there was no correctness reason to block.

Fix: remove the `await` on `recipeLoadPromise` during initial hydration, matching project-switch behaviour.

## Lesson

Hydration-blocking `await`s are a common source of startup latency. For any async load during hydration, ask: does the UI correctness of the first paint depend on this data? If the data populates in the background without causing visible inconsistency (recipes appear once loaded, not needed for initial render), it should be fire-and-forget.

Verify consistency between the initial hydration path and the project-switch path — they should have the same async/await semantics for equivalent operations.

## Applies when

- Adding new data loading to the hydration path in `stateHydration/index.ts` — default to fire-and-forget unless the data is needed for the first render frame.
- Any `await` in the hydration sequence: justify why it blocks `isStateLoaded`.
