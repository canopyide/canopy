## Context

Canopy injects the agent launch command (e.g., `gemini` or `claude`) directly into the PTY after spawning the terminal. On systems with shell initialization that changes the Node version (e.g., nvm with `.nvmrc`), the wrong Node version was active when the command ran.

## Problem

Users with nvm saw agents fail to launch because the shell hadn't yet sourced `.nvmrc`/`.nvm/nvm.sh` by the time the command was injected. The PTY was open but shell initialization (including nvm hooks in `.zshrc`/`.bashrc`) hadn't completed. Gemini CLI installed for Node 24 would fail to run because nvm hadn't yet switched away from Node 16 (the base version).

## Root cause

The agent command was written to the PTY immediately after spawn with no delay for shell init. There's a fundamental race between:

1. PTY process spawned (fd opened, shell process starts)
2. Shell reads `.zshrc`/`.bashrc`, including nvm initialization which may `chnode`
3. Shell prints the first prompt (indicating readiness)

Steps 2–3 can take hundreds of milliseconds on systems with nvm, rbenv, pyenv, or other shell-level version managers.

## Lesson

Never inject commands into a PTY immediately after spawn. Wait for the shell to emit its first prompt (via `waitForShellReady()`) before writing. Use a data listener that resolves on any output containing prompt characters (`$`, `%`, `>`, `#`) with a fallback timeout (10s) that injects anyway so the feature degrades gracefully on non-standard shells.

The `waitForShellReady()` pattern:

```typescript
await waitForShellReady(terminalId, { timeoutMs: 10_000 });
await writeToTerminal(terminalId, command);
```

## Applies when

Any code that programmatically writes to a PTY/terminal immediately after spawn — agent launches, dev preview command injection, initial shell configuration. Particularly important when users may have nvm, rbenv, pyenv, or asdf in their shell init.
