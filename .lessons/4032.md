## Context

IPC handler and listener accumulation was a suspected source of memory leaks and "ghost" event processing. There was no automated test to catch regressions.

## Problem

Without a way to count active IPC handlers and listeners from tests, listener leaks would only surface as memory issues or mysterious duplicate event handling after many terminal/panel cycles.

## Root cause

No observability into IPC handler counts from the test layer.

## Lesson

Expose `ipcDiagnostics.getHandlerCount()` and `getListenerCount()` through the preload bridge so E2E tests can assert that counts remain stable after N open/close cycles. The pattern:

1. Record baseline counts after app launch
2. Perform N cycles of the operation under test
3. Assert counts are within a small delta of baseline

This catches both renderer-side listener leaks and main-process handler leaks. The preload diagnostic bridge is the correct mechanism — don't rely on renderer `window` introspection for main process state.

## Applies when

- Any new IPC channel added without corresponding cleanup verification
- Debugging IPC handler accumulation or "handler already registered" warnings
- Adding E2E coverage for operations that register/deregister IPC listeners
