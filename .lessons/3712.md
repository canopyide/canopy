## Context

`TerminalOutputIngestService` flushed PTY output to xterm.js on fixed timers: 8ms for standard output and 25ms for ink redraw patterns. These constants don't adapt to system load.

## Problem

On a fast machine with few terminals, 8ms is unnecessarily conservative. On a loaded machine with many simultaneous agent terminals, 8ms is too aggressive — the xterm.js parser can fall behind and buffer up queued work, causing UI freezing for high-throughput output (e.g., an agent running `cat` on a large file).

## Root cause

Fixed timer intervals are a lowest-common-denominator approximation. The real signal — parser readiness — was available in xterm.js 6.0 via the `onWriteParsed` event but wasn't being used.

## Lesson

xterm.js 6.0 exposes `onWriteParsed`, which fires at most once per animation frame after the parser has processed all pending data. This is the correct backpressure signal for terminal output ingestion:

- Write data to xterm.js
- Wait for `onWriteParsed` before writing the next batch
- When buffer exceeds high watermark (128KB), pause upstream writes
- Resume when buffer drains below low watermark (32KB)

This naturally adapts to system load: fast machines process near-instantly, loaded machines naturally throttle. `onWriteParsed` must be exposed from `TerminalInstanceService` so the ingest service can subscribe to it.

The ink redraw detection path should remain as a separate concern — it needs different timing for real-time CLI tools regardless of parser throughput.

## Applies when

Terminal output ingestion performance; any place that writes batched data to xterm.js; adding new terminal features that need to know when xterm.js has caught up with pending output.
