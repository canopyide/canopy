## Context

Canopy has a drag-and-drop terminal grid. Some panels contain `<iframe>` or Electron `<webview>` elements (Browser panel, Dev Preview panel).

## Problem

When a user dragged a terminal panel near or over a browser/dev-preview panel, the drag would freeze. The cursor stopped tracking, the drag overlay locked in place, and the user had to release and restart the drag.

## Root cause

iframes (and Electron `<webview>` elements, which behave identically for pointer event routing) create an isolated browsing context. Once the pointer enters an iframe boundary, the browser routes `pointermove` events to the iframe's document rather than the top-level drag handler. dnd-kit loses the pointer and freezes.

This is a well-known browser limitation with no DOM-level workaround — the iframe itself must be prevented from receiving events during a drag.

## Lesson

The idiomatic fix for dnd-kit + iframe coexistence is a CSS `pointer-events: none` rule applied to all `webview` and `iframe` elements during a drag:

1. At drag start in the `DndProvider`, set `data-dragging` on `document.documentElement`.
2. In global CSS: `[data-dragging] webview, [data-dragging] iframe { pointer-events: none }`.
3. Remove the attribute at drag end and drag cancel.

This is simpler and more robust than injecting overlay divs over each iframe, and covers Electron `<webview>` elements without special-casing.

Ensure the attribute is removed even when the drag ends outside the application window — wire a `window.pointerup` fallback to handle orphaned drags.

## Applies when

Any dnd-kit (or similar drag library) integration where the layout contains `<iframe>` or Electron `<webview>` panels.
