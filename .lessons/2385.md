## Context

The worktree sidebar shows the current branch name for each worktree. Previously, a full `discoverAndSyncWorktrees` poll was the only way the branch name could update — up to a 10-second delay for background worktrees.

## Problem

Users switching branches saw stale sidebar labels for several seconds after the change, breaking the mental model that the sidebar reflects reality.

## Root cause

Spawning `git` inside the polling loop is too slow and too coarse. The branch info is trivially available in a file: `gitDir/HEAD` for the active branch reference, and the per-worktree reflog at `gitDir/logs/HEAD` changes on every checkout. The file watcher already fires on `HEAD` mutations — we just were not reading the file on the hot path.

## Lesson

For branch-name updates, read `gitDir/HEAD` directly inside `updateGitStatus()` instead of spawning a git process. This gets branch changes to the UI in one file-watcher debounce cycle (~300ms) instead of one polling cycle (up to 10s). For linked worktrees, also watch `gitDir/logs/HEAD` — not just the shared `commonDir/logs/HEAD` — or branch changes in linked worktrees will be missed. Normalize `gitDir`/`commonDir` with `path.normalize` before comparing to tolerate trailing-slash and separator differences. When fast-pathing a branch change, also re-extract any branch-derived metadata (issue number, PR number) and clear stale values — mirroring the slow-path `syncMonitors` behaviour — or the UI shows new branch with old badges.

## Applies when

Optimizing git status polling, adding any git-based metadata to the worktree sidebar, or extending `GitFileWatcher` to cover a new watch surface.
