## Context

The Electron main process is I/O-bound — IPC dispatch, file operations, service orchestration. V8's default optimization strategy favors raw execution speed via JIT compilation, which produces larger code and data structures.

## Problem

The main process held excessive V8 heap from JIT-compiled code that didn't benefit the actual workload (mostly async I/O waiting, not CPU-intensive computation).

## Root cause

V8 defaults to performance-optimized compilation. For a process spending most time awaiting I/O, this wastes memory on optimized machine code that rarely executes on hot paths.

## Lesson

`--optimize-for-size` instructs V8 to prefer compact code/data over raw execution speed. For I/O-bound processes like Electron's main process, this is a clear memory win with negligible throughput cost.

Apply via `app.commandLine.appendSwitch('js-flags', '--optimize-for-size')` alongside the existing `--max-old-space-size` flag. Both flags can coexist in the same `--js-flags` string.

Importantly: do NOT apply `--optimize-for-size` to the PTY host utility process. The PTY host does real-time stream parsing where CPU speed directly affects terminal responsiveness and latency.

## Applies when

- Electron main process memory profiling shows large V8 heap from JIT code
- Any Electron utility process that is I/O-bound vs CPU-bound
- Choosing between `--optimize-for-size` and default V8 optimization — evaluate whether the process is throughput-critical
