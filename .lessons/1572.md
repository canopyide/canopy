## Context

Terminal output flows through two transports — a SharedArrayBuffer ring buffer (low-latency, the renderer polls it) and an IPC fallback (`terminal:data` events). Agent terminals were explicitly excluded from SAB writes via a `skipVisualStream` flag, so they went down the IPC path. When the renderer was in SAB polling mode, however, it acked and discarded IPC `terminal:data` frames.

## Problem

Agent output was dropped on the floor: producer wrote to IPC, consumer threw away IPC because it was polling SAB. Users saw truncated Claude/Codex responses with no error.

## Root cause

The exclusion rule mixed two concerns. The reason to skip SAB was agent backpressure — chatty agents could fill the ring buffer and starve the consumer. But the decision was keyed on terminal `kind === "agent"`, not on actual polling state. Once the renderer's ack logic was updated to check polling state instead of terminal kind, the transports became consistent again.

## Lesson

When you have two transports for the same data, the producer's routing rule and the consumer's consumption rule must reference the same single source of truth. If the producer decides "agents skip SAB" but the consumer decides "SAB-polling terminals ignore IPC", the overlap drops data. Route by runtime transport state (is SAB polling active? is the ring buffer suspended?), not by static attributes of the terminal (kind, agentId, type). When suspending one transport, always preserve the analysis/state-detection path — otherwise idle/working detection breaks in parallel with the visual drop.

## Applies when

Any dual-transport system where a consumer can switch modes: SAB vs IPC, websocket vs polling, main-thread vs worker-thread handoff. Always write an end-to-end test that exercises "producer routes to A while consumer listens on B" to catch the mismatch.
