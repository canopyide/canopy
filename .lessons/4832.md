## Context

`WorkspaceClient.getAllStatesAsync()` has 11 call sites across worktree handlers, copyTree handlers, artifact handler, and `ResourceProfileService`. Multiple can fire concurrently from a single user action.

## Problem

Two distinct performance issues:

1. **No request deduplication** — opening the CopyTree modal while project pulse is visible triggers two simultaneous `getAllStatesAsync(windowId)` calls that each independently IPC to the same workspace host. Under load, this multiplies IPC round-trips by the number of concurrent callers.

2. **Sequential host iteration** — the no-windowId path queried workspace hosts with a `for...of` + `await` loop, so N hosts = N serial round-trips.

## Root cause

No caching or deduplication was ever added to the client; the no-windowId path was written sequentially without considering parallelism.

## Lesson

**Singleflight pattern for IPC clients**: when a client method can be called concurrently by multiple callers that all want the same data, cache the in-flight Promise:

```ts
const inflight = new Map<string, Promise<T>>();

async function getAllStatesAsync(windowId?: number): Promise<T> {
  const key = windowId?.toString() ?? "all";
  if (inflight.has(key)) return inflight.get(key)!;

  const promise = doActualWork().finally(() => {
    // Evict after TTL so stale data doesn't linger
    setTimeout(() => inflight.delete(key), 150);
  });
  inflight.set(key, promise);
  // Evict immediately on error to allow retries
  promise.catch(() => inflight.delete(key));
  return promise;
}
```

Key details:

- Cache key must include the windowId (or `"all"`) to prevent cross-project contamination.
- Evict on error immediately; don't cache failures.
- Use `Promise.allSettled` for the multi-host fan-out path, not sequential `await`.
- A 150ms TTL after resolution is enough to collapse a burst of concurrent callers without serving stale data to user-triggered actions.

## Applies when

Any IPC client method with multiple call sites that may fire concurrently from a single user action. Especially relevant for workspace state, file system status, and other read-heavy operations that are cheap to cache briefly.
