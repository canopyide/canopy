## Context

Every xterm.js terminal instance loaded `@xterm/addon-web-links`, which runs regex on every visible line during rendering to detect and make URLs clickable. Simultaneously, the backend `UrlStyler` wraps URLs with OSC 8 hyperlink sequences before they reach the renderer.

## Problem

Both systems detected URLs — the backend in the PTY host (once, on raw output), and the frontend addon (on every rendered line). With 40+ simultaneous agent terminals producing high-throughput output, the frontend regex was significant CPU overhead in the renderer main thread.

## Root cause

The WebLinksAddon predated the backend `UrlStyler`. When `UrlStyler` was added with an explicit comment "Clickable without WebLinksAddon", the addon was not removed. Dual detection went unnoticed because both systems produce working clickable links.

## Lesson

When adding backend URL processing (OSC 8) to replace frontend detection, also remove the frontend addon to avoid double work. The backend path is strictly better: it processes data once in the PTY host; the result flows through IPC and renders natively in xterm.js with zero renderer-side cost.

**Caveat**: `UrlStyler` skips text already containing ANSI escape sequences (to avoid breaking colored output). URLs embedded within ANSI-styled text won't be OSC 8-linked by the backend. This is an acceptable trade-off — URLs in colored output are rare in practice.

OSC 8 links use xterm.js native link handling, not the `openLink` callback passed to `WebLinksAddon` — verify click behavior (Cmd+click vs plain click) is preserved via native handlers.

## Applies when

Adding or auditing terminal link detection; reviewing xterm.js addon loading; any dual-detection pattern where backend and frontend both process the same data.
