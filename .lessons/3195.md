## Context

`AgentStateMachine` tracks agent state via output heuristics (busy/prompt/completion events) with a few high-confidence transitions driven by user input. The `failed` state was treated as a terminal sink with no outgoing transitions.

## Problem

Once an agent hit `failed` (e.g. Claude errored out mid-response), the user could type a new prompt but the state machine stayed stuck on `failed` forever. The agent was actually responding, but the UI showed it as broken.

## Root cause

Heuristic events are noisy — letting `busy` or `prompt` clear `failed` would mean spurious output could silently hide a real error. The solution isn't to loosen heuristics; it's to add a single high-confidence transition gated exclusively on the `input` event (confidence 1.0), matching the pattern already used for `completed → working` and `waiting → working`.

## Lesson

When designing agent/task state machines, no state should be a true sink. Always keep a `input → working` escape hatch gated on explicit user action (not output heuristics), and mirror it across every place that tracks state (main service, PTY service, renderer worker). The rule: heuristic signals get blocked from terminal states; explicit user events always flow through.

## Applies when

Adding or modifying any agent state machine, terminal state tracking, or error recovery UI. Also applies when debugging "agent stuck" reports — check whether the current state has a valid user-input transition.
