## Context

`BulkCreateWorktreeDialog` orchestrates a batch worktree creation flow: the user selects issues, clicks "Create Worktrees", and the component runs them through a queue. A `progressReducer` tracks per-item state, and a `batchTrackingRef` ref holds a Map of tracking data parallel to the reducer state.

## Problem

Clicking "Create Worktrees" could trigger the entire batch twice. After the second pass, the progress counter reset to "0 of N created" even though all worktrees existed on disk. Users saw the spinner spin twice and then land on a broken state.

## Root cause

Two independent bugs compounded each other:

1. **No execution guard on the async gap**: `handleCreate` dispatches `START` (which hides the Create button), then `await`s `runBatch`. Between the synchronous dispatch and the first `await`, the Create button is still technically mounted and clickable for one render cycle. A rapid double-click fired two invocations before the UI updated. Fix: add an `isExecutingRef` boolean guard checked at the top of `handleCreate`.

2. **`batchTrackingRef` never cleared on reset**: The `progressReducer`'s `RESET` action clears the `items` Map in state, but `batchTrackingRef.current` (a plain ref, invisible to the reducer) retained entries from the prior batch. If a second execution occurred, stale tracking data caused the second run to skip worktree creation steps while the freshly-reset progress reducer showed all items as pending, producing the "0 of N" display. Fix: explicitly clear `batchTrackingRef.current` in `handleClose` and `handleDone`.

## Lesson

When a component maintains **both** reducer state and a parallel ref (for perf or non-reactive bookkeeping), the ref is invisible to `dispatch(RESET)`. Every cleanup path (`handleClose`, `handleDone`, any RESET-triggering action) must manually clear the ref too. Forgetting to synchronize ref cleanup with reducer reset is a silent divergence bug that only surfaces under specific timing.

Additionally, `async` event handlers that initially hide their trigger element are still vulnerable to double-invocation: the button hides on the _next render_, not synchronously. An `isExecutingRef` guard at the function entry is the correct defence — not relying on UI state alone.

## Applies when

- Dialog or wizard components that run an async batch operation on button click
- Any component that uses a ref as a parallel data store alongside reducer state
- Operations with "one-shot" semantics that must not re-execute within a session
