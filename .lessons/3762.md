## Context

When a terminal panel is closed it enters a 20-second trash grace period (supporting undo). The trash timeout is tracked in memory only via `setTimeout`. If the app or PTY host crashes during this window, the timeout is lost and the PTY process becomes a permanent orphan.

## Problem

On macOS, orphaned PTY processes consume one of ~511 available PTY file descriptors system-wide. Multiple crashes during the trash window could accumulate enough orphans to exhaust the system limit. Normal startup had no mechanism to detect these orphans.

## Root cause

`TerminalRegistry.trash()` created an in-memory `setTimeout` for the 20-second kill. Nothing persisted the "pending kill" state to disk, so a crash erased all knowledge of which PTYs should have been killed.

## Lesson

Any cleanup operation with a time-delayed kill must persist its pending state to disk if the delay spans a potential crash window. The `TrashedPidTracker` persists PIDs to a JSON file in the app data directory and runs orphan cleanup on startup. PID reuse safety is critical: validate the process command name before killing to avoid killing an unrelated process that was assigned the same PID after a restart.

Cross-platform kill: use `SIGKILL` on macOS/Linux, `taskkill /F /PID` on Windows. The `global.gc()` call after batch kills (PR 3794) complements this — both address cleanup of freed PTY resources.

## Applies when

- Any resource with an in-memory deferred cleanup (grace period timers, pending deletes)
- Startup initialization — always check for orphaned resources from a previous crashed session
- PID-based process tracking — always validate command name before killing to handle PID recycling
