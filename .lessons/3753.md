## Context

During high-throughput agent output (code generation, large diffs, log streams), xterm.js was processing each incoming data chunk as a separate render cycle. The output worker already batches ~8ms of output per postMessage, but each `terminal.write()` call still triggered an independent paint.

## Problem

Without synchronized output, rapid terminal writes produce visual tearing — users see partial lines between frames — and redundant GPU render passes. With 8-16 panels active during agent runs, this creates measurable jank.

## Root Cause

xterm.js 6.0 natively supports DEC Mode 2026 (Synchronized Output) but it must be explicitly signalled by the writer. Without wrapping each write in `\x1b[?2026h` ... `\x1b[?2026l` escape sequences, xterm.js renders each write independently rather than batching all mutations into a single atomic frame.

## Lesson

Always wrap `terminal.write()` calls with DEC Mode 2026 begin/end sequences to eliminate visual tearing during high-throughput output:

- Begin: `\x1b[?2026h`
- End: `\x1b[?2026l`

xterm.js 6.0 handles nested DEC 2026 sequences correctly, so agents that emit their own synchronized output sequences are safe. node-pty passes these sequences through transparently.

For Uint8Array data paths, allocate a new buffer with 16 extra bytes (prefix + suffix) rather than string concatenation — this avoids encoding overhead.

Overhead is negligible: 6 bytes per write at ~750 writes/sec = ~4.5 KB/sec.

## Applies when

- Writing output to any xterm.js 6.0 terminal instance
- Any `terminal.write()` call that processes batched or multi-line data
- Debugging visual tearing/jank during agent output
