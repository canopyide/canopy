## Context

The "Move to New Worktree" feature (added in PR #4767) attempted to transfer agent context by resuming the existing agent session in the new worktree directory.

## Problem

Agent session resume is directory-scoped. When an agent starts in one directory, its session is stored under a path derived from the working directory. Starting the agent in a different directory (the new worktree path) means it can't find the session:

```
➜  feature-test git:(feature/test) claude --resume 9c93a63a-89c7-43e7-94c8-f401c8d5dab7
No conversation found with session ID: 9c93a63a-89c7-43e7-94c8-f401c8d5dab7
```

This is not fixable per-agent because each stores sessions differently:

- **Claude Code** — sessions in `~/.claude/projects/{path-encoded}/`. No `--cwd` flag to override.
- **Gemini** — sessions scoped via `~/.gemini/projects.json` path mapping.
- **Codex** — has `-C <DIR>` flag (works), but is the exception.

## Root cause

Session resume is fundamentally path-coupled in every agent except Codex. The initial implementation assumed resume was path-agnostic.

## Lesson

Cross-directory agent session resume is not possible for Claude Code or Gemini. The correct approach for transferring context across a directory boundary is **terminal buffer injection**:

1. Before tearing down the old panel, capture the xterm.js output buffer via `TerminalInstanceService.captureBufferText(instanceId)`.
2. Strip ANSI escape sequences from the captured text.
3. Truncate to a reasonable size (with a note when history was trimmed).
4. Start a fresh agent in the target directory and inject the captured history as a context prompt.

This trades "seamless session continuation" for "awareness of prior context" — agents understand the history without needing the session file.

## Applies when

Any feature that transfers or moves an agent terminal across git worktree directories. Never assume agent session IDs are portable across paths.
