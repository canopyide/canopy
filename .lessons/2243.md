## Context

Deleting a worktree would remove it from the sidebar briefly, then the deleted entry would reappear on the next refresh. The cache had a TTL but something was refilling it with stale data.

## Problem

A cache-with-promise pattern (store the promise while the request is in flight so concurrent callers dedupe) has a subtle race: if the cache is invalidated WHILE a request is in flight, and that in-flight promise resolves after the invalidation, the resolver writes the stale result back into the cache.

## Root cause

`refresh()` cleared the cache, then listWorktreesFromGit() from an earlier call finished and its `.then(result => cache.set(key, result))` ran. The completing promise had no idea its owner had been invalidated. On the next read, the cache looked "fresh" with resurrected deleted entries.

## Lesson

Any cache-with-in-flight-promise pattern needs ownership checks. Stash a token (identity of the entry/promise) on the entry when you start the fetch. In the `.then` handler, verify the token still matches before writing the result. If it does not match, drop the result on the floor. Also prefer keyed invalidation over global `clear()` so unrelated in-flight requests survive. Create-then-invalidate and delete-then-invalidate paths must use the same key shape.

## Applies when

Writing or reviewing any cache with async refill. Git status caches, GitHub API caches, project metadata caches. Debugging "deleted item comes back" or "stale data after action" bugs. Any place `cache.set` happens inside a `.then` that could outlive an invalidation.
