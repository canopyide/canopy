## Context

The project switcher palette loaded stats (worktree count, branch info, last opened) for all projects by making N individual IPC calls — one per project — in rapid succession.

## Problem

With a large project list (10+ projects), opening the project switcher triggered an IPC storm that caused the renderer to freeze. The user reported memory overload and UI becoming unresponsive.

## Root cause

Each `project:get-stats` IPC call crosses the contextBridge boundary, serializes/deserializes, and runs a git operation (branch info, worktree enumeration). Multiplied by every project, and triggered on every keystroke (or mount), this overwhelmed the IPC channel and caused the V8 heap in the renderer to spike.

## Lesson

Any UI component that needs data for N items should batch-fetch via a single IPC call rather than N parallel calls. Pattern:

1. Add a `getBulkStats` IPC endpoint that returns an array/record in one round-trip
2. Cache the bulk result on the renderer side with a TTL (30s is reasonable for project metadata)
3. Poll on a timer instead of re-fetching on every user interaction

The renderer-side TTL cache (`projectSwitchRendererCache.ts`) prevents any re-fetch within the cache window even if the component remounts.

Also: mark any IPC handler that performs git operations as "potentially slow" in code review, and prefer batch endpoints for list-view scenarios.

## Applies when

Any palette/list UI that shows per-item data for an unbounded collection (projects, worktrees, branches). The threshold for concern is ~5+ items with individual IPC calls.
