## Context

After migrating the app UI from `BrowserWindow.webContents` to a `WebContentsView` (child of `win.contentView`), the native Electron application menu broke for edit and zoom operations. This PR is a follow-up to #4798 which fixed DevTools and zoom — this one fixed edit roles (undo/redo/cut/copy/paste/selectAll).

## Problem

Electron's built-in `role` property on menu items dispatches commands to `BrowserWindow.webContents`. After the WebContentsView migration, `BrowserWindow.webContents` is an empty shell with no rendered content. The actual app UI lives in a child `WebContentsView`. Result: all `role`-based menu items silently no-op or appear greyed out:

- `toggleDevTools` — grayed out
- `resetZoom / zoomIn / zoomOut` — grayed out
- `undo / redo / cut / copy / paste / selectAll` — unreliable; broke OS-level menu integration

## Root cause

Electron's `role` dispatch is hardwired to `BrowserWindow.webContents`. It has no awareness of child `WebContentsView` instances. This is a fundamental limitation of the role system — it cannot be configured to target a different webContents.

Edit roles are additionally tricky because the correct target depends on what has focus: the main app view, a portal webview, or a browser panel. Using `webContents.getFocusedWebContents()` handles this correctly.

## Lesson

After migrating to `WebContentsView`, **every Electron menu `role` that targets webContents must be replaced with an explicit `click` handler**. Window-level and app-level roles (`close`, `minimize`, `quit`, `hide`, etc.) are unaffected — only content-targeting roles break.

Two distinct resolution patterns:

1. **App-specific actions** (DevTools, zoom): resolve `getAppWebContents(win)` to get the primary `WebContentsView` webContents.
2. **Focus-sensitive actions** (cut/copy/paste/undo/redo/selectAll): use `webContents.getFocusedWebContents()` so the action routes to whichever WebContents currently has focus (app, portal, browser panel).

Also: `openDevTools` must be called with `{ mode: 'detach' }` inside a custom `contentView` hierarchy — docked mode is unstable.

## Applies when

Any Electron app that uses `WebContentsView` (formerly `BrowserView`) for its main UI instead of rendering directly into `BrowserWindow.webContents`. The symptom is greyed-out or silently broken menu items after a migration.
