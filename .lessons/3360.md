## Context

`ProcessMemoryMonitor` tracked per-process memory pressure via Electron's `process.getProcessMemoryInfo()` return shape, using `workingSetSize` for threshold comparisons and leak-trend analysis.

## Problem

On macOS, memory pressure alerts fired under normal operation. The trend detector also produced sawtooth noise that made slow-leak detection unreliable — growth signals were drowned out by GC cycles.

## Root cause

macOS aggressively compresses pages that haven't been touched recently. The kernel reports the compressed size as part of `workingSetSize`, so a process that allocated a stable amount of memory can appear to shrink and grow as pages enter and leave the compressor. `privateBytes` reports private, uncompressed resident memory and is stable across compression activity. V8's own sawtooth (mark-sweep GC) on top of compression produces double-noise when watching `workingSetSize`.

## Lesson

For Electron memory thresholds and leak detection, always compare against `privateBytes`, never `workingSetSize`. For trend detection, use bucket minimums over a rolling window (e.g. 30 minutes of 1-minute buckets, take the min of each bucket) before feeding into an EMA — this filters both compression waves and GC sawtooth while still surfacing genuine growth. Suppress trend evaluation for the first 10-15 minutes after startup to absorb JIT compilation and module-init allocation bursts.

## Applies when

Any `process.getProcessMemoryInfo()` consumer that compares values against thresholds or looks for growth trends. Especially applies on macOS — on Linux/Windows the two fields are closer, but `privateBytes` is still the more honest signal.
