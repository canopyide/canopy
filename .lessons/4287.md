## Context

Electron's shutdown sequence runs a chain of async cleanup operations (flush buffers, close PTY processes, save state, close DB). If any cleanup step hangs (e.g., a PTY process that never exits, a network call that never resolves), the app freezes indefinitely and never quits.

## Problem

No hard timeout existed on the shutdown cleanup chain. A hang in any cleanup phase would prevent the process from exiting, requiring a force-kill from the OS or the user.

## Lesson

**Every async cleanup chain must have a hard timeout.** The pattern is `Promise.race([cleanupChain, hardTimeout])`. If the timeout fires first, call `app.exit(1)` (not `app.quit()` — `quit` re-triggers lifecycle events, while `exit` is immediate).

**Track the current phase** to log which step was still running when the timeout fired. Update a `currentPhase` variable as you enter each cleanup step so the timeout handler can emit a useful diagnostic message.

**10 seconds is a reasonable hard limit** for graceful shutdown. Users notice hangs after 2-3 seconds; 10 seconds is long enough for real cleanup (flushing DB, closing PTY) but short enough to not feel frozen.

**Clear the timeout** on normal completion so the process doesn't receive a spurious `exit(1)` call after a successful shutdown.

The timeout is a backstop, not a replacement for per-operation timeouts. Individual operations (e.g., waiting for a subprocess to exit) should still have their own shorter timeouts.

## Applies when

- Electron `before-quit` / `will-quit` handlers with multi-step async cleanup
- Any cleanup chain where a single hung step would deadlock the entire shutdown
- Service `stop()` methods that rely on external processes or network I/O
