## Context

`EditorService.openFile` on macOS uses `execa` with `{detach: true}` to spawn `open` as a fallback when no editor is configured. The spawned process promise was not given a `.catch()` handler.

## Problem

If the `open` command exits non-zero after `openFile` has already returned (e.g., bad file association, permission denied, corrupted Launch Services DB), the rejected promise floats outside the surrounding `try/catch`. In Electron 37+, unhandled rejections in utility processes cause a hard crash of that process.

## Root cause

`execa` with `{detach: true}` returns a `ChildProcess` promise. When `openFile` returns before the child process exits, any subsequent rejection is unhandled — the original `try/catch` has already exited. The adjacent `launchEditor` helper in the same file already had a `.catch(() => {})` for this reason, but the `open` fallback path missed it. The Electron 37 change that makes unhandled rejections crash utility processes exposed this latent bug.

## Lesson

All detached `execa` child processes (spawned with `{detach: true}`) must attach a `.catch(() => {})` immediately, regardless of surrounding try/catch. The try/catch only covers synchronous throws and rejections that happen before the first `await` after function return — it does not protect against post-return rejections from floating promises. Verify the same pattern in every callsite where `execa` is used with detach. Also: never use `shell.openPath()` in main-process services — it blocks the renderer process.

## Applies when

- Using `execa` with `{detach: true}` or any fire-and-forget child process
- Platform-specific fallback spawn paths (macOS `open`, Windows `start`, etc.)
- Electron utility processes where unhandled rejections are fatal (Electron 37+)
