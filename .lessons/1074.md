## Context

The PTY host runs as an Electron utility process. If it crashes (OOM, native segfault, assertion) the main process restarts it, but terminals on the renderer silently froze: no output, no error, stale UI. Users didn't know anything was wrong.

## Problem

Three compounding bugs: (1) crash listeners were attached too late, missing the race where the host died during construction; (2) the old MessagePort was never closed on restart, leaking handles and occasionally delivering stale messages into the new session; (3) after renderer reset, xterm rendered white-on-white text until the next viewport change.

## Root cause

Recovery is a multi-step process and each step has a race window. Listeners must be attached synchronously in the PtyClient constructor, not inside an async init, or the first crash after app start escapes detection. MessagePorts are GC-rooted from both sides — if you replace your reference without calling `.close()`, the port lingers and the next `postMessage` can land in the old handler. The white text after `resetRenderer()` is an xterm rendering glitch that goes away only if you send a soft reset (DECSTR, `\x1b[!p`) to reinitialize terminal attributes before resetting the renderer.

## Lesson

Crash recovery needs atomic, synchronous setup. Attach crash/ready listeners in the PtyClient constructor. On every backend ready event: (1) close the old MessagePort before replacing it; (2) cancel any pending recovery timer (a second crash while recovering from the first will leak timers); (3) for each live terminal, write the DECSTR soft reset sequence, then call xterm's `resetRenderer()`, then recalculate dimensions and resend. Make `resetRenderer()` non-throwing so one bad terminal doesn't block the rest. Wrap `MessagePort.postMessage` in try/catch with automatic fallback to IPC — a closing port throws synchronously. Track backend connection state in the store (connected/disconnected/recovering) and block user input via an overlay during disconnected/recovering so typed keys aren't silently dropped.

## Applies when

Any feature built on Electron utility processes with MessagePort-based data channels. Adding crash/restart recovery to workspace-host, pty-host, or a new utility host. Debugging "terminals look stuck after backend restart", white/invisible xterm text after a reset, or leaked MessagePort handles in heap snapshots.
