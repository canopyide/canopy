## Context

`better-sqlite3` is a native module that must be compiled for Electron's ABI, not the system Node.js ABI. On Windows 11, `npm run rebuild` silently produced the wrong binary — all E2E tests failed with `ERR_DLOPEN_FAILED` at startup.

## Problem

`better-sqlite3` uses `prebuild-install` as its npm install script. When `electron-rebuild` ran, `prebuild-install` downloaded a prebuilt binary matching the **system Node.js ABI** (e.g. 137 for Node 24) rather than the **Electron ABI** (e.g. 143 for Electron 40). The `.forge-meta` file was updated to mark the rebuild as complete, but the binary was wrong. No validation caught this — the app loaded fine in Node.js but crashed in Electron.

`afterPack.cjs` validated `node-pty` binaries but had no check for `better-sqlite3`.

## Root cause

`prebuild-install` resolves the target ABI from the runtime environment, not from `electron-rebuild`'s target flags. On Windows the `electron-rebuild` invocation didn't force a source build, so `prebuild-install`'s prebuilt binary (for Node ABI) silently "won".

## Lesson

- **Add ABI validation to `afterPack.cjs` for every native module**, not just `node-pty`. Use `process.dlopen` inversion or `node-abi` to verify the compiled ABI matches the Electron version before packaging.
- **Add a smoke check in `environment.ts`** for startup: `new Database(':memory:').prepare('SELECT 1').get()` — catches ABI mismatches at launch rather than during E2E runs.
- **`prebuild-install` can silently win against `electron-rebuild`** on Windows when prebuilt binaries are available for the Node ABI. Force source compilation with `--force-rebuild` or verify by checking the binary's ABI after rebuild.
- If Windows CI fails with `ERR_DLOPEN_FAILED` for a native module, the first suspect is ABI mismatch from `prebuild-install`, not a broken source build.

## Applies when

Packaging any Electron app that includes native modules which use `prebuild-install` (better-sqlite3, argon2, canvas, etc.). The afterPack validation step is critical insurance.
