## Context

Canopy pastes text into agent terminals then sends Enter to submit. For Claude and Codex this works via bracketed-paste mode, but Gemini's CLI doesn't support bracketed paste, so Canopy just writes the bytes and waits. Large pastes into Gemini were submitting with only part of the text.

## Problem

With bracketed paste the terminal knows when the paste starts and ends, so Enter can be sent immediately. Without it, the terminal echoes each byte back asynchronously, the agent reads its own stdin in chunks, and if Enter arrives before the echo/readline finishes, the buffer is cut mid-stream. A fixed sleep (e.g. 200ms) is brittle: it's too long when the system is fast and too short under load or when the paste is very large and backs up in OS pipes.

## Root cause

Submission timing was decoupled from actual terminal progress. The fix anchors submit to "output stopped changing for N ms" with a hard max wait, and anchors the settle window to the start of submit so stale pre-paste output can't prematurely satisfy the check.

## Lesson

For any terminal that doesn't support bracketed paste, use an output-settle debounce instead of a fixed delay: wait until there's been no new PTY output for ~200ms (capped at ~2s), then send Enter. Anchor the debounce timestamp to when the submit starts, not to the last-ever output, otherwise unrelated prior output makes the wait terminate instantly. Check a `wasKilled`/`disposed` flag on each poll so a terminal death during the wait doesn't hang the submit.

## Applies when

Adding a new agent whose CLI doesn't advertise bracketed-paste support, extending the input pipeline to support new paste modes, or debugging "last few characters of pasted input get lost" on Gemini-like terminals.
