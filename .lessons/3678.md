## Context

In Electron's renderer process, unhandled promise rejections and uncaught errors silently disappeared — no UI feedback, no logs. This is distinct from the main process where Electron 37+ crashes utility processes on unhandled rejections.

## Problem

Silent async failures in the renderer are difficult to debug and invisible to users. An agent action failing asynchronously would leave the app in an inconsistent state with no user notification.

## Root cause

`window.addEventListener('unhandledrejection')` and `window.addEventListener('error')` were never registered. Unlike Node.js (which has `process.on('unhandledRejection')`), browser-context error handling requires explicit listener registration.

## Lesson

Electron renderer processes need explicit global error handler registration — no automatic crash or log like the main/utility processes. Register both handlers early in `src/main.tsx` before app mount:

```ts
window.addEventListener("unhandledrejection", handler);
window.addEventListener("error", handler);
```

Key implementation details:

- Filter `AbortError` rejections — these are expected (e.g., fetch cancellation)
- Re-entrancy protection — logging errors from within the error handler can recurse
- Route to the error store for UI surfacing, not just console
- Cleanup on HMR dispose to avoid double-registration during development

The handler should be resilient to logger failures — if `logError` itself throws, the handler must not propagate that error.

## Applies when

Setting up a new Electron renderer app; reviewing error observability gaps; any async operation in the renderer that could fail silently.
