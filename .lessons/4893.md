## Context

The preload script (`electron/preload.cts`) is bundled by esbuild at build time, which inlines its `shared/` dependencies. The `typecheck` npm script also ran `tsc -b` on the preload tsconfig.

## Problem

`tsc -b` (build mode) emits output files. Running it on `electron/tsconfig.preload.json` produced a bare `.cjs` that overwrote the esbuild bundle in `dist-electron/`. The overwritten file had no bundled `shared/` dependencies, so subsequent `WebContentsView` loads in dev mode hit `module not found: ../shared/utils/trustedRenderer.js`.

## Root cause

`tsc -b` vs `tsc --noEmit -p` distinction: `-b` (build mode) always emits output even without `--emitDeclarationOnly`. The fix is `tsc --noEmit -p electron/tsconfig.preload.json` — type-checks without touching the filesystem, leaving the esbuild output intact.

The main electron tsconfig correctly uses `tsc -b` because it intentionally emits `.js` files. The preload is special: esbuild owns the output file.

## Lesson

When esbuild bundles an entry point and deposits it in the dist folder, any `tsc -b` invocation on that tsconfig will silently clobber the bundle. Use `tsc --noEmit -p <tsconfig>` for type-checking-only targets that are owned by a separate bundler step.

The symptom (missing shared module) is misleading — it looks like a missing dependency rather than a stale/overwritten file.

## Applies when

- Any preload or UtilityProcess entry point that is bundled by esbuild but also type-checked in the `typecheck` script.
- Adding a new tsconfig project to the build chain: verify whether it should emit (`tsc -b`) or only type-check (`tsc --noEmit -p`).
