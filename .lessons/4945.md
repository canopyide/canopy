## Context

Terminals can live in either the grid or the dock. Moving between them involves `TerminalPane` unmounting/remounting, `XtermAdapter` detaching/reattaching, and `terminalRegistry` state updates.

## Problem

Multiple corruption scenarios on dock interaction and project switch:

1. Double-clicking a docked terminal to move it back to grid would fail or corrupt state
2. A stale dock detach could background/detach an xterm instance already claimed by a new grid panel
3. After project switch restore, terminals with persisted `location: "dock"` had their location overwritten with the tab group's stale `"grid"` location
4. `moveTerminalToDock` (single-panel path) left `isVisible` and `runtimeStatus` in inconsistent states vs the grouped path

## Root cause

Four independent bugs:

- **Click/double-click race**: the first click opened a popover whose teardown raced the double-click's grid-move action
- **No ownership check on unmount**: `TerminalPane` unmount cleanup ran unconditionally, detaching xterm instances that had already been claimed by a new mount
- **Hydration location override**: `hydrateTabGroups` overwrote `location: "dock"` with the group's `location`, discarding the persisted dock state
- **Single vs grouped move inconsistency**: the single-panel `moveTerminalToDock` path omitted `isVisible`/`runtimeStatus` updates present in the group path

## Lesson

- Double-click handlers on items with single-click side effects need click debouncing/suppression. When a double-click fires, suppress the preceding single-click action.
- `XtermAdapter` and `TerminalPane` cleanup must check ownership/generation before detaching. Use a `mountGeneration` counter â€” stale callbacks from a previous mount cycle are no-ops.
- `hydrateTabGroups` must never overwrite persisted `location: "dock"` â€” always check if the terminal already has a specific location before applying the group default.
- Keep single-panel and grouped move paths in sync for any state field they're expected to update.
