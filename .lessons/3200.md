## Context

`ActivityMonitor` watches PTY output bytes to decide whether an agent is "working" vs "waiting". Any byte counted as activity resets the idle debounce and can push the state back to `working`.

## Problem

Agents that display a spinner or elapsed-time counter ("Thinking... 00:14") kept retriggering the working state indefinitely — the monitor saw a continuous byte stream that was actually just the same line being repainted. Resize reflows also replayed buffered content and looked like new activity.

## Root cause

Terminal spinners and progress counters use carriage-return overwrites (`\r<new text>` or `\x1b[G<new text>` or `\x1b[<n>D` then text) so the visible line appears to update in place. The underlying byte stream isn't empty — every frame writes the full line again — but semantically nothing has changed. Resize reflows emit multi-line replays of buffered content using cursor positioning sequences, which also bump byte counters.

## Lesson

Before counting a PTY chunk as "real activity", strip ANSI escape sequences and check whether the chunk is a pure CR-overwrite with no semantic content change. Two cheap heuristics catch the bulk of cases:

1. **Cosmetic redraw:** chunk contains `\r` or `\x1b[G` followed only by text that renders on the current line, with no line-feeding. Compare the stripped payload against the previous chunk's stripped payload — if they are the same shape (e.g. both "Thinking... HH:MM:SS") at different values, it's a redraw.
2. **Resize reflow:** multiple cursor-positioning sequences in a burst without any net new content. Characterised by repeated `\x1b[<row>;<col>H` or `\x1b[<n>A` sequences.

Pass-through advancing content (progress bars that actually grow, new text lines, shell prompts) must still count. Write explicit tests for spinner frames, elapsed timers, resize reflows, and real progress-bar growth.

## Applies when

Any heuristic that uses raw PTY byte volume as a proxy for "work happening": activity monitors, idle detectors, agent state machines, notification triggers. Especially relevant for agents (Claude, Gemini, Codex) that emit status-line UIs.
