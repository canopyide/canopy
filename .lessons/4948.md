## Context

`hardenedGit.ts` provides a stripped-down git instance for passive background operations (status polling, worktree monitoring) that intentionally omits credential helpers to prevent accidental auth prompts in background processes.

## Problem

Private repo clone and push operations failed because they used the same hardened (no-credential) git instance as background polling.

## Root cause

The original `createHardenedGit()` was designed as a single git factory. As user-initiated operations (clone, push) were added, they inherited the hardened config that explicitly blanks `core.askpass`, `credential.helper`, and `core.sshCommand`.

## Lesson

Maintain two distinct git factories with documented security boundaries:

- `createHardenedGit(baseDir)` — strips all credential/SSH config; used by `WorkspaceService` and background polling; never prompts
- `createAuthenticatedGit(baseDir)` — inherits process environment (system keychain, `gh auth`, SSH agent); used for explicit user-initiated operations

Route by intent, not by operation type: clone, push, fetch = authenticated; status, log, diff (background) = hardened. Document the boundary in `hardenedGit.ts` so future operations are added to the correct factory.

The `WorkspaceService` should always use hardened git even when adding new background operations.

## Applies when

- Adding new git IPC handlers that involve network operations or credentials
- Modifying `hardenedGit.ts`
- Any `git-write.ts` operations (push, commit, tag)
