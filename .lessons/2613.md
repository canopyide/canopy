## Context

A CodeMirror 6 auto-size extension measured line height once by querying the first `.cm-line` DOM element via `getBoundingClientRect()` and using that value in its growth math.

## Problem

The hybrid input bar grew 2-3 extra lines tall when users added a newline to multi-line wrapped content with a slash/file/image chip on the first line, then snapped back on the next edit. A visible jump followed by a correction.

## Root cause

When a CodeMirror mark/widget decoration is present on a line (e.g., a chip widget for `@file` or `/command`), that line's rendered height can exceed the baseline text line-height — 28px instead of 20px in this case. Using this inflated value as the per-line increment caused `computeAutoSize` to over-count visual lines. A secondary geometry-change pass then re-entered and corrected to the real value, producing the visible snap-back.

A secondary gotcha: writing `overflowY` unconditionally on every update invalidated layout and triggered extra measurement passes.

## Lesson

Never measure line height from a DOM element that may contain decorations. Either:

- Read CodeMirror's own `view.defaultLineHeight` (the canonical, decoration-free value the view uses for scroll math), or
- Measure a synthesized probe element with the same font metrics outside the document tree.

Guard style writes in update listeners so they only set `overflowY`/height when the value actually changes — unconditional writes on every update cause layout thrash and re-entry.

## Applies when

Writing any CodeMirror 6 extension that computes editor height or line counts. Adding decorations (marks, widgets) that change rendered line height. Debugging auto-size widgets that flicker or over-grow. Reviewing any code that reads `.cm-line` via `getBoundingClientRect`.
