## Context

`PullRequestService` refreshes PR metadata on three paths: periodic poll, debounced resolve, and explicit `refreshAll`. Each used an exponential backoff on failure, capped at 30 minutes.

## Problem

Worktree sidebar cards showed stale PR state and wrong issue numbers for long stretches — up to 30 minutes after transient GitHub hiccups. The debounced-resolve path didn't check the circuit breaker at all, so new refreshes could complete mid-backoff but still race against stale writes from the still-running poll path.

## Root cause

Three independent problems:

1. **Backoff ceiling too high:** 30 minutes meant any transient GitHub outage caused half an hour of stale UI.
2. **No auto-recovery:** the breaker only reset on an explicit manual refresh — one bad minute cost the rest of the session.
3. **Guard missing on debounce path:** poll path respected the breaker, debounce path did not, so concurrent paths raced.
4. **No cancellation on refreshAll:** pending debounce timers kept firing after a full refresh landed, overwriting fresh data with stale data.

## Lesson

A production-ready circuit breaker needs four things together — missing any one of them creates the stale-data failure mode:

1. A low cap on backoff (5 minutes is usually enough; 30 is too long for UI-facing data).
2. Success-counted auto-recovery: a single successful response resets the failure counter. Don't require N successes; one is enough.
3. The breaker check applied to _every_ path that might trigger the operation (poll, debounce, manual, IPC).
4. Cancellation of pending debounce/throttle timers when a full refresh runs, so the refresh result isn't overwritten.

Write tests for each of these specifically — it's the combination that breaks.

## Applies when

Any service with multiple refresh paths against a flaky remote (GitHub API, language-server, AI provider). If you have both polling and debounced reactive refreshes, they must share the breaker and cancel each other on full refresh.
