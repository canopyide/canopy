## Context

E2E tests for panel drag-and-drop reordering (using @dnd-kit) were unreliable in CI/Xvfb environments when using Playwright's built-in `dragTo()` or simple `mouse.move()` calls.

## Problem

@dnd-kit's drag detection relies on observing a sequence of `pointermove` events. A single `mouse.move()` from source to destination — even with `steps` — didn't reliably trigger the drag activation threshold in headless/Xvfb environments where frame timing differs from headed mode.

## Root cause

@dnd-kit activates drag on a pointer move that exceeds the activation distance (default 5px). In headless environments, if the move happens too fast or the DOM doesn't update between pointer events, the library may not register the activation. The fix is a `smoothDrag()` helper that performs many small intermediate steps (`mouse.move()` in a loop with small increments), giving @dnd-kit time to process each `pointermove` event and properly activate.

```ts
// e2e/helpers/dragDrop.ts
export async function smoothDrag(page, from, to, steps = 20) {
  await page.mouse.move(from.x, from.y);
  await page.mouse.down();
  for (let i = 1; i <= steps; i++) {
    await page.mouse.move(
      from.x + ((to.x - from.x) * i) / steps,
      from.y + ((to.y - from.y) * i) / steps
    );
  }
  await page.mouse.up();
}
```

## Lesson

When writing Playwright E2E tests for @dnd-kit drag interactions, use a stepped `smoothDrag()` helper rather than `dragTo()` or a single `mouse.move()`. The default Playwright drag simulation moves too fast for @dnd-kit's event-based activation to reliably fire in headless/Xvfb environments. 20 steps is a good default; increase for CI flakiness.

## Applies when

- Writing E2E tests for any @dnd-kit sortable/draggable UI
- Panel reordering, worktree list reordering, or any drag-to-reorder feature
- CI environments using Xvfb (Linux headless)
