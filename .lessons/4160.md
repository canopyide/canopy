## Context

Exiting maximize mode caused the previously maximized panel to shrink to a tiny size while other panels expanded disproportionately, breaking the restored layout.

## Problem

Three cleanup effects in `ContentGrid.tsx` had `preMaximizeLayout` in their dependency arrays. When layout restoration began, these effects re-fired because `preMaximizeLayout` changed — which cleared the saved layout data before it was fully applied, breaking the restore mid-process.

A secondary issue: `repeat(N, 1fr)` in `gridTemplateColumns` doesn't allow children to shrink below their content size, causing uneven distribution.

## Root cause

Classic React dependency cycle: an effect that reads and modifies the same value it also depends on. The `preMaximizeLayout` state was both the trigger for restoration and the value being consumed/cleared, so the effect interrupted itself.

## Lesson

When effects need to _read_ a value without _reacting_ to its changes, replace the state reference with a `useRef` mirror. The ref always reflects current value but doesn't trigger re-runs. Also use `repeat(N, minmax(0, 1fr))` instead of `repeat(N, 1fr)` in CSS grid to allow children to shrink below intrinsic content width. Add `min-w-0` on flex/grid children that shouldn't impose minimum widths.

## Applies when

- Layout save/restore patterns in ContentGrid
- Effects that "use" a piece of state but shouldn't re-run when it changes
- CSS grid columns where panels need to shrink below content size
