## Context

Tailwind v4's `@theme inline { ... }` block defines CSS custom properties that become Tailwind tokens. The codebase had `--color-accent-primary` defined twice inside the same block, once bound to a dark foreground and once (later) bound to a light foreground via a `.dark` chain.

## Problem

Primary buttons rendered with near-white text on the green accent background — ~2.37:1 contrast, far below WCAG AA. The first (correct) declaration was being overridden by a second declaration further down in the same file.

## Root cause

Duplicate CSS custom property declarations follow the standard cascade: the later declaration wins. There is no warning from the CSS engine or Tailwind compiler — both declarations parse cleanly, and the "wrong" one resolves through a chain of `var()` references that looks legitimate. The cascade is silent, and the token only manifests when applied via `text-primary-foreground` etc., making the root cause invisible from devtools (which just shows the final computed value).

## Lesson

Never declare the same CSS custom property twice in `@theme inline` (or any shared token block). Treat the theme block as a registry: each token gets exactly one declaration. When adding a token that "feels similar" to an existing one, grep the file for any prior definition before adding.

For audits: a quick grep for duplicate `--color-*:` lines in `src/index.css` catches this class of bug. Consider a lint rule or a CI check that parses the theme block and fails on duplicates.

Tailwind v4's `@theme inline` does not deduplicate or warn — it trusts the CSS cascade, which means human diligence is the only guard.

## Applies when

Editing `src/index.css` or any theme token file. Adding new semantic tokens or overriding existing ones. Debugging an element that shows the "wrong" color despite a token appearing to be set correctly. Reviewing PRs that touch `@theme inline` blocks.
