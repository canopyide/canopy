## Context

The `SoundService` relied on `SOUNDS_DIR` resolved relative to the compiled output directory (`dist-electron/electron/services/`). This worked neither in dev (source lives at `electron/resources/sounds/`) nor in packaged builds (files inside ASAR are inaccessible to external OS processes like `afplay`, `paplay`, and PowerShell).

## Problem

Sound playback was silently failing. `existsSync` returned false so `playSound` returned a no-op without any logging. Two separate bugs compounded:

1. Relative path resolution pointed to the wrong directory in both dev and production
2. Sound files were not in `asarUnpack` / `extraResources`, so external OS audio processes couldn't read them from inside the ASAR archive

## Root cause

Path resolution used a compile-relative path rather than runtime-aware resolution. Even if the path had been correct for the compiled layout, ASAR packaging would still have broken external process access because `afplay`/`paplay`/PowerShell cannot read files inside `app.asar`.

## Lesson

For any files that need to be read by **external OS processes** at runtime:

1. Use `app.isPackaged` + `process.resourcesPath` to locate files in production; use `path.join(__dirname, ...)` relative to the source tree in dev
2. Add the files to both `asarUnpack` and `extraResources` in `package.json` so they land outside the ASAR archive
3. Add an explicit error log (not just a silent `existsSync` guard) so path failures are visible during development

The `asarUnpack` requirement is non-obvious: files inside `app.asar` are readable by Node/Electron but not by arbitrary OS processes spawned via `child_process.spawn`.

## Applies when

- Any service that shells out to OS processes (`afplay`, `paplay`, PowerShell, `open`, etc.) to access app-bundled files
- Resources that must survive ASAR packaging (fonts loaded by native modules, sound files, native binaries)
- Silent failure patterns where `existsSync` guards swallow errors instead of logging them
