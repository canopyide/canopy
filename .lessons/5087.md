## Context

Theme switching gained a View Transitions API-based circular clip-path animation that expands from the click coordinates. `document.startViewTransition()` takes a callback that performs the actual DOM mutation.

## Problem

If the Zustand store state is read inside the transition callback via a closure captured at the time `startViewTransition` is called, it can be stale by the time the callback fires — particularly when hover preview has modified the active scheme id between when the user clicked and when the transition actually executes.

## Root cause

`document.startViewTransition(callback)` is asynchronous — the browser waits for the current frame to finish painting before invoking the callback. Any Zustand state accessed via a closure formed before `startViewTransition` is called reflects the state at closure formation time, not at callback execution time. In the theme picker, the hover preview path frequently changes the active scheme id, so the snapshot in the closure would be pointing to the last-previewed scheme rather than the committed scheme.

The fix: resolve state from a fresh `getState()` call **inside** the transition callback, not from a closure. This ensures the transition always applies the scheme that is current at execution time.

## Lesson

Inside `document.startViewTransition(callback)`, never close over React state or Zustand values captured before the call. The callback executes asynchronously after the current paint, so any closured values may be stale. Always use `store.getState()` (or equivalent) inside the callback to read current values.

## Applies when

- Wrapping state mutations in `document.startViewTransition()`
- Any async animation callback (rAF, setTimeout, requestIdleCallback) that needs to apply current store state
- Store values that can change between user gesture and its visual application (hover preview patterns)
