## Context

Canopy embeds webviews for browser and portal panels, using `persist:` partitions so each panel has a persistent session. Users reported Web Storage (localStorage/sessionStorage) data being lost even after setting the correct partition on the `<webview>` tag.

## Problem

Two independent bugs caused storage data loss:

1. **Partition not propagated in `will-attach-webview`**: The handler validated `params.partition` against an allowlist and hardened `webPreferences` (sandbox, contextIsolation, etc.) but never set `webPreferences.partition = params.partition`. In Electron 40+, modifying `webPreferences` in the `will-attach-webview` handler without explicitly re-assigning the partition causes the webview to silently fall back to an ephemeral (non-persistent) session.

2. **Storage flush race on tab destruction**: `PortalManager.destroyView()` and `destroyHiddenTabs()` called `view.webContents.close()` without first flushing pending writes. If `localStorage.setItem()` was called recently, the in-memory write may not reach disk before the renderer terminates.

## Root cause

For bug 1: `will-attach-webview` lets you modify `webPreferences` to harden security, but Electron does NOT automatically carry `params.partition` through to `webPreferences` â€” you must set it explicitly. This is a silent footgun.

For bug 2: `session.flushStorageData()` returns `Promise<void>` at runtime but Electron 40's TypeScript type declarations declare it as returning `void`. Using `await` directly causes a TypeScript error, requiring a cast: `await (session.flushStorageData() as unknown as Promise<void>)`.

## Lesson

- In `will-attach-webview`, always explicitly set `webPreferences.partition = params.partition` after any mutation to `webPreferences`. Electron will NOT automatically preserve the partition from `params`.
- Before calling `webContents.close()` on a tab/view that may have pending Web Storage writes, await `session.flushStorageData()`.
- Watch for Electron type definitions that are wrong at runtime â€” use `as unknown as Promise<void>` pattern when types lag reality.

## Applies when

- Implementing browser/portal panels with persistent webview sessions
- Any `will-attach-webview` handler that mutates `webPreferences`
- Destroying webviews or BrowserViews that the user has interacted with
- Handling Electron session/storage APIs where type defs may not match runtime
