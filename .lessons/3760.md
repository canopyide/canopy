## Context

`TerminalHeaderContent` is wrapped in `React.memo`, but it was re-rendering on every activity update because `buildPanelProps` constructed an inline object literal `{ headline, status, type }` on every call. `Object.is()` comparison always fails for different object references, so the memo never bailed out.

## Problem

The terminal store already had an early-exit guard in `updateActivity` that prevented store updates when activity fields hadn't changed. But this protection didn't help because `buildPanelProps` reconstructed a new activity object on every `useMemo` recomputation triggered by any terminal field change — not just activity fields.

## Root cause

The mismatch between "data stored as flat fields on TerminalInstance" and "data assembled into an object at the props layer." The assembly always produced a new reference. `React.memo`'s shallow comparison saw a new object on every render, defeating the optimization entirely.

## Lesson

When passing assembled objects as props to `React.memo` components, the object must be cached by its constituent values, not constructed inline. Use a per-terminal cache map (`activityCache: Map<id, object>`) and return the cached reference when field values are unchanged.

This is a common pattern where `React.memo` is added but doesn't help because upstream code creates new object references. Audit all object-literal props in `buildPanelProps`-style functions — each one is a potential memo defeat. The `null` ↔ object transition is a special case: `null` is a stable reference, so only the object side needs caching.

## Applies when

- `React.memo` on any component that receives assembled object props (not primitive props)
- Profiler shows components re-rendering despite "unchanged" activity/state data
- `buildPanelProps` or similar prop-building functions used inside `useMemo` with a terminal in deps
