## Context

The Portal feature allows multiple browser tabs implemented as `WebContentsView` instances. Each WebContentsView is a full Chromium renderer process. Without eviction, opening many portal tabs keeps all of them alive indefinitely.

## Problem

Each background WebContentsView tab consumes ~100–200 MB of Chromium renderer memory regardless of whether the user is looking at it. No eviction policy meant memory grew proportionally with tab count.

## Root cause / Design decision

The fix implements LRU eviction in `PortalManager`: only the active tab plus the 2 most recently used tabs (`MAX_LIVE_VIEWS = 3`) remain alive as `WebContentsView` instances. Less-recently-used tabs are destroyed via `webContents.close()`. Their URL is preserved so they can be seamlessly recreated when switched back to.

LRU order is tracked with a `lruOrder` array updated on `switchTab()` and `createTab()`. The `evictLru()` method determines which views to destroy. A new `portal:getEvictedTabs` IPC channel lets the renderer know which tab IDs are evicted so the UI can show a "loading" state on switch.

## Lesson

WebContentsView tabs need LRU eviction at the main-process level — the renderer cannot evict them because only the main process can call `webContents.close()`. Keep an in-memory LRU tracking array alongside the view map. Expose which tabs are evicted via IPC so the renderer can show appropriate loading UI on re-visit. The threshold of 3 live views is a balance between responsiveness (frequent tabs stay loaded) and memory (inactive tabs are freed).

## Applies when

- Adding new multi-tab WebContentsView features
- Debugging high memory usage with many open portal tabs
- Implementing any main-process-managed view pool that needs bounded size
- Extending `PortalManager` with new eviction strategies
