## Context

PR #4598 added ack-based backpressure to the PTY host → renderer MessagePort path using transferable `ArrayBuffer`s. It was fully reverted because it caused all agent terminals to render blank until clicked.

## Problem

Two bugs, both caused by binary `ArrayBuffer` transfer over MessagePort:

1. **Node.js Buffer pool corruption**: `Buffer.from(data)` for small strings uses a shared 8KB slab pool — multiple buffers share the same underlying `ArrayBuffer`. Transferring `bytes.buffer` as a transferable detaches the _entire_ slab, corrupting every other `Buffer` in the PTY host that shared that pool.

   ```js
   const b1 = Buffer.from("hello");
   const b2 = Buffer.from("world");
   console.log(b1.buffer === b2.buffer); // true — same 8192-byte slab
   ```

Fix requires copying into a dedicated `new ArrayBuffer(n)` — never transfer `Buffer.buffer` directly.

2. **Blank terminals after Buffer fix**: Even after patching the slab issue, blank terminals persisted. The full PR revert was needed. Root cause traced to backpressure gating (`portQueueManager.isAtCapacity()`) interacting badly with React StrictMode's mount/unmount/remount double-cycle — the renderer's `onData` callback registration was arriving after the host had already queued and stalled.

## Root cause

Using transferable `ArrayBuffer` with Node `Buffer.from()` corrupts the shared slab pool. Additionally, backpressure capacity checks that gate terminal output interact with the renderer's async callback registration lifecycle in ways that cause blank terminals.

## Lesson

- **Never transfer `Buffer.buffer` directly** over MessagePort. Always copy: `new Uint8Array(Buffer.from(data)).buffer` or a fresh `new ArrayBuffer(n)`.
- For terminal output transport, **strings are sufficient and safe**. VS Code uses strings over MessagePort for their terminal PTY host — no binary transfer, no SAB. AI agent output is <5 KB/s; xterm.js parsing (~25-35 MB/s) is the bottleneck.
- **React StrictMode doubles mount/unmount**: any capacity-based backpressure that stalls before `onData` registers will silently black out a terminal. The `earlyDataBuffer` in `terminalClient.ts` was added to handle this race.
- The reimplementation (this PR) uses string `"ack"` messages in a capacity queue (32 slots) — simple, correct, and avoids all shared memory/transferable semantics.

## Applies when

Implementing binary MessagePort transport in Electron's PTY host, or any backpressure mechanism gating output to renderer callbacks that may not be registered yet (especially under React StrictMode).
