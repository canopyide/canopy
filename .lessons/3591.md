## Context

Canopy defers background work (terminal snapshot restores, idle resize operations) using `requestIdleCallback`. This runs in the renderer process under Electron, where there is continuous workload from terminal output, IPC messages, Zustand updates, and animation frames.

## Problem

`requestIdleCallback` only fires when the main thread has **no pending tasks**. In a live IDE, the main thread never truly idles. The 1200ms timeout fallback eventually fires, but delivers a synchronous burst of work at the worst possible moment — under load rather than at a genuinely quiet time.

## Root cause

`requestIdleCallback`'s semantics are "fire only when truly idle." The 1200ms timeout ensures it eventually runs, but that timeout-triggered call is a synchronous burst, not a gentle background task. Under sustained IDE workload the callback can starve until the timeout fires.

## Lesson

Use `scheduler.postTask({ priority: 'background' })` for deferred non-critical work in Electron renderer processes. It is available on `window.scheduler` in Chromium 94+ (Electron 40 / Chromium 144+) and provides the same deferral semantics as `requestIdleCallback` but with starvation resistance — it is interruptible, de-prioritized relative to user interaction, and guaranteed to run.

Retain a `setTimeout(fn, 0)` fallback for defensive completeness, but `scheduler.postTask` is the right primary mechanism in Electron 40+.

Key constraint: `scheduler.postTask` exists only on `window.scheduler`, not in Node.js main process or utility process contexts. Usage must be scoped to renderer process code only.

`AbortController` can be passed as the signal for cancellation, which is useful for resize-supersession patterns where a queued resize becomes stale before it fires.

## Applies when

Scheduling deferred non-critical work (snapshot restores, idle operations) in the Electron renderer where `requestIdleCallback` starvation is a risk.
