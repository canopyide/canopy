## Context

Canopy migrated from a single `BrowserWindow` renderer to per-project `WebContentsView` instances, each with its own V8 context and independent Zustand stores. The entire E2E suite broke after this migration.

## Problem

Numerous E2E test failures across core and full suites after the WebContentsView migration.

## Root cause — multiple issues

Several distinct bugs were uncovered simultaneously:

**1. Cross-renderer state drift**: Each WebContentsView has its own Zustand store. A project rename in the originating view never reached cached project views. Fix: broadcast `project:updated`/`project:removed` IPC events to all views.

**2. `getAllAppWebContents` missed cached views**: The broadcast helper iterated only the "active" webcontents, ignoring project views cached in `viewToProject`. Cached views never received broadcasts.

**3. E2E `getActiveAppWindow` returned the wrong page**: The welcome `appView` is always first in `contentView.children`, so it shadowed the real active project view. Fix: iterate top-to-bottom and prefer URLs containing `projectId=`.

**4. `useSearchablePalette` stale closure**: `defaultGetItemId` was recreated on every render (not memoized), causing the results memo to recompute every render, and a `useEffect` watching result changes kept resetting `selectedIndex`. Action palette arrow-key navigation was broken for all users post-migration.

**5. `cachedProjectViews` LRU eviction during E2E**: Default LRU size of 1 evicted the onboarding wizard view the moment a second project was added. In E2E mode, defaulted to 4.

**6. Stale `ctx.window` in test helpers**: After every `selectExistingProject` call, the Playwright page reference pointed to the old WebContentsView. Every project-switch helper must re-fetch the active window.

## Lesson

When migrating to multi-renderer architecture (WebContentsView), audit every assumption that "there is one renderer": state broadcasts, IPC sends, E2E page references, and memoized hooks that capture renderer-global singletons. Each `WebContentsView` is a fully isolated JS context — stores, closures, and `getAllWindows()[0].webContents` all break.

The `useSearchablePalette` stale closure is a separate React lesson: if a memo key function is recreated on every render (e.g., an inline arrow function or a non-memoized callback), any `useEffect` that watches the memo output will fire on every render, not just when data genuinely changes.

## Applies when

- Implementing any IPC broadcast that must reach all open project views
- Writing E2E tests that switch between projects (always refresh `ctx.window` after switch)
- Adding memoized selectors/palettes in React (ensure key functions are stable references)
- Setting LRU cache sizes — test environments may need larger limits than production defaults
