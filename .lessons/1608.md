## Context

TUI applications (vim, htop, OpenCode, Claude Code) switch xterm.js into its alternate screen buffer. On terminal resize, the main-buffer code path needs to reflow history, but the alt-buffer path should leave the alt screen alone — TUI apps redraw their own content. The resize controller checked a boolean `isInAlternateBuffer` flag to branch.

## Problem

The resize path cleared the alternate screen every time, wiping TUI state on any container resize. The alt-buffer guard was never firing.

## Root cause

Two independent mechanisms tracked alt-buffer state: a DECSET/DECRST parser handler setting `isInAlternateBuffer`, and xterm.js's own `onBufferChange` event setting `isAltBuffer`. The resize controller read `isInAlternateBuffer`, but nothing ever set that property to `true` — the parser handler had been bypassed or renamed. The two trackers disagreed silently because TypeScript happily accepted both property names on the same object.

## Lesson

When xterm.js exposes first-party state (`onBufferChange`, `buffer.active.type`), treat it as the single source of truth — do not maintain a parallel DECSET/DECRST handler to shadow it. Parallel state trackers drift the moment one side is refactored, and typos in property names between read and write sites are invisible to TypeScript when both properties exist on a loosely-typed object. Consolidating to one source prevents the "nothing ever sets this flag" class of bug. When converting, delete the old property outright rather than leaving it as dead state — unused state becomes a future footgun.

## Applies when

- Integrating with xterm.js events where the library already tracks the state you need (buffer mode, cursor, selection)
- Reviewing code that has both a DECSET/DECRST handler and an xterm.js event listener for the same escape sequence
- A feature "silently stops working" after a refactor that renamed a property without a TypeScript error
