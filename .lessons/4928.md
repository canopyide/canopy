## Context

The recipe store merges three sources: in-repo (`.canopy/recipes/`), project-local, and global. On name collision the intended priority is in-repo > project-local > global, since in-repo recipes are the git-tracked team-shared source of truth.

## Problem

Project-local recipes shadowed in-repo recipes of the same name. A team recipe in `.canopy/recipes/deploy.json` would be hidden by a user's local `deploy` recipe, making the shared recipe invisible.

## Root cause

The filter logic in `mergeRecipes()` was backwards — it was filtering out in-repo recipes that matched a project-local name, rather than filtering out project-local recipes that matched an in-repo name.

## Lesson

When implementing a priority-ordered merge of multiple sources (A > B > C), the filter must exclude lower-priority items that collide with higher-priority items — not the reverse. Write explicit unit tests that assert the dominant source wins on collision from the very first implementation; the bug is undetectable without them. Also ensure the UI reflects the merged result (RecipeManager must filter shadowed recipes from its display list).

## Applies when

Any multi-source merge with a defined priority order in Canopy's recipe, settings, or config system.
