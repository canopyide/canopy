## Context

Terminals receiving output while the user is scrolled up needed a visual indicator. The implementation required an overlay that doesn't interfere with terminal mouse event handling.

## Problem

A naive overlay positioned over the terminal steals mouse events from xterm.js (text selection, middle-click paste, right-click context menu). The entire terminal interaction model breaks if mouse events are captured by an overlay.

## Root Cause

Not a bug — architectural constraint. Standard overlay pattern intercepts pointer events.

## Lesson

For overlays on top of xterm.js terminals: apply `pointer-events: none` to the overlay container and `pointer-events: auto` only to the specific interactive button within it. This allows mouse events to pass through to the terminal except precisely where the button is. The `useUnseenOutput` hook should listen to xterm's `onScroll` and `onData` events to determine scroll state — do not attempt to measure scroll via DOM APIs on the terminal element. Use `data-testid` on the indicator element for reliable E2E targeting.

## Applies when

- Any overlay UI (badges, indicators, action buttons) placed on top of an xterm.js terminal
- Any floating button positioned inside the terminal pane
