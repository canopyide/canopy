## Context

The production CSP was emitted as a `<meta http-equiv="Content-Security-Policy">` tag from vite.config.ts. It included `frame-ancestors 'none'` alongside the other directives. Renderers logged a CSP warning on every load.

## Problem

The browser warning was correct: `frame-ancestors` is one of several directives (along with `sandbox`, `report-uri`, `report-to`) that only take effect when delivered via an HTTP response header. In a meta tag, and especially under Electron's `file://` protocol, there is no HTTP header to carry them. The directive does nothing and pollutes the console.

## Root cause

CSP directive applicability differs by delivery mechanism. It is easy to copy a "complete" CSP from web docs without noticing that a subset silently no-ops in meta form.

## Lesson

When authoring CSP in a meta tag, include only directives that work there: default-src, script-src, style-src, img-src, connect-src, font-src, media-src, object-src, form-action, base-uri, etc. Drop frame-ancestors, sandbox, report-uri, report-to. If you need frame-ancestors in Electron, enforce it via `webRequest.onHeadersReceived` or by setting the header when serving renderer assets over a custom protocol — not in the HTML.

## Applies when

Editing CSP in vite.config.ts, index.html, or any renderer entry. Also during security hardening reviews when tempted to "copy a stricter CSP" from generic web guides into the meta tag.
