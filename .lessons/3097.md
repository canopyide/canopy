## Context

`listCommits` used `git log --format=...` with `|` as the field separator to parse structured commit data (hash, author, date, subject, body).

## Problem

Any commit whose body contained a `|` character shifted all subsequent fields out of position, corrupting the date value and crashing the commits dropdown with `RangeError: Invalid time value`.

## Root cause

Printable characters can legitimately appear in commit bodies, so they are unsafe field delimiters for `git log --format`. Pipes are common (tables, CLI examples, pipelines).

## Lesson

When parsing structured `git log --format` output, use a null byte (`\0`, `%x00`) as the field delimiter — git allows it and it cannot appear in commit content. Also use `-z` for record separation when possible. Combine with defensive guards downstream: validate dates with `isNaN(date.getTime())` before formatting, so one malformed row does not take down the whole list view.

## Applies when

Parsing any `git log --format=...` output into structured records, or generally picking a delimiter for tool output that may contain arbitrary user text.
