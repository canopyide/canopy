## Context

The renderer and main process had V8 heap limits set to 4 GB each, and both had `--optimize-for-size` enabled. The pty-host also had a 4 GB heap limit despite being an I/O forwarding process.

## Problem

`--optimize-for-size` degrades JIT compilation throughput. It was designed for embedded/constrained environments where code size matters more than execution speed. In a desktop IDE with a hot React/Zustand render loop, this flag causes slower UI rendering without meaningful memory savings. The 4 GB heap limits allowed unconstrained allocation growth before GC pressure increased.

## Root cause

`--optimize-for-size` was likely added as a blanket "reduce memory" heuristic without understanding its performance trade-off. For the pty-host specifically, the 4 GB limit was inherited from the renderer configuration — but the pty-host only does I/O forwarding; its ring buffers are off-heap (via `node-pty`), so an in-heap limit of 512 MB is more than sufficient and creates GC pressure sooner.

## Lesson

Do not use `--optimize-for-size` in Electron renderer or utility processes. It trades JIT speed for code size — the wrong trade-off for a desktop IDE. Correct heap limits by process role: renderer ~1536 MB (React/Zustand have many short-lived allocations; add `--max-semi-space-size=64` to encourage earlier minor GC), pty-host ~512 MB (I/O only, off-heap buffers), main process at OS default. Always remove `--optimize-for-size` when found.

## Applies when

- Reviewing V8 flags in `electron/main.ts` or `electron/setup/environment.ts`
- Adding memory reduction flags — check whether the flag applies to server/embedded (code-size) or desktop (throughput) contexts
- pty-host memory pressure — remember its allocations are mostly off-heap
