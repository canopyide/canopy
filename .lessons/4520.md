## Context

AI coding agents running in Canopy terminals (Claude Code, Gemini CLI, Codex) have documented memory leaks that grow 500 MB–1 GB/hour and can reach 10–30 GB before crashing. Users don't notice until the machine degrades or the agent crashes and loses session state.

## Problem

The existing process metric polling (via `ProcessTreeCache`, 2.5s intervals) provided RSS data per terminal, but there was no consumer that analysed trends for leak patterns. Distinguishing a genuine leak from legitimate spikes (indexing, large diffs) required a sliding-window heuristic, not a simple threshold.

A simple "RSS > N GB" threshold produces false positives during codebase indexing (spikes then plateaus). An auto-restart must also not fire while the user is actively typing.

## Root cause

No detection existed. The RSS data pipeline was already in place (via issue #4149) but nothing consumed it for anomaly detection.

## Lesson

**Leak detection heuristic:** require N consecutive polling intervals (10 = ~25 seconds) of monotonic growth above a minimum threshold (500 MB) before flagging. "Near-monotonic" tolerance allows minor fluctuations. This eliminates spike-then-plateau false positives.

**Auto-restart guard:** check for active user input before auto-restarting. If input is detected, block the restart and require explicit confirmation instead of silently killing the process.

**Cooldown + session suppression:** after alerting, apply a 5-minute cooldown before re-alerting the same terminal. If the user dismisses, suppress for the entire session. This prevents notification storms if the leak is slow.

**Implementation pattern:** detection lives in a React hook (`useMemoryLeakDetection`) mounted in the app bootstrap, subscribing to process metrics from the store and dispatching notification actions. Keeps all UI-layer concerns in the renderer; the main process only needs to report metrics.

## Applies when

- Implementing any trend/anomaly detection over time-series process metrics
- Designing auto-restart or auto-recovery features that interact with user input state
- Deciding where to place detection logic (renderer hook vs main-process service)
- Handling the "spike then plateau" vs "unbounded growth" distinction
