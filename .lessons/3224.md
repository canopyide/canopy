## Context

Several agents (Gemini, Codex in some modes) set the terminal title repeatedly to display progress — e.g. `"⠋ Thinking"`, `"⠙ Thinking"`, `"⠹ Thinking"`. The state machine mapped title patterns to agent states (working/waiting/directing).

## Problem

The mapped state flapped rapidly between `working` and `waiting` as the spinner frames cycled past the pattern matchers, producing a visibly jittery dock indicator and spamming state-transition events to subscribers.

## Root cause

xterm's `onTitleChange` fires on every title update, and title-based pattern matchers are naturally fuzzy — different spinner glyphs match different patterns. There is no hysteresis built into either xterm or the agent state machine, so a 10 Hz spinner produces 10 transitions per second. A second contributor: premature `clearDirectingState()` on Enter caused a guaranteed `directing → waiting → working` flash on every prompt submission instead of letting natural debounce handle it.

## Lesson

When deriving agent state from `onTitleChange`, apply per-terminal hysteresis: hold a proposed state for a dwell period (150-300 ms) before committing it. If a different state is proposed during the dwell, cancel and restart — only stable states get through. Also resist the urge to "help" the state machine by clearing directing state on Enter — let the normal debounce expire, because preemptive clears always create a visible flash.

```ts
// Per-terminal dwell timer
if (proposed === lastCommitted) return;
clearTimeout(this.titleDwellTimers.get(id));
this.titleDwellTimers.set(
  id,
  setTimeout(() => {
    commitState(id, proposed);
  }, TITLE_HYSTERESIS_MS)
);
```

## Applies when

Any UI state derived from a high-frequency signal: terminal title changes, PTY output bursts, file-watcher events, websocket heartbeats. If the source can fire at frame rate, you need dwell-based hysteresis — not just debounce.
