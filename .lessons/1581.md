## Context

Panels are serialized to disk on exit and rehydrated on startup. `AppStateTerminalEntrySchema` required `type` (shell type) and `cwd` (working directory) as non-optional. When browser and notes panels were added, they inherited the schema but had no `type` or `cwd` — Zod validation rejected them, and they were silently dropped from the restored state.

## Problem

Users would set up a workspace with mixed panel kinds (terminals, browsers, notes) and on restart only the terminals would come back. Panel ordering was also lost because the restoration code ran PTY panels first then non-PTY, rather than preserving the saved order.

## Root cause

The schema was written before the panel kind union was introduced. When new kinds arrived, the schema's required fields were never loosened, and older saved states (from before the `kind` field existed) had no way to be parsed at all. This is the classic "add a new variant to a discriminated union without migrating the serializer" regression.

## Lesson

When a schema outlives its original use case, required fields are liabilities. For panel-like discriminated unions: make per-kind fields optional at the schema layer, then validate per-kind in a refinement (`z.object({...}).superRefine(...)`) that requires `type`/`cwd` only when `kind` is PTY-backed. Add a kind-inference fallback for older saved states that lack `kind` entirely (e.g. if `browserUrl` is present, it's a browser). Use a single-pass restoration loop that preserves saved ordering rather than a two-pass `PTY-then-others` split — ordering is user-visible state and splitting it across two loops always loses it.

## Applies when

Adding a new panel kind, new recipe type, or any new variant to a persisted discriminated union. Also the general lesson: check that the persistence schema, runtime type, and restoration order-preservation all agree whenever you widen a union. A round-trip test (save → clear → hydrate → assert identity and order) catches this before it ships.
