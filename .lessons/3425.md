## Context

Git PATH discovery on Windows only checked a small set of standard install locations. Users installing Git via Scoop, Chocolatey, or the 32-bit installer on 64-bit Windows had Canopy fail to find Git, breaking all git operations.

## Problem

Users on Windows with non-standard Git installs saw "Git not found" errors on launch even with Git correctly installed and working in their system shell.

## Root cause

The candidate path list missed:

- `C:\Program Files (x86)\Git\cmd\git.exe` (32-bit installer on 64-bit OS)
- Scoop shims (`%USERPROFILE%\scoop\shims\git.exe`)
- Chocolatey bin (`C:\ProgramData\chocolatey\bin\git.exe`)

Additionally, `~/.local/bin` was included in the Windows candidate list — this is a Unix/XDG path that never exists on Windows.

## Lesson

Windows Git discovery should include a **Registry lookup** as the most authoritative source, plus all common package manager paths:

```ts
// Registry (most authoritative):
reg query HKLM\SOFTWARE\GitForWindows /v InstallPath

// Candidate paths:
'C:\\Program Files\\Git\\cmd\\git.exe',
'C:\\Program Files (x86)\\Git\\cmd\\git.exe',
path.join(homedir(), 'scoop', 'shims', 'git.exe'),
'C:\\ProgramData\\chocolatey\\bin\\git.exe',
```

Never include Unix-only paths (`~/.local/bin`) in Windows discovery. Always check the Registry before falling back to filesystem paths — it's updated by any official Git installer.

## Applies when

Any cross-platform tool that needs to locate Git (or other developer tools) on Windows. Package manager diversity on Windows is far higher than on macOS/Linux.
