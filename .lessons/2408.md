## Context

Dev servers (Vite, Next.js, Webpack) print startup URLs wrapped in OSC 8 hyperlink escape sequences and occasionally use 8-bit C1 control bytes. UrlDetector strips ANSI before regex-matching the URL.

## Problem

30-50% of dev preview sessions showed a blank webview. The URL appeared in the terminal but the Electron webview never transitioned to `running`. Output buffer scanning silently missed the URL.

## Root cause

Two grammar gaps in the ANSI stripper. (1) OSC 8 hyperlinks can be terminated by ST (`ESC \`) as well as BEL; the BEL-only regex left the ESC-terminated form intact, so the URL character class captured ESC/BEL bytes and broke the match. (2) C1 8-bit OSC (0x9D) was not handled at all. The URL-character class included `\x80-\x9f`, letting C1 control bytes pose as URL characters. Also: the 4096-char rolling buffer was too small — verbose webpack progress bars could push the URL out of the detection window before scanning.

## Lesson

When stripping ANSI to extract data from PTY output, handle both OSC 8 terminators (BEL and ST), 8-bit C1 OSC (0x9D), and exclude `\x80-\x9f` from any character class that represents "printable" payload. Use `[^\x07\x1b]*` — not `.*` — inside OSC bodies so unterminated sequences do not over-strip. Size rolling detection buffers for the noisiest dev server, not the quietest.

## Applies when

Adding or modifying PTY output scanners, URL detectors, prompt detectors, or anything that decodes ANSI/OSC sequences from agent or dev server terminals.
