## Context

Canopy has an integrated browser panel backed by a webview. The `will-navigate` handler in `protocols.ts` was intended to block unsafe navigations (non-HTTP/HTTPS schemes, e.g., `javascript:`, `file:`).

## Problem

OAuth/OIDC flows and any web app that redirects to an external domain silently broke. The browser appeared to do nothing when clicking "Sign in with Google" or any button that triggered a cross-origin redirect.

## Root cause

The `will-navigate` handler's condition was too aggressive — it blocked all navigations where the destination origin differed from the source origin. Cross-origin navigation via `window.location.href`, `window.location.assign()`, and `<a target="_self">` are all completely normal browser behaviors. The intent was only to block non-HTTP/HTTPS scheme navigations.

## Lesson

In Electron's `will-navigate` handler, block only navigations to dangerous schemes (`javascript:`, `file:`, `data:`, `vbscript:`), not cross-origin navigations. Cross-origin HTTP/HTTPS redirects must be allowed.

When a navigation is blocked, emit an IPC event to the renderer (`browser:navigation-blocked` with the URL and reason) so the UI can display an inline warning banner. Silent failures in webviews are extremely hard to diagnose — always surface blocked navigations to the user.

```ts
// Allow cross-origin HTTP/HTTPS; block dangerous schemes only
const url = new URL(details.url);
if (!["http:", "https:"].includes(url.protocol)) {
  event.preventDefault();
  // notify renderer
}
```

## Applies when

- Integrated browser panels with webviews
- Any `will-navigate` or `will-redirect` handler in Electron
- OAuth flows, OIDC redirects, or any multi-step auth that redirects externally
