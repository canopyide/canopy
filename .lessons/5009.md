## Context

Canopy uses `WebContentsView` per project. When switching projects, the outgoing view is evicted (destroyed) to free memory. Before eviction, any unsaved renderer state must be persisted. Panel state uses a 500ms debounce write to disk.

## Problem

Browser and dev-preview panels lost their state on project switch. After switching away and back, those panels were gone. Terminal/agent panels survived because they have a live PTY process in the main process as an anchor; browser/dev-preview panels are purely renderer-side state.

## Root cause

The project switch flow was fire-and-forget from the renderer (`projectClient.switch()` with no await). The main process immediately evicted the old `WebContentsView`. The renderer's `beforeunload` handler attempted an async IPC flush, but the webContents was destroyed before the IPC could complete. The 500ms debounce window also hadn't fired for recent changes.

## Lesson

When switching `WebContentsView` projects, never rely on `beforeunload` + debounced IPC flushes to persist renderer state — the view can be destroyed mid-flight. Instead, capture the outgoing state **synchronously** in the renderer before initiating the switch, embed it in the switch IPC payload, and have the main-process handler write it **before** destroying the old view. This makes persistence synchronous from the renderer's perspective, with no race window.

Pattern:

1. Renderer captures outgoing snapshot synchronously (no debounce)
2. Snapshot travels in the `switchProject` IPC request payload as `outgoingState`
3. Main-process handler writes `outgoingState` to disk first, then swaps views

## Applies when

Any project-switch or view-teardown flow where renderer state must be preserved. Equally applicable to tab/workspace concepts that destroy renderer contexts.
