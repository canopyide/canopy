## Context

Sound playback in Canopy previously spawned OS-level processes: `afplay` on macOS, `paplay` on Linux, and a PowerShell command on Windows. This had high latency, poor mixing support, and was fragile across OS versions.

## Problem

The OS process approach meant:

- ~200ms+ latency from spawn overhead before audio starts
- Sounds can't be mixed/layered cleanly
- Different code paths per platform are hard to maintain
- Process spawn errors are hard to surface to the user

## Root cause (architecture decision)

Sound synthesis produces WAV files at build time. Playing WAVs natively requires either OS-level process spawning or a Web Audio API. Since Canopy already has a renderer process (Chromium), Web Audio API is the better fit — pre-decoding the buffer at startup gives near-instant playback.

## Lesson

- Route sound playback through the renderer via `sound:trigger` / `sound:cancel` IPC channels rather than spawning OS processes from main.
- `WebAudioService` pre-decodes all WAV buffers at renderer startup so playback is synchronous (no decode latency).
- Sound files must be served via a custom protocol (`app://`) so Web Audio `fetch()` can load them — `file://` is blocked by Electron's CSP in the renderer.
- Main retains an OS fallback path for when no renderer window is available (e.g., sounds triggered during app startup before a window is ready).
- `AudioContext` auto-suspends when idle — no need to manually pause/resume for battery efficiency.

## Applies when

- Adding new sounds or audio feedback to the app
- Any cross-platform media playback in an Electron renderer
- Routing main-process events that need renderer-side side effects (sounds, toasts)
