## Context

Creating a worktree from a GitHub PR failed for fork PRs because the branch wasn't available locally. The UI previously skipped or disabled fork PRs entirely, treating them as unsupported.

## Problem

Fork PR branches don't exist in the upstream remote's normal branch namespace. Attempting to resolve `headRefName` as a local branch fails with "branch not found." The app either showed an error or silently fell back to the main branch.

## Root cause

`git fetch origin <branchName>` only works for same-repo branches. Fork PR branches live in the contributor's fork, not in the upstream remote's branch refs.

## Lesson

- **The correct fetch refspec for any PR (same-repo or fork) is `pull/{number}/head:{branchName}`**: GitHub exposes all PR heads (including forks) at this synthetic ref. This is the same mechanism `gh pr checkout` uses internally.

  ```
  git fetch origin pull/123/head:feature-branch-name
  ```

- **Fetch before branch resolution**: call `fetchPRBranch(prNumber, headRefName)` before attempting worktree creation. The branch then exists locally and resolves normally.
- **Remove fork-PR gates in the UI**: once fetch-before-create is in place, the hover icon, context menu item, and bulk dialog no longer need `!isForkPR` guards. Fork PRs work identically to same-repo PRs.
- This refspec also works for draft PRs and closed PRs (the branch is still fetchable via the PR number even after merge).

## Applies when

Creating local worktrees or branches from GitHub PRs, particularly fork PRs. Any feature that requires the PR's branch to exist locally before git operations can proceed.
