## Context

Canopy persists window size and position. When the app is closed on an external monitor and reopened without that monitor attached, the restored window may have dimensions larger than the current display.

## Problem

A window saved at 2560×1440 on an external monitor was being restored on a 1920×1080 display. `win.center()` was called after clamping the position, but Electron/macOS collapses the window to its minimum size when `center()` is called on a window whose set size exceeds the display's work area.

## Root cause

`clampToDisplay()` only clamped `x`/`y` coordinates — it did not reduce `width`/`height` to fit within the work area. The off-screen recovery path triggered `win.setSize()` and `win.center()` only for windows that were mostly off-screen, not for windows that were oversized but positioned at the origin (0,0). A window at origin that exceeds display bounds appears "on screen" by position but is still broken.

## Lesson

When restoring window state, clamp `width` and `height` to the display's work area dimensions first, then clamp position. The off-screen recovery condition must also trigger when the window is oversized relative to the current display, regardless of whether the window's origin position is on-screen. Call `win.setSize()` with the clamped dimensions before `win.center()`.

The two-step fix: (1) `clampToDisplay` reduces dimensions to work area, (2) off-screen recovery triggers on `oversized || mostlyOffScreen`.

## Applies when

- Restoring persisted window state across monitor configurations
- Any `win.center()` call preceded by `win.setSize()` where the size may exceed the display
