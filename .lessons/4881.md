## Context

`TerminalRegistry` tracks which project each terminal belongs to. It had a `lastKnownProjectId` fallback field intended to help match terminals during project transitions.

## Problem

After a project switch, agent terminals from the old project appeared in the new project's terminal list ("phantom" terminals). The fallback logic that was meant to smooth transitions was instead allowing cross-project contamination permanently.

## Root cause

`terminalMatchesProject()` and `terminalBelongsToProject()` both fell back to `lastKnownProjectId` when the primary `projectId` check failed. The `lastKnownProjectId` was set in `onProjectSwitch()` as a bridge for the transition window, but there was no expiry or cleanup — so it persisted indefinitely, matching terminals to a project they no longer belonged to.

The fix removed the `lastKnownProjectId` field entirely. The two matching methods now only check the terminal's actual `projectId`. Without the stale fallback, no terminal can match a project it was never assigned to.

## Lesson

Fallback/bridge state intended for use only during a transition window is dangerous if it has no TTL or cleanup path. If a fallback is needed during project switch, scope it to the duration of the switch and clear it explicitly on completion, rather than relying on subsequent operations to overwrite it.

Prefer removing a leaky fallback over patching it — if the correct `projectId` is available at assign time, there's no legitimate need for a stale fallback.

## Applies when

- Any registry pattern that uses a "last known" or "previous" ID as a fallback.
- Project-switch code that sets transitional state: always define how and when that state gets cleared.
