## Context

Canopy uses a two-way sync mechanism for worktree activation: the renderer calls `setActive` IPC, the backend processes it, then emits `WORKTREE_ACTIVATED` back to the renderer so that the `onActivated` listener can apply the selection. This same echo is also used for backend-initiated activations (sidecar commands, auto-switch on removal).

## Problem

When the renderer initiated the activation and the backend unconditionally echoed the event back, the echo could arrive after the renderer had already switched to a _different_ worktree. The async round-trip delay caused `WORKTREE_ACTIVATED` for worktree A to land after worktree B was already selected, switching the UI back to A. During bulk creation (3+ worktrees) this created a visible cascade of incorrect switches.

The equality check in `selectWorktree` normally made the echo a no-op, but the async gap broke that invariant.

## Root Cause

`WorkspaceClient.setActiveWorktree()` unconditionally called `sendToRenderer(CHANNELS.WORKTREE_ACTIVATED, { worktreeId })` after every activation, regardless of who initiated it. The renderer listened and re-applied selection on every such event. Neither side knew whether the other had initiated the change.

## Lesson

When both sides of an IPC pair can initiate the same operation, unconditional echo-back creates a race. Tag the initiator at call time (`source: "renderer"` vs omitted for backend) and suppress the echo when the initiator is the same side that will receive it. The renderer needs the echo only for changes it did not initiate. For renderer-initiated changes, the invoke return value provides confirmation.

Backend-initiated paths (sidecar, auto-switch on removal) must still emit the event because the renderer has no prior knowledge of those changes.

## Applies when

- An IPC call triggers an event that the caller is already subscribed to
- Rapid sequential operations where the async round-trip can deliver stale events
- Any bidirectional sync where echo suppression prevents re-entry loops
