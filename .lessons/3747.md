## Context

All GitHub GraphQL/REST API calls and simple-git operations had no explicit timeout. During API outages or slow networks, these calls could hang indefinitely, freezing UI operations that depended on their results.

## Problem

Without timeouts, a single slow GitHub API call or a git network operation (fetch, push, clone) could stall the entire feature depending on it — leaving the user with no feedback and no way to recover except restarting the app.

## Root Cause

The initial implementations had no timeout configuration. simple-git's constructor accepts a `timeout` option that was not being used. Octokit/GraphQL calls had no `AbortController` signal attached.

## Lesson

Apply timeouts at both layers:

**GitHub API (Octokit/GraphQL):**

```typescript
const controller = new AbortController();
setTimeout(() => controller.abort(), 15_000);
await client.graphql(query, { ...vars, signal: controller.signal });

// For REST fetch calls, use the static helper:
await fetch(url, { signal: AbortSignal.timeout(15_000) });
```

**simple-git:**

```typescript
simpleGit(cwd, {
  timeout: { block: 30_000 }, // 30s for local commands
});
```

Timeout errors should flow through the existing `parseGitHubError` handler so timed-out requests serve stale cached data instead of showing an error — maintaining UI responsiveness during transient outages.

Pass `AbortSignal` through `createClient()` as an optional parameter so the timeout controller is created at the call site (where context exists) rather than buried in the auth layer.

## Applies when

- Any new GitHub API call (GraphQL or REST) in `electron/services/`
- Any new `simpleGit()` construction (always provide timeout config)
- Network operations that should degrade gracefully (serve stale data) rather than hang
