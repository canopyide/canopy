## Context

Canopy moves terminals between activity tiers (foreground, background, hibernated) to save CPU. Each transition tears down and re-creates xterm.js renderer state. Agent output continues to arrive from the PTY host while these transitions happen.

## Problem

After a tier transition — particularly `BACKGROUND → FOCUSED` via `wakeAndRestore()` — the xterm.js buffer kept receiving data but the display stopped repainting. Output was accumulating internally but the user saw a frozen screen. The only user-level recovery was to click away and back, which triggered the browser's visibility change and forced a repaint.

## Root cause

xterm.js's DOM/canvas renderer can land in an internally consistent but visually stale state when the renderer backend is swapped or when the element's visibility changes mid-write. There is no public API to detect this — the internal state is not stuck from the library's perspective, so no error or event fires. Once stuck, only a forced `refresh()` or visibility-driven repaint recovers.

## Lesson

When swapping xterm.js renderer backends (webgl/dom/canvas) or when tier changes cause the terminal element to toggle visibility, force a `term.refresh(0, term.rows - 1)` after every tier transition completes — success, failure, or skipped paths all need it. For long-running agent terminals, add a cheap periodic health refresh (once every ~2s) while the terminal is known to be receiving data, gated on working/active state so idle terminals are untouched. This is a workaround, not a fix for the underlying xterm.js stuck state — the library has no public hook to detect it — so defensive refreshes are the reliable option.

## Applies when

- Implementing terminal activity tiers, hibernation, or renderer-backend swaps
- Users report "terminal frozen until I click it" symptoms
- Adding any code path that calls `wakeAndRestore`, detaches/reattaches xterm, or changes `display`/`visibility` on a parent element mid-stream
