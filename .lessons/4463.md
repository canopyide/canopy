## Context

`canopy-cli.sh` is a bundled shell script that allows opening projects in Canopy from the terminal. It needs to locate the Canopy binary to pass the project path to the running app. The `--status` flag worked, but the project-opening path failed on all Linux installs.

## Problem

The Linux fallback used `command -v canopy-app` to locate the binary. This fails in both distribution formats:

- **AppImage**: The binary runs from a FUSE-mounted path (`/tmp/.mount_*/canopy-app`) — not in `$PATH`. The `$APPIMAGE` environment variable holds the actual path, but the script didn't use it.
- **.deb**: Installs to `/opt/Canopy/canopy-app` — the package created no symlink in `/usr/bin/` so `command -v` found nothing.

The failure message was "Canopy executable not found" with no indication of which lookup strategy failed.

## Root cause

Single-strategy binary discovery assumes `$PATH` is always populated — valid on macOS (the .app bundle is in a well-known location) but wrong for Linux's diverse distribution formats.

## Lesson

Linux binary discovery needs a **multi-strategy ordered candidate list**:

```bash
for candidate in \
  "${APPIMAGE}" \
  "/opt/Canopy/canopy-app" \
  "/usr/lib/canopy-app/canopy-app" \
  "${HOME}/.local/share/canopy-app/canopy-app"; do
  if [[ -n "$candidate" && -x "$candidate" ]]; then
    "$candidate" ...
    exit 0
  fi
done
command -v canopy-app  # final PATH fallback
```

For `.deb` packages, always include a `postinst.sh` that creates a `/usr/bin/canopy-app` symlink, and a `postrm.sh` that cleans it up. Wire both into `electron-builder`'s deb config. Without a PATH symlink, users must know the exact install path.

## Applies when

Any Electron app distributed on Linux via AppImage and/or .deb. The `$APPIMAGE` env var is set automatically by AppImage's runtime — always check it first in any binary self-reference.
