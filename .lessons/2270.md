## Context

The hybrid input bar computes how many lines its content occupies by dividing `scrollHeight` by a constant `LINE_HEIGHT_PX` and uses the result to size the collapsed height.

## Problem

At non-100% browser zoom (50%, 125%, 150%, 200%) the empty input bar rendered taller than one line and grew unevenly as lines were added. Scrollbars flickered near the max boundary.

## Root cause

Browser zoom scales computed `lineHeight` by a fractional factor — `20px` becomes `25.333px` at 125%. Dividing `scrollHeight` (also fractional) by a hardcoded integer `LINE_HEIGHT_PX = 20` returns more than 1 for a single empty line due to rounding error. `Math.ceil` then inflates the displayed line count. Empty documents in particular had `scrollHeight > 0` from baseline metrics even with no text.

## Lesson

For any DOM-based line or row counting: read the actual line height from the element via `getComputedStyle(...).lineHeight` (or measure a test element), never hardcode pixel constants. Add a small epsilon (~2px) when comparing fractional measurements so rounding error does not tip a count over a threshold. Short-circuit the empty-document case to return 1 line (or 0) directly — do not depend on the measurement path to produce that answer. Use `Math.round` rather than `Math.ceil` when snapping to line boundaries, and align scroll-state thresholds with the same epsilon to avoid scrollbar flicker at max.

## Applies when

Implementing auto-resizing inputs, scroll-to-line logic, or any DOM measurement that must be accurate across user zoom levels and display DPI.
