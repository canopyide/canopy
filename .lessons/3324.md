## Context

`useConsoleCaptureStore` retained per-pane message entries forever because its only cleanup path was the React component unmount in `BrowserPane.tsx`. `onTerminalRemoved` in `terminalStore` knew how to orchestrate teardown but never called into the capture store.

## Problem

Programmatic panel removal (project reset, close-all, drag-out) never unmounted the component, so captured messages leaked indefinitely in memory across sessions.

## Root cause

Relying on component unmount as the cleanup trigger only works for user-driven close actions. Any path that destroys state without rendering (reset, migration, restore) bypasses the cleanup entirely. The store orchestration layer (`onTerminalRemoved`) is the single source of truth for "this terminal/panel is going away" — every peer store that holds entity-keyed data needs to be notified from there.

## Lesson

Entity teardown must be coordinated from the store that owns the lifecycle (`terminalStore.onTerminalRemoved` in this codebase), not from React component unmounts. Every peer store with entity-keyed data needs a `remove(id)` call from that single choke point. Component unmount cleanup is fine as belt-and-suspenders, but never the primary path. Use a dynamic import (`import('./consoleCaptureStore')`) if you need to avoid a circular dependency, but call it synchronously — do not `await` it, the cleanup must complete in the same tick as the removal.

## Applies when

Adding a new Zustand store that holds per-terminal/per-panel/per-worktree state. Audit `onTerminalRemoved` (and similar `onXxxRemoved` orchestrators) and add your cleanup call there. Also applies when investigating memory leaks during rapid open/close cycles.
