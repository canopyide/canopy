## Context

The terminal service forwarded every `onData` chunk to the agent state machine as "user input", which is what drives the `directing` state (the user is typing a prompt).

## Problem

Clicking into an agent terminal, or the terminal merely receiving focus, bounced the agent state into `directing` and back. Click-to-focus alone was enough to flash the wrong state indicator.

## Root cause

xterm's `onData` handler is a single pipe for everything the terminal emits toward the PTY — not just keyboard input. When mouse tracking is enabled by the running program (e.g. Claude, nano, vim) or when `FocusEvent` reporting is on, xterm writes escape sequences through the same `onData` channel:

- X10 mouse: `\x1b[M...` (three-byte positional payload follows)
- SGR mouse: `\x1b[<...`
- URXVT mouse: `\x1b[<digits>;<digits>;<digits>M/m`
- Focus in/out: `\x1b[I` / `\x1b[O`

All of these are "data from xterm to PTY" and arrive through `onData`, indistinguishable by transport from a real keystroke.

## Lesson

Filter non-keyboard escape sequences before treating an `onData` chunk as user input, but still forward the raw bytes to the PTY so mouse/focus protocols keep working:

```ts
function isNonKeyboardInput(data: string): boolean {
  return (
    /^\x1b\[M/.test(data) || // X10 mouse
    /^\x1b\[<\d+;\d+;\d+[Mm]/.test(data) || // SGR mouse
    /^\x1b\[\d+;\d+;\d+[Mm]/.test(data) || // URXVT mouse
    data === "\x1b[I" ||
    data === "\x1b[O" // focus in/out
  );
}
```

Only gate the "treat as user typing" side of `onData` behind this filter — do not drop the data.

## Applies when

Any xterm `onData` handler that infers semantic user intent (activity monitors, state machines, draft-input persistence, idle detection). If the terminal has mouse tracking or focus reporting enabled, you will see these sequences.
