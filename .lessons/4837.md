## Context

Electron exposes `app.setUserDataPath()` and `v8.enableCompileCache()` to enable V8 bytecode caching. The main process bootstrap already used this, but the two UtilityProcesses (`pty-host` and `workspace-host`) had no compile cache, so V8 re-parsed and re-compiled their import trees on every app launch.

## Problem

Both UtilityProcesses paid the full V8 parse/compile cost on every cold start. For long-running services like the PTY host this adds measurable startup latency.

## Root cause

`enableCompileCache()` must be called before any imports, at the very start of the entry point. The existing `pty-host.ts` and `workspace-host.ts` had substantial import trees that prevented retroactively adding the call — it had to be the first line of a new bootstrap file.

Additionally, the main process `bootstrap.ts` was using the OS temp dir default for cache storage, which can cause permission conflicts in multi-user environments.

## Lesson

`v8.enableCompileCache()` must be called before any `import`/`require` to be effective. The only clean way to retrofit it is a thin bootstrap wrapper that:

1. Reads `CANOPY_USER_DATA` from `process.env`
2. Calls `enableCompileCache(path.join(userData, 'compile-cache/<process-name>'))`
3. Dynamic-imports the real entry point

Each process should use a dedicated subdirectory under `userData` (not OS temp) to avoid permission issues and to keep caches scoped per process.

The Rollup build must add the bootstrap files as explicit entry points.

## Applies when

- Adding a new UtilityProcess entry point — always create a bootstrap wrapper with `enableCompileCache()` first.
- The `userData` path must be passed via environment variable since `app.getPath('userData')` is only available in the main process.
