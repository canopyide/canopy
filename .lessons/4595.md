## Context

The project resource badge shows memory health status. PR 4595 evolved it into a full diagnostics panel with real-time process metrics (PID, type, memory, CPU%), V8 heap progress bar, and trend indicators.

## Problem

Displaying meaningful memory trend data requires more than just a current snapshot. A single reading can't tell you if memory is growing, stable, or declining. Showing a raw MB number also gives users no actionable context.

## Solution patterns worth recording

**Conditional polling rate**: The diagnostics popover polls at 3-5s intervals, but **only while the popover is open**. When closed, polling stops entirely. This prevents continuous IPC overhead for a feature most users never open.

**Linear regression for trend arrows**: Rather than comparing just the last two samples (noisy), a 12-sample sliding window with least-squares linear regression produces a stable growth rate in MB/min. This eliminates false "growing" signals from GC fluctuations.

**State-driven badge coloring**: Three states with clear thresholds:

- Normal: < 1.5GB total
- Elevated: > 1.5GB OR growing > 3MB/min
- Critical: > 2.5GB OR V8 heap > 85%

## Lesson

- Poll expensive diagnostics (process list, heap stats) only when the UI displaying them is open — wire polling start/stop to popover `onOpenChange`.
- Use a sliding window with linear regression rather than delta-between-last-two for trend indicators. With 5s polling, a 12-sample window gives a 60s trend that's resistant to momentary GC spikes.
- Separate the IPC channels for different data freshness needs: `diagnostics:get-process-metrics` (changes per second, needs fast polling) vs `diagnostics:get-heap-stats` (changes per few seconds, can be slower).
- Express memory thresholds in absolute MB, not percentages — available system RAM varies wildly between machines.

## Applies when

- Building any live diagnostics or monitoring UI in Electron
- Implementing trend/sparkline indicators from sampled data
- Deciding polling frequency and when to pause polling
