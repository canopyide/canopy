## Context

After committing in a worktree, the sidebar worktree card kept showing "uncommitted changes" for several refresh cycles before finally updating.

## Problem

The git status cache had a fixed ~15 second TTL, but active worktrees poll every 2 seconds and background ones every 10. Every poll within the TTL window returned the exact same cached result, so 6-7 polls in a row showed stale data. The cache looked like it was working (high hit rate) but was silently defeating the polling.

## Root cause

Cache TTL was chosen in isolation from the caller's polling cadence. Whenever a cache's job is to dedupe requests that happen faster than the refresh cadence, its TTL must be strictly less than the cadence, otherwise the cache absorbs refreshes instead of speeding them up. The caller and the cache had been written by different features and nobody checked they were compatible.

## Lesson

When a cache sits between a poller and an expensive operation, its TTL must be `pollInterval - epsilon`, not a constant. Ideally the cache takes the TTL per-key from the caller (e.g. `get(key, { ttlMs })`), or the caller passes `forceRefresh: true` on every poll and relies on in-flight dedupe only. Guard against non-finite intervals (NaN, Infinity) before subtracting from them. Write a test that commits, polls, and asserts the status updates within one poll cycle — pure cache-hit-rate tests will happily pass while the UI stays stale.

## Applies when

Reviewing any cache-with-TTL where the consumer is a poller (git status, GitHub stats, file system watchers, metrics, health checks). Adding new worktree-scoped data. Debugging "UI shows old data even though I changed something" bugs with no obvious stale read.
