## Context

PTY host and renderer each track a terminal's "activity tier" (foreground/visible/background/hibernated). The PTY host backgrounds terminals when their project is inactive to reduce CPU; the renderer applies matching throttling to xterm.js. When a project is switched back, the renderer reconnects to existing PTY processes.

## Problem

After switching away and back, agent terminals froze — output arrived but the renderer would not pump it to xterm.js. The two sides silently disagreed about the current tier.

## Root cause

On reconnect the renderer assumed fresh terminals start in the "active" tier. The PTY host, however, had already backgrounded them during the away period and was emitting throttled data consistent with a background tier. The renderer's wake logic refused to process data it believed should be arriving at a higher rate, and nothing in the reconnect snapshot carried the backend's current tier. The two processes walked out of sync the moment the project switched, and reconnect did not reconcile them because the IPC snapshot schema had no `activityTier` field.

## Lesson

Whenever two processes maintain parallel copies of the same state (tier, visibility, focus, activity), the reconnect/hydration path must ship the authoritative value alongside the handle. Do not default to optimistic values ("assume active") on reconnect — take the backend's word. Extend the snapshot type (`BackendTerminalInfo`, `TerminalPublicState`) to carry every piece of state the renderer will assume, then have the renderer apply the backend value instead of re-initializing. The same fix needs a companion callback during the project switch so that live tier transitions after reconnect stay synchronized. Generally: if a field influences renderer behavior and the backend owns it, it must be in every IPC message that crosses the boundary after a disconnect.

## Applies when

- Adding any per-terminal state that both main and renderer need to agree on
- Reviewing reconnect/hydration code paths — audit every field the renderer assumes about a freshly-reconnected resource
- Debugging "works on first open, freezes on second open" symptoms after project/worktree switches
