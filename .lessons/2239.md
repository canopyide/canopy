## Context

After switching away from a project while a webview panel (browser or dev preview) was loading and then switching back, the webview got stuck showing the loading overlay forever. The URL was fine, pages loaded, but the UI never hid the "Loading..." state.

## Problem

The React component gated its loading UI on an `isWebviewReady` state that was initialized to false and only flipped to true on the `dom-ready` event. When the component remounts against a webview that is already dom-ready (because the underlying WebContentsView was preserved), `dom-ready` has already fired and will not fire again. The gate stays closed forever.

## Root cause

`dom-ready`, `did-finish-load`, and friends are edge events, not queryable state. Remounting React components that listen to them needs to probe the webview's current state on mount (via `getURL()`, `isLoading()`, etc.) because the events that transitioned it into that state already fired — possibly while the previous component instance was unmounted, possibly while the webview lived in a different window. Additionally, gating the listener setup on "ready" creates a chicken-and-egg: you cannot learn that it is ready without listening.

## Lesson

For any Electron webview or `<webview>` element: on mount, check the current state explicitly (`getURL()`, `isLoading()`) and seed local state accordingly, BEFORE attaching event listeners. Then attach listeners unconditionally so they catch future transitions. Add a timeout-based recovery (reload after ~30s stuck in loading) for cases where events truly never fire. Track the webview element itself in effect dependencies so listeners are re-attached when the element is recreated (e.g. after project switch or WebContentsView recycle).

## Applies when

Building or debugging any React component wrapping an Electron webview, BrowserView, WebContentsView, or `<webview>`. Project-switch bugs where UI state does not match actual webview state. Any "stuck loading" bug on remount. LRU-evicted views being reattached.
