## Context

Canopy protects the main worktree from deletion and treats it specially in several flows. Originally `isMainWorktree` was set to `worktrees.length === 0` during discovery — i.e. "first entry returned by git worktree list".

## Problem

In PR review workflows the main worktree's terminals were being closed as if it were a secondary worktree scheduled for cleanup.

## Root cause

Git does not guarantee the main worktree is always first in `git worktree list`. The order depends on internal git state, refs/worktrees directory traversal, and platform filesystem ordering. Relying on output order is a heuristic that fails under stress. Even a correct "first entry" rule breaks on Windows when comparing paths whose case differs between git's output and the stored project root, or when one side is a symlink and the other is a realpath.

## Lesson

Identify the main worktree by comparing canonicalized paths to the project root — never by git output order. The comparison must: resolve symlinks via `fs.realpathSync` on both sides; apply case-insensitive comparison on Windows; normalize trailing slashes. Update `isMainWorktree` on every monitor sync so the flag stays correct across worktree add/remove cycles, and do not fall back to git-order logic when the project root is unavailable — fail closed instead.

## Applies when

Touching worktree identification, any cleanup handler that decides what to remove, or path-comparison code that crosses platform boundaries (macOS/Linux vs Windows) or symlink boundaries.
