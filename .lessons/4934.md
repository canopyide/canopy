## Context

E2E test "open 3 terminals" setup was spawning terminals in a loop with fixed `T_SETTLE` delays between each spawn to wait for the panel to register.

## Problem

The test non-deterministically landed at 2 or 4 panels instead of 3, especially on Windows where process spawn timing is slower and more variable.

## Root cause

Fixed `T_SETTLE` waits are inherently racy — they assume a maximum latency that doesn't hold on all platforms. Additionally, `clickToolbarButton` without `noWaitAfter: true` caused Playwright to wait for post-click network/navigation activity, which raced with the button's action handler and could trigger a double-dispatch (spawning two terminals from one click).

## Lesson

Never use fixed `T_SETTLE` / `sleep` waits between sequential operations in E2E tests. Use poll-until-confirmed helpers like `spawnTerminalAndVerify()` that wait for each step to complete before proceeding. When clicking toolbar buttons that have action handlers (not navigation links), pass `noWaitAfter: true` to prevent Playwright's built-in post-click wait from interfering with the handler.

## Applies when

Writing E2E tests that spawn multiple resources sequentially. Any `clickToolbarButton` call that triggers a client-side action (not a page navigation).
