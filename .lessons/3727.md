## Context

Webview guest `WebContents` can navigate freely after attachment. Canopy validates `params.src` at attachment time via `will-attach-webview`, but post-attachment navigations were completely unrestricted. This is a TOCTOU (time-of-check-time-of-use) gap.

## Problem

The main window already had a `will-navigate` handler blocking untrusted URLs, but the webview guest WebContents created via `web-contents-created` did not. A webview could pass initial attachment validation with a trusted localhost URL, then navigate to `file:///` or an external origin after attachment.

`setWindowOpenHandler` only blocks `window.open()` — it does nothing for in-page navigations via `location.href`, anchor clicks, form submissions, or `loadURL`.

## Root cause

The navigation restriction was applied to the main `BrowserWindow` WebContents but not to the dynamically-created webview guest WebContents. Each needs its own `will-navigate` listener since they are independent WebContents instances.

## Lesson

When setting up webview security in Electron, attachment-time validation is insufficient. You must also attach a `will-navigate` handler to the guest WebContents _after_ `web-contents-created` fires. The main window restriction does not propagate to guest WebContents — they are independent and require their own listeners.

Pattern:

```ts
// In web-contents-created handler for webview type:
webContents.on("will-navigate", (event, url) => {
  const { hostname } = new URL(url);
  if (hostname !== "localhost" && hostname !== "127.0.0.1") {
    event.preventDefault();
  }
});
```

Also: `setWindowOpenHandler` only blocks `window.open()`, not same-window navigation.

## Applies when

Adding or auditing security for Electron webviews; any `web-contents-created` handler that validates initial `src` but not subsequent navigations; reviewing security hardening that touches `will-attach-webview`.
