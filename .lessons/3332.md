## Context

`WorkspaceService` polls git status per worktree with an adaptive backoff strategy. Consecutive errors trip a circuit breaker that suppresses polling for a cooldown period.

## Problem

If a user deleted a worktree directory externally (`rm -rf`, Finder delete, tool-driven cleanup), the sidebar card never went away. The git commands inside the poll failed, tripped the circuit breaker, and then the breaker suppressed the very polls that would have detected the deletion — an indefinite stale card.

## Root cause

The circuit breaker was applied uniformly to "git operations on this worktree", but deletion detection is not a git operation — it's a filesystem existence check. By gating everything behind the breaker, the cheapest and most reliable signal (the directory is gone) was the thing that got suppressed first.

## Lesson

When a circuit breaker protects expensive operations, pre-check the cheap observable state that the breaker cannot mask. For worktrees that means `fs.existsSync(path)` at the very top of `updateGitStatus`, with an early exit to the external-removal handler:

```ts
async updateGitStatus(wt) {
  if (!fs.existsSync(wt.path)) {
    return this.handleExternalWorktreeRemoval(wt);
  }
  if (this.breaker.isOpen(wt.id)) return;
  // expensive git operations...
}
```

Add structured logs at every stage of the removal chain (path check, handler entry, scope-id guard, IPC emit) so the next silent failure is visible without re-instrumentation.

## Applies when

Any polling loop with a circuit breaker that also needs to detect "the resource disappeared" — worktrees, watched files, remote endpoints, PTY processes. If the cheap existence check is gated behind the same breaker as the expensive ops, you will miss deletions.
