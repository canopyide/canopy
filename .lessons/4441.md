## Context

`getWorktreeChangesWithStats()` runs three git commands per worktree (`git status`, `git diff --numstat`, `git log`). It is called both on a polling interval and when a filesystem watcher fires on the worktree directory.

## Problem

When a watcher event fired while a poll was already in progress for the same worktree, a second (and sometimes third) parallel invocation would queue up, running redundant git operations and creating unnecessary load — particularly noticeable in large repositories or with many open worktrees.

## Root cause

The function had no mechanism to detect or coalesce concurrent calls for the same path. Each call was independent, so events that arrived during an in-progress operation would launch a full duplicate run rather than waiting for the result already being computed.

## Lesson

Use an in-flight promise map to deduplicate concurrent operations keyed by a natural identifier (e.g. worktree path):

```ts
const inFlight = new Map<string, Promise<Result>>();

function getWorktreeChanges(path: string): Promise<Result> {
  if (inFlight.has(path)) return inFlight.get(path)!;
  const promise = doWork(path).finally(() => inFlight.delete(path));
  inFlight.set(path, promise);
  return promise;
}
```

Always clean up via `.finally()` so a rejected promise doesn't permanently block the path. Provide a `forceRefresh` parameter that bypasses deduplication for explicit user-triggered refreshes.

This pattern is especially important for operations that are triggered by both periodic polling AND filesystem events — these two triggers are likely to fire simultaneously.

## Applies when

Any function that is called from both a polling interval and an event listener (filesystem watcher, IPC message), where concurrent invocations for the same resource produce redundant work.
