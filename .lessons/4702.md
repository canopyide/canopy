## Context

The IPC handler for `git.getFileDiff` had a path traversal security check to reject paths containing `..` components, preventing a renderer from reading arbitrary files outside the repo.

## Problem

The file diff view silently failed for any file in a nested directory. `components/Button.tsx` works fine, but any path that happened to contain `..` as a substring (e.g. `src/layout/..utils/helpers.ts`) would be rejected. The renderer received empty data and rendered a blank modal with no error message.

## Root cause

The check used `path.includes('..')` — a substring match against the full path string. This is both too broad (rejects legitimate paths that contain the two-character sequence `..` in a directory name) and theoretically bypassable (encoded variants). The correct check is segment-by-segment: split on `/` and `\`, then reject only if any segment is exactly `..`.

## Lesson

Path traversal guards must operate on path segments, not the raw string. The correct implementation:

```ts
const segments = filePath.split(/[\\/]/);
if (segments.some((seg) => seg === "..")) {
  throw new Error("Path traversal not allowed");
}
```

`path.includes('..')` is a common anti-pattern that creates both false positives (breaks valid paths) and potential false negatives (encoded or mixed-separator bypasses).

Additionally, diff failures should propagate as errors to the renderer — not silently return empty data — so the UI can display a meaningful error state rather than a blank panel.

## Applies when

- Validating file paths received over IPC from the renderer
- Any security check on user-controlled path strings
- Building file diff / file read IPC handlers
