## Context

The `canopy-file` custom Electron protocol was registered with `bypassCSP: true` to ensure images loaded via that protocol would render correctly. This privilege completely disables CSP enforcement for all content loaded from that scheme.

## Problem

`bypassCSP: true` is a nuclear option — it eliminates all CSP protections for the entire scheme, not just for images. Any content loaded via `canopy-file://` could load resources from arbitrary origins, defeating the point of having a CSP at all.

## Root Cause

The initial implementation used `bypassCSP: true` as the quick fix to make images work through the custom protocol. The correct solution — adding `canopy-file:` to the `img-src` CSP directive — was not implemented, leaving a broad privilege in place longer than intended.

## Lesson

When registering a custom Electron protocol via `protocol.registerSchemesAsPrivileged`, prefer explicit CSP directives over `bypassCSP: true`. The `bypassCSP` flag disables ALL CSP enforcement for that scheme unconditionally. Instead, add the scheme to the specific CSP source directives where it is actually needed (`img-src`, `media-src`, etc.). These go in both `DEV_CSP` and `PROD_CSP` strings in `vite.config.ts`.

The retained privileges (`secure`, `supportFetchAPI`) allow the protocol to behave like HTTPS without bypassing security policy.

## Applies when

- Registering new custom Electron protocols
- Debugging "image not loading" issues with custom schemes — the fix is a `img-src` CSP addition, not `bypassCSP`
- Auditing `registerSchemesAsPrivileged` calls for overly broad privileges
