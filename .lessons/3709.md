## Context

`ActionPaletteItem` and `QuickSwitcherItem` render lists of items during keyboard navigation and query typing in palettes. Every keystroke updated the query state, triggering re-renders of all visible items even when the item content hadn't changed.

## Problem

Without memoization, all N visible palette items re-render on every keystroke. For palettes with 50-200 visible items, this is measurable render overhead, especially on lower-end hardware.

## Root cause

`React.memo` was missing on the item components. Additionally, the `onClick` callback in `QuickSwitcher` was recreated on every render because it lacked `useCallback`, defeating any memo on the child even if it had been added.

## Lesson

`React.memo` on list item components only works if **all props are stable**. Wrapping the item in `memo` without also stabilizing the parent's callback props with `useCallback` is ineffective — the new function reference on every render causes memo to miss.

Two-step checklist:

1. Wrap item component: `export default React.memo(ItemComponent)`
2. Stabilize parent callbacks: `const handleClick = useCallback(() => ..., [deps])`

Both steps are required. If only one is done, profiling will show memo is never hitting.

The `WorktreeCard` memoization in Canopy is the established pattern to follow.

## Applies when

Any list component where items re-render during parent state changes (typing, navigation) that don't affect item content; performance profiling showing high component render counts in palette/list views.
