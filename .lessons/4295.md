## Context

Canopy writes session snapshots, terminal state backups, and config files continuously during normal operation. Running out of disk space can silently corrupt these files.

## Problem

No proactive warning existed. Users would hit corruption or silent write failures only after their disk was already full.

## Root cause

Not a bug fix — this is a pattern lesson from the implementation design.

## Lesson

**Hysteresis is required for threshold-based warnings.** Without it, a disk hovering around the warning boundary (e.g., 502MB free) generates a notification storm: warn → user frees space → falls back below → warn again. The `DiskSpaceMonitor` uses a hysteresis band: once you warn at <500MB, don't warn again until usage drops back to >600MB.

**Suppress non-essential writes at critical level** rather than showing another notification. At <100MB free, PTY session snapshots and terminal backups are skipped via a shared `diskSpaceLevel` state propagated to `TerminalProcess` and `terminalSessionPersistence`. This preserves the write budget for critical config saves.

**Cross-process propagation**: The disk level is set in the main process but PTY runs in a separate host process. The level must be forwarded via `PtyClient` IPC, not accessed via shared memory.

`fs.statfs()` (Node.js 18+) is the correct API for available bytes; it's available in Electron's main process without additional native modules.

## Applies when

- Any threshold-based monitoring (disk, memory, CPU)
- Deciding whether to skip non-critical writes during degraded conditions
- Propagating degraded-mode flags to the PTY host process
