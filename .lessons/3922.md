## Context

The PTY host is a dedicated utility process managing all PTY processes. It uses SharedArrayBuffers for zero-copy terminal output to the renderer via MessagePort. The `cleanup()` function disposes managers and pool but did not release module-level SAB references.

## Problem

`visualBuffers` (SharedRingBuffer[]), `visualSignalView` (Int32Array backed by SAB), and `analysisBuffer` (SharedRingBuffer) were initialized once at startup and never nullified in `cleanup()`. SharedArrayBuffers are special in V8's memory model — they represent fixed, non-movable allocations shared across process boundaries and are not collected while any reference exists in any thread. The `ipcDataMirrorTerminals` Map holding per-terminal buffer routing state was also never cleared.

Meanwhile, `ptyPool` was already correctly nullified after disposal — the SAB variables were inconsistently handled.

## Root Cause

The cleanup function was written incrementally; each newly added resource was disposed but the SAB variables predated the dispose pattern and were never retroactively included.

## Lesson

SharedArrayBuffers must be explicitly nullified in cleanup code — V8 will not GC them as long as any module-level reference exists, even in a utility process. Follow the same pattern as `ptyPool`: dispose, then assign to empty/null. Existing null-guards (`if (visualBuffers.length > 0)`, `if (analysisBuffer)`) protect against post-cleanup late writes, so nullifying is safe. This matters most for hot-reload scenarios and potential future PTY host restart capability.

## Applies when

- Any code that allocates SharedArrayBuffers at module level
- PTY host cleanup or any utility process that uses SABs for IPC
- Auditing cleanup functions: compare disposed resources against all module-level references
