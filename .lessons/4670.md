## Context

This is the 8th occurrence of cross-project worktree contamination (#2315, #2346, #2358, #2703, #2741, #2770, #4532, #4642). Each previous fix added another downstream guard. PR #4647 introduced per-project workspace host processes, but contamination persisted through a different vector.

## Problem

After switching projects, worktrees from the previous project appeared in the sidebar alongside the current project's worktrees.

## Root cause

Three compounding bugs, all in the project switch path:

1. **Double `forceReinitializeWorktreeDataStore` call**: `projectStore.switchProject` called it in a `.then()` callback AND `useProjectSwitchRehydration` called it again on `PROJECT_ON_SWITCH`. The first call was discarded (generation mismatch), but the second set up **new IPC listeners at the current generation counter** — right as old-host events could still be in flight.

2. **`routeHostEvent()` never validated `projectScopeId`**: The old workspace host (still alive during the 5-second `CLEANUP_GRACE_MS`) continued polling git and emitting `worktree-update` events. `routeHostEvent()` routed these events by `windowId` alone, never checking the `projectScopeId` field that was already present in the event type definition but never read.

3. **Snapshot validation only checked the main worktree**: `isSnapshotValidForProject()` validated only the primary worktree path. A contaminated snapshot (written on switch-away while the store held foreign worktrees) would pass validation and get restored on switch-back, making contamination appear persistent across multiple switches.

## Lesson

- **Generation counters alone don't protect against in-flight events from old processes**. The generation matches the new listeners; old events passing the same guard is the exact failure mode.
- **Validate `projectScopeId` in `routeHostEvent()`** — route by project identity, not just window identity. Having the field in the type and not reading it is a latent bug.
- **Never call `forceReinitializeWorktreeDataStore` from two places** in the same flow. Idempotence assumptions break when both callers share a generation counter.
- **Snapshot validation must be holistic**: checking only the main worktree lets contaminated branch worktrees survive into restored snapshots.
- The fix: `routeHostEvent()` validates `event.projectScopeId`, `projectStore.switchProject` no longer calls reinitialize directly (defers entirely to `PROJECT_ON_SWITCH`), and `isSnapshotValidForProject()` checks all worktree paths.

## Applies when

Adding any new IPC event routing, project switch orchestration, or snapshot/cache restoration logic involving the workspace host.
