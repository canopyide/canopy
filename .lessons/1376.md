## Context

The GitHub search query used a variable named `$query` to pass the search string through to `search(query: $query, ...)`. Every toolbar search (issues, PRs) failed with `"query" cannot be used as variable name`.

## Problem

`@octokit/graphql` reserves the key `query` for the operation document itself. When you pass `{ query: "...", query: "search text" }` as variables, the variable collides with the request body key. Even if GraphQL spec-wise a variable can be called anything, the octokit client's request shape makes `query` unusable.

## Root cause

Tooling-level reservation rather than a GraphQL spec restriction. Copy-pasting a "canonical" GraphQL search snippet will hit this immediately because the spec-level variable name is almost always `$query`.

## Lesson

Never name a GraphQL variable `query` when using @octokit/graphql — rename it (`$searchQuery`, `$q`) in both the operation and the variables object. More generally, treat any key that is also a top-level HTTP/client request field as reserved: `query`, `mutation`, `headers`, `url`, `method`.

## Applies when

Writing new GraphQL operations against GitHub via octokit, porting example GraphQL snippets from GitHub docs, or debugging "cannot be used as variable name" errors in the toolbar issue/PR search.
