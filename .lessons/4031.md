## Context

Memory leaks from terminal churn and error store accumulation had no automated detection. Issues would only manifest after extended use.

## Problem

Without memory baseline tests, terminal open/close cycles could silently accumulate heap and external memory (from node-pty, native buffers) across sessions. The error store had no upper bound, allowing unbounded growth from repeated injected errors.

## Root cause

No memory regression gate in CI.

## Lesson

Memory leak tests require IPC exposure of `process.memoryUsage()` from the main process (not renderer `window.performance.memory` which only sees renderer heap). Key thresholds established: <20MB heap growth after 20 terminal cycles, <10MB external memory after 10 cycles, error store bounded at 50 entries after 100 injections.

Run memory tests in `e2e/nightly/` only — not on PRs — because they're slow and timing-sensitive. Use a dedicated workflow triggered after the core nightly suite. Expose `window.__errorStore` internals in `App.tsx` only in test environments for bounded-growth assertions.

## Applies when

- Suspecting memory leaks from terminal churn (node-pty external memory)
- Error store unbounded growth
- Setting thresholds for new nightly memory tests
