## Context

Multiple components used `hover:text-text-inverse` (or similar `*-inverse` tokens) on interactive elements. In light themes this produced legible hover contrast. In dark themes like Daintree, `text-inverse` resolves to a dark color — rendering text invisible against the dark hover background.

## Problem

`text-text-inverse` is semantically "the opposite of normal text color" — in a dark theme, that means light-on-dark inverts to dark-on-dark on hover backgrounds that are also dark.

## Root cause

Token name mislead: `text-inverse` reads as "high-contrast alternative" but actually means "inverted from normal text" — which in a dark theme produces dark text. Using it for hover states on dark backgrounds is incorrect.

## Lesson

- **Use `text-canopy-text` for readable text in hover states**, not `text-text-inverse`. `text-canopy-text` is the semantic token for "text that is readable in the current theme context."
- **`text-text-inverse` has a specific meaning**: it is for text that appears on inverted/contrasting backgrounds (e.g., a light-colored button in a dark UI). It is not a general "make text visible" token.
- **Affected components in this PR**: `ArtifactOverlay`, `PortalLaunchpad`, `ProblemsContent`, `FileChangeList`, `FileStageRow` — all had `hover:text-text-inverse`.
- **Add contrast regression tests**: after the fix, contrast-focused unit tests were added for each component to verify correct token usage and prevent future regressions.

## Applies when

Adding hover states to any component, especially in new components that may not be tested against all built-in themes. Prefer `text-canopy-text` over any `*-inverse` variant for readable hover text.
