## Context

PR #3659 removed `@xterm/addon-web-links` as "redundant" after adding a backend `UrlStyler` that wraps URLs with OSC 8 hyperlink escape sequences. This assumed the backend fully replaced the addon's functionality. It did not.

## Problem

URLs in agent terminal output became plain, non-clickable text. The backend `UrlStyler` has known fragility: it operates on PTY output chunks (susceptible to chunk-splitting), skips URLs already in OSC 8 context, and can miss URLs surrounded by ANSI color sequences. The `linkHandler` option on the xterm Terminal only handles clicks on already-formatted OSC 8 sequences; it does not detect plain-text URLs.

## Root Cause

The backend `UrlStyler` and the frontend `WebLinksAddon` serve **complementary, not redundant** roles:

- Backend: regex on PTY output → OSC 8 wrapping (catches URLs before ANSI processing, but fragile)
- Frontend: regex on xterm.js buffer lines via ILinkProvider (client-side safety net for anything the backend missed)

Removing the addon left a single point of failure. A previous patch (`a11a6a8e`) had already fixed one UrlStyler ANSI regression — evidence that the backend-only approach was unreliable.

## Lesson

In xterm 6.0, `@xterm/addon-web-links` is still required for plain-text URL detection even when backend OSC 8 wrapping is in place. The two layers are complementary. When re-adding: route clicks through `TerminalLinkHandler` and restrict to `http://` and `https://` schemes only. The backend UrlStyler retains value for visual styling in headless/scrollback contexts. Never remove the WebLinksAddon as "redundant" to a backend URL processor.

## Applies when

- Any refactor of terminal URL handling or link detection
- Removing any xterm addon that "seems" covered by a backend equivalent
- The backend UrlStyler is the first line; WebLinksAddon is the safety net — both must be present
