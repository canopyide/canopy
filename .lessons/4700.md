## Context

Canopy uses `fix-path` (a package that calls a login shell to inherit the correct PATH) for production builds, where Electron is launched via the dock or Finder and has a minimal environment. It also has a `CliAvailabilityService` that checks whether tools like `git`, `gh`, etc. are accessible.

## Problem

Two separate PATH-related bugs:

1. In dev mode (`npm run dev`), `fixPath()` was running unconditionally, overwriting the PATH that was already correctly inherited from the terminal that launched the dev server.
2. `CliAvailabilityService` checked availability using whatever PATH was current at module load time — before any PATH fixup had run — so tools could be reported as missing in production.

## Root cause

1. `fixPath()` launches a login shell to extract its PATH, then replaces `process.env.PATH`. In dev, the terminal's PATH is already correct and richer. Overwriting it could drop tools only available in the dev environment.
2. Module load order: `CliAvailabilityService` imported and captured PATH-dependent utilities before `fixPath()` had been called in the startup sequence. The fix is lazy: call `refreshPath()` exactly once before the first `checkAvailability()` invocation, not at module load time.

## Lesson

- Guard `fixPath()` (and any login-shell PATH fixup) behind `app.isPackaged`. In dev, the inherited PATH is correct by definition.
- Services that check PATH-dependent CLI tools must refresh PATH lazily before their first check, not at construction time. A one-shot `hasRefreshed` flag ensures `refreshPath()` runs exactly once regardless of how many concurrent checks are triggered at startup.

## Applies when

- Any Electron service that calls `which`/`where` or executes shell commands
- Startup ordering where PATH fixup and CLI checks are initialized in parallel
- Dev/prod environment differences in Electron apps
