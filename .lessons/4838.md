## Context

Canopy runs multiple polling services in the main process (WorkspaceService git status, ProjectStatsService, ProcessTreeCache). These poll at fixed intervals regardless of whether the app window is focused or the device is on battery.

## Problem

Background polling consumed unnecessary CPU/battery when the user was working in another app. All polling services ran at full rate even when the window was minimized or out of focus.

## Root cause

No backgrounding/throttling logic existed. The polling services had a binary `setPollingEnabled()` suspend/resume (for when the window is completely hidden) but no intermediate throttle for "backgrounded but still alive" state.

## Lesson

Electron provides `BrowserWindow` `blur`/`focus`/`minimize`/`restore` events and `powerMonitor` battery events as the canonical mechanism for background throttling. The pattern:

1. Create a `WindowFocusThrottle` class that listens to these events.
2. Register polling services with it; they implement `setIntervalMultiplier(n)`.
3. On blur/minimize, apply a ~5x slower multiplier; on focus restore, reset multiplier and fire a one-shot refresh.
4. Layer on top of the existing binary suspend/resume — don't replace it.

For UtilityProcess services (PTY host), drive the multiplier via an IPC message since the UtilityProcess doesn't share event subscriptions with the main window.

The one-shot refresh on refocus is critical for UX: users expect stale data to update immediately when they switch back.

## Applies when

- Adding a new polling service: register it with `WindowFocusThrottle` and implement `setIntervalMultiplier()`.
- Any service that polls at a fixed interval — background throttling should be the default, not an afterthought.
