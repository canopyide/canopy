## Context

Standard terminals (non-agent) use xterm.js's DOM renderer. Agent terminals use WebGL and are structurally unaffected. `TerminalHibernationManager` serialises scrollback and replaces live Terminal instances with fresh ones on unhibernate.

## Problem

Two distinct failures surfaced together:

1. Standard terminals went blank/frozen silently. No error, no user action — the terminal was just visually stuck despite the process still running.
2. After unhibernation, terminals were in a zombie state: buffer restored, but nothing rendered because the DOM was never bound.

## Root cause

**Blank/frozen:** xterm.js 6.0's DOM renderer installs an `IntersectionObserver` internally. Any layout change, CSS transform, or `clip-path` that briefly drops `isIntersecting` below the threshold sets `_isPaused = true` on the renderer. The renderer never self-recovers — it waits for the next intersection event that may never come if the element stays technically visible but the observer misfired.

**Zombie on unhibernate:** `TerminalHibernationManager.unhibernate()` created a fresh `Terminal` instance and restored the buffer, but never called `terminal.open(hostElement)`. Without `open()`, the DOM is never bound and xterm never paints. An additional guard was needed: defer `open()` when `clientWidth/clientHeight === 0` (element not yet in layout) and let the `attach()` path handle it once dimensions are available.

## Lesson

When using xterm.js 6.0's DOM renderer, never assume a visible terminal is a rendering terminal. The `IntersectionObserver`-based pause mechanism can trigger on layout side-effects and will not recover on its own. Build layered recovery: (1) trigger `fit()` + a `forceXtermReflow` call on `onWriteParsed`, (2) run a periodic heartbeat (e.g. 3s), (3) respond to `window.focus` and `document.visibilitychange`. Also: `terminal.open(host)` is mandatory before any rendering — always call it when creating a fresh instance for an existing host element, but guard with a dimension check (`clientWidth > 0`) to avoid binding to an element not yet in layout.

## Applies when

- Creating or restoring Terminal instances (hibernation, panel remount, split-pane rearrangement)
- Configuring the DOM renderer for standard terminals
- Adding any CSS transform/clip-path/layout animation that touches a terminal's ancestor chain
- Implementing any terminal lifecycle that serialises/deserialises xterm state
