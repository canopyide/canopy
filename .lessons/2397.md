## Context

Canopy's copy-tree path writes `text/uri-list` buffers to the clipboard so users can paste files into file managers. It also uses `fs.watch` to observe git worktree directories.

## Problem

Pasting into Nautilus/Dolphin/Thunar silently failed. Separately, on busy repositories the file watcher stopped firing with no actionable error.

## Root cause

Two distinct Linux issues. (1) `text/uri-list` requires RFC 2483 format: each URI must be a proper `file:///` URL (not a raw absolute path) and each line must be terminated with CRLF (`\r\n`). Using plain paths with `\n` is rejected by GTK and Qt file managers. (2) `fs.watch` on Linux consumes inotify watches from two limits — `fs.inotify.max_user_watches` AND `fs.inotify.max_user_instances`. When either is exhausted, watch creation throws `ENOSPC` synchronously OR the watcher emits an `error` with that code. Silent catches left watchers dangling and users with no clue why nothing updated.

## Lesson

For Linux clipboard `text/uri-list`: always produce URIs via `pathToFileURL()` from `node:url`, and terminate each entry with CRLF. For `fs.watch` on Linux: handle `ENOSPC` in both the synchronous throw path AND the async `error` event, close and remove the dangling watcher from internal state, and log an actionable message that mentions both `fs.inotify.max_user_watches` and `fs.inotify.max_user_instances` with the `sysctl -w` transient fix and `/etc/sysctl.d` persistent fix.

## Applies when

Writing to the system clipboard with file URI payloads on Linux, or creating any `fs.watch` usage that may run on developer machines with large repositories.
