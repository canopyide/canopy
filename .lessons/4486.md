## Context

The `delete active worktree` E2E test was consistently failing on Windows but passing on macOS and Linux. The test opened a Radix UI dropdown context menu and clicked the delete menu item, but the dropdown would dismiss before the click registered.

## Problem

On Windows, Playwright's synthesized click events trigger focus changes before the click action completes. Radix UI's dropdown listens to `onOpenChange` and dismisses when it detects a blur/focus event, so the dropdown was closing as the click was being synthesized — before the click landed on the menu item.

## Root cause

Windows has different focus event ordering during synthesized input: focus events fire during the click sequence rather than after it. Radix UI interprets the intermediate focus change as a "click outside" dismissal. This is a Windows-specific timing difference in Electron's `webContents` event synthesis.

## Lesson

Two mitigations are needed together:

1. **Ensure window focus before opening the menu**: Call `ensureWindowFocused(ctx.app)` before any interaction that opens a Radix dropdown. This prevents the initial click from triggering a spurious focus event that dismisses the menu.

2. **`hover()` before `click()` on menu items**: Adding `await menuItem.hover()` before `await menuItem.click()` ensures the element is fully interactive and hovered before the click fires, giving Radix time to settle focus state.

```ts
await ensureWindowFocused(ctx.app);
await actionsButton.click();
const deleteItem = page.getByRole("menuitem", { name: /delete/i });
await deleteItem.hover(); // stabilize before click
await deleteItem.click();
```

Both steps are no-ops on macOS/Linux, so adding them doesn't affect those platforms.

## Applies when

- Any E2E test using Radix UI dropdowns/context menus that is flaky on Windows
- Playwright tests that open menus via click and then click menu items
- Windows-only E2E failures involving dropdown dismissal
