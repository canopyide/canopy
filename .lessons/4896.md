## Context

Browser and Dev Preview panels persist their `browserUrl` across sessions so the panel restores to the same page. Dev Preview panels also have an optional dev server command.

## Problem

After reloading the app: (1) Dev Preview panels with no dev server configured would silently reload the stale URL from last session, connecting to a server that no longer exists. (2) Both panel types showed Chromium's generic error page on `ERR_CONNECTION_REFUSED` with no explanation of why.

## Root cause

On mount, Dev Preview panels didn't clear the persisted `browserUrl` when no dev server command was configured. Neither panel type checked whether the very first navigation after restore resulted in a connection-refused error, so there was no contextual messaging.

## Lesson

On panel mount, if a persisted URL is only meaningful alongside a running process (e.g., a dev server), clear it if that process is not configured. Track an `isInitialLoad` ref to detect when the first post-restore navigation fails with error code `-102` (ERR_CONNECTION_REFUSED) and surface an inline message explaining the saved URL is no longer reachable. Scope the special handling to the initial load only — don't interfere with subsequent user navigations.

## Applies when

Any panel type that persists a URL or resource location across sessions and depends on an external process being active (dev server, local API, etc.).
