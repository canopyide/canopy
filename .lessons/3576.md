## Context

`ActivityMonitor` detects when an agent terminal is working vs idle by pattern-matching terminal output. State transitions are debounced. This affects the visual working/waiting indicator shown on worktree cards and panel headers.

## Problem

Two distinct failure modes:

1. **False entry**: A burst of output — shell prompt clearing/redrawing, a progress bar from a non-agent command, two carriage returns in quick succession — triggered the working indicator when no real agent task had begun.

2. **Premature exit**: The 2500ms idle debounce was shorter than the 1–5 second silence gaps that occur during LLM API calls. The debounce fired mid-response, transitioning the terminal to "waiting" while the agent was still generating.

## Root cause

Entry and exit thresholds were symmetric and too low (2500ms exit, 2 CRs in 500ms entry). A more robust design requires sustained evidence to enter working and sustained silence to exit.

## Lesson

Apply hysteresis to both entry and exit:

- **Exit (idle debounce)**: Raise from 2500ms to 6000ms to hold through typical LLM API call gaps (network + first-token latency can reach 4–5s).
- **Entry guard**: Require two consecutive signals within a window (or 3 CRs in 800ms for line-rewrite detection) before `becomeBusy()` fires. A single carriage-return burst is not enough evidence.

The 6000ms debounce must be paired with fast exit on prompt detection (~500ms) to avoid the indicator staying "working" too long after the agent genuinely finishes.

These values should be exposed in `agentRegistry` per-agent config so different agents (Claude Code, Gemini, Codex) with different output patterns can tune independently.

## Applies when

Any heuristic state machine driven by terminal output patterns where both false positives and false negatives matter. The same hysteresis pattern applies to any debounced entry/exit threshold.
