## Context

When a user is typing a prompt in an agent terminal (directing state) and presses Enter, the state should immediately transition to working (prompt submitted). Previously it reverted to waiting first, causing a visible flicker.

## Problem

Pressing Enter in directing state reverted to waiting before transitioning to working, creating a brief wrong-state flash. The transition was not immediate.

## Root cause

The `onData` handler was not distinguishing Enter from regular keystrokes at the state controller level. The directing debounce would expire and emit waiting before the agent's response began triggering the working transition from the main process.

## Lesson

`Enter` in directing/waiting state is semantically different from all other keystrokes — it submits the prompt and should immediately emit `working` without waiting for the debounce.

Two important distinctions for the Enter detection:

1. **Plain Enter vs Shift+Enter**: Plain Enter is `\r`. Shift+Enter in Kitty Keyboard Protocol is `\x1b[13;2u`. Kitty Protocol encodes modifier keys in the escape sequence. Only plain Enter should trigger the immediate `working` transition — Shift+Enter is a multi-line continuation.

2. **IME composition guard**: CJK input methods may send `\r` to confirm a composition candidate, not to submit the whole prompt. An IME composition flag must suppress the transition while composition is active.

The transition should only apply to agent terminals in `directing` or `waiting` canonical states — shell terminals are unaffected.

## Applies when

Any agent terminal state controller that interprets user input. Whenever distinguishing Enter from other keys, account for Kitty Protocol modifier encoding and IME composition state.
