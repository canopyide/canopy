## Context

`resetAllStoresForProjectSwitch()` resets 11 Zustand stores during a project switch, but `useTerminalInputStore` was omitted. The store holds per-terminal Maps for command history, draft inputs, and editor states.

## Problem

1. Memory grew unboundedly over multi-hour sessions with multiple project switches.
2. `commandHistory` used bare `terminalId` keys — if terminal IDs collided across projects, command history from project A bled into project B's terminals.

## Root Cause

Two separate issues combined:

- The per-terminal cleanup path (`clearTerminalState()`) was fixed in PR #3154 for permanent terminal removal, but the project-switch reset path was a separate code flow never updated to include the input store.
- `commandHistory`, `historyIndex`, and `tempDraft` used bare `terminalId` keys while `draftInputs` already used composite `projectId:terminalId` keys. This inconsistency meant history could not be scoped to a project even after cleanup was added.

## Lesson

When adding cleanup to a store for one code path (individual terminal removal), audit other code paths that remove terminals in bulk (project switch, batch close). They are separate flows and must each be updated.

When some Maps in a store use composite keys and others use bare keys, the inconsistency is a future bug: the bare-keyed collections cannot be selectively cleared per project and will either leak or over-clear. Standardize on composite keys `projectId:terminalId` for any per-terminal state that needs per-project scoping.

Also: `preserveTerminalIds` semantics must be respected in project-switch cleanup — terminals carried across the switch should keep their state.

## Applies when

- Adding a new per-terminal Map to `terminalInputStore`
- Auditing store cleanup completeness after a `resetAllStoresForProjectSwitch` change
- Choosing a key format for state that is both terminal-scoped and project-scoped
