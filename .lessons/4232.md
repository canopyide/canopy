## Context

Running Canopy from within a VS Code terminal (or any environment where VS Code's extension host is active) sets `ELECTRON_RUN_AS_NODE=1` in the shell environment. Child processes spawned by npm scripts inherit this variable.

## Problem

When `ELECTRON_RUN_AS_NODE=1` is inherited, Electron runs as a plain Node.js process — it skips `browser_init.js` entirely. As a result `process.type` is `undefined`, and `require('electron')` returns an empty object with no `BrowserWindow`, `app`, or any other API. ESM `import { BrowserWindow } from 'electron'` throws: `SyntaxError: The requested module 'electron' does not provide an export named 'BrowserWindow'`. Both the dev server (`npm run dev`) and smoke tests fail this way.

## Root cause

`scripts/dev.mjs`, `scripts/run-smoke.mjs`, and `e2e/helpers/launch.ts` all spawned Electron child processes without explicitly sanitizing the inherited environment. Since `ELECTRON_RUN_AS_NODE` is set by VS Code's extension host and propagates through `process.env` into child processes, any script that doesn't explicitly delete it will pass the poisoned value to Electron.

## Lesson

Any script that spawns an Electron process must explicitly remove `ELECTRON_RUN_AS_NODE` from the child process environment before spawning. This is not Electron-version-specific — it affects every Electron app launched from within VS Code terminals. Playwright's `_electron.launch()` also inherits the variable and needs the same treatment.

```js
const env = { ...process.env };
delete env.ELECTRON_RUN_AS_NODE;
spawn(electronPath, args, { env });
```

## Applies when

- Writing or debugging npm scripts that spawn Electron processes
- Adding new E2E test launch helpers
- Debugging "BrowserWindow is not exported" or "process.type is undefined" errors in Electron
- Running any Electron app from within VS Code integrated terminal
