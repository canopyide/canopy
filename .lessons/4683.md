## Context

Canopy's `ProjectViewManager` calls `setBackgroundThrottling(true)` on deactivated `WebContentsView` instances to reduce CPU usage for cached-but-hidden views. Two global Chromium flags had been set at app startup:

- `disable-renderer-backgrounding`
- `disable-background-timer-throttling`

## Problem

Background throttling set via `setBackgroundThrottling(true)` had no effect on cached views. Deactivated project views continued consuming CPU as if they were active.

## Root cause

`app.commandLine.appendSwitch('disable-renderer-backgrounding')` is a process-wide flag applied before app ready. It overrides any per-renderer throttling APIs, including `setBackgroundThrottling()`. The per-view call was silently a no-op.

## Lesson

Global Chromium command-line switches applied via `app.commandLine.appendSwitch()` take precedence over per-instance WebContents APIs. If you use `setBackgroundThrottling(true/false)` or `Page.setWebLifecycleState` (CDP) for per-view throttling, you must NOT have `disable-renderer-backgrounding` or `disable-background-timer-throttling` set globally.

These flags were likely added to prevent UI jank during development but became counterproductive once per-view throttling was implemented. Remove them and rely on `ProjectViewManager` to handle throttling at the correct granularity.

## Applies when

- Per-view throttling via `setBackgroundThrottling()` in Electron
- Using CDP `Page.setWebLifecycleState` for freeze/resume
- Any global Chromium flag that may interact with per-instance behavior
