## Context

Canopy creates a new `WebContentsView` for each project. Originally each view used a unique `persist:project-${projectId}` Electron session partition.

## Problem

Each unique partition name creates a separate Electron session with its own V8 bytecode cache, HTTP cache, and cookie store. With multiple cached project views, this multiplied cache overhead N times for identical renderer code.

## Root cause

The `persist:` prefix was intended to give each project an isolated cookie/storage namespace. However, the renderer JS bundle is identical across all projects — duplicating the V8 bytecode cache for it wastes tens of MB per additional project. The per-project isolation was only needed for the browser and portal webviews (which actually visit different URLs), not for the main renderer view.

## Lesson

Use a single shared `persist:canopy-app` session for all project `WebContentsView` renderers. The session's V8 bytecode cache and HTTP cache are shared, saving significant memory with multiple cached views. Per-project isolation for storage/cookies is a concern only for webviews that browse external sites — handle those separately with their own partition.

The `session-created` handler in `security.ts` should target only the `canopy-app` partition by name, not all partitions, to avoid accidentally applying security headers to webviews that have different CSP requirements.

## Applies when

- Multiple `WebContentsView` instances loading the same renderer bundle
- Any Electron app with project/workspace-scoped views
- Auditing session/partition strategy for memory efficiency
