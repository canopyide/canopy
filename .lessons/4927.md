## Context

The worktree delete dialog warns users before deleting a worktree that has uncommitted changes, since `git worktree remove` will refuse to delete it and force-delete will lose work.

## Problem

The warning appeared even when the worktree only had untracked files and no tracked changes. `git worktree remove` succeeds with untracked files (it only blocks on tracked modifications), so the warning was a false alarm that confused users.

## Root cause

`WorkspaceService` reported a single `changedFileCount` that lumped tracked changes (modified, staged, deleted, renamed, conflicted) and untracked files together. The dialog used this combined count to decide whether to show the warning.

## Lesson

`git worktree remove` blocks only on tracked changes, not on untracked files. When surfacing deletion warnings, split the file count into `trackedChangeCount` and `untrackedFileCount` separately. Use only `trackedChangeCount` to gate the uncommitted-changes warning. Reflect the distinction in the force-delete label so users know what they're actually losing.

## Applies when

Any git worktree deletion UI. Any dialog that warns about "unsaved work" in a git context — untracked files are a different risk level than tracked modifications.
