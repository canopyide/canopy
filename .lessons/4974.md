## Context

`ActivityMonitor` watches CPU usage to detect when an agent terminal transitions from `working` to `waiting/idle`. It uses a debounced reschedule loop when CPU is elevated.

## Problem

Agent terminals would get permanently stuck in the `working` state, requiring a manual restart. This happened when: (a) a background process (like a dev server) kept CPU elevated indefinitely, or (b) `stopActivityMonitor()` was called mid-flight while a debounce timer was pending.

## Root cause

Three separate bugs:

1. **No upper bound on CPU-high deferral**: The reschedule loop had no maximum duration — if any descendant process held CPU up, the idle transition could be deferred forever.
2. **Disposed-check ordering**: The `disposed` flag was checked _before_ emitting the pending `working → waiting` transition, so calling `stopActivityMonitor()` during the debounce window silently dropped the state update.
3. **Stale pattern buffer**: A buffered "working" regex match could survive into the idle check window and block the transition even after activity had genuinely stopped.

## Lesson

- CPU-high reschedule loops **must** have an absolute upper bound (`MAX_CPU_HIGH_DURATION_MS`). Five minutes is a reasonable cap.
- When an activity monitor is disposed mid-flight, emit any pending state transition _before_ checking the disposed flag — the state must be correct even during teardown.
- Clear pattern buffers on timeout to prevent stale matches from blocking subsequent state transitions.

## Applies when

- Modifying `ActivityMonitor.ts` debounce or reschedule logic
- Any pattern that defers a state transition while waiting for a condition that may never clear
- PTY process teardown where in-flight async operations reference the disposed flag
