## Context

Canopy uses a WebGL context pool for xterm.js terminal rendering, with lease management to avoid exceeding the browser's WebGL context limit. Standard terminals and agent terminals share the pool.

## Problem

Standard terminals would occasionally show blank content. The blank content was caused by the WebGL context release/reacquire cycle leaving the terminal with no active renderer for a window. Additionally, a race condition could fire a context-loss handler after the terminal was disposed, triggering a reattach on a dead instance.

## Root cause

Standard terminals were competing for WebGL leases that they don't actually need for their use case. The release/reacquire cycle (needed for agent terminals to reclaim contexts) would transiently leave standard terminals with no active renderer, causing blank frames. The disposed-terminal check in `onContextLoss` happened after the reattach attempt rather than before.

## Lesson

Reserve WebGL exclusively for agent terminals; standard terminals should use DOM rendering unconditionally. Mixing terminal kinds in the WebGL pool creates a class of transient blank-renderer bugs that are difficult to reproduce. Implement the exclusion in `acquireWebGLLease()` by checking `panelKind` before attempting allocation.

Also: always check `isDisposed` at the top of async renderer callbacks (context loss, context restore) before taking any action. Callbacks can fire after `dispose()` is called.

## Applies when

- Adding new panel kinds that use xterm.js
- Modifying `TerminalInstanceService.ts` WebGL lease logic
- Any `onContextLoss` / `onContextRestore` handler
- Changing the WebGL context pool tier logic (FOCUSED/BURST/VISIBLE/BACKGROUND/HIDDEN)
