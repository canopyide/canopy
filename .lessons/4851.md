## Context

`TerminalRestoreController.restoreFromSerializedIncremental()` writes scrollback in chunks. Each chunk write races a 5-second fallback timeout against the `terminal.write()` callback. On success, the write callback wins the race.

## Problem

Timer handles were never cleared when the write callback resolved successfully. For a 256 KB restore (8 chunks of 32 KB), this left 8 dead 5-second timers lingering per restore. On repeated hibernation/wake cycles these accumulated.

## Root cause

The timeout ID was not captured — `setTimeout(...)` was called without storing the return value, so there was no handle to pass to `clearTimeout()` in the success branch. Classic "set and forget" timer leak.

## Lesson

Every `setTimeout` in a race-against-callback pattern must:

1. Capture the timeout ID in a variable scoped to the same closure as the callback.
2. Call `clearTimeout(timeoutId)` in the callback's success branch.

This is especially important in code paths that execute per-chunk across multiple iterations (restore, incremental sends, retry loops) — a per-iteration leak multiplies with usage.

Jest's fake timer infrastructure can catch these: `expect(jest.getTimerCount()).toBe(0)` after a complete restore is the correct assertion.

## Applies when

- Any `setTimeout` that races against an async callback — always clear it in the callback path.
- New restore/streaming paths in `TerminalRestoreController.ts` — use the same `writeChunkWithTimeout` helper, which now clears correctly.
