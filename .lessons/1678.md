## Context

On macOS, the Option (Alt) key is a dead-key modifier that transforms the typed character before it reaches the browser. Pressing `Alt+P` produces `event.key === "π"`, not `"P"`. Pressing `Alt+1` produces `"¡"`. This is how macOS users type special characters — it is not a browser quirk.

## Problem

Keybinding matching that used `event.key` to identify the pressed letter could never match a user-defined shortcut like `Cmd+Alt+P` on macOS, because by the time the keydown fires `event.key` is `"π"`. The project switcher shortcut silently did nothing.

## Root cause

`event.key` reflects the character after modifier transformations. On macOS, Option intentionally rewrites the character. `event.code`, by contrast, is the physical key location (e.g., `"KeyP"`, `"Digit1"`) and is stable across modifier state and keyboard layouts with Latin-script alphabet positions.

## Lesson

When matching `Alt`/`Option` chords on macOS, derive the key identifier from `event.code` (strip the `Key`/`Digit` prefix), not `event.key`. Gate this behavior to macOS only — on Windows and Linux, AltGr combinations should continue to use `event.key` because Linux keyboard layouts legitimately remap letters through AltGr and you want the remapped character. The fix lives in a single normalization function used by both the matcher and the chord recorder so that the stored binding and the runtime comparison agree.

## Applies when

- Registering any `Alt`/`Option` + letter or digit keybinding that must work on macOS
- A keybinding works on Linux/Windows but silently fails on macOS
- Recording chord sequences where the user holds Option
- Writing key normalization utilities that will be fed arbitrary keyboard events
