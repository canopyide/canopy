## Context

`useOrchestrationMilestones` and `useGettingStartedChecklist` used a module-level `observerInitialized` guard and set up their Zustand store subscriptions at module scope, outside of any React effect.

## Problem

Subscriptions set up at module scope are never cleaned up when components unmount. More dangerously, async callbacks inside those subscriptions can fire after the component has unmounted and act on stale state — updating state that no longer exists, or triggering effects in a component tree that has been torn down.

## Root cause

The module-level initialization pattern (`let observerInitialized = false; if (!observerInitialized) { subscribe(...); observerInitialized = true; }`) was used to avoid re-subscribing on re-renders. This works for setup, but there is no corresponding teardown. In strict mode or during fast-refresh, the component may remount, and in test environments, subscriptions from a previous test bleed into the next one.

## Lesson

All Zustand subscriptions initiated from a React hook must live inside a `useEffect` with a cleanup function that calls the returned unsubscribe handle:

```ts
useEffect(() => {
  const unsub = store.subscribe(handler);
  return () => unsub();
}, []);
```

If the subscription contains async logic, add a `disposed` flag:

```ts
useEffect(() => {
  let disposed = false;
  const unsub = store.subscribe(async (state) => {
    if (disposed) return;
    await doSomething(state);
  });
  return () => {
    disposed = true;
    unsub();
  };
}, []);
```

The same applies to `window` event listeners registered from hooks.

## Applies when

- Any hook that calls `someStore.subscribe()` directly (not via `useStore` selector)
- Hooks that attach `window` / `document` event listeners
- Any async callback registered inside a subscription — always guard with a `disposed` flag
