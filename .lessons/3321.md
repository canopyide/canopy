## Context

The BrowserWindow was created with `frame: false` and a CSS `-webkit-app-region: drag` zone for all platforms, giving macOS and Windows a custom frameless chrome. Linux was inheriting the same treatment.

## Problem

On GNOME/KDE, `frame: false` strips the window manager's title bar, which breaks standard window controls, tiling shortcuts, snapping, and workspace moves. Users lost maximise/minimise affordances and the WM couldn't drag-tile the window.

## Root cause

Electron's frameless mode removes the WM-drawn decorations. On macOS and Windows this is fine because the app draws its own chrome. On Linux the window manager owns window decorations, tiling, and drag semantics via the title bar — stripping it doesn't give you a clean slate, it just leaves the window undraggable and unresizable in the WM's eyes.

## Lesson

Always branch window chrome by platform. Pass `frame: !isLinux` and only set `titleBarStyle: "hiddenInset"` on macOS. Skip the `-webkit-app-region: drag` toolbar region on Linux since the native title bar handles dragging. Detecting Linux in the renderer needs a guard: `typeof navigator !== "undefined" && /Linux/.test(navigator.userAgent)` — tests and SSR contexts have no `navigator`.

## Applies when

Electron apps shipping to Linux. Any time you reach for `frame: false`, `titleBarStyle`, or CSS `-webkit-app-region: drag`, you need a Linux carve-out. Also applies to custom traffic-light / close-button rendering.
