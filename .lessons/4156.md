## Context

`PortalManager.navigate()` called `loadURL()` without error handling, causing unhandled promise rejections when navigation failed (DNS errors, network issues, aborted navigations).

## Problem

In Electron 37+, unhandled promise rejections in utility processes crash the process. `loadURL()` rejects for many common reasons: network unavailable, DNS failure, user-aborted navigation, redirect errors. The `createTab()` method already had a `.catch()` — `navigate()` was an oversight.

## Root cause

Inconsistent error handling between two related methods in the same class. `createTab()` pattern was correct; `navigate()` was written without checking the pattern.

## Lesson

Every `webContents.loadURL()` call must have a `.catch()` handler. In Electron 41, unhandled rejections from `loadURL()` can surface as crashes or warning noise. Treat `loadURL()` like any other async API that can fail on network issues. When adding new navigation methods, follow the pattern established in `createTab()`.

## Applies when

- Any `BrowserView`, `WebContentsView`, or `webContents.loadURL()` call
- Portal navigation, preview loading, external URL opening
- Electron services where network operations can fail silently
