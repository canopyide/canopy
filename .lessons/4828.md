## Context

When a terminal is hibernated, its xterm.js DOM container must be cleared. The hibernation manager was using a while-loop to remove child nodes one at a time.

## Problem

```ts
// Old: crosses JS-to-C++ boundary on each iteration
while (el.firstChild) el.removeChild(el.firstChild);
```

Each `removeChild` call crosses the JS-to-C++ boundary and triggers a layout invalidation. For a terminal container that may have many DOM nodes (canvas elements, rows, scrollbars from xterm.js), this is N separate layout invalidation passes.

## Root cause

The standard "safe empty a node" pattern from older JS idioms — written before `replaceChildren()` was widely available.

## Lesson

Use `element.replaceChildren()` (no arguments) to empty a DOM node. Chromium 86+ supports it; Electron 41 ships Chromium 146, so there are no compatibility concerns in this codebase.

```ts
// New: single internal ContainerNode::RemoveChildren call
hostElement.replaceChildren();
```

`replaceChildren()` batches all child removal into one internal C++ operation with a single layout invalidation pass. This is especially impactful when clearing containers with many children (terminal DOM trees, list renderers, virtual scroll containers).

Note: this optimization is safe here because `terminal.dispose()` is called during the hibernate step _before_ DOM clearing, ensuring WebGL contexts and event listeners are properly cleaned up first. The DOM clear is purely structural.

## Applies when

Any place that empties a DOM container with a while-loop or `innerHTML = ''` alternative. The `replaceChildren()` approach is always preferable for multi-child containers in Electron 36+ (Chromium 86+).
