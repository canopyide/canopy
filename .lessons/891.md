## Context

Docked terminals are rendered inside a Radix `Popover.Content`. Users pressing Escape expected the key to reach the terminal process (exit vim, cancel prompts, clear input) but instead the popover closed.

## Problem

Radix Popover's default Escape behavior is to close the popover. The keydown is swallowed before xterm's DOM ever sees it. The terminal process never receives ESC.

## Root cause

Radix listens for Escape at the overlay level and calls close(). You can stop the close with `onEscapeKeyDown` by calling `e.preventDefault()`, but you must NOT `stopPropagation()` — xterm still needs the DOM event to bubble so its own handlers can forward the byte to the PTY.

## Lesson

When embedding any input surface inside a Radix overlay (Popover, Dialog, DropdownMenu), decide per-key whether the overlay should consume the key or let it pass. For terminals in a popover: add `onEscapeKeyDown={(e) => e.preventDefault()}` (or a helper like `handleDockEscapeKeyDown`) so the popover stays open and the key propagates to xterm. Keep click-outside as the dismissal affordance. This pattern also applies to CodeMirror/Monaco/any rich input surface where Escape has semantic meaning. There's an enforcement test in `dockPopoverShadow.test.tsx` that checks the handler still uses a helper, not an inline preventDefault-only version — keep that invariant if you touch the popover.

## Applies when

Embedding xterm or any editor inside a Radix Popover/Dialog; adding a new dock/popover variant that hosts terminals; users report Escape closing the UI instead of interacting with the embedded content.
