## Context

Canopy had per-terminal flow control that paused the PTY when unacknowledged bytes exceeded a watermark (~100KB). Renderers sent acknowledgments after xterm.js consumed output. Worked fine for visible terminals, broke for any terminal without an attached renderer.

## Problem

Background terminals (minimized, in another workspace, in trash, on tab not yet mounted) never had an xterm instance to emit acks. Unacked bytes reached the watermark, the PTY paused, and since no ack would ever come, it never resumed. Agents silently froze.

## Root cause

ACK-based flow control conflates "data delivered" with "data rendered by xterm". Those are two different things and only the second needs a renderer. When the transport is SharedArrayBuffer, a successful ring-buffer write is already a delivery receipt — no renderer participation needed. The old scheme kept per-terminal ACK gating even under SAB, which turned SAB's own global backpressure into dead code for backgrounded terminals.

## Lesson

Flow control must gate on whether data was accepted by the transport, not on whether a downstream consumer has rendered it. If SAB is active, skip per-terminal ACK-based pausing and let SAB ring-buffer backpressure be the sole throttle. When transitioning a terminal into SAB mode dynamically, force-resume any currently paused PTY immediately — otherwise terminals that hit the old watermark stay frozen forever. IPC fallback still needs its own bounded queueing (see lesson 1510), but that must not depend on UI attach state either: if a renderer may never mount, flow control cannot require it to.

## Applies when

Designing or touching PTY flow control, adding a new terminal transport, debugging "terminal froze in background", or changing the rules for when a PTY is paused/resumed. Also when adding any feature that keeps terminals alive without mounted xterm instances (detached panels, headless agents, pooled terminals).
