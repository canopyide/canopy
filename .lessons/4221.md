## Context

`hardenedGit.ts` maintains a `HARDENED_GIT_CONFIG` array of `-c` flag overrides passed to all git invocations. The intention is to neutralize dangerous repo-level config (external tools, hooks, SSH commands). One entry was `diff.external=` (empty string).

## Problem

Setting `diff.external` to an empty string does NOT disable external diff — git interprets it as "execute this (empty) program as the diff tool", failing with `error: cannot run : No such file or directory` / `fatal: external diff died`. Every `git diff` invocation on a tracked modified file fails silently in the UI.

The same class of bug applies to any other empty-string override in the hardened config (e.g., `core.sshCommand=`).

## Root cause

In git, there is no safe way to "unset" a config key via `-c`. Setting a key to empty string is not equivalent to unsetting it — for keys that name an executable, empty string means "run the empty string as a program". The only safe strategy for neutralizing external tool config is to pass `--no-ext-diff` on diff commands (a git flag that ignores `diff.external` regardless of config), rather than trying to neutralize the config key itself.

## Lesson

Do not use empty-string `-c key=` overrides in hardened git config to "disable" executable-valued git settings. Empty string is not a no-op — it becomes an invalid executable path. Instead:

- Remove the empty-string override from `HARDENED_GIT_CONFIG`
- Add `--no-ext-diff` to every `git diff` call site as defense-in-depth
- Audit all other empty-string config overrides for the same class of bug

## Applies when

- Adding new security hardening to git invocations
- Debugging `fatal: external diff died` or `cannot run :` errors
- Reviewing or extending `HARDENED_GIT_CONFIG` in `electron/utils/hardenedGit.ts`
- Adding new git diff call sites anywhere in the codebase
