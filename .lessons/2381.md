## Context

Canopy builds the renderer with Vite and the main process with esbuild, then packages everything into an Electron ASAR.

## Problem

Anyone who extracted the `.asar` from a distributed build could recover the complete, readable TypeScript source — both renderer and main process.

## Root cause

Three separate exposures. (1) `vite.config.ts` had `sourcemap: true` in production, so the ASAR shipped `.js.map` files with original TypeScript embedded. (2) `scripts/build-main.mjs` used an `isProd` flag for the esbuild `sourcemap` option, but the `build:main` script did not set `NODE_ENV=production`, so `isProd` was false and main-process maps were emitted. (3) `dist-electron/electron` was not cleaned between builds, so stale `.map` files from earlier dev builds were getting packaged even when the current build did not emit them.

## Lesson

For a production Electron release: (1) `sourcemap: false` in `vite.config.ts` for the renderer build; (2) `cross-env NODE_ENV=production` in the `build:main` script so the esbuild `sourcemap: !isProd` gate actually activates; (3) clean `dist-electron/electron` before every production build to prevent stale artifacts sneaking into the package. Without all three, source maps leak. Verify by extracting your built `.asar` and grepping for `.ts` or for known symbol names after each release candidate.

## Applies when

Configuring Vite/esbuild for Electron production builds, reviewing release candidates for source exposure, or changing the build scripts.
