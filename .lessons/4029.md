## Context

Concurrent IPC operations (multiple terminal spawns, simultaneous trash+restore, rapid create/destroy) had no test coverage for race conditions.

## Problem

UI debouncing normally prevents users from triggering truly concurrent operations, hiding potential race conditions in the underlying IPC layer. Without bypass, tests couldn't exercise the concurrent paths that real agents or scripts might trigger.

## Root cause

Standard UI interaction tests serialize operations. Race conditions only manifest under genuine concurrency.

## Lesson

Use `page.evaluate()` with `Promise.all()` to bypass UI debouncing and invoke IPC calls truly concurrently in E2E tests. Key patterns:

- Concurrent spawns with same terminal ID should produce exactly one running terminal (idempotency)
- Spawn queue overflow (51+ concurrent) should return errors, not hang
- Tests should assert final state, not intermediate ordering (timing-insensitive)
- Verify main process is alive after stress with a simple IPC ping

This is the correct approach to smoke-test IPC concurrency safety without requiring OS-level thread tools.

## Applies when

- Testing new IPC handlers that could receive concurrent calls
- Any spawn/create operation that should be idempotent
- Validating queue overflow behavior
