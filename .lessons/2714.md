## Context

Cmd+W was bound to `terminal.close` in the renderer's keybinding system but had no effect. A main-process `before-input-event` handler was calling `event.preventDefault()` to stop Electron's native "Close Window" behavior.

## Problem

The keybinding never fired in the renderer. `terminal.close` was registered correctly, but the key event never reached React's keydown listener.

## Root cause

`webContents.on('before-input-event', (event, input) => event.preventDefault())` does TWO things: it suppresses the native menu accelerator AND it prevents the event from propagating further to the renderer. Once `preventDefault()` runs, the renderer's JS event listeners never see the key. This is not clearly documented — the event name suggests it's just a hook for the native shortcut layer.

The correct API is `webContents.setIgnoreMenuShortcuts(true)`: it disables native menu shortcut interception without swallowing the raw key event, so the renderer still receives the keydown and can dispatch its own handler.

## Lesson

To block an Electron native menu accelerator without stealing the keystroke from the renderer, call `webContents.setIgnoreMenuShortcuts(true)` inside the `before-input-event` handler (based on a match predicate) instead of `event.preventDefault()`. Reserve `preventDefault()` for cases where you truly want to drop the key entirely.

Test every main-process key interception by confirming the renderer sees the keydown — a silent "nothing happens" is the telltale symptom of a preventDefault swallowing the propagation.

## Applies when

Adding a main-process handler to intercept a shortcut that conflicts with a native menu accelerator (Cmd+W, Cmd+Q, Cmd+M, Cmd+R, etc.). Debugging a renderer keybinding that's configured correctly but never fires. Migrating keyboard handling between main and renderer processes.
