## Context

Dev preview panels got stuck on "Loading preview..." because the DevPreviewService never saw the dev server URL. The service listens to PtyClient "data" events to extract URLs from stdout, but the events never fired.

## Problem

When SharedArrayBuffer (SAB) is the active path for PTY output, data is written directly into the shared ring buffer and PtyClient.emit("data") is never called. Main-process listeners that rely on PtyClient data events silently get nothing even though the terminal is receiving output normally in the renderer.

## Root cause

SAB is the primary path for performance. IPC events are only a fallback when SAB is unavailable. Any main-process consumer that assumes PtyClient "data" events will always fire is wrong whenever SAB is active. This is a silent failure: output flows, UI looks fine, but monitors that wanted to observe it see nothing.

## Lesson

PtyClient "data" events are NOT a reliable observation channel while SAB is enabled. If a main-process service needs to watch PTY output (URL detection, error detection, state inference, etc.), it must opt-in to IPC data mirroring on the specific terminals it cares about, and disable mirroring on detach. Mirroring also needs to be re-armed after a PTY host restart and must respect backpressure (skip suspended/backgrounded terminals). Always think: "where does this event come from when SAB is active?"

## Applies when

Adding any main-process observer/monitor of PTY output (URL detection, error scanning, state machines). Anything in electron/ that subscribes to PtyClient "data". Reviewing why a main-process listener works in dev but not prod, or only works for unpooled terminals.
