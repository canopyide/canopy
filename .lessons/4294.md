## Context

Canopy persists session state (panel layout, open files, etc.) to disk and hydrates it on startup. Corrupted state can cause the app to crash immediately on every launch, trapping the user in an infinite crash-relaunch loop.

## Problem

If session state becomes corrupted in a way that causes a crash during hydration, the app relaunches, tries to hydrate the same corrupt state, crashes again — indefinitely.

## Root cause

State hydration was unconditional: the app always attempted to restore previous session state. There was no detection mechanism for repeated abnormal exits.

## Lesson

**Crash loop detection requires a persistent counter with a stability timer.** The counter must be stored to disk (synchronously, using `writeFileSync`) so it survives process death. A "stability timer" (5 minutes of uptime) resets the counter, preventing false positives from unrelated one-off crashes over time.

**Safe mode must skip state hydration entirely** — not just show a warning banner. If corrupted state causes the crash, partial hydration still triggers the crash. The `stateHydration/index.ts` checks `isSafeMode` from the IPC app state before running.

**Clean shutdown must mark itself explicitly** (a separate marker file or flag) so that normal app quits don't increment the crash counter. Only abnormal exits (those that don't mark clean exit before quitting) increment the counter.

**Stop-relaunch threshold** (5 consecutive crashes): after this many crashes, don't even attempt relaunch — the auto-relaunch itself becomes a denial-of-service against the user's system.

Counter file corruption (invalid JSON) must be handled gracefully — treat it as a fresh start rather than crashing the crash guard.

## Applies when

- Any startup code that reads potentially corrupted state (settings, session, DB)
- Implementing crash recovery or safe mode for Electron apps
- Designing graceful shutdown sequences that distinguish clean vs. abnormal exit
