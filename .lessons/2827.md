## Context

`sendError()` (main process) delivers errors to the renderer via IPC so they surface in the notification center.

## Problem

During startup, crash recovery, or window recreation, the main window is destroyed or not yet `did-finish-load`. Errors sent in that window were silently dropped, including critical config/filesystem errors that were exactly the ones the user needed to see.

## Root cause

The send path assumed a live `WebContents`. Any gap — early startup, main-window recreation on a new project view, crash during reload — meant the error vanished.

## Lesson

Any main-process error/telemetry pipe that targets the renderer must buffer when no window is available, not drop. Use an in-memory ring (cap it — this PR used 100) plus disk persistence (`electron-store`) for critical categories so errors survive a full crash. Flush on `did-finish-load` and expose a pull-on-demand IPC (`ERROR_GET_PENDING`) so the renderer can drain the buffer at any point in its lifecycle. Dedupe between memory and disk when flushing so reloaded errors are not duplicated. The same pattern applies to notifications, agent events, and any "fire and forget" main-to-renderer channel.

## Applies when

Adding or debugging any main-to-renderer one-shot delivery path (errors, notifications, toasts), or investigating "the error happened but nothing showed up" problems during startup or window recreation.
