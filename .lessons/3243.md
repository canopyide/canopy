## Context

The first `TerminalWebGLManager` implementation held a single WebGL lease: attach the addon on focus, detach and destroy on blur. With many terminals open, switching between them destroyed and recreated a WebGL context on every focus change.

## Problem

Every focus switch flashed blank for 1-2 frames then cost 85-230 ms while shaders recompiled. Users perceived Canopy as janky during rapid terminal switching even though nothing was actually working hard.

## Root cause

`@xterm/addon-webgl` compiles its shader program when the context is first created. That compilation is not cached by the browser — destroying the addon throws away the compiled shaders, and the next attach has to recompile them from source. There is no fast-attach API; the cost is paid on every `loadAddon(new WebglAddon())`.

## Lesson

Don't tie WebGL addon lifetime to focus. Keep a pool of live contexts keyed by terminal ID for every FOCUSED/BURST/VISIBLE tier terminal and only release on BACKGROUND-or-lower tiers. Use LRU eviction when the pool hits Chromium's limit (see #3358 — cap at 12, not 16). When handling `webglcontextlost`, capture the addon reference at registration time and verify it's still the pool's current entry before reacting, so stale events from evicted addons don't corrupt live terminals:

```ts
const ownAddon = addon;
addon.onContextLoss(() => {
  if (pool.get(id)?.addon !== ownAddon) return; // stale event
  // ...
});
```

## Applies when

Any long-lived WebGL/WebGPU consumer that's tempted to release contexts on visibility changes. The shader recompile cost dominates everything else; pool aggressively and evict lazily.
