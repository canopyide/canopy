## Context

PR detection on worktree cards is driven by services that run inside the workspace-host utility process (PullRequestService). Those services need a GitHub token. The token is stored via Electron's `safeStorage` API in the main process.

## Problem

The utility process called `GitHubAuth.getToken()` and got nothing. PR detection silently failed. safeStorage returned empty/errored because encryption keys are bound to the main process.

## Root cause

`safeStorage.encryptString` / `decryptString` are documented as main-process APIs and rely on OS keychain access that is not available from Electron `UtilityProcess`. The utility process has its own module-level GitHubAuth instance which attempted to hit the disk-backed SecureStorage. Even if you wire it up, you don't want the utility process decrypting secrets — that expands the attack surface of a process that's running less-trusted code (git parsing, remote calls).

## Lesson

Any secret stored via safeStorage stays in the main process. For utility processes that need the secret, add a memory-only mode: the auth module keeps an in-memory token and never touches SecureStorage when running in a utility process. The main process pushes the token to the utility process over IPC on the utility's "ready" event (not "spawn" — spawn fires before listeners are attached, so the first message can be dropped). Re-push on every utility restart. Clear the in-memory token when the user signs out in main, and reset any dependent state (PR cache, etc.) to avoid stale data. The in-memory token must never be persisted by the utility process to disk.

## Applies when

Adding any feature in `workspace-host.ts`, `pty-host.ts`, or any other Electron `UtilityProcess` that needs credentials, tokens, or OAuth secrets. Also when reviewing why `safeStorage`-backed values work in dev (single process) but fail in a new utility-hosted service, or when porting a main-process service into a utility process.
