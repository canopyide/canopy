## Context

`TerminalWebGLManager` attaches `@xterm/addon-webgl` to terminals for fast rendering. On machines where Chromium falls back to software GL (llvmpipe, ANGLE CPU path, or the user disabled hardware acceleration), "WebGL" still works — it's just emulated on the CPU.

## Problem

On software-GL systems, attaching the WebGL addon was slower than the DOM renderer it replaced. The addon reported successful context creation, so there was no fallback, and terminals janked badly on scroll.

## Root cause

A `WebGLRenderingContext` is still obtainable when Chromium's GPU process is doing the rendering in software. From the xterm addon's point of view the feature is "available"; there is no runtime API on the `WebGLRenderingContext` to say "this is CPU-emulated". The truth lives in Electron's main-process API `app.getGPUFeatureStatus()`, which returns strings like `"enabled"`, `"disabled_software"`, `"unavailable_software"`, `"disabled_off"`.

## Lesson

Before calling `terminal.loadAddon(new WebglAddon())`, check `app.getGPUFeatureStatus().webgl2` (main process) and forward the boolean to the renderer via state hydration. Skip WebGL attach when the status contains `"software"` or is anything other than `"enabled"`. Keep the `CANOPY_DISABLE_WEBGL` env override as an always-on escape hatch:

```ts
// Main:
const status = app.getGPUFeatureStatus();
const gpuWebGLAvailable = status.webgl2 === "enabled";
// Renderer: if (!gpuWebGLAvailable || env.CANOPY_DISABLE_WEBGL) return; // DOM renderer
```

Do this hydration once at startup before the first terminal is constructed — the addon decision is sticky per terminal.

## Applies when

Any Electron app attaching `@xterm/addon-webgl` (xterm 6.0 dropped the canvas renderer, so DOM is the only fallback). Also applies to other WebGL-backed renderers (e.g. custom chart libs) where software GL is a regression compared to Canvas2D/DOM.
