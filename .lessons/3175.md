## Context

Agents (Claude, Gemini, Codex) launch with CLI flags — e.g. `claude --dangerously-skip-permissions`. On crash recovery or hot-reload, `buildResumeCommand()` reconstructed the launch from the agent name and session ID but did not include the original flags.

## Problem

Resumed Claude sessions prompted for every permission instead of running autonomously. Users in dangerous-mode had to re-opt-in after every restart. The bug was invisible in logs because the resume "worked" — it just ran with different capabilities than the original session.

## Root cause

Only the session ID was round-tripped through IPC and persistence. Launch flags were derived ephemerally at launch time from current settings, so resume always re-derived them from _current_ settings, not the settings that were active when the session started. If the user disabled dangerous-mode in the meantime, the resumed session didn't honour the original intent either.

## Lesson

Whenever a feature persists a resumable state (session ID, process handle, remote URL), it must also persist the arguments that shaped the original invocation. Store `agentLaunchFlags` on the panel at launch time, round-trip it through every layer — persistence, IPC (`PtyPanelData`, `TerminalState`), hydration — and prepend it to the resume command:

```ts
// buildResumeCommand now takes flags:
function buildResumeCommand(agent, sessionId, flags: AgentLaunchFlags) {
  return [...flags, agent.resumeSubcommand, sessionId];
}
```

Test every resume path (cold-start hydration, live reconnection, crash recovery) with and without flags.

## Applies when

Any "resume", "reconnect", or "reattach" API that reconstructs a process or request from stored state. If the original invocation took arguments that affected behaviour, those arguments must travel with the resumable token — never re-derive them from current settings at resume time.
