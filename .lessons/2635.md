## Context

Panel-header context menus were built with Electron's `Menu.buildFromTemplate` and IPC round-trips, using `sublabel` to display keyboard shortcut hints next to menu items.

## Problem

Electron's native menu renderer always places `sublabel` on a second line below the main label, producing tall, two-line menu items with the shortcut wrapped under the text. There is no native option to position it inline on the right (the way every modern UI shows shortcuts).

## Root cause

`sublabel` in Electron's native menu is rendered by the OS's native menu widget (macOS `NSMenu`, Windows context menu). Neither native widget exposes an inline-shortcut slot that Electron maps to; the Electron API hands you `label` + `sublabel` and lets the OS decide placement. On macOS, `sublabel` becomes a second line. There is no workaround via template config, accelerator config, or role — you cannot get an inline right-aligned shortcut column in native menus.

## Lesson

If you need menu items with inline shortcut hints (the standard VS Code / modern app look), do not use Electron's native `Menu.buildFromTemplate` — render menus with a DOM-based library like `@radix-ui/react-context-menu` that gives you full control over the layout. The tradeoff: native menus match OS look, Radix menus match in-app look. Canopy chose Radix because consistency with other menus and inline shortcuts mattered more than OS chrome.

Bonus: DOM menus avoid the main↔renderer IPC round-trip per open, which was a minor perf win.

## Applies when

Designing any menu surface that should show keyboard shortcuts inline. Evaluating whether to use native vs. DOM menus. Implementing context menus that match in-app menu primitives (dropdowns, command palette). Migrating away from `Menu.buildFromTemplate` for UX reasons.
