## Context

The main process already called `v8.setFlagsFromString("--expose_gc")` and used `global.gc()` during memory pressure. The PTY host utility process — which manages the most memory-intensive resources (node-pty native bindings, headless xterm parsers, output buffers) — lacked the `--expose-gc` flag, making the existing `ResourceGovernor.engageThrottle()` GC call a silent no-op.

## Problem

After batch terminal kills (project hibernation, bulk kill), V8's lazy GC might not reclaim freed native PTY bindings and parser memory promptly, leaving dead allocations in the heap until the next natural GC cycle. The `ResourceGovernor` already called `global.gc()` but it did nothing.

## Root cause

The PTY host is spawned as a utility process with `execArgv: ['--max-old-space-size=...']` only. `--expose-gc` was never added, so `global.gc` was undefined. The ResourceGovernor code had no error path for this case.

## Lesson

When spawning utility processes, audit which V8 flags are needed for the process's workload. The PTY host handles native bindings that benefit from explicit GC after bulk operations, but `--expose-gc` must be in `execArgv` — it cannot be set after process start.

Critically: `global.gc()` is synchronous and pauses the event loop. Call it only after batch operations during natural pauses (post-hibernation, post-bulk-kill), never during active terminal streaming. A 100ms `setTimeout` deferred call is safer than immediate invocation to ensure the event loop is idle.

Note: `--optimize-for-size` (PR 3793) was NOT added to the PTY host — the PTY host does real-time stream parsing where speed matters more than in the main process.

## Applies when

- Utility processes managing native bindings (node-pty, node-addon-api modules)
- Batch resource cleanup operations where explicit GC would help
- Debugging why `global.gc()` calls in a utility process are silently doing nothing
