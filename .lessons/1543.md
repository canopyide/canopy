## Context

Sensitive environment variable detection used `/\b(key|secret|token|password)\b/i` to decide whether to mask a value before displaying or logging it. Variables like `GITHUB_TOKEN`, `API_KEY`, and `AWS_SECRET_ACCESS_KEY` leaked through unmasked.

## Problem

The regex looked correct — it even had case-insensitive matching — but it still failed on the exact variable names we were trying to catch.

## Root cause

In ECMAScript regex, `\b` is a word boundary between a word character (`[A-Za-z0-9_]`) and a non-word character. Underscore is a word character, so there is no boundary between `TOKEN` and `_`. `GITHUB_TOKEN` has no boundary next to `TOKEN` on the left, so `\btoken\b` does not match. Same for `API_KEY`, `AWS_SECRET_...`, `DATABASE_PASSWORD`, etc. This is ECMAScript spec behaviour, not a bug.

## Lesson

Never use `\b` to delimit tokens in identifiers that contain underscores — env var names, snake*case symbols, or any SCREAMING_SNAKE identifier. Use substring matching (`/(key|secret|token|password)/i`) or explicit separator classes (`/(?:^|[*-])token(?:$|[_-])/i`) if you need word-like matching. Write test cases that include the exact identifier formats you expect in production (`GITHUB_TOKEN`, not just `token`).

## Applies when

Building any redaction, filter, or classifier over env var names, config keys, or any underscore-heavy identifier namespace. This regex pattern recurs across logger redaction, settings UIs, telemetry scrubbing, and diagnostic exporters.
