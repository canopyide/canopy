## Context

Activity patterns for Claude/Codex/Gemini anchor the "esc to interrupt" hint at end-of-line so they only match when the hint is actually at a terminal row boundary. The scan buffer fed to these regexes contains multiple lines joined with `\n`. Long agent status descriptions (120+ chars) were wrapping and the patterns stopped matching, so the activity monitor lost track of working state.

## Problem

In JavaScript's default regex mode, `$` only matches end-of-string, not end-of-line. A pattern like `esc to interrupt[^)\n]*\)?$` applied to a buffer of many joined rows will only match if the hint happens to land at the very end of the whole buffer — otherwise it fails silently. Without the `m` (multiline) flag, the anchor is useless for multi-row scan buffers.

## Root cause

The compiler called `new RegExp(pattern, "i")` instead of `new RegExp(pattern, "im")`. The authoring style of using `$` in the agentRegistry entries assumed multiline semantics that the compile step never enabled.

## Lesson

Any regex that uses `^` or `$` against a multi-line scan buffer must be compiled with the `m` flag. In Canopy this is enforced centrally in `compilePatterns` in `terminalActivityPatterns.ts` — new pattern sources should go through that helper, not call `new RegExp` directly. When adding patterns to `agentRegistry.ts`, it is safe to assume `i` and `m` are set; when adding patterns elsewhere, double-check the compile site.

## Applies when

Adding or tuning agent detection patterns in `shared/config/agentRegistry.ts`, writing new pattern matchers for terminal scan buffers, or debugging "pattern is obviously right but never matches" on buffers containing multiple lines.
