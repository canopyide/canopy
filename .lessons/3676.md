## Context

Active tab state was stored in two places: `activeTabByGroup` in the focus slice and `tabGroups[groupId].activeTabId` in the registry slice. A 40-line manual sync override in `terminalStore.ts` kept them in sync, including async persistence via `import().then()`.

## Problem

The async persistence used `get().tabGroups` inside an async callback. If state changed between the `set()` call and the async `import().then()`, stale data was saved to disk. Any code calling `setState({ activeTabByGroup: ... })` directly would bypass the sync, creating split-brain state. Hydration at startup required manually syncing registry → focus layer; silent divergence if this failed.

## Root cause

Dual state storage with manual synchronization is inherently fragile. The registry slice was the natural single source of truth (it already had `activeTabId` per group), but a focus slice was layered on top for reactive reads without the sync being automatic.

## Lesson

In Zustand, never maintain the same value in two slices with a manual sync override. The sync logic creates async race conditions and bypass risks. Instead, designate one slice as the single source of truth and add a selector to it:

```ts
// In registry slice — single source of truth:
getActiveTabId: (groupId) => {
  const group = state.tabGroups[groupId];
  if (!group) return state.terminalMap.get(groupId)?.id ?? null; // virtual groups
  return group.activeTabId;
};
```

Remove the manual sync override entirely. Components that previously read from the focus slice update to use the registry selector.

**Virtual groups** (single panels without an explicit `TabGroup`) must be handled in the selector — they need special-casing since they have no `TabGroup` entry.

## Applies when

Any Zustand store with dual state; reviewing persistence code that uses async `import().then()` with `get()` after a `set()`; diagnosing tab state inconsistencies after rapid switching.
