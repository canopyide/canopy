## Context

Canopy parses GitHub clone URLs to extract `owner/repo` for issue, PR, and release queries. The original parser used a regex along the lines of `github\.com[:/]([^/]+)/([^.]+)` that truncated the repo name at the first dot.

## Problem

Any repo whose name contains a dot (`my.repo`, `awesome.site`, `next.js`, or the `.github` organizational repo) had its name truncated at the first dot. `foo/my.repo` became `foo/my`. All GitHub integration silently broke for those projects — issues wouldn't list, PRs wouldn't link, CI wouldn't show — with no user-facing error to explain why. Trailing slashes on URLs and `.git` suffixes interacted with the regex in similar subtle ways.

## Root cause

Regex negative character classes for "stop at dot" assumed the dot only ever separated the `.git` extension. GitHub actually allows dots in repo names. The URL has enough structure to parse with `new URL()` + path splitting — which strips trailing slashes, handles percent-encoding, and doesn't care about dots.

## Lesson

Parse URLs with `new URL()` or a dedicated library, never with a hand-rolled regex. For GitHub specifically: strip a trailing `.git` only at the end, strip trailing slashes, then split the pathname. Support both HTTPS (`https://github.com/owner/repo`) and SSH (`git@github.com:owner/repo`) forms — the SSH form is not a valid URL and needs its own branch. Write table-driven tests with repos named `.github`, `my.repo`, `repo.with.dots`, and URLs with and without `.git`, trailing slash, and extra path segments.

## Applies when

Adding any new GitHub-integrated feature, or any feature that parses git remote URLs. Also applies to npm packages (scoped names with slashes), Docker image names (registry/namespace/name:tag), and any other identifier namespace with optional trailing qualifiers.
