## Context

`index.html` carried a single `<meta http-equiv="Content-Security-Policy">` tag that had to allow `localhost:5173` and `ws://localhost:5173` so Vite HMR could connect in dev. That same permissive CSP shipped into production builds, silently weakening the runtime security posture of the packaged app.

## Problem

A dev-friendly CSP that references localhost or `ws://` bypasses one of Electron's main defenses against remote-content attacks. But editing `index.html` between runs, or maintaining two copies, is fragile — developers forget, and CI happily ships the dev version.

## Root cause

Static CSP in `index.html` has no concept of build mode. Vite does, via the `transformIndexHtml` hook on plugins: `ctx.server` is present in dev and absent in production, so a plugin can swap the CSP meta tag's `content` attribute at build time.

## Lesson

Define `DEV_CSP` and `PROD_CSP` constants in `vite.config.ts`, and register a `transformIndexHtml` plugin that matches the meta tag with a case-insensitive regex and rewrites `content`. Crucially, have the plugin **fail the build** if the meta tag is missing — a silent failure here would ship an empty CSP, which is worse than the dev version. Allow both single and double quotes in the attribute regex since formatters can rewrite them.

## Applies when

Any time you need environment-specific CSP, HTML meta tags, or inline script nonces in a Vite-based Electron renderer. The same pattern handles dev-only hot-reload hooks, production-only CSP hashes, and localhost allow-lists that must never reach production.
