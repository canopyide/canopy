## Context

Dock panels (browser, dev-preview) host Electron `<webview>` elements. When the dock popover closed, React unmounted the panel subtree, which destroyed the `<webview>` and lost scroll position, navigation history, form inputs, and zoom level. Reopening the dock rebuilt the webview from scratch and reloaded the URL.

## Problem

A webview inside a component that conditionally renders via `{open && <Panel/>}` gets fully destroyed every close/open cycle. No user state survives the cycle because the `<webview>` tag only holds state while attached to the DOM.

## Root cause

Electron `<webview>` has no equivalent of React's `hidden` that preserves its renderer process across unmount. Once the DOM node is removed, the guest page is torn down. Conditional rendering in React therefore cannot be used to "hide" a webview while keeping it alive.

## Lesson

To keep a webview alive while visually hidden, render it into a stable offscreen React portal target that is always mounted. Move the panel between the offscreen container and the visible popover by switching which portal target receives it — never toggle mount state. Use a tiny offscreen size (1x1px, `position: absolute`, `pointer-events: none`) rather than `display: none` so the guest page keeps receiving layout and can run background work if the app needs it. Using a single stable portal target for both visible and hidden states avoids the remount that happens when React flips between two different parents.

## Applies when

- Hiding/showing Electron `<webview>` or any element whose teardown cost is non-trivial (iframes with navigation history, canvas renderers, media players)
- Implementing dock/minimize/background-tab semantics where the underlying view must survive visibility toggles
- A panel loses scroll position, form data, or network state every time it is reopened
