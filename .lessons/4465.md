## Context

Canopy's browser panel renders a `<webview>` with a dynamically-built Content Security Policy. The CSP has a special localhost mode that relaxes restrictions when the URL is a local dev server. This regression was introduced in v0.5.0.

## Problem

Two separate CSP gaps broke local dev server access:

1. **IPv6 loopback not recognised**: `isLocalhostUrl()` only matched `localhost` and `127.0.0.1`. URLs like `http://[::1]:5173/` were not treated as localhost, so the relaxed CSP was never applied and `connect-src` blocked all requests.

2. **`unsafe-eval` missing from `script-src`**: Vite's Hot Module Replacement uses `eval()` to apply module updates in-browser. Without `'unsafe-eval'` in `script-src`, HMR silently failed, breaking the live-reload experience for any Vite-based dev server.

## Root cause

The `isLocalhostUrl()` helper was written for the IPv4 case only. IPv6 loopback has a different URL format (`[::1]`) that requires a dedicated check. The `unsafe-eval` omission was a conservative CSP default that didn't account for eval-dependent dev tooling.

## Lesson

- Always include `[::1]` alongside `127.0.0.1` and `localhost` in any localhost detection utility. IPv6 loopback is the default on many modern systems (Node.js, macOS networking stack).
- Vite's dev server (and other bundler dev servers) require `'unsafe-eval'` in `script-src` for HMR to function. Add it to the CSP when the origin is localhost — the security tradeoff is acceptable for local-only origins.
- Centralise localhost detection into a single shared helper (`urlUtils.ts`) used by both CSP builders and URL validators so both sites are updated together.

## Applies when

Building Content Security Policies for webviews or iframes that need to load local dev servers, especially Vite, webpack-dev-server, or any bundler that uses eval-based HMR.
