## Context

The terminal inside the dev preview console drawer was blank when the drawer was expanded. Toggling it closed and re-opened did not fix it.

## Problem

XtermAdapter was conditionally rendered based on `drawerOpen`. When drawerOpen flipped to true, the xterm.js instance initialized inside a container whose layout had not yet stabilised — it saw 0x0 dimensions and cached those as its initial size. The subsequent ResizeObserver event sometimes fit the display to the correct size, but rendering was already broken for the initial frame and any subsequent text arrived into a zero-row buffer.

## Root cause

xterm.js `FitAddon` relies on non-zero container dimensions being available at construction/open time. If the element exists only inside a conditional branch, React renders it in the same tick that the parent expands, and the browser has not yet flushed layout. The FitAddon reads `getBoundingClientRect()` and gets zeros.

## Lesson

Do NOT conditionally mount xterm.js via `{open && <Xterm />}`. Mount it unconditionally and hide/show it with CSS (`visibility`, `z-index`, height animation, `display: none` for hidden siblings). This keeps dimensions stable across toggles and lets FitAddon fit against the real layout. Also set `aria-hidden` and `inert` on the hidden state for a11y, and manage refresh tier (VISIBLE/BACKGROUND) explicitly so xterm throttles when hidden. The same pattern applies to any drawer, tab, collapsible, or maximize transition that contains an xterm.

## Applies when

Adding a new surface that hosts an xterm instance (drawers, modals, tabs, splits, sidecars). Debugging "terminal is blank" after toggle, layout change, or panel move. Reviewing panel kind module changes. Working with @xterm/xterm 6.0 which removed the canvas renderer and changed the viewport implementation — layout bugs behave slightly differently than in older xterm versions.
