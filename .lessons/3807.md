## Context

Bulk worktree creation from the GitHub issues panel was failing for the last several items in large batches (10+). A retry mechanism existed (2 retries, 1s/2s delays) but was insufficient.

## Problem

PQueue was configured with `concurrency: 4` and no inter-operation delay. With 16+ items, 4 concurrent `git worktree add` + branch creation operations competed for the same `.git/` lock files. Git's locking is not designed for high concurrency from the same repository — even 2-3 concurrent `git worktree add` operations can collide. The built-in retries helped individual items but kept perpetuating lock pressure by immediately queuing more operations at the same high concurrency.

## Root Cause

The 4-concurrent assumption is fine for network/API operations but wrong for filesystem-locking git operations. The retry mechanism is insufficient when the system keeps creating new contention at the same rate it's retrying.

## Lesson

For git operations (especially `git worktree add`, `git checkout`, branch creation), limit PQueue concurrency to 2 and add a minimum inter-operation interval (300ms). Use PQueue's `intervalCap: 1`, `interval: 300`, `strict: true` for minimum spacing between task starts. Extract these as named constants (`QUEUE_CONCURRENCY`, `QUEUE_INTERVAL_CAP`, `QUEUE_INTERVAL_MS`) for clarity. Retry logic is a safety net, not a substitute for appropriate concurrency limits. Reliability matters more than speed for bulk worktree creation — a 10-20 second batch is fine.

## Applies when

- Any bulk operation that runs concurrent `git worktree add`, `git checkout`, or other git lock-acquiring commands
- Any PQueue used for git operations — default concurrency 4 is likely too high
