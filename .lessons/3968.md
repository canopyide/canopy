## Context

`ActivityMonitor` detects agent state via pattern matching (`PromptDetector`) and a 6-second idle timeout (`IDLE_DEBOUNCE_MS`). Pattern matching catches known prompt formats fast (500ms). Unknown formats fall through to the timeout, making agents appear stuck in "working" for 6 seconds.

## Problem

Pattern-based detection has an inherent blind spot: new agents, version updates, or natural-language clarifying questions (e.g., `"Which file should I modify?"`) never match configured patterns. The system had no catch-all mechanism — the only fallback was the 6-second idle timeout with low confidence (0.6).

## Root cause

The design correctly used pattern-first detection but had no intermediate heuristic layer. The 6-second timeout was conservative to avoid false positives, but this penalized all unrecognized prompts equally.

## Lesson

Implement a three-tier prompt detection hierarchy:

1. **Pattern match** — highest confidence, 500ms fast-path
2. **Stall + lexeme heuristic** — medium confidence (0.7), fires after 3s of quiet if the last visible line contains `?`, `:`, `y/n`, `[Y/n]`, or `(yes/no)` without a trailing newline
3. **Idle timeout fallback** — lowest confidence (0.6), 6 seconds, all other cases

The lexeme heuristic is the pexpect/SWE-agent pattern: "output stall + incomplete line + prompt indicator." Key conditions to prevent false positives: suppress it when a working pattern is already active, reject lines with carriage returns (progress bars rewrite lines), and reject pure progress indicators (`%`, `eta`). Always add the lexeme check AFTER existing pattern detection to avoid changing existing behavior.

## Applies when

Agents display novel prompt formats or clarifying questions that don't match configured patterns. Also relevant when tuning the balance between false-positive rate and detection latency.
