## Context

`HibernationService` decides which projects to hibernate by computing inactivity duration as `now - (project.lastOpened || 0)`. The `|| 0` fallback was intended as a safe default for missing timestamps.

## Problem

If `project.lastOpened` is `0`, `null`, or `undefined`, the `|| 0` fallback evaluates to `now - 0`, producing an inactivity of ~55 years (~1.7 trillion ms). Since the hibernation threshold is 24 hours, the project is immediately hibernated — killing all its terminals.

This happened because SQLite's `NOT NULL` constraint on `lastOpened` is not retroactively enforced on existing rows added via migration. A database migration that added the column could leave existing rows with `null` or `0` for `lastOpened`.

## Root cause

`|| 0` is the wrong fallback for a timestamp used in duration arithmetic. Zero epoch means "opened 55 years ago" not "opened just now". The correct behavior for missing data is to skip the project (conservative default), not to treat it as maximally stale.

## Lesson

When writing duration-based logic using timestamps that may be falsy, **never use `|| 0` as the fallback**. Zero epoch produces maximal duration, which is almost always the worst possible default. Instead, skip or treat as "just now":

```ts
// Wrong — || 0 means "55 years ago":
const inactive = now - (project.lastOpened || 0);

// Right — skip falsy timestamps entirely:
if (!project.lastOpened) continue;
const inactive = now - project.lastOpened;
```

SQLite schema `NOT NULL` constraints don't retroactively enforce on rows pre-dating the constraint (added via `ALTER TABLE` migration). Always handle falsy timestamps defensively at the application layer.

## Applies when

Any duration or age computation involving timestamps from persistent storage; hibernation logic; SQLite migration scenarios where columns are added to existing tables.
