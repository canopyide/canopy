## Context

The diagnostics log viewer and its search filter called `JSON.stringify` directly on arbitrary log context. Anywhere a BigInt appeared in a log payload — filesystem inode numbers, `process.hrtime.bigint()`, certain native module stats — the whole log viewer crashed with `TypeError: Do not know how to serialize a BigInt`.

## Problem

`JSON.stringify` was called without a replacer and without a try/catch on a completely unknown data shape. A single hostile entry took down the UI.

## Root cause

The JSON spec has no BigInt representation, so ECMAScript throws rather than guess. Circular references also throw. Symbols and functions silently become undefined. Error objects serialize to `{}` because their properties are non-enumerable. Every one of those cases is real and shows up in production log context eventually.

## Lesson

Never call `JSON.stringify` directly on data you did not construct. Use a `safeStringify` helper that (a) passes a replacer converting BigInt to string, (b) tracks visited objects in a WeakSet for `[Circular]`, (c) unwraps Error objects to `{name, message, stack}`, (d) has a top-level try/catch with a fallback string so a hostile Proxy or throwing `toString()` cannot take down the caller. This is mandatory for anything that touches user-facing diagnostic surfaces (log viewers, error banners, telemetry exports) where one bad entry must not break the whole view.

## Applies when

Any renderer code that formats log entries, error reports, IPC payloads, or user-configurable context objects. Also applies to `structuredClone` and `postMessage` — they each have their own non-serializable-value behaviour worth checking.
