## Context

The `TerminalRefreshTier` system (BURST/FOCUSED/VISIBLE/BACKGROUND) already manages WebGL context release and scrollback size per tier. However, FileLinksAddon and ImageAddon were loaded once at terminal creation and left active for the entire terminal lifetime regardless of tier.

## Problem

With 30-40+ background agent terminals receiving data, FileLinksAddon runs its `FILE_PATH_REGEX` against every visible line during rendering for terminals the user never sees. ImageAddon can hold significant memory for image data in background terminals. The cumulative cost across all background instances is non-trivial.

## Root Cause

The tier-aware lifecycle infrastructure existed (via `TerminalWebGLManager`) but wasn't extended to FileLinksAddon and ImageAddon. These addons were treated as permanent rather than tier-managed resources.

## Lesson

xterm.js addons that do per-render work (regex scanning, image processing) should follow the same tier-aware dispose/recreate lifecycle already used by the WebGL addon:

- `BACKGROUND` tier → dispose both addons
- `VISIBLE` or `FOCUSED` tier → recreate and reload them if missing

xterm.js doesn't have a public `unloadAddon()` API, but `addon.dispose()` cleanly unregisters hooks. FileLinksAddon specifically registers via `terminal.registerLinkProvider()` which returns an `IDisposable` — call that disposable, then re-register when returning to visible tier.

Make addon fields nullable (`| null`) in type definitions to support the dispose/recreate cycle safely.

SearchAddon, FitAddon, and SerializeAddon should remain loaded at all times — they're needed regardless of tier.

## Applies when

- Adding any new xterm.js addon that does per-render or per-data work
- Optimizing memory/CPU for apps with many concurrent background terminals
- Extending the tier transition logic in `applyRendererPolicyImmediate()`
