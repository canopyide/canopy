## Context

`getBulkStats` in the project switcher retrieved stats for all projects at once. Internally it called `getTerminalAsync` once per terminal to build agent counts, then ran per-project stats lookups.

## Problem 1 — N+1 IPC pattern

With 10 projects × 8 terminals each, `getBulkStats` triggered ~180 IPC round-trips to the pty-host every time it ran (which was every 10 seconds while the project switcher was open). Each call crossed the IPC boundary between the renderer host and the pty utility process.

**Fix:** Replace the per-terminal loop with a single `getAllTerminalsAsync` call, then group agent counts by project ID in memory. Per-project stats run in parallel via `Promise.all`.

## Problem 2 — Silent isTrashed bug

The pty-host's `get-all-terminals` and `get-terminals-by-state` handlers read `isTrashed` off the raw `TerminalInfo` object, but that field isn't on `TerminalInfo` — it lives in the registry. Result: `isTrashed` was always `undefined`, silently making trashed terminals appear as live.

**Fix:** Read `isTrashed` from the registry map, not from the serialized terminal object.

## Lesson

Two separate lessons here:

1. When aggregating data that requires iterating over a collection, always batch the underlying fetch first, then aggregate in memory. A loop of async calls that each cross a process boundary (IPC, network) is a classic N+1 pattern — easy to miss because each individual call looks innocuous.

2. When reading fields from deserialized IPC objects, verify the field actually exists on that type. TypeScript won't catch this if the field is optional or comes from a different object shape with the same name. The pty-host registry and the serialized `TerminalInfo` both exist for valid reasons but have different shapes.

## Applies when

- Any bulk-stats handler that loops over terminals/windows/processes
- Reading derived state (trashed, active, paused) from IPC objects — always trace which object shape you're reading from
