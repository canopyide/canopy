## Context

The "directing" state (pencil indicator) appears when the user is typing a prompt into an agent terminal. A debounce timer clears it when typing stops. Previously a flat 2500ms timeout was used for all input.

## Problem

For users typing multi-sentence prompts — especially across LLM interfaces that accept long instructions — the 2500ms timeout was too short. The directing indicator disappeared mid-composition, suggesting the user had abandoned their input.

## Root cause

The same debounce (2500ms) was applied regardless of whether the user had typed 1 character or 50 characters. Short accidental presses need a fast timeout; deliberate multi-sentence compositions need a much longer one.

## Lesson

Use a two-phase debounce based on composition length:

- **Phase 1** (0–4 chars): 1500ms — handles accidental presses, arrow-key slips, quick corrections
- **Phase 2** (5+ chars): 10000ms — handles deliberate multi-sentence prompts

Track composition count per terminal ID. Backspace (`\x7f`) decrements the count (floor 0) and can move the state back to Phase 1. Ctrl+U (`\x15`) resets the count to 0 (line clear).

Bracketed paste (xterm.js 6.0 delivers as a single chunk) jumps immediately to Phase 2 if ≥5 chars.

Callers that only signal keep-alive (no raw data) should always use Phase 1 timeout — they are refreshes, not real keystrokes, and should not escalate the debounce window.

Clean up composition counts in all four exit paths: Enter, Escape, agent response received, terminal destroyed.

## Applies when

Any directing/composing state backed by a debounce where input length is a meaningful signal for the user's intent.
