## Context

The `ActivityMonitor` uses multiple interacting timing windows to detect when an agent transitions from working to waiting/idle. These constants work together: `WORKING_HOLD_MS`, the prompt fast-path quiet threshold, and `IDLE_DEBOUNCE_MS`.

## Problem

Agents flickered visually between "working" and "waiting" states during active work, especially between tool calls. The state indicator would briefly flip to "waiting" and immediately return to "working" multiple times in a single continuous session.

## Root Cause

Five interacting causes:

1. `WORKING_HOLD_MS` was only 200ms — a single >200ms pause between tool calls (common during API latency) expired the hold and opened a path to idle.
2. Prompt fast-path quiet threshold was 1000ms — a 1-second pause between tool invocations is normal; if a prompt pattern was momentarily visible, this path fired.
3. `AgentPatternDetector` scans only the last 10 lines — when an agent clears its status line between tool calls, the working pattern disappeared from the scan window, making `isWorkingPattern` go false.
4. `WorkingSignalDebouncer` recovery delay (1500ms) created asymmetric response: fast exit from working but slow re-entry, amplifying perceived jitter.
5. `IDLE_DEBOUNCE_MS` at 300ms was too short.

## Lesson

Timing constants in activity detection interact multiplicatively, not additively. A 200ms hold + 1000ms fast-path + 300ms idle debounce creates compounding failure modes where multiple guards can fire in sequence during normal API latency gaps. When tuning these constants, consider the realistic pause cadence of each supported agent (Claude bursts, Gemini spinner rewrites, Codex longer gaps) and set thresholds above the common inter-tool-call pause duration.

Fix: `WORKING_HOLD_MS` 200→800ms, prompt quiet threshold 1000→2500ms, `IDLE_DEBOUNCE_MS` 300→600ms.

## Applies when

- Agent state appears unstable or flickering during active multi-tool-call sessions
- Adjusting any single ActivityMonitor timing constant without considering interactions
- Adding a new agent type with a different output cadence
