## Context

`CrashRecoveryService` writes a `running.lock` marker at session start and removes it on clean exit. If the marker is present on next launch, it's treated as a crash. On Windows, nodemon restarts Electron via `taskkill /F` (TerminateProcess), which kills the process without invoking signal handlers or `before-quit`, orphaning the marker every hot-reload cycle.

## Problem

Every hot-reload in dev on Windows surfaced the crash recovery dialog, because `taskkill /F` bypasses all graceful shutdown hooks. SIGTERM/SIGINT handlers on Windows don't fire for forced process termination.

## Root cause

Windows `TerminateProcess` (used by taskkill /F and therefore nodemon) is not a signal — it's a kernel-level termination that gives the process no opportunity to clean up. No Node.js shutdown hooks, `before-quit`, or signal handlers fire. This is fundamentally different from macOS/Linux where SIGTERM is delivered and handled.

## Lesson

For dev-mode resilience, write an `isPackaged: false` field into the crash marker at session start. On startup, if both the stale marker and the current session are dev-mode **and** the marker has no crash log attached, silently discard the marker rather than showing the recovery dialog.

```ts
// On startup, discard orphaned dev markers:
if (!marker.isPackaged && !app.isPackaged && !marker.crashLogPath) {
  // Nodemon/hot-reload orphan — discard silently
  return;
}
```

Keep signal handlers for macOS/Linux (they work correctly there). The `isPackaged` field in the marker acts as a universal fallback for platforms where signals don't fire. Always handle legacy markers lacking the field (treat as production to be safe).

## Applies when

Any Electron app using crash/session markers in dev with nodemon or electron-reload on Windows. The `taskkill /F` problem affects all nodemon-based hot-reload setups.
