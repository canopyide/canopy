## Context

`TerminalRestoreController` has two restore paths: a small path (single `terminal.write()`) and an incremental path (multiple chunks). The incremental path guards against stale callbacks using a `restoreGeneration` counter — if the terminal was unhibernated between scheduling the write and the callback running, the generation mismatch causes the callback to bail out.

## Problem

The small restore path only checked object identity (`terminal === managed.terminal`), not `restoreGeneration`. If unhibernation fired between scheduling the write and the callback, the callback passed the identity check and applied a scroll offset calculated from the old terminal buffer against the new one, corrupting the display.

## Root cause

The two restore paths were written at different times. The incremental path added `restoreGeneration` protection as a later safety measure, but the fix was not backported to the small path. The small path looked safe (object identity check) but the check was insufficient: after unhibernation the managed terminal is a new xterm instance, so identity fails — but if the old instance is somehow retained (edge case in the wake timing), identity can pass with stale state.

## Lesson

When a safety mechanism (generation counter, epoch check) is added to one code path, audit all sibling paths for the same vulnerability. The incremental and small restore paths share the same race window; generation checks should be consistent across both.

Pattern: capture `const gen = ++managed.restoreGeneration` before the async operation, then check `managed.restoreGeneration === gen` in the callback. This is safe even if the counter wraps (extremely unlikely at write frequency).

## Applies when

- Adding new async write operations to `TerminalRestoreController.ts` — always capture and validate `restoreGeneration`.
- Any pattern where a callback is scheduled on an object that can be replaced before the callback fires — generation counters are the correct solution.
