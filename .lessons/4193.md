## Context

`AutoUpdaterService.initialize()` had a Windows portable build guard (checking `PORTABLE_EXECUTABLE_FILE`) but no equivalent Linux guard. On `.deb`-installed Linux systems, `electron-updater` throws "APPIMAGE env is not defined" every 4 hours (matching `CHECK_INTERVAL_MS`).

## Problem

Every `.deb`-installed Linux user got persistent error noise in logs and confusing UI error notifications ("Unable to check for updates") every time the update interval fired.

## Root cause

electron-updater's Linux AppImage updater requires `process.env.APPIMAGE` to be set (it contains the path to the running AppImage). On `.deb` or other non-AppImage Linux distributions, this variable is absent, and electron-updater throws rather than gracefully no-oping.

## Lesson

When implementing auto-update for Electron apps, the guard pattern must cover all non-updatable package types:

- Windows portable: check `process.env.PORTABLE_EXECUTABLE_FILE`
- Linux non-AppImage: check `!process.env.APPIMAGE`
- Potentially also: snap packages, Flatpak, etc.

The Windows guard was added first, and the Linux case was missed because Linux initially targeted only AppImage. As `.deb` distribution was added, no one revisited the updater initialization path.

## Applies when

Any code path that calls electron-updater on Linux. Cross-platform guard logic should be written defensively for all distribution formats at the time the feature ships, not patched in later.
