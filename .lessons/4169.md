## Context

The dev preview panel spawns a terminal, runs `npm run dev` (or equivalent), and waits for a localhost URL to appear in the output via `UrlDetector`. Fast bundlers (Vite, Turbopack) emit the URL within ~300ms.

## Problem

On every first spawn, the dev server URL was missed by the detector, triggering the `STALE_START_RECOVERY_MS` (10s) timeout, a kill/restart cycle, and then successful detection on the second boot. Effective startup time: 15-20s instead of 2-3s.

## Root cause

A timing gap between PTY spawn and IPC data mirror readiness. The sequence:

1. `spawnSessionTerminal()` spawns PTY and calls `attachTerminal()`
2. 100ms `setTimeout` then submits the dev command
3. Dev server boots in ~300ms and emits URL
4. But `handleData()` (which feeds `UrlDetector`) is only wired up after the IPC data mirror is fully piped through

The URL output arrived in the window where the PTY was running but the data listener wasn't yet subscribed. `replayRecentOutput()` existed but only ran on re-entry (second attempt), not on first spawn.

## Lesson

When detecting output from a process that may emit critical data very early (within the first second), don't rely solely on a real-time data listener. Add a startup replay mechanism:

1. After spawn, start a polling loop (every 500ms for up to 5s) that calls `getRecentLines()` from the PTY buffer
2. Feed each poll result into the detection logic (`UrlDetector.scanOutput()`)
3. Stop polling once detected or timeout reached (the stale-start recovery still acts as the last-resort fallback)

This tolerates any listener attachment timing gap without changing the architecture of the data pipeline.

## Applies when

Any feature that needs to detect content in terminal output within the first few seconds of a process starting — dev server URL detection, agent auth prompts, compilation success messages. The pattern is: real-time listener + startup replay polling as belt-and-suspenders.
