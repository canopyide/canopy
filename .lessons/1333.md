## Context

GitHubStatsCache used a helper `getUserDataPath()` whose fallback, if both `CANOPY_USER_DATA` and `app.getPath("userData")` were unavailable, returned `process.cwd()`. Under some startup paths this fallback hit, and `github-stats-cache.json` was written into whichever directory the user had opened Canopy from — including project repos, polluting their working tree.

## Problem

`process.cwd()` is attacker-ish territory for a desktop app. It can be anything — the user's home dir, a git repo, `/`, a mounted volume. Writing cache or config there will corrupt unrelated projects, leak information, or surprise users into committing the file. It also lacks any guarantee of writability.

## Root cause

The fallback was written as "safe" with no thought for when it would fire. In practice, module-level initialization, tests, and pre-`app.ready` code can all reach the cache path before Electron has set userData, and none of those contexts have a meaningful cwd.

## Lesson

Never use `process.cwd()` as a fallback for userData, config, cache, or log paths. The correct fallback chain is: (1) explicit env override like `CANOPY_USER_DATA`, validated as absolute; (2) Electron's `app.getPath("userData")` when `app.isReady()`; (3) platform-specific defaults built from `os.homedir()` — `~/Library/Application Support/Canopy` on macOS, `%APPDATA%\Canopy` on Windows, `$XDG_CONFIG_HOME/Canopy || ~/.config/Canopy` on Linux. Validate that every env var is an absolute path before trusting it. If none are available, throw — do not guess.

## Applies when

Adding a new cache/config/log file in main process services, writing helper utilities that need a persistent directory, or reviewing PRs that add a new `fs.writeFileSync` with a path computed from anything that isn't explicitly `app.getPath(...)`.
