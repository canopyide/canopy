## Context

Lifecycle commands (postCreate, preDelete, etc.) were spawned with `detached: true` and killed on timeout via `process.kill(-pid, "SIGTERM")`. This worked on macOS/Linux but silently broke every path on Windows.

## Problem

On Windows, lifecycle timeouts threw `ESRCH`, child trees were orphaned, and `detached: true` popped up a stray console window. The command's spawn env also pointed PATH at `/usr/bin:/usr/local/bin` as a fallback — meaningless on Windows — so any subprocess lookup could fail when inheriting an empty PATH.

## Root cause

Three separate Windows-isms combined: (1) there are no POSIX process groups, so `process.kill(-pid, ...)` always fails; (2) `detached: true` on Windows means "new console window", not "new process group"; (3) Windows tooling expects `SystemRoot` and `USERPROFILE` in the env, plus a Windows-shaped PATH (`C:\\Windows\\System32;...`).

## Lesson

Branch spawn options and kill strategy by platform at every step:

```ts
const isWin = process.platform === "win32";
const opts: SpawnOptions = { detached: !isWin, env: buildEnv(isWin) };
// Kill path:
if (isWin) {
  execSync(`taskkill /T /F /PID ${child.pid}`, { stdio: "ignore" });
} else {
  process.kill(-child.pid!, "SIGTERM");
  setTimeout(() => {
    try {
      process.kill(-child.pid!, "SIGKILL");
    } catch {}
  }, 5000);
}
```

Build a Windows-appropriate env fallback (`SystemRoot`, `USERPROFILE`, Windows PATH) whenever you synthesise env for a child — never hardcode Unix paths.

## Applies when

Any main-process code that spawns user-configured commands with a timeout: lifecycle hooks, dev-server runners, `simple-git` wrappers, E2E helpers. If you write `detached: true` or `process.kill(-pid, ...)`, you owe a Windows branch.
