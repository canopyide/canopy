## Context

node-pty has documented file descriptor leaks on macOS — master FDs for `/dev/ptmx` aren't always closed when child processes exit or are force-killed. macOS defaults `kern.tty.ptmx_max` to ~511. With 40+ terminal sessions created and destroyed over a long IDE session, leaked FDs can exhaust the system-wide PTY limit, causing all PTY creation to fail across the _entire system_ (not just Canopy).

## Problem

When the macOS `/dev/ptmx` limit is exhausted: new node-pty spawns fail with `posix_spawnp failed` / `ENOENT` or `EMFILE`; macOS Terminal.app also fails with `[forkpty: Device not configured]`. The failure is system-wide and requires manual process cleanup.

The `ResourceGovernor` monitored heap memory but had no FD awareness.

## Root cause

Calling `ptyProcess.kill()` from JavaScript often leaves pending event loop references that keep the native FD open. This is a documented issue affecting Gemini CLI, GitHub Copilot CLI, and others — not specific to Canopy.

## Lesson

The safe detection strategy is read `/dev/fd` (macOS) or `/proc/self/fd` (Linux) via `fs.readdirSync()` — lightweight and requires no shell-out. Expected FD count is `(active_terminals * 2) + safety_margin + baseline`. Deviation indicates leaks.

**Critical constraint**: Do NOT auto-close leaked FDs. node-pty's native bindings may still hold references — force-closing causes crashes. Detect and warn only; cleanup happens when the PTY host process restarts.

Windows is a no-op (ConPTY uses different handles).

Cache `kern.tty.ptmx_max` at startup — it's a syscall (`sysctl kern.tty.ptmx_max`) and doesn't change at runtime.

Integrate into the existing `ResourceGovernor` 2-second loop rather than a new monitoring cycle.

## Applies when

Adding PTY lifecycle management; diagnosing `EMFILE` / `ENOENT` PTY spawn failures; reviewing `TerminalProcess.kill()` sequences; any long-running IDE session that creates and destroys many PTYs.
