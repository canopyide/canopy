## Context

Electron apps use `app.requestSingleInstanceLock()` to prevent multiple instances from running simultaneously. The second instance should call `app.quit()` and exit without doing anything else.

## Problem

Launching a second Canopy instance (e.g., double-clicking the AppImage on Linux while already running) wiped all log files of the already-running instance. The second instance exited immediately, but not before clearing the logs.

## Root cause

`initializeLogger()` called `clearSessionLogs()` — truncating all `.log` files to zero bytes — before `app.requestSingleInstanceLock()` was called. The startup order was:

```
initializeLogger()  // clears logs ← too early
...
const gotTheLock = app.requestSingleInstanceLock()
if (!gotTheLock) { app.quit() }  // exits, but logs already gone
```

## Lesson

**Any initialization that has side effects on shared resources must come after the single-instance lock check.** The second instance should quit before touching anything the first instance owns: log files, socket files, lock files, temp directories.

The correct order:

```ts
const gotTheLock = app.requestSingleInstanceLock();
if (!gotTheLock) {
  app.quit();
} else {
  initializeLogger(app.getPath("userData")); // safe: only first instance reaches here
  // ... rest of startup
}
```

This pattern extends to any startup side effect: clearing caches, writing PID files, acquiring ports. All of these must be guarded by the lock check.

## Applies when

- Electron main process startup sequence
- Any initialization that modifies files also accessible by other instances
- Log rotation, cache clearing, or session initialization in Electron apps
