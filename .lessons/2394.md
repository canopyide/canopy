## Context

Several main-process paths treated absolute-path detection and the home directory using POSIX assumptions.

## Problem

On Windows, multiple flows broke: git util returned wrong results for `C:\...` and UNC paths; default terminal `cwd` fell back to an undefined home; and Git-for-Windows binaries could not be found in fresh installations.

## Root cause

Three common landmines. (1) `result.startsWith("/")` is not a valid absolute-path check on Windows — drive-letter (`C:\...`) and UNC (`\\server\share`) paths return false. The correct API is `path.isAbsolute()`. (2) `process.env.HOME` does not exist on Windows; the equivalent is `USERPROFILE`. Using `os.homedir()` is the only cross-platform safe way. (3) Git-for-Windows installs to `%LOCALAPPDATA%\Programs\Git\cmd` or `C:\Program Files\Git\cmd` but does not add itself to the system PATH by default. Even `fixPath()` (which pulls the login shell's PATH) does not help if the user never opened a shell.

## Lesson

On Windows: use `path.isAbsolute()` for absolute checks; use `os.homedir()` never `process.env.HOME`; after calling `fixPath()`, prepend known Git-for-Windows install directories to `process.env.PATH` if they exist on disk (case-insensitive dedupe so `PATH` entries differing only in case are not both added).

## Applies when

Writing any path logic that runs on Windows, spawning child processes that depend on Git being on PATH, or reading user-home-based config files in main.ts or pty-host.
