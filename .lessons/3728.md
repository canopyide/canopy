## Context

`ActivityMonitor.resetDebounceTimer()` used a recursive setTimeout pattern: the callback checked conditions (active children detected, working pattern visible) and, if conditions held, called `resetDebounceTimer()` again to keep monitoring. This created a self-perpetuating timer chain.

## Problem

If the terminal process died without `ActivityMonitor.dispose()` being called (e.g., PTY host crash, abnormal terminal exit), the recursive timer chain continued running indefinitely, polling a dead terminal. No check existed to detect the disposed or dead state.

## Root Cause

The timer callback performed state checks (`lastPatternResult.isWorking`, `hasActiveChildrenSafe()`) but had no `disposed` guard. The existing "working silence" safety timeout would transition state to idle, but after transitioning, the next `resetDebounceTimer()` call could re-enter working state if stale pattern data remained — the chain never stopped.

## Lesson

Any class with a recursive or self-rescheduling timer must include a `disposed` flag check at the very top of the timer callback:

```typescript
private disposed = false;

private resetDebounceTimer() {
  this.debounceTimer = setTimeout(() => {
    if (this.disposed) return;  // guard must be FIRST
    // ... rest of logic
  }, DEBOUNCE_MS);
}

dispose() {
  this.disposed = true;
  clearTimeout(this.debounceTimer);  // clear any in-flight timer
  // ... rest of cleanup
}
```

The `clearTimeout` in `dispose()` handles timers that fired before the flag was set. The flag handles callbacks already executing when `dispose()` is called (re-entry guard).

This pattern applies to any recursive/self-rescheduling timer in a service that has a `dispose()` lifecycle, including polling intervals, debounce chains, and watchdog timers.

## Applies when

- Any service using `setTimeout` in a recursive or self-scheduling pattern
- Any `dispose()` method that doesn't also clear in-flight timers
- Debugging CPU usage from services that should have shut down
