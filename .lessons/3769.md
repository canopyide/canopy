## Context

Electron's IPC uses structured clone to serialize values across the process boundary. Custom error properties (error codes, recovery hints, correlation IDs) are stripped during this serialization — a thrown `Error` with `code: 'FILE_LOCKED'` arrives in the renderer as a generic `Error` without the `code` property.

## Problem

46 IPC handler files used 4 inconsistent error-handling patterns: direct throws, result objects, Zod validation, and silent catches. The renderer couldn't reliably distinguish error types, and custom error fields were lost in transit.

## Root cause

Electron's structured clone algorithm does not preserve non-standard properties on Error objects. Even though `throw new MyError({ code: 'X' })` works within a single process, crossing the IPC boundary via `ipcRenderer.invoke` strips everything except `message` and `stack`.

## Lesson

To preserve custom error properties across Electron's IPC boundary, use an `IpcEnvelope` discriminated union pattern: wrap responses as `{ ok: true, data }` or `{ ok: false, error: serializedError }` and unwrap transparently in the preload bridge. The preload's `_unwrappingInvoke` detects envelope responses, unwraps success data, or throws properly deserialized Error instances with all custom fields restored.

`serializeError` must strip non-serializable fields (functions, circular refs) before sending. In production builds, sensitive fields should be redacted by `sanitizeField`. This pattern also catches the inconsistency where some handlers throw and some return result objects — the envelope normalizes the contract.

## Applies when

- Any IPC call that needs to distinguish error types in the renderer (e.g., FILE_LOCKED vs PERMISSION_DENIED vs TIMEOUT)
- Adding new IPC handlers where errors need structured information beyond message strings
- Debugging why renderer error handling doesn't see expected error properties
