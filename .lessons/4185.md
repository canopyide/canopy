## Context

The hybrid input bar (CodeMirror editor embedded in the terminal pane) used hardcoded CSS custom property references for its colors. When a user selected a standalone terminal color scheme (Solarized, Dracula, etc.) via `useTerminalColorSchemeStore`, the input bar retained the app-level theme colors and clashed visually.

## Problem

Terminal-specific color schemes (which set an `ITheme` on xterm.js) didn't propagate to the CodeMirror input bar. The input bar looked wrong whenever a terminal-specific scheme diverged from the app theme.

## Root cause

CodeMirror themes are configured at extension build time, not CSS variables at render time. The original implementation used CSS custom properties which reflect the app theme, but xterm's `ITheme` values are JavaScript objects passed to xterm — they don't automatically become CSS variables.

## Lesson

When embedding a CodeMirror editor adjacent to an xterm terminal, derive the CM theme from `ITheme` programmatically:

1. Build a `buildInputBarTheme(theme: ITheme)` function that maps `ITheme` fields to CodeMirror `EditorView.theme()` styles
2. Store the theme in a CodeMirror `Compartment` so it can be swapped at runtime
3. Subscribe to `useTerminalColorSchemeStore` in the React component and call `view.dispatch({ effects: compartment.reconfigure(buildInputBarTheme(newTheme)) })` on change

CodeMirror compartments are the correct mechanism for hot-swapping any configuration (theme, keymap, language) without recreating the editor.

## Applies when

Any CodeMirror editor that lives inside or adjacent to a themed xterm terminal. Also applies to other embedded editors (Monaco, etc.) where the host color scheme is dynamic.
