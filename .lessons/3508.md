## Context

The main `BrowserWindow` was created with `backgroundThrottling: false` to keep the renderer running at full speed when Canopy was hidden. A `useGlobalSecondTicker` hook fired every second regardless of window visibility. `XtermAdapter` could also receive resize events while the window was invisible, producing zero-size fit calls.

## Problem

Three sources of unnecessary CPU usage when Canopy was backgrounded:

1. `backgroundThrottling: false` prevented Chromium from throttling JS timers and RAF loops
2. The 1Hz global ticker kept polling for state updates with no user visible
3. xterm fit calls on hidden windows could produce zero-dimension layouts

## Root cause

`backgroundThrottling: false` is a blunt override that disables Chromium's built-in power-saving for the entire renderer. It was added to prevent timer drift but is far too aggressive — PTY I/O routes through a UtilityProcess and is unaffected by renderer throttling, so the override provided no benefit.

## Lesson

- **Remove `backgroundThrottling: false`** unless there is a concrete, measured reason to keep it. PTY output is handled in a UtilityProcess; renderer throttling only affects JS timers and RAF in the renderer.
- **Gate polling hooks on Page Visibility API**: use `document.visibilitychange` + `document.hidden` to pause/resume any periodic work that serves no purpose while the window is hidden.
- **Guard `fit()` calls** in `XtermAdapter` (and any layout-triggering code) with a `document.hidden || container.offsetWidth === 0` check to prevent layout thrash on resize events that fire before the window is visible.

## Applies when

Any Electron app with periodic renderer-side polling, timers, or layout recalculations. Especially relevant for xterm.js fit addon calls, which should always validate container dimensions before executing.
