## Context

Terminal scrollback is persisted to a `.restore` file via a debounced write. Hibernation (and app-quit) kill idle PTY processes to reclaim memory, then reconstruct them on demand using the last snapshot.

## Problem

Hibernation silently discarded scrollback. Users who returned to a hibernated terminal saw an empty buffer, even though the persistence system "worked" and the `.restore` file path was known.

## Root cause

Two bugs compounded. First, `kill()` cancelled the pending debounce timer (correctly freeing resources), so the most recent output never reached disk. Second, `persistSessionSnapshot()` was guarded by `wasKilled === false` — so even manually calling it after kill was a no-op. The `wasKilled` guard was originally intended to prevent writes after the process exited naturally, but it also blocked the pre-kill flush path. Net result: the last in-memory state was always lost to the debounce/guard interaction.

## Lesson

For any debounced persistence tied to a process lifecycle, add a synchronous `flushBeforeKill()` path that bypasses any `isDead`/`wasKilled` guard and runs BEFORE the kill. Call it once per live resource from the orchestrator (hibernation service, before-quit handler) before invoking the kill loop:

```ts
// In hibernation/quit path:
for (const term of project.terminals) {
  term.flushSessionSnapshotBeforeKill(); // synchronous, bypasses wasKilled
}
await gracefulKillByProject(project.id);
```

Guards like `wasKilled` should gate _future_ writes, not the explicit flush-on-shutdown path. Test the sequence explicitly: flush writes snapshot, then kill sets the guard, then post-kill `persistSessionSnapshot` is blocked.

## Applies when

Any debounced write tied to a resource that can be killed: terminal scrollback, unsaved editor drafts, pending telemetry events, in-flight log batches. Also applies to the `before-quit` and hibernation paths of Electron apps — both are the last chance to get state to disk.
