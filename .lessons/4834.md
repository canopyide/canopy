## Context

The EventInspector is a diagnostics tool that records every internal event in the app. During active agent work or file-watching bursts, the system can generate hundreds of events per second.

## Problem

Every event record was broadcast individually to all subscribed WebContents via IPC. Each send triggers:

- A structured-clone serialization call
- An IPC message dispatch
- A renderer-side deserialization

At hundreds of events/second this created significant CPU overhead — each event crossing the process boundary independently.

## Root cause

The initial implementation had no batching; it was written for low-frequency events and the design didn't account for burst scenarios during agent activity.

## Lesson

**IPC batching for diagnostic/observability channels**: when a channel can emit high-frequency events but the consumer doesn't need sub-50ms granularity (logging, diagnostics, analytics, non-real-time UI), batch events into a buffer and flush on a timer:

- Use per-subscriber buffers + timers so one slow subscriber can't affect others.
- 50ms flush window is imperceptible for diagnostic tools but reduces IPC volume by 90%+ during bursts.
- Preserve the existing lazy-subscription pattern — only create flush timers when a subscriber is actually connected.
- Change the channel payload from a single event to an event array (`EVENT_INSPECTOR_EVENT_BATCH`), update the renderer handler to iterate the array.
- Clear the timer on subscriber disconnect to avoid memory leaks.

The existing lazy subscription approach (EventBuffer subscription only created when a WebContents subscribes) should be preserved alongside the batching.

## Applies when

Any diagnostic, logging, or observability IPC channel that can emit bursts of individual events. The 50ms window is appropriate for non-real-time consumers; reduce it (e.g., 16ms) only if the consumer drives UI that must feel responsive.
