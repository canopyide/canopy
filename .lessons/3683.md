## Context

Several Zustand selectors used `.filter()` inside `useShallow()` wrappers, creating new array references on every store change. This caused components consuming these selectors to re-render on unrelated state changes (focus changes, MRU updates, command queue changes).

## Problem

`useShallow` prevents re-renders by doing shallow comparison of the selector result. But `.filter()` always returns a **new array reference**, so `useShallow` must compare every element on every state change — O(N) work per selector per store update, even when the filtered items didn't change. With 5 affected selectors and frequent store updates, this was measurable overhead.

## Root cause

The `useShallow` + `filter` pattern looks correct but defeats the purpose: `useShallow` can't help if the reference is always new. The fix requires `createSelector` from `reselect` so filtering only reruns when the input reference (`state.terminals`) actually changes.

## Lesson

**Anti-pattern** — `useShallow` + inline `.filter()` does NOT prevent re-renders:

```ts
// Runs O(N) work on every store update:
useTerminalStore(useShallow((s) => s.terminals.filter((t) => t.waiting)));
```

**Correct pattern** — `createSelector` for reference memoization:

```ts
const selectWaiting = createSelector([(s: State) => s.terminals], (terminals) =>
  terminals.filter((t) => t.waiting)
);
useTerminalStore(useShallow(selectWaiting));
```

For counting selectors that only return primitives, a loop-based approach is better than `.filter().length` (no intermediate arrays).

For ID lookups, use `Map.get(id)` over `Array.find()` — O(1) vs O(N).

Cross-store selector dependencies (e.g., `worktreeIds` from a separate store) need `useShallow` on the cross-store input to prevent metadata-change invalidation.

## Applies when

Any Zustand selector that calls `.filter()`, `.map()`, or `.find()` before passing to `useShallow`; performance profiling showing high re-render frequency in terminal-heavy views.
