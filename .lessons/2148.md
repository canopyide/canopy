## Context

Dev preview servers started in different worktrees were reporting identical ports/URLs. The user could not tell which worktree's changes they were actually seeing.

## Problem

PtyPool warms terminals ahead of time in the home directory (or whatever defaultCwd was configured with). When a spawn request arrives, the pool hands out a pre-warmed PTY and the caller issues a "cd <worktree>" afterwards. For cwd-sensitive services this races: the dev server could execute from the home directory, or two worktrees could simultaneously pull pooled PTYs that briefly share state, and the first "cd" wins.

## Root cause

Acquire-then-cd is not atomic. A service that MUST start in a specific cwd cannot safely use the pool because the pool's defining trade-off is that cwd is chosen at warm time, not acquire time. The dev-preview case made this visible because each worktree needs a distinct port and the symptom was impossible to mistake.

## Lesson

PtyPool is a latency optimization for shell-style terminals where an extra "cd" on acquire is harmless. Any terminal whose behaviour depends on the initial cwd — dev servers, any command that reads files relative to cwd before the first "cd" runs, anything that forks subprocesses before the cd — must bypass the pool and spawn fresh with the correct cwd from the start. Enforce this with a runtime assertion on the spawn path (e.g. assert kind !== "dev-preview" when acquiring from the pool) so it cannot regress silently. Also log panelId/ptyId/generation/cwd for anything worktree-scoped so future pool-reuse bugs are diagnosable.

## Applies when

Adding a new panel kind that spawns a PTY with side effects tied to cwd. Reviewing PtyPool changes. Debugging "why does this terminal act like it is in the wrong directory?" Debugging duplicate ports/URLs across worktrees.
