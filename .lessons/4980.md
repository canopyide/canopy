## Context

Dev-preview runs a user's app in an Electron webview with external navigations blocked. OAuth flows redirect to external identity providers (Keycloak, Auth0, Google, GitHub), which breaks because the callback can't return to the webview.

## Problem

External OAuth redirects would abandon the PKCE `code_verifier` stored in the webview's `sessionStorage`. The system browser completing auth had no way to hand the token back to the in-app webview.

## Root cause

Two structural incompatibilities: (1) the embedded webview context (with PKCE state) is isolated from the system browser that handles the callback; (2) using a popup `BrowserWindow` violates RFC 8252 §8.12 and Google blocks `Electron/` user-agent in embedded auth.

## Lesson

OAuth in Electron dev-preview requires an RFC 8252 loopback server pattern, not a popup BrowserWindow:

1. Capture `sessionStorage` (PKCE verifier/state) from the webview via CDP before redirecting
2. Spawn an ephemeral HTTP server on `127.0.0.1:0` (random port), rewrite `redirect_uri`
3. Open system browser via `shell.openExternal` — IdP sees a native client, not Electron
4. Loopback server captures the `code`, shuts down, restores `sessionStorage` into webview via CDP `Page.addScriptToEvaluateOnNewDocument`
5. Use CDP `Fetch` interceptor to rewrite `redirect_uri` in the token exchange POST body (RFC 6749 §4.1.3)

The IdP must allow `http://127.0.0.1:*` as a valid redirect_uri — supported by Keycloak, Auth0, Google, GitHub, Okta, Azure AD for public clients.

Also: `window.open` / `target="_blank"` OAuth URLs must be intercepted at the protocol level (`protocols.ts`) and routed through the same banner flow, not sent directly to `shell.openExternal`.

## Applies when

- Implementing any OAuth/OIDC flow in the dev-preview panel
- Using CDP Fetch interception in Electron webviews
- Handling `window.open` in a controlled WebContentsView context
