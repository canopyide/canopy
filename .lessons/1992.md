## Context

A streamed message that contained only tool calls (no text) never appeared in the UI while it was streaming. It would pop in all at once when the stream finalized, making the app look stuck.

## Problem

The streaming item had a stable key like `"__streaming__"`. Virtuoso (and any virtualizing list) uses item keys to decide whether a row needs to re-render. With a stable key it concluded the row hadn't changed and skipped the render, even though the underlying object (the tool-call array) had grown.

## Root cause

Virtualized lists intentionally avoid re-rendering "unchanged" items — that's their whole performance story. "Unchanged" is judged by key identity, not by deep equality. For an item whose content mutates over time (a streaming buffer, a progressive image, anything incrementally appended), a fixed key is actively wrong: it tells the virtualizer "cache me forever."

## Lesson

For items that mutate in place during streaming, the key MUST encode enough of the mutable state to force re-renders when it matters. Something like `` `__streaming__:${toolCalls.length}:${toolCalls.map(t=>t.status).join(",")}` ``. This is ugly but correct. Alternatively, use a counter that bumps on every chunk. Also use functional `setState(prev => ...)` inside finalizers so you do not read a stale ref captured at stream-start. On error, explicitly clean up the streaming item or the loading UI stays stuck.

## Applies when

Any use of Virtuoso, react-window, react-virtual, or any virtualizer with items that mutate in place (streaming output, progressive loading, live-updating cells). Reviewing lists that show "nothing" during streaming and "everything at once" at the end. Building a new chat/log/timeline UI.
