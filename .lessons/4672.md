## Context

After per-project workspace host processes (#4647) solved the backend contamination, a parallel problem remained: the shared React renderer (a single `BrowserWindow` webContents) had to reset all Zustand stores and re-hydrate state on every project switch. This required complex client-side guards (`resetStores`, `projectSwitchRendererCache`, store generation counters) and took 500–1500ms.

## Problem

A single renderer V8 context shared across all projects meant project switching required store resets, hydration round-trips, and fragile sequencing. Every guard eventually failed because shared state + async orchestration is inherently racy.

## Root cause

Same architectural anti-pattern as the workspace host singleton: sharing a V8 context across projects makes isolation a software concern instead of a hardware guarantee.

## Lesson

- **WebContentsView-per-project gives each project its own V8 heap** — no shared stores, no reset logic, no hydration race conditions. Switching projects becomes GPU composite (<16ms) rather than a JS store teardown + IPC round-trip.
- **`ProjectViewManager` manages view lifecycle**: create on first open, LRU eviction (2 cached views for memory efficiency), session partition per project for complete isolation.
- **Background view throttling matters**: creating a view immediately in the background on app start caused startup overhead. Throttle background pre-warming.
- **Session partitions per project**: each view gets its own Chromium session, preventing cookie/storage bleed.
- **`BrowserWindow.fromWebContents(event.sender)` fails for WebContentsView senders**: all 28 callsites needed replacing with a `webContentsRegistry.ts` that maps `webContents.id` → parent `BrowserWindow`. Add `getWindowForWebContents()` with a fallback lookup when building a multi-view architecture.
- **Net code change**: +1,382 / −3,153 across 54 files. The migration _deleted_ significantly more code than it added — all the guard layers, reset utilities, and store switch machinery.
- Store switch test suites (projectStore.switching, worktreeDataStore.projectSwitchRace, resetStores.atomicSwap) — 1,500+ lines — were deleted entirely because isolation by construction made them unnecessary.

## Applies when

Building an Electron app where multiple independent V8 contexts (projects, workspaces, user sessions) must be isolated without fragile application-level state management.
