## Context

Added a stable/nightly update channel setting. Nightly builds have higher version numbers (or different version schemes) than stable releases. When a user switches back from nightly to stable, the auto-updater would refuse to "downgrade" unless explicitly configured to allow it.

## Problem

Without `allowDowngrade: true`, `electron-updater` refuses to install a version that is lower than the currently installed version. A user on a nightly build (e.g., `1.5.0-nightly.20260409`) who switches to the stable channel would see no updates offered, or the updater would silently skip the stable release.

## Root cause

`electron-updater` defaults to blocking downgrades as a safety measure. This is correct behavior for a single channel but breaks the stable/nightly switching UX because from the updater's perspective, going from nightly → stable is a version downgrade.

## Lesson

When implementing multiple update channels:

1. Set `allowDowngrade: true` on non-stable channels (or whenever the channel changes):

   ```ts
   autoUpdater.allowDowngrade = channel !== "stable";
   ```

2. Set it when the channel is initialized AND when the channel is changed via `setChannel()` — the flag must be live at the time the update check runs.

3. The channel URL swap (nightly URL vs stable URL) alone is insufficient. `electron-updater` checks the new version number against the installed version number, regardless of which feed it came from.

4. Test the nightly→stable downgrade path explicitly — it's the path most likely to be silently broken.

## Applies when

- Any Electron app with multiple update channels (stable, beta, nightly, canary)
- Auto-updater configurations where version numbers between channels are not strictly monotonically increasing relative to the installed version
