## Context

When switching projects, the workspace host process must be replaced. A naive kill-then-spawn approach creates a gap window where neither host is running, and rapid switching (A → B → C) can leave orphaned processes.

## Problem

Naive kill-then-spawn:

1. Creates a gap where events may be dropped or arrive with no registered handler
2. Rapid project switching (A → B → C) could leave an intermediate host for B running after C's host started
3. `git` child processes spawned by the old host were not tracked, leaving `.git/index.lock` files if the host process died mid-operation

## Root cause

Process lifecycle without handoff semantics creates observable gaps and unbounded in-flight state.

## Lesson

- **Blue-green swap**: spawn new host → wait for ready signal → hand off connection → kill old host. The new host is ready before the old one dies, eliminating the gap window.
- **Generation counters for rapid switching**: each swap increments a counter. If a newer swap starts before the current one completes, the superseded swap's cleanup logic detects the mismatch and aborts, preventing stale processes from outliving the intended switch.
- **Track git child processes in the workspace host**: register child PIDs in `process.on('exit')` and `process.on('disconnect')` handlers. On exit, send `SIGKILL` to all tracked children. This prevents `.git/index.lock` files from orphaned `git` subprocesses when the host process dies unexpectedly.
- **Crash-during-swap fallback**: if the new host crashes before signaling ready, fall back to keeping the old host running rather than leaving the window hostless.

## Applies when

Replacing any long-lived process (workspace host, language server, build daemon) that must be hot-swapped with no observable gap, especially when rapid user actions can trigger multiple swaps in quick succession.
