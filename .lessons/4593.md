## Context

Canopy recipes can contain `secureEnvVars` — sensitive environment variables like API keys. PR 4593 added the ability to mirror recipes to `.canopy/recipes/{name}.json` in the project repo so they can be committed to version control and shared with the team.

## Problem

If recipes were written to disk verbatim, any `secureEnvVars` would end up in committed files. Developers who share their repo would inadvertently leak API keys and tokens to collaborators or public repositories.

## Root cause

The in-repo persistence path needed a sanitization step that the clipboard export path did not. The key insight is that `.canopy/recipes/*.json` files are tracked in git, so the standard for what is "safe to write" is stricter than for a local-only clipboard export.

## Lesson

- Implement `sanitizeRecipeForRepo()` that strips `secureEnvVars` before writing to any tracked file path. The in-memory store retains the full recipe; only the disk representation is sanitized.
- On import from file, merge sanitized file data with existing secure vars from the store if a recipe with the same ID already exists — this handles the "team member imports a shared recipe and supplies their own keys" workflow.
- Recipe file names must be safe for all filesystems. Use a `recipeFilename.ts` utility that converts recipe names to lowercase, replaces spaces with hyphens, strips non-alphanumeric chars, and caps length at ~50 chars.
- Document this security property in the `.canopy/recipes/` directory so developers understand the invariant.

## Applies when

- Any feature that writes user-configured data (settings, recipes, templates) to tracked git paths
- Designing import/export flows for data that may contain credentials
- Building "sync to repo" features for team-shared configuration
