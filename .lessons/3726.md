## Context

Multiple palette components (CommandPicker, ActionPalette, QuickSwitcher, PromptHistoryPalette) each maintained their own manual debounce timers (150–200ms) using `useRef`/`setTimeout` to delay filtering after user input.

## Problem

Manual debounce timers introduce fixed delays regardless of actual render load. They also create boilerplate in each palette hook and require cleanup logic to avoid stale closures. Across four components, this was ~25 lines of repetitive timer management.

## Root cause

Pre-React 19 idiom. React 19's `useDeferredValue` is the idiomatic replacement: it keeps the input instantly responsive while deferring the expensive filtering work to a lower-priority render pass.

## Lesson

In React 19, `useDeferredValue` replaces manual debounce for search/filter scenarios. The key insight: `useDeferredValue` adapts to actual render load — on a fast machine it's near-instant, on a loaded machine it naturally defers. Fixed ms timers don't adapt.

Pattern: centralize in the shared hook layer (`useSearchablePalette`), not per-component:

```ts
const deferredQuery = useDeferredValue(query);
const items = useMemo(() => filterItems(deferredQuery), [deferredQuery]);
```

The input field binds to `query` (instant), while filtering binds to `deferredQuery` (deferred). Tests that used `jest.advanceTimersByTime()` for debounce need to shift to React Testing Library's `act()` or `waitFor()`.

## Applies when

Any search/filter UI that currently uses `useRef` + `setTimeout` debounce in React 19; new palette or picker components; migrating existing debounce patterns.
