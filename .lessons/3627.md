## Context

Event loop lag monitoring (main) and long task monitoring (renderer via PerformanceObserver) were gated behind `CANOPY_PERF_CAPTURE=1`. They only ran when a developer explicitly enabled them, so jank in production was never observed or logged.

## Problem

Performance regressions in production were invisible. The monitoring infrastructure existed but only developers using the perf capture mode saw it.

## Lesson

Always-on performance monitoring requires two safeguards to avoid being more harmful than helpful:

1. **Startup suppression** (5 seconds): app startup is intentionally heavy — suppress warnings during the initial load to avoid flooding logs with false positives.
2. **Rate limiting** (1 warning per 10 seconds): sustained jank should produce one log line, not thousands. Rate limiting prevents log flooding during worst-case scenarios.

The `CANOPY_PERF_CAPTURE=1` detailed file output can remain opt-in; only the warn-on-threshold behavior needs to be always-on.

For the renderer, the PerformanceObserver long task threshold should be raised to 100ms for always-on mode (vs a lower threshold in capture mode).

The main process and renderer are separate monitoring domains: `startEventLoopLagMonitor()` in the main process uses a setInterval-based lag detector; `startLongTaskMonitor()` in the renderer uses `PerformanceObserver` with `"longtask"` entry type.

## Applies when

- Diagnosing unexplained jank or slow interactions that only happen in production
- Adding a new performance monitoring probe to either process
- Setting the always-on threshold for a new type of performance warning
