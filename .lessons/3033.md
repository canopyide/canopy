## Context

`WorktreeMonitor` polls each worktree and needs to detect when a worktree has been removed externally (by an agent, terminal, or script) so the dashboard card can disappear instead of lingering in an error state.

## Problem

When a user ran `git worktree remove`, git deregistered the worktree but left the directory in place. The monitor kept polling the still-present directory, got a "not a git repository" error, and stuck the card in a permanent error state instead of removing it.

## Root cause

`WorktreeRemovedError` only detected ENOENT-style ("directory gone") failures. The git-deregistered case produces a different error text (`not a git repository`) with the filesystem path still valid, so the removal codepath never triggered.

## Lesson

When detecting resource-removal errors from a shelled-out tool, enumerate all the distinct failure shapes the tool can produce — deletion via filesystem (ENOENT) and deletion via the tool's own command (`not a git repository`, `fatal: '<path>' is not a working tree`, etc.) are different error strings for the same logical condition. Also ensure the monitor actually calls `onRemoved()` and `stop()` on detection, not just emits an error state — otherwise the polling loop keeps firing. Add a regression test for every new error pattern you accept, and keep them in one pattern list so future additions are obvious.

## Applies when

Touching `WorktreeRemovedError` detection, `WorktreeMonitor`, or any monitor that classifies shell-tool errors as "resource gone" vs "transient failure".
