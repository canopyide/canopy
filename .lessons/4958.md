## Context

`useProjectSettingsForm` manages the project settings dialog form state, including auto-save on field changes. The hook uses `useEffect` to initialise the form when the dialog opens and to reset it when the dialog closes.

## Problem

Settings changes made in the project settings dialog were not persisting. Values would appear correct while the dialog was open but revert after closing and reopening.

## Root cause

React effect ordering bug: the reset effect was declared _after_ the init effect in the hook. When the dialog opened, React fired both effects — init ran first (set `projectIsInitialized: true`), then reset ran second (set `projectIsInitialized: false`), suppressing all subsequent auto-saves. The bug was introduced when two previously separate effects were split/reordered during a dialog consolidation refactor.

## Lesson

When a hook has multiple `useEffect` calls that share state variables as flags (like `projectIsInitialized`), their declaration order determines execution order on the same render cycle. Splitting init/reset into separate effects creates an ordering dependency that is invisible and fragile. Consolidate them into a single effect keyed on `[projectId, isOpen]` so init and reset are a single atomic operation with explicit conditional branches.

Also: scope Zustand subscriptions to only the fields the form actually reads. Subscribing to the entire store slice causes spurious re-renders from unrelated fields (`isLoading`, `error`, etc.) that can re-trigger effects.

## Applies when

- Writing form hooks with `useEffect` for initialisation and reset lifecycles
- Any hook where multiple effects share a boolean flag (`isInitialized`, `isReady`, etc.)
- Modifying `useProjectSettingsForm.ts` or similar dialog-backed form hooks
