## Context

Canopy validates IPC sender origin against `isTrustedRendererUrl` to prevent untrusted frames from invoking main-process handlers. The original implementation only wrapped `ipcMain.handle` (request/response) and missed `ipcMain.on` (fire-and-forget) channels.

## Problem

Extending sender validation to `ipcMain.on` naively breaks every caller that later calls `ipcMain.removeListener(channel, listener)` or `ipcMain.off(...)` — because the listener that was actually registered on the EventEmitter is the _wrapped_ function, not the original reference the caller stored.

## Root cause

EventEmitter's removal APIs match by reference equality. When you patch `ipcMain.on` to wrap every listener, the caller still holds the unwrapped reference; `removeListener(unwrapped)` silently no-ops and listeners accumulate as a memory leak or cause double-invocation bugs after module reload.

## Lesson

When wrapping EventEmitter-style registration, maintain a `Map<channel, Map<originalListener, wrappedListener>>`, patch `removeListener`/`off` to resolve the wrapped function via the map, alias `off` to the patched `removeListener`, and patch `removeAllListeners` to clear map entries for the channel. Guard the channel check with explicit `channel !== undefined` so empty-string channels are handled (not falsy-skipped). Fire-and-forget channels have no error return path — drop untrusted messages silently with a `console.warn`, not a throw.

## Applies when

Adding security wrappers around `ipcMain.on`, monkey-patching any EventEmitter method, or debugging IPC listeners that "won't unregister".
