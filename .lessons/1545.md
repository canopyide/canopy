## Context

`redactSensitiveData` walked log context objects to mask secret-shaped keys before writing to the log. Passing any object with a cycle (e.g. a React fiber node, an Electron event payload, a Zustand store reference) crashed the logger with RangeError: Maximum call stack exceeded. `safeStringify` already had cycle detection; `redactSensitiveData` did not.

## Problem

Two bugs stacked. First, no cycle detection meant circular objects recursed forever. Second, the recursive type check `typeof value === "object"` descended into `null` because `typeof null === "object"` in JavaScript, causing downstream property access to fail or loop.

## Root cause

Both bugs are classic JavaScript traps. `typeof null === "object"` is a spec quirk from JS 1.0 that will never be fixed. Any recursive object walker needs `value !== null && typeof value === "object"` as its guard and a WeakSet (not a Set — objects may not be hashable/comparable and WeakSet doesn't retain them) to track visited nodes.

## Lesson

Any function that recursively walks unknown objects needs both: (1) explicit `!== null` check before `typeof === "object"`, and (2) WeakSet cycle detection returning a sentinel like `"[Circular]"`. If you have multiple walkers (e.g. `safeStringify` and `redactSensitiveData`), they must apply the same sensitive-key rules and cycle handling, or data that takes one path bypasses redaction available on the other.

## Applies when

Writing or reviewing any recursive object traversal — serializers, redactors, diff tools, schema validators, deep-clone helpers. Log context in particular gets fed arbitrary shapes from every part of the app and is the most likely place a circular ref will land.
