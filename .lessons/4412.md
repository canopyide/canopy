## Context

Canopy forces `color-scheme: dark` on the document root (`:root`) to ensure the Electron chrome and all UI components render in dark mode when a dark theme is active. The browser panel uses an Electron `<webview>` element to display external web content.

## Problem

Chromium's built-in viewers (JSON viewer, error pages, `about:blank`) rendered dark backgrounds with hardcoded light-mode syntax colors — making JSON API responses visually unreadable on dark themes.

## Root cause

`color-scheme` is an inherited CSS property. Setting `color-scheme: dark` on `:root` causes the value to cascade into the `<webview>` element. Chromium's built-in page renderers (which are not affected by userland CSS) read `color-scheme` from the host document's context and apply it to native UI like the JSON syntax highlighter — where the colors are hardcoded for one mode only.

The fix is to break the inheritance chain at the webview boundary by explicitly setting `color-scheme: normal` on the `webview` selector, restoring OS-default behavior for the embedded content.

## Lesson

Any Electron `<webview>` or embedded iframe that displays external or built-in content must explicitly reset `color-scheme` to `normal` (or `light dark`) to prevent inheriting forced dark/light mode from the parent document. Add this to the global stylesheet:

```css
webview {
  color-scheme: normal;
}
```

This is a non-obvious gotcha because the symptom (dark-on-dark text) only appears in Chromium's built-in viewers, not in normal web pages that define their own colors.

## Applies when

Any Electron app that forces `color-scheme` on `:root` while also embedding a `<webview>` that displays external content or Chromium built-in viewers.
