## Context

Browser and dev-preview panels create their `<webview>` element immediately on mount, regardless of whether the panel is visible. Docked panels and panels in the background grid spawn full Chromium renderer processes even if the user never opens them.

## Problem

Each `<webview>` spawns a separate Chromium renderer process. Background or docked panels that the user has never visited would consume memory and CPU for page loading that is completely unnecessary.

## Root cause / Design decision

Solved with a `useHasBeenVisible` hook backed by `IntersectionObserver`. The hook tracks whether a sentinel `<div>` inside the panel container has ever entered the viewport. The webview element is only rendered (`createWebview = true`) once that first-visibility event fires. After first render, the webview is never re-hidden — the "lazy" gate is one-way: open once, stay open.

This does not interfere with the existing webview eviction system (`useWebviewEviction`) — lazy-load controls initial creation, eviction controls teardown under memory pressure.

## Lesson

Use `IntersectionObserver` + a `hasBeenVisible` boolean gate (not a reactive visibility toggle) to defer webview creation to first use. The gate must be one-way (once true, stays true) to avoid recreating the webview on scroll. This pattern is composable with separate eviction hooks — lazy creation and pressure-based destruction are orthogonal concerns.

IntersectionObserver cleanup in the hook is critical: call `observer.disconnect()` in the cleanup function to avoid memory leaks when panels are removed before ever becoming visible.

## Applies when

- Adding new panel types that embed a webview
- Implementing lazy initialization for any expensive resource tied to panel visibility
- Combining with `useWebviewEviction` for full webview lifecycle management
- Debugging unnecessary Chromium process spawning on app start
