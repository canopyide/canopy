## Context

Next.js 15 dev servers started in webpack mode (the default) produce blank or completely unstyled pages when loaded inside Canopy's Electron webview. The pages appear to load but render without CSS — or are entirely blank due to Next.js's FOUC prevention (`body{display:none}`) never being cleared.

## Problem

Next.js webpack mode uses `style-loader` to inject CSS via JavaScript at runtime. `style-loader` works by creating `<style>` tags via DOM manipulation. This mechanism fails silently inside Electron webviews — the CSS injection call returns without error but the styles are never applied, leaving the page visually broken.

## Root cause

webpack's `style-loader` uses `document.head.appendChild` to inject styles, which appears to succeed inside the webview but the styles don't take effect. This is a known incompatibility between webpack's runtime CSS injection and Electron's webview security model.

The fix: `--turbopack` switches Next.js to Turbopack, which uses a different (compatible) CSS delivery mechanism that works correctly inside Electron webviews.

## Lesson

When a Next.js dev server shows blank/unstyled pages in Canopy's webview but works fine in a regular browser:

- The cause is almost certainly webpack's `style-loader` CSS injection failure
- The fix is to run `next dev --turbopack` instead of `next dev`

The auto-injection is implemented at two layers:

1. **`DevPreviewSessionService`** — runtime normalization (`normalizeNextjsDevCommand`) as a safety net when the command is already running
2. **`findDevServerCandidate`** — upfront enhancement so the UI display shows the correct augmented command

Injection is idempotent (checks for existing `--turbopack` flag before adding) and handles all package managers (`npm run`, `pnpm`, `yarn`, `bun`) plus direct invocations (`next dev`).

## Applies when

- Any Next.js project with blank/unstyled integrated browser rendering
- Adding support for other frameworks that use webpack runtime CSS injection — may need similar treatment
- Writing dev server detection logic: always normalize commands at both detection time and runtime
