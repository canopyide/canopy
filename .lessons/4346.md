## Context

When the Electron renderer process crashed, the app hung silently with a blank window. Users had no path to recovery except force-quitting the entire application. In a multi-worktree IDE where each project has its own `WebContentsView`, renderer crashes are a meaningful risk.

## Problem

No crash handling was implemented in `createWindow.ts`. Electron fires `render-process-gone` on `WebContents` but nothing listened to it. The renderer becoming unresponsive also had no production handler.

## Root cause

Renderer crash recovery requires distinguishing between a one-off crash (recoverable via `webContents.reload()`) and a crash loop (repeated crashes within a short window indicating a persistent corruption). A simple auto-reload without loop detection would infinitely reload a broken state, consuming CPU and never presenting the user with a recovery option.

## Lesson

Implement crash recovery in `createWindow.ts` with two-tier logic:

1. **First crash:** Auto-reload `webContents.reload()` — covers most OOM/transient crashes.
2. **Crash loop (2 crashes within 60s):** Load a static recovery HTML page (`public/recovery.html`) with no React dependencies — React itself may be broken. The recovery page offers reload, workspace reset, and diagnostics via a minimal IPC channel (`recovery:resetAndReload`).

The recovery page must be static HTML/JS with zero dependencies on the main app bundle. Wire `unresponsive`/`responsive` events in production only (not dev) to avoid false positives during HMR. Always track crash timestamps to detect loops. `getMainWindow()` must be exported from `windowServices.ts` for the recovery IPC handler to reach it.

## Applies when

- Any `WebContentsView` or `BrowserWindow` that runs the main renderer
- Electron 37+ where utility process crashes are also fatal
- Implementing "reset workspace" recovery that clears persisted state before reload
