## Context

`twoPaneSplitStore` persists the user's split ratio when resizing two panels side-by-side, so the ratio is remembered when the same panels are reopened.

## Problem

Opening a different pair of panels on the same worktree would show the ratio from a previous, unrelated pair. E.g., resize A+B to 60/40, close both, open A+C and get 60/40 instead of the default 50/50.

## Root cause

`resolveEffectiveRatio()` keyed stored ratios by worktree ID alone. Any two panels on the same worktree shared a single stored ratio regardless of which panels they were. The stored value didn't record which panel IDs produced that ratio.

## Lesson

When persisting UI state that applies to a specific pair/combination of entities, key the store by the pair (e.g., a sorted tuple of both panel IDs) not by an outer container like the worktree. Return `undefined` — triggering the default — when the stored pair doesn't match the current pair. Never silently apply a stale value that doesn't belong to the current context.

## Applies when

Storing per-pair, per-combination, or per-slot UI state (split ratios, scroll positions, column widths) in any Zustand store.
