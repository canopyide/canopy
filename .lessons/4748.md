## Context

The project switcher had an `onMouseEnter` handler on each list item that called `prewarmHost()` — this spawned a UtilityProcess workspace host for the hovered project and prefetched its settings and runner detection results.

## Problem

Hovering over 5 project items spawned 5 UtilityProcess workspace hosts. Each host had a 3-minute cleanup grace period before it was torn down. This meant moving the mouse over the project list created long-lived background processes burning CPU and memory, even if the user never opened any of those projects.

## Root cause

The prefetch was added to speed up perceived project switching, but Canopy's multi-view architecture already makes the actual switch fast (< 1 second cold). The hover prefetch was solving a problem that didn't exist for most users, while creating a real memory/CPU problem for all users.

## Lesson

Before adding hover/focus prefetch logic:

1. Measure whether the cold path is actually slow enough to justify the overhead.
2. Account for the worst-case scenario: what if the user scans through many items without selecting any?
3. UtilityProcess instances are not cheap — each one is a separate Node.js process with its own startup cost and memory footprint.
4. Cleanup grace periods on background processes multiply the cost of "abandoned" prefetches.

The `prewarmHost` startup call in `App.tsx` was also removed — it was eagerly spinning up a workspace host on app launch regardless of whether the user would ever open the project switcher.

## Applies when

- Any hover or focus handler that triggers background work (IPC calls, process spawns, network requests)
- App-startup prewarming calls — validate that the warm path is measurably faster before paying the startup cost
