## Context

HibernationService has two hibernation paths: `checkAndHibernate()` (scheduled, time-based) and `hibernateUnderMemoryPressure()` (triggered by memory monitor). The memory-pressure path already guarded against killing active agents. The scheduled path did not.

## Problem

`checkAndHibernate()` killed terminals for projects that had been inactive by timestamp, even if those projects had actively running AI agents (Claude/Gemini mid-operation). It also had no awareness of in-progress git operations, meaning a hibernation could interrupt an active rebase, merge, or commit, orphaning sentinel files like `.git/index.lock`, `MERGE_HEAD`, `REBASE_HEAD`, etc.

The symptom: user switches projects, leaves an agent running in Project A, waits 24 hours — the scheduled hibernation kills the agent mid-operation. The memory-pressure path would have protected it; the scheduled path did not.

## Root Cause

`ACTIVE_AGENT_STATES` checking and the logic pattern existed in `hibernateUnderMemoryPressure()` but was never applied to `checkAndHibernate()`. The two paths diverged silently.

Git sentinel detection was absent from both paths entirely. `WorktreeMonitor` was aware of `index.lock` errors for its own retry logic, but this wasn't exposed to hibernation.

## Lesson

When a safety guard exists on one code path, audit ALL sibling paths for the same guard. Hibernation had two entry points; only one was protected. Add a `hasActiveGitOperation()` check that scans `.git/` and all linked worktree git dirs for sentinel files (`index.lock`, `MERGE_HEAD`, `REBASE_HEAD`, `CHERRY_PICK_HEAD`, `rebase-merge/`, `rebase-apply/`). Stale locks (older than inactivity threshold) should be ignored to prevent permanently blocking hibernation.

## Applies when

- Adding new hibernation triggers or code paths
- Any "safe shutdown" logic that must respect agent and git state
- Multi-worktree projects: sentinel files exist per-worktree; scan all linked worktree git dirs, not just the main one
