## Context

The PTY host previously held a single `rendererPort` — one MessagePort connection to the renderer. With multi-window support, each window needs its own independent terminal data stream from the shared PTY host process.

## Problem

A single port means all terminal output goes to one window. A second window has no channel to receive terminal data or send keystrokes.

## Root cause

The PTY host was designed for single-window and held the renderer port as a singleton field.

## Lesson

- **Replace singleton port with `Map<windowId, RendererConnection>`**: each window connects with its `windowId` in the initial `connect-port` message. The PTY host registers the connection under that ID and routes output to the correct window.
- **Per-window project tracking**: `set-active-project` and `project-switch` messages carry `windowId`. The host maintains a `windowId → activeProjectPath` map and filters terminal data emissions — a window only receives output for terminals belonging to its active project.
- **Clean up on `close`**: when a `MessagePort` fires `close` (window destroyed), remove the entry from the map and clean up per-connection backpressure state. Failure to do this leaks connection state indefinitely.
- **Per-connection backpressure**: extend `portQueue.ts` to manage a queue per `windowId` rather than a single global queue. Each window's backpressure state is independent.
- The `connect-port`, `set-active-project`, and `project-switch` message types all needed `windowId` added to their shapes in `shared/types/pty-host.ts`.

## Applies when

Adding multi-window support to any Electron utility process (PTY host, workspace host) that previously held a singleton connection to a single renderer.
