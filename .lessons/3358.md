## Context

`TerminalWebGLManager` uses a pool of live WebGL contexts (one per focused/visible terminal) with LRU eviction. The pool cap was set to 16, matching Chromium's documented limit.

## Problem

When terminals filled the pool, the first non-terminal WebGL consumer in the renderer — a Notes editor embed, a dev-preview overlay, a future chart, anything — would force Chromium to drop an existing context via `webglcontextlost` on a random terminal. The evicted terminal silently fell back to DOM rendering mid-session.

## Root cause

Chromium enforces a hard limit of 16 simultaneously active `WebGLRenderingContext`s per renderer process. The cap is per-process, not per-document, and it's shared with any other WebGL consumer in the same renderer. Hitting 16 triggers Chromium's own LRU eviction (not ours), which fires `webglcontextlost` on the oldest context regardless of whether it's terminal or not. `<webview>` partitions have their own budget, but anything in the main renderer document shares this pool.

## Lesson

Never claim Chromium's full WebGL budget in a single consumer. Reserve headroom (4 slots is a reasonable default) so other renderer-resident WebGL code can attach without triggering eviction of your pool. Document the Chromium limit inline so the next person doesn't bump the cap back up:

```ts
// Chromium enforces a hard limit of 16 WebGL contexts per renderer process.
// Leave 4 slots free for other consumers (Notes, dev-preview, future charts).
private static readonly MAX_CONTEXTS = 12;
```

## Applies when

Any renderer-process code that keeps long-lived WebGL contexts: terminal renderers, canvas chart libs, 3D previews. `<webview>` and `WebContentsView` each have their own renderer process with an independent budget, so the concern only applies within one document.
