## Context

A "paint-first" startup order created the window and loaded the renderer URL before all IPC handlers were registered. Some services initialized late (e.g., workspace host over IPC), so a subset of handlers landed after the renderer's first frame.

## Problem

Cold start produced ~25 "No handler registered for 'channel'" errors as the renderer fired IPC calls before their handlers existed. Some were benign retries; others silently failed and left the UI in a partial state.

## Root cause

The renderer attaches listeners and dispatches IPC eagerly on mount. Any handler not yet registered via `ipcMain.handle()` at that moment produces an error. "Paint-first" optimistically assumed the renderer would lazily probe — but React 19 / Zustand initialization is aggressive and fires dozens of calls in the first tick.

## Lesson

Register every IPC handler synchronously before `mainWindow.loadURL()` / `loadFile()`. If a handler depends on an async-initialized service, use a shared mutable `HandlerDependencies` object and have handlers read from it lazily at call time — don't defer registration. Make any "already active" handler (e.g., `project:reopen`) idempotent rather than throwing, since the renderer can legitimately re-invoke on hot reload.

The pattern:

1. Create sync-only services.
2. Register all handlers (late-bound via deps object).
3. Await async service init (workspace host, pty host).
4. Then load the renderer.

## Applies when

Adding a new IPC handler whose backing service initializes asynchronously. Seeing "No handler registered" in the console during cold start or hot reload. Reordering main-process bootstrap for any reason.
