## Context

React 19.2 introduced the Activity API for preserving component tree state across visibility transitions (e.g., tab switching). This PR evaluated whether Activity could be used for instant project UI switching in Canopy.

## Problem

Vestigial `key={activeProjectId}` props on `<ProjectScopeProvider>` and `<AppShell>` in `App.tsx` forced a full component tree remount on every project switch, adding unnecessary teardown/remount cost.

## Root cause / finding

The Activity API is designed for single-renderer apps where multiple "views" share the same React tree. Canopy's per-project `WebContentsView` architecture gives each project its own renderer process — there is no shared tree to preserve with Activity. The API is simply not applicable.

The `key={activeProjectId}` props were a leftover pattern from before per-project renderers existed, when project switching genuinely needed to reset all component state. They now cause unnecessary full remounts with no benefit.

## Lesson

`key` props on top-level component trees force complete React subtree remounts. In Electron apps where project isolation is achieved via separate `WebContentsView` renderer processes, `key`-based remounting at the app root is redundant and should be removed.

Before adopting a React feature for state preservation across tab/view switches, confirm whether the app is actually single-renderer or multi-renderer. The React 19 Activity API only applies to single-renderer scenarios.

## Applies when

- Evaluating React 19 Activity / `<Activity>` component for Electron apps
- Auditing `key` props on top-level components after architectural changes
- Migrating from single-renderer to multi-renderer project isolation
