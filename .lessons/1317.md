## Context

When a user types a prompt into an agent terminal, each keystroke is echoed back through the PTY. The ActivityMonitor's output-volume heuristic saw those echo bytes and flipped the agent into "working" state while the user was still composing their message.

## Problem

Purely output-based busy detection cannot distinguish "agent is doing work" from "terminal is echoing my keystrokes". Character echo is indistinguishable from tiny agent progress updates at the byte level. Any heuristic that looks only at output bytes will false-positive on typing.

## Root cause

`becomeBusyFromOutput()` had no cross-check against whether the child process was actually doing work. The fix correlates output with child-process CPU activity: if output arrives but the agent's CPU is idle, it's almost certainly echo, not work, so no state transition fires. Input-based transitions (Enter key) bypass the CPU check because those are unambiguous user intent.

## Lesson

Output-based busy detection needs a second signal before flipping state: either CPU activity on the child process, a pattern match against a known "working" indicator, or output volume well above plausible echo rates. A fail-open fallback is important — if no CPU validator is wired up (e.g. in tests), don't block transitions entirely, or you'll break unrelated flows. Keep input signals (keystrokes that correspond to submission, Enter) immediate so user-initiated transitions stay snappy.

## Applies when

Tuning ActivityMonitor heuristics, adding a new signal source for agent state detection, or debugging "agent shows working while I'm still typing" / "agent shows working during scroll or mouse events".
