## Context

`ActivityMonitor` uses a hierarchy of guards in `resetDebounceTimer()` to stay in the "working" state for legitimately long-running tasks. If a child process is still alive, if a working pattern is still visible, or if high output activity was recent, the timer reschedules indefinitely rather than transitioning to idle.

## Problem

A hung or stalled agent — alive in the process tree but producing no output — could hold the terminal in "working" state indefinitely. The user had no way to distinguish a legitimately busy agent from one that silently failed minutes ago.

## Root Cause

`hasActiveChildren()` returning true caused unconditional timer reschedule with no iteration limit and no elapsed-time check. `lastDataTimestamp` was tracked but never enforced as an upper bound when child processes were present. The guard designed for correctness (don't exit working while children are alive) had no escape hatch.

## Lesson

Any "keep trying" guard loop that checks liveness of a subprocess needs an absolute maximum duration based on data recency, not just process existence. A process can be alive but stalled. The fix adds `WORKING_SILENCE_TIMEOUT_MS` (3 minutes): if the elapsed time since `lastDataTimestamp` exceeds the threshold, transition to idle even if child processes appear alive.

Separately, PTY EOF (process exit) should immediately resolve stuck working state without waiting for any timeout. Add an `onProcessExit()` hook that forces idle on PTY close so the safety timeout is never the only resolution path.

## Applies when

- Any polling or timer-rescheduling loop conditioned on subprocess liveness
- Agent state that may be stuck with no clear output-based termination signal
- Building safeguards for long-running operations that could silently stall
