## Context

Crash recovery writes a `running.lock` marker on startup and clears it from a `before-quit` handler that depends on `mainWindow` still being alive.

## Problem

After applying an update, users saw a false crash-recovery dialog on next launch — the lock file was never cleared, so the app thought it had crashed.

## Root cause

On macOS, `autoUpdater.quitAndInstall()` closes all windows before firing `before-quit`. By the time the handler runs, `mainWindow` is already null and the cleanup early-returns, leaving the lock marker on disk.

## Lesson

Do not rely on `before-quit` for cleanup that must run before `autoUpdater.quitAndInstall()`. Call the cleanup synchronously right before `quitAndInstall()` in the quit-and-install IPC handler, wrapped in `try/catch` so the update proceeds even if cleanup throws. More generally: any teardown that depends on renderer/window state must run before the quit path that tears the window down, not after. This also applies to session flushes, store persistence, and any telemetry that needs a live `WebContents`.

## Applies when

Touching `AutoUpdaterService`, adding shutdown hooks that depend on `mainWindow`, or debugging "clean exit didn't actually clean up" problems on macOS after updates.
