## Context

`TerminalResizeController` was reading cell dimensions via `(terminal as any)._core._renderService.dimensions.css.cell` — a private internal API of xterm.js. This was needed because xterm 6.0 removed the canvas renderer addon and no stable public API for cell dimensions was available at the time.

## Problem

The `as any` cast bypasses TypeScript's type system entirely and is fragile across xterm.js upgrades. If the internal path changes, the failure is silent at compile time and only surfaces at runtime as `undefined` cell dimensions, causing misaligned PTY output.

## Root cause

No stable public API exists in xterm 6.0 for reading computed cell dimensions. The internal path `_core._renderService.dimensions.css.cell` is an implementation detail that may change without notice.

## Lesson

When accessing unavoidable private APIs in xterm.js (or any third-party lib), use a **structural interface** instead of `as any`:

```ts
interface XtermInternals {
  _core?: {
    _renderService?: {
      dimensions?: { css?: { cell?: { width: number; height: number } } };
    };
  };
}
```

Cast to the structural type instead of `any`. Add a comment citing the tracking issue for the public API. Also **validate the returned values** (reject zero, negative, NaN, Infinity) and fall back to safe defaults — defensive coding prevents misaligned PTY output if the path returns unexpected data.

## Applies when

Any code that needs to read cell dimensions or other internal xterm.js state. The structural type approach gives TypeScript-level documentation of the shape while still being honest that this is private API access.
