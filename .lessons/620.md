## Context

Context-copy generated a file named after the current branch (`canopy-feature/issue-596-context.md`). Git branch names legitimately contain forward slashes (`feature/`, `bugfix/`, `release/`), plus `@`, `#`, `.`, and other characters that git allows but filesystems don't.

## Problem

`feature/issue-596` in a filename caused `fs.writeFile` to treat `feature` as a directory component and throw ENOENT when that directory didn't exist. Simpler cases like `bug#123-fix` or `hotfix@v2.0` survived on macOS but failed on Windows or when the filename was later passed through tools that couldn't handle the character.

## Root cause

Branch names are not filesystem-safe strings. Git only prohibits a short list of characters, but that list is a strict subset of what every filesystem allows. Treating a branch name as a filename component is a cross-platform landmine — it may work on your dev machine and fail in CI or on a user's machine.

## Lesson

Any time a branch name (or any user-derived git identifier) is going to be embedded in a filename or path component, run it through a strict allowlist sanitizer: replace everything outside `[a-zA-Z0-9_-]` with `-`, collapse consecutive dashes, trim leading/trailing dashes, cap length (100 chars is a safe default — Windows MAX_PATH pressure), and fall back to a fixed token like `head` when the result is empty. Don't try to preserve slashes as subdirectories "because it looks nicer" — that explodes the problem space.

## Applies when

Generating any file whose name contains a branch, tag, commit message slug, or PR title: context copy, diff exports, snapshot names, log files, worktree directories. If the user can type it and git will accept it, assume the filesystem won't.
