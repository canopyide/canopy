## Context

The embedded side-panel browser feature was called "Sidecar" — a name that conflicts with Apple's Sidecar feature (iPad as external display). It was renamed to "Portal" across 87 files (~1179 additions, ~1195 deletions).

## Why this rename required a single atomic PR

The Canopy codebase has tightly coupled cross-layer dependencies: action IDs in `shared/types/actions.ts`, IPC channels in `electron/ipc/channels.ts`, type imports, store references, preload bridge namespaces, and component imports all form a connected graph. A partial rename produces a compile error because TypeScript types and IPC channel string constants reference each other across process boundaries. The entire rename had to land in one PR or the codebase would be broken at every intermediate state.

## Persistence key migration

The Zustand store persistence key changed from `"sidecar"` to `"portal"`. Any existing persisted Portal tab state under the old key is silently orphaned — users lose their open tabs on first launch after the rename. This was accepted as a trade-off since tabs are session-ephemeral. **Lesson**: when renaming a Zustand persisted store, document whether to migrate existing state (via `migrate()`) or accept data loss. For ephemeral UI state, data loss is usually fine; for settings or persistent user data, write a migration.

## Electron session partition migration

`PortalManager` uses `persist:portal` as the WebContentsView session partition (was `persist:sidecar`). The old partition's cookies, cache, and storage are orphaned but not deleted — they remain as dead data in the Electron userData directory. If the old partition held auth tokens for web apps, users may need to re-authenticate. For security-sensitive state, consider explicit partition cleanup rather than silent orphaning.

## Lesson

Large mechanical renames in Electron apps must be atomic because IPC channel strings, TypeScript types, Zustand store keys, Electron partition names, and CSS custom properties are all part of a cross-layer contract. Plan for:

1. Zustand persistence key migration (migrate vs. accept data loss)
2. Electron partition orphaning (document for users, optionally clean up old partitions)
3. DOM element IDs, data-testid attributes, and E2E selectors that reference the old name
4. CLAUDE.md / docs that describe the IPC namespace — these need updating for future agents to work correctly

## Applies when

- Renaming any cross-cutting feature that spans shared types, IPC channels, stores, and components
- Any Zustand store with `persist` middleware being renamed
- Any `WebContentsView` with a named Electron session partition
