## Context

Canopy runs `git status` on all worktrees every 2-10s via a polling queue. To prevent git hooks and credential helpers from executing arbitrary code, a `HARDENED_GIT_CONFIG` array overrides dangerous git settings for all spawned git commands.

## Problem

`HARDENED_GIT_CONFIG` included `core.fsmonitor=false`, which disabled Git's built-in FSMonitor daemon (introduced in Git 2.36). This was added as a precautionary measure alongside the genuine security overrides (no hooks, no credentials, no SSH, no proxy). The result was every `git status` performed a full working tree scan regardless of what had changed. At 40 worktrees polling every 2-10s, this created significant unnecessary filesystem I/O and compounded queue contention.

Additionally, `core.untrackedCache` was not enabled, meaning untracked file detection (typically the slowest part of `git status`) was never cached between calls.

## Root cause

`core.fsmonitor` is not a security feature — it runs a git-controlled daemon that uses OS-level filesystem events (FSEvents on macOS, inotify on Linux). It does not execute user hooks or external commands. It was mistakenly grouped with the actual security restrictions.

## Lesson

- **`core.fsmonitor` is safe to enable in hardened contexts.** It is a git-internal performance optimisation, not a hook execution point. Disabling it is overly conservative.
- **`core.untrackedCache=true`** caches untracked file lists between status calls — enable it alongside fsmonitor for maximum polling performance.
- When building a hardened git config, document _why_ each override exists. Performance features bundled with security features will eventually be disabled unnecessarily.
- Both fsmonitor and untrackedCache gracefully no-op on older Git versions or environments where they're not supported (CI, containers) — zero regression risk.

## Applies when

Any codebase that runs frequent `git status` operations across many repositories or worktrees. The pair `core.fsmonitor + core.untrackedCache` can reduce status time from hundreds of milliseconds to single-digit milliseconds on a warm cache.
