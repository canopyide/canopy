## Context

The panel grid renders many terminal instances simultaneously. During heavy agent activity, each terminal streams output continuously, triggering frequent style/layout recalculations that could cascade across the entire grid.

## Problem

Heavy terminal output caused elevated "Recalculate Style" and "Layout" times in the browser DevTools performance panel. Style changes in one terminal panel were causing the browser to recalculate styles and reflow layout for the entire grid.

## Root cause

Without containment hints, the browser doesn't know that a panel's internal layout changes cannot affect sibling panels. It must conservatively re-run style/layout calculations for the whole grid on any change inside any panel.

## Lesson

Apply CSS `contain` to panel wrapper elements to signal to the browser that each cell's layout and style are independent:

- `contain: layout style` on grid cell wrappers — isolates layout and style recalculations without clipping. Safe for tooltips, context menus, and overlays that need to overflow.
- `contain: style` only (not `layout`) on wrappers with absolutely positioned children that must overflow the wrapper bounds — `layout` containment would clip them.
- Do NOT use `contain: paint` or `contain: strict` if children need to overflow (dropdowns, tooltips, context menus).

This is especially impactful in terminal-heavy UIs where dozens of xterm.js instances update simultaneously. Each terminal is a contained subtree, so style invalidations from xterm's internal DOM don't propagate outward.

## Applies when

- Grid layouts with many independently-updating children (terminal panels, log viewers)
- Any component that updates frequently and is visually isolated from siblings
- Drag-and-drop grid containers where individual cells animate independently
