## Context

`EditorService.ts` constructs macOS/Linux-style paths (e.g., `/Applications/WebStorm.app/Contents/MacOS/webstorm`) using `path.join()`. Tests mock `process.platform` as `'darwin'` to test macOS code paths.

## Problem

On Windows, `path.join('/Applications', 'WebStorm.app', 'Contents/MacOS/webstorm')` returns a path with backslashes, not forward slashes. Tests that mock `process.platform = 'darwin'` but do not mock `path.join` fail on Windows with assertion errors like `expected '\\Applications\\WebStorm.app\\...' to be '/Applications/WebStorm.app/...'`. This caused 9 test failures on Windows. Since CI is Ubuntu-only for PRs, the regression was invisible in automated CI.

## Root cause

`path.join()` always uses the host OS path separator regardless of what `process.platform` is mocked to. Mocking `process.platform` does not change `path`'s behavior — the OS's Node.js build determines the separator at compile time.

## Lesson

When constructing paths for a specific target OS (not necessarily the host OS), use the explicit platform path module:

- `path.posix.join()` for macOS/Linux paths (always uses `/`)
- `path.win32.join()` for Windows paths (always uses `\`)
- Select based on the target platform, not `path.join()` which uses the host

This is especially important in services that contain cross-platform path logic (editor discovery, shell paths, etc.) that must be testable on all developer machines regardless of their OS.

## Applies when

- Writing tests that mock `process.platform` and assert on file paths
- Implementing editor or tool discovery with hardcoded platform-specific base paths
- Debugging test failures that only reproduce on Windows
- Any service that constructs paths for an OS different from the current host
