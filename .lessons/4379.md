## Context

When using voice-to-text tools (specifically Wispr Flow) to dictate text into a terminal, pressing Enter either did nothing (swallowed) or arrived at the PTY before the composed text, resulting in a blank submission followed by the text appearing on the next line.

## Problem

xterm 6.0's custom `onKey` handler intercepted Enter keypresses without checking `event.isComposing`. During and immediately after IME composition (including voice dictation, which uses the composition API), the Enter keypress should be handled by xterm's built-in `CompositionHelper` to finalize and submit the composed text — not by the custom handler.

## Root cause

The Web Composition API sets `event.isComposing = true` on keyboard events that occur while an IME or voice composition session is active. Custom `onKey` handlers in xterm that process Enter without this guard will either consume the Enter before composition finalizes (swallowing it) or deliver it to the PTY before the composed text arrives. The fix: check `event.domEvent.isComposing` in the custom handler and return early, yielding to xterm's `CompositionHelper`.

## Lesson

Any custom `onKey` handler in xterm that intercepts Enter (or Shift+Enter) must guard with `if (event.domEvent.isComposing) return`. Track composition state via `compositionstart`/`compositionend` listeners on the terminal element as a backup for rapid cycles where `isComposing` might have already reset. This affects not just traditional CJK input but any tool that uses the Composition API including voice-to-text assistants.

## Applies when

- Adding or modifying xterm `onKey` handlers that process Enter/Shift+Enter
- Users report Enter being swallowed or text appearing out of order after voice dictation
- Supporting non-Latin input methods (CJK, Arabic, voice-to-text) in terminal panels
