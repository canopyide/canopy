## Context

`cleanupOrphanedPtys()` killed stale PTY roots with `process.kill(pid, "SIGKILL")`. On Windows this only terminates the root pty process, leaving the shell and any child commands as orphans that continue to hold file locks and eat memory.

## Problem

After a crash/unclean exit on Windows, reopening Canopy left previous-session shells and commands running in the background. Nothing visibly failed — the cleanup loop reported success while the tree was still alive.

## Root cause

Node's `process.kill` on Windows maps to `TerminateProcess(pid)`, which acts only on that single PID. Windows has no process-group concept, so there is no equivalent of `process.kill(-pgid)` to take down a tree. Only `taskkill /T /F /PID <pid>` walks the child tree via the job/snapshot mechanism.

## Lesson

On Windows, terminate PTY (and any spawned-tree) processes with `spawnSync("taskkill", ["/T", "/F", "/PID", String(pid)])`. Treat exit code `0` and `128` as success — `128` means the process was already gone. Keep `process.kill` as a last-resort fallback for access-denied or unusual cases, and always gate the taskkill path on `process.platform === "win32"`.

## Applies when

Any code path that kills spawned child processes on Windows: PTY cleanup, dev-server termination, lifecycle-command timeouts, `runSingleCommand`-style helpers. If the process can spawn children, you need `/T`.
