## Context

`FdMonitor.readPtmxLimit()` reads the system's maximum PTY count to warn users before exhaustion. It had a macOS path (via `sysctl`) but returned `null` on Linux, making PTY exhaustion much harder to diagnose on Linux.

## Problem

Linux users hitting PTY limits got no proactive warning — `FdMonitor` simply couldn't report the ceiling. This made debugging "too many open files" errors harder since the diagnostic info was absent.

## Root cause

The initial implementation only targeted macOS (the primary dev platform). Linux support was simply not added.

## Lesson

Linux exposes PTY limits via `/proc/sys/kernel/pty/max` (default kernel value: 4096). Reading it is a straightforward `readFileSync` with integer parse. The platform-specific paths are:

- **Linux**: `parseInt(fs.readFileSync("/proc/sys/kernel/pty/max", "utf8").trim(), 10)`
- **macOS**: `sysctl kern.maxptys`
- **Windows**: N/A (no equivalent kernel limit)

When adding diagnostic/resource-monitoring features, explicitly handle all three platforms. A `null` return is acceptable for unsupported platforms, but Linux and macOS should both have implementations since they're the primary deployment targets.

## Applies when

Any code that reads kernel resource limits for diagnostic purposes. PTY exhaustion is a real risk for multi-agent workflows with many concurrent terminals.
