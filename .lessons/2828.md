## Context

`AppDialog` needs a Tab focus trap so keyboard users cannot escape the modal, but it also contains Radix popovers (dropdowns, selects, comboboxes) that render their content in a portal outside the dialog DOM tree.

## Problem

A naive focus trap intercepts every Tab key inside the dialog and cycles focus through its own focusable children — which breaks nested popovers because the portaled popover content is not a descendant of the dialog.

## Root cause

Portaled popovers live in `document.body`, not inside the dialog, so "closest focusable inside dialog" misses them entirely. The trap hijacks Tab before the popover can handle it.

## Lesson

When implementing a focus trap for a component that may contain portaled popovers, check `event.target.closest('[aria-modal="true"]')` — if the closest modal ancestor is not the dialog itself, skip the trap and let the nested popover handle Tab. Auto-focus the first focusable child on open via `requestAnimationFrame` (not immediately — the dialog may not be fully mounted), with a fallback to `tabIndex={-1}` + `outline-none` on the container so it can receive focus when the dialog has no focusable children. This pattern works with Radix, Headless UI, and any popover library that portals.

## Applies when

Adding a focus trap to any dialog/modal, or debugging "Tab behaves weirdly inside a dropdown inside a dialog" bugs.
