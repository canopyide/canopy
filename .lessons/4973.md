## Context

On app startup, `stateHydration` restores panels from the previous session. Agent terminals attempt to reconnect to their PTY processes. A "phantom agent guard" was added to filter out stale agent panels that no longer have a live PTY.

## Problem

All agent terminal panels were silently dropped on cold app restart. Users lost their entire saved layout on every restart.

## Root cause

The phantom-agent guard checked `reconnect === "not_found"` and discarded the panel. During cold start, _every_ PTY reconnect returns `not_found` because the PTY host hasn't started yet — but `not_found` on cold start means "needs to be respawned", not "this panel is phantom/stale". The guard had no way to distinguish cold-start `not_found` from project-switch `not_found`.

The fix: check `_switchId !== undefined` before activating the guard. `_switchId` is only set during a live project switch, not during initial hydration.

## Lesson

When hydration logic runs in two contexts (cold app start vs. live project switch), guards that are valid in one context can be catastrophic in the other. Use an explicit context discriminator (like a `_switchId` parameter) to gate context-specific behaviour rather than relying on observable state that happens to differ between contexts.

PTY `reconnect: "not_found"` means different things depending on context:

- **Cold start**: PTY host not running yet → panel needs respawning
- **Project switch**: PTY genuinely gone → panel should be discarded (phantom guard fires)

## Applies when

- Modifying `src/utils/stateHydration/index.ts` phantom-agent or reconnect logic
- Any hydration path that differentiates cold-start from project-switch restore
- Adding new reconnect states to the agent state machine
