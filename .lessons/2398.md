## Context

`getDefaultShell()` selects the shell for new PTY terminals on each platform. On Windows the original implementation returned `process.env.COMSPEC || "powershell.exe"`.

## Problem

Users always got cmd.exe on Windows even when PowerShell was installed. The PowerShell fallback was dead code.

## Root cause

`COMSPEC` is set by Windows itself on every session to `C:\Windows\System32\cmd.exe`. It is effectively never undefined, so the `||` fallback was unreachable. Developers expecting "prefer PowerShell, fall back to cmd" had the logic exactly backwards.

## Lesson

`COMSPEC` on Windows is a system-managed variable pointing to cmd.exe — do not use `env.COMSPEC || ...` as a "prefer modern shell" pattern. To prefer PowerShell, actively probe the PATH for `pwsh.exe` then `powershell.exe` (via `execFileSync("where", ...)` with a short timeout so a hung PATH lookup cannot stall Electron startup), and only fall back to `cmd.exe` if neither is found. Related landmine: classifying agent exits via `shell.includes("cmd")` is brittle — a path like `C:\src\cmd-tools\bash.exe` matches. Use basename exact-match (`"cmd"` or `"cmd.exe"`).

## Applies when

Detecting or defaulting shells on Windows, or matching shell identity by substring for lifecycle/exit classification.
