## Context

Project switches had a visible overlay that was blocked by terminal scrollback deserialization. The `getSerializedStates` IPC fetch was on the critical path inside `stateHydration`, meaning the switch overlay stayed visible until all scrollback data had been fetched and deserialized — even for terminals in non-visible worktrees.

## Problem

`stateHydration/index.ts` issued a batch IPC call for all terminal scrollback states upfront before the UI rendered. This added significant latency to every project switch because scrollback deserialization is expensive and was done synchronously in the critical path before the overlay could dismiss.

## Root cause

The hydration logic conflated two distinct concerns: restoring layout/panel state (needed immediately) and restoring scrollback history (only needed when the user actually looks at a terminal). Both were bundled into one synchronous blocking operation.

## Lesson

Separate critical-path UI restoration from deferred background work. The pattern used here:

1. Restore screen state and layout immediately (critical path — allows overlay to dismiss)
2. Use a `TerminalRestoreController` to schedule scrollback fetches at idle priority via `requestIdleCallback` for the active worktree's terminals
3. Defer non-visible worktree terminals to first-activation lazy restore

A `batchScheduler` extended with prioritized idle-callback scheduling keeps the logic centralized. Each terminal instance tracks a `ScrollbackRestoreState` so callers know whether restore is pending, in-progress, or complete.

## Applies when

- Any IPC operation that blocks UI rendering during a transition or switch
- Multi-worktree scenarios where non-active worktrees' data is fetched eagerly
- Terminal or panel hydration that mixes layout reconstruction with expensive data loading
