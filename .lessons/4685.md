## Context

Canopy's Portal feature hosts webviews for external sites. The `will-attach-webview` handler validates the requested partition, and `PortalManager` evicts hidden tabs to reclaim memory.

## Problem

Two separate bugs:

1. Webviews did not persist their session across reloads — they lost cookies and localStorage, appearing to log users out.
2. Recent `localStorage` writes were occasionally lost when a portal tab was evicted.

## Root cause

1. The `will-attach-webview` handler validated `params.partition` but never set `webPreferences.partition` on the `webPreferences` object being returned. Electron 41 requires the partition to be explicitly set in `webPreferences` — it does not inherit the `partition` attribute from the `<webview>` tag when `will-attach-webview` is used. Without it, Electron silently falls back to an ephemeral (non-persistent) session.

2. `PortalManager.destroyView()` called `webContents.close()` synchronously. Chromium buffers `localStorage` writes and flushes them asynchronously. Destroying the webContents before the flush completed discarded buffered writes. Fix: `await session.flushStorageData()` before calling `webContents.close()`.

## Lesson

In `will-attach-webview`, always explicitly set `webPreferences.partition` from the validated partition value — do not assume Electron will copy it from the `<webview>` element attributes.

When destroying a WebContents that may have pending storage writes, `await session.flushStorageData()` first. This applies to tab eviction, LRU cache eviction, and any forced teardown of a webview.

## Applies when

- `will-attach-webview` handler in Electron
- Any webview eviction or forced teardown (LRU cache, memory pressure)
- Portal/browser panel tab lifecycle management
