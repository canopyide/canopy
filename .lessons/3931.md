## Context

Layout undo/redo snapshots panel positions and restores them. Terminal fit (xterm `fit()` call) is triggered by a timeout after grid layout changes. The grid CSS uses a `200ms` transition on `grid-template-columns`.

## Problem

**Grid capacity bypass on undo:** `applySnapshot()` validates terminal IDs still exist but never calls `getMaxGridCapacity()`. If the window was resized to a smaller size after the snapshot was taken, restoring could place more panels in the grid than the current capacity supports. Panels would render below the 380px minimum width and 200px minimum height, breaking terminal readability.

**Fit timing misaligned:** The batch-fit effect used a 150ms timeout. The CSS transition on `grid-template-columns` is 200ms. Running `fit()` 50ms early means xterm measures fractional pixel dimensions mid-animation, producing incorrect terminal column/row counts. A separate two-pane split path used an even shorter 50ms timeout.

## Root cause

`applySnapshot()` was written to validate terminal existence but not layout validity for the current window state. The fit timeouts were set independently from the CSS transition duration with no shared constant, so they drifted apart.

## Lesson

Any code path that restores layout state (undo, project switch, session restore) must call `getMaxGridCapacity()` and clamp panel count before applying. Excess panels should overflow to the dock rather than violate dimension constraints. Always define CSS transition durations as shared constants (`GRID_TRANSITION_MS`) and use the same constant in JS timeout values. Without a `transitionend` listener, timeout-based fit calls must use the transition duration exactly — not shorter.

## Applies when

Adding new layout restoration paths or modifying grid transition durations. If terminals appear mis-sized after undo/redo, suspect this timing issue first.
