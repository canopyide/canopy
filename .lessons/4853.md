## Context

`TerminalOutputIngestService.coalesceBatch()` joins all queued string chunks before calling `terminal.write()`. This coalescing reduces xterm parse overhead during normal streaming. However during backpressure drain bursts, many chunks accumulate in the queue.

## Problem

After a backpressure pause, all queued chunks drained at once with no size limit. During agent streaming or build logs this produced multi-hundred-KB or multi-MB strings in a single `terminal.write()` call, causing:

- Large allocation spikes
- Long blocking xterm parse cycles on the renderer main thread
- GC pressure from the oversized string

## Root cause

`coalesceBatch()` did `chunks.join("")` with no size cap. The assumption was that the watermark system would prevent excessive queuing, but during drain bursts the queue genuinely accumulates many chunks before the flow resumes.

Fix: `COALESCE_BATCH_CAP_BYTES = 256 KB`. Fast path: single join when total is within cap. Slow path: accumulate until cap is reached, leave remainder in queue for the next tick.

## Lesson

String coalescing optimisations must be bounded. `Array.join("")` on an unbounded queue can produce strings much larger than intended. The xterm `terminal.write()` function blocks the main thread proportionally to input size — a 2 MB write is ~16× worse than eight 128 KB writes spread across ticks.

The 256 KB cap is a reasonable ceiling for a single synchronous xterm parse operation. Spreading larger payloads across ticks keeps the UI responsive during drain bursts.

## Applies when

- Any batching loop that coalesces data for a synchronous consumer (xterm, canvas, etc.) — always impose a size cap and defer remainder.
- Tuning: if 256 KB causes visible latency on slow machines, lower the cap further.
