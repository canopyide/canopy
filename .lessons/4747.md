## Context

The project switcher used Fuse.js for fuzzy search. A `useMemo` created a Fuse index from `sortedProjects`. The `projectStatsStore` pushed updates every 10 seconds.

## Problem

Every stats push triggered a re-render. `sortedProjects` was derived from a selector that produced a new array reference even when the underlying projects hadn't changed — only their stats changed. This caused the `useMemo` to see a new dependency and rebuild the entire Fuse index on every stats update, even during idle typing.

## Root cause

The memo dependency was `sortedProjects` (an array), but array reference equality always fails in JavaScript. The Fuse constructor call is expensive (it builds an inverted index), so rebuilding it every 10 seconds during active use meant every keystroke in the search field could be racing against a Fuse rebuild.

The fix was to use a stable project-ID–based memo key so the Fuse index only rebuilds when actual project identity changes (add/remove/rename), not when stats change.

## Lesson

When memoizing expensive objects derived from a collection:

- Use a stable identity key (sorted IDs joined to a string, or a count + checksum) rather than the collection reference itself
- Stats/status fields that update frequently should not be part of the dependency for structural computations (index building, sorting)
- This is especially important in Electron apps where push-based store updates fire on a clock, not on user interaction

Fuse.js specifically: rebuild only when the searchable items themselves change, never when their metadata (status, counts) changes.

## Applies when

- Any `useMemo` that wraps a search index, sorted list, or other expensive derivation
- Any Zustand store subscription that pushes updates on a timer — check whether dependent memos use stable keys
