## Context

Canopy intentionally disables renderer backgrounding (`--disable-renderer-backgrounding`) so terminals continue processing agent output when the window is minimized. This correct decision means Chromium's built-in background memory reclamation never fires, so Blink caches accumulate indefinitely.

## Problem

The main process had memory pressure mitigation (`ProcessMemoryMonitor`) that cleared `session.defaultSession.clearCache()` and called `global.gc()`. But the renderer's Blink-level caches (compiled scripts, decoded images, font glyph caches, layout data) were never cleared. These are typically the largest memory consumers, and `webFrame.clearCache()` is the only API that reclaims them back to the OS.

Key distinction: `global.gc()` frees V8 JS objects but V8 retains the freed heap pages (RSS doesn't drop). `webFrame.clearCache()` clears Blink's internal caches and the OS reclaims the physical memory. Both are needed for effective memory reduction.

## Root Cause

`webFrame.clearCache()` is a preload/renderer-only API. The main process memory monitor could not call it directly — it requires an IPC channel to notify the renderer. This indirect path was never wired up.

## Lesson

Wire up renderer Blink cache clearing via IPC: main process sends `reclaim-renderer-memory` after a 5-second debounce on minimize (cancel if restored before the timeout). The debounce prevents cache thrashing on quick minimize/restore cycles. Also trigger the same IPC from the memory pressure path in `windowServices.ts`. In the preload, call `webFrame.clearCache()` and optionally `gc()` if `--expose-gc` is passed to the renderer. `webFrame.clearCache()` only affects rendering caches, not IPC or data processing, so terminals continue unaffected.

## Applies when

- Any app-wide memory pressure mitigation strategy
- Any IDE/long-running Electron app that disables renderer backgrounding for background tasks
