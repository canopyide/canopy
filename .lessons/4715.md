## Context

Canopy supports multiple simultaneous project windows, each showing its own worktree data. Before this rewrite, all renderer views shared a single global Zustand store (`worktreeDataStore`) with window-ID-based scoping to route updates to the correct view.

## Problem

Cross-project worktree contamination in multi-window scenarios. One project's worktrees would appear in another window. Despite 9 rounds of incremental fixes — adding scope IDs, generation counters, `isSwitching` flags, `projectMismatch` short-circuits — the contamination persisted. The root structural problem was a global singleton that once polluted stayed polluted.

## Root cause

Window-ID-based scoping is fragile in a multi-view architecture. `WebContentsView` IDs don't map cleanly to window IDs when multiple views share a window. Fallback-to-aggregate code paths meant any routing miss fell back to sending updates to all listeners. Global Zustand singletons live for the lifetime of the renderer process — there's no clean reset boundary between project switches.

## Lesson

When data must be strictly isolated per-view (not per-window), use MessagePorts to create direct private channels between each renderer view and its backend process. A `WorktreePortBroker` in main creates a dedicated `MessageChannel` per `WebContentsView`, transfers one port to the view via `webContents.postMessage`, and keeps the other connected to the workspace host. The renderer creates a per-view Zustand store instance (`createWorktreeStore`) wired to its port. There is no shared global store — isolation is structural, not enforced by guards.

Incremental scope-checking fixes to a global singleton don't solve multi-view contamination. The architecture itself must isolate stores per view.

Cached views need rehydration on wake: attach a `visibilitychange` listener that triggers a silent refresh when the view becomes visible again after being backgrounded.

## Applies when

- Multiple renderer views share a window and need isolated data stores
- Worktree/project data must never bleed between windows
- Building per-view state in an Electron multi-WebContentsView architecture
