## Context

`PtyClient.waitForReady()` returned a promise that resolved when the PTY host utility process emitted its `ready` event. On any failure path — `utilityProcess.fork()` throwing synchronously, the host crashing before emitting ready, a missing binary — the promise never settled. The app froze on its splash screen until force-quit.

## Problem

The promise had a resolver for the happy path and nothing for failure. Exit events arriving before `ready` were ignored. Late `ready` events after host death were accepted and resolved the promise to a dead client.

## Root cause

Electron utility processes can fail in three distinct ways and each needs an explicit handler:

1. `utilityProcess.fork()` throws synchronously (missing binary, bad args, OS limits)
2. The process starts but crashes before sending its IPC ready message (startup script throws)
3. The process starts, forks OK, but its stdin/stdout pipes fail

If any of these are not wired to reject the ready promise, the awaiter blocks forever.

## Lesson

Every `waitForReady()` over a child/utility process needs all of: (a) synchronous try/catch around `fork()` that rejects, (b) an `exit`/`error` handler that rejects if the promise hasn't settled yet, (c) a guard flag that ignores late `ready` events arriving after death, (d) paired cleanup of both resolve and reject so neither leaks when the promise settles. Pair this with `Promise.allSettled` at the call site so a single service failure degrades the feature area instead of hanging the whole app — gate dependent work (MessagePort distribution, default terminal spawn) on a per-service `ready` flag.

## Applies when

Reviewing any child process, worker thread, utility process, or remote IPC client that exposes a ready promise. If the code path doesn't reject on fork/spawn errors, exit-before-ready, and already-settled late events, it will eventually hang a user's app. This also applies to WebSocket connections, MessagePort handshakes, and anything with a "wait for first message" contract.
