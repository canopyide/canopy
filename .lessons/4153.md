## Context

Agent panels were showing a "failed" state during normal operation — after routine exits or during phases where agents emit stderr or error-like output.

## Problem

The `AgentStateMachine` had an unconditional `error -> failed` transition: any `error` event sent the agent to the `failed` state regardless of context. Additionally, `isRoutineExit()` in `AgentStateService` did not include exit code 1 (common success-adjacent exit for many CLI tools) or exit code 137 (SIGKILL — often used by orchestration layers to stop agents intentionally).

## Root cause

The state machine was designed assuming `error` events are always fatal. In practice, many agents emit `error` events during normal operation (output parsing false positives, stderr that isn't fatal). The routine exit whitelist was too narrow.

## Lesson

In an agent state machine driven by PTY output heuristics, `error` events should generally be no-ops — they are too noisy to be reliable crash indicators. Reserve `failed` transitions for: explicit crash signals (SIGSEGV/SIGABRT), unrecoverable exit conditions diagnosed by forensic analysis, and explicit `agent:killed` events. Exit codes 0, 1, and 137 should all be treated as routine.

## Applies when

- `AgentStateMachine` transitions
- `isRoutineExit()` classification in `AgentStateService`
- Debugging false positive "failed" badges on agent panels
