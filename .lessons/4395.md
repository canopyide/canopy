## Context

Multiple subsystems in the pty-host independently called `ptyProcess.pause()` and `ptyProcess.resume()` on the same terminal: `ResourceGovernor` (memory/CPU pressure), `BackpressureManager` (IPC queue depth), and `IpcQueueManager` (throughput throttling). Each caller treated pause/resume as a binary toggle on the shared PTY object.

## Problem

State desync: if `ResourceGovernor` called `resume()` because memory pressure lifted, it would resume the PTY even if `BackpressureManager` still had a pause vote active. The net effect was a terminal that continued outputting data into an already-saturated IPC queue, causing data loss or dropped output.

## Root cause

`node-pty`'s `pause()`/`resume()` methods are not reference-counted — they operate as raw on/off. When three independent subsystems each manage the same flag without coordination, any one of them can accidentally lift a hold placed by another. This is a classic "multiple owners of shared mutable state" concurrency bug.

## Lesson

Any time multiple independent subsystems need to pause/resume the same PTY (or any resource with binary on/off control), use a reference-counted coordinator rather than direct calls. The `PtyPauseCoordinator` pattern: `pause(terminalId, callerName)` increments a vote count; `resume(terminalId, callerName)` decrements it; the actual `ptyProcess.resume()` is only called when the count reaches zero. Each subsystem holds its own named vote and cannot accidentally lift another's hold. Always call `releaseAll(terminalId)` on terminal destroy.

## Applies when

- Multiple subsystems (backpressure, memory governor, throttle) need to control the same flow-control mechanism
- Adding a new reason to pause PTY output — always go through the coordinator, never call `ptyProcess.pause()`/`ptyProcess.resume()` directly
- Any binary resource gate (pause, lock, mute) with multiple independent callers
