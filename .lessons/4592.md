## Context

Canopy runs many agent terminals concurrently. After an agent session completes, its xterm.js instance stayed at the VISIBLE tier indefinitely, holding scrollback buffers, addons (fit, search, weblinks), and canvas allocations in memory.

## Problem

Completed agent terminals were never downgraded from VISIBLE to BACKGROUND tier. The BACKGROUND tier has a lower scrollback limit and can be eligible for full hibernation (destroying the xterm instance and reclaiming all memory). With 10+ completed agents, this held tens of MB of xterm state unnecessarily.

Additionally, `ProcessMemoryMonitor` thresholds were too relaxed — warning only fired at 1.5GB and critical at 2GB, giving too little headroom before the renderer approached OS memory limits.

## Root cause

The hibernation system used an idle timer approach: terminals were only considered for downgrade after a period of inactivity. Completed agent terminals were not treated as a special case that should immediately drop to BACKGROUND tier regardless of their last activity time.

The fix introduces `hibernateCompletedAgentTerminals()` on `TerminalInstanceService`, called from `terminalStore` as a direct reaction to agent session completion. This is a **push** model rather than a **poll** model — the terminal store knows exactly when an agent finishes and can trigger reclamation proactively.

## Lesson

- xterm.js instances are expensive: each holds scrollback (configurable line buffer), addon state, and canvas context. Downgrading to BACKGROUND tier frees most of this.
- Don't rely solely on idle timers for memory reclamation when you have precise lifecycle events (agent completed, panel hidden) — use those events to trigger immediate tier downgrades.
- Add a `hibernateCompletedAgentTerminals()` sweep after any agent lifecycle event (completion, exit, error). The terminal store already tracks agent state per terminal session; use it.
- Memory monitor thresholds should leave a 500MB+ buffer before the renderer's V8 heap limit.

## Applies when

- Adding new agent lifecycle events (completion, cancellation, error)
- Any code that checks `AgentStateMachine` state transitions
- Tuning `ProcessMemoryMonitor` thresholds for different workloads
