## Context

Three components — `TerminalContextMenu`, `DndProvider`, and `useAccessibilityAnnouncements` — were subscribing to Zustand stores with selectors that returned arrays or objects. Each store update caused these components to re-render even when the underlying data hadn't changed, because a new array/object reference was returned from the selector.

## Problem

Selectors like `(s) => s.terminals` return the same array reference when the store changes any key, but components using these selectors re-render on every store update regardless. Selectors using `.find()` inline create a new closure on each call, also producing unstable references.

## Root cause

Zustand's default equality check is reference equality (`===`). Any selector that returns a new object or array on every call (even if the contents are identical) will trigger a re-render. The `useShallow` helper from `zustand/react/shallow` performs a shallow comparison and prevents re-renders when the array/object contents haven't changed.

## Lesson

**Always wrap array and object Zustand selectors with `useShallow`**. Bare selectors are only safe when selecting primitive values (strings, numbers, booleans).

```ts
// Bad — new array reference on every store update:
const terminals = useTerminalStore((s) => s.terminals);

// Good — stable reference when contents haven't changed:
const terminals = useTerminalStore(useShallow((s) => s.terminals));
```

For inline `.find()` selectors, wrap the whole selector:

```ts
const terminal = useTerminalStore(useShallow((s) => s.terminals.find((t) => t.id === id)));
```

This is especially impactful in components mounted for every terminal (like context menus) or hooks that power drag-and-drop, where the re-render cascade can be significant.

## Applies when

Any Zustand selector that returns an array, object, or derived value. Review all `useXxxStore((s) => s.someArray)` patterns during code review.
