## Context

ActivityMonitor infers when an agent becomes "busy" by watching user input for an Enter keypress. Users pasting multi-line text into a terminal triggered a spurious busy transition on every newline in the paste, even though the agent hadn't actually started working.

## Problem

Pasted newlines look identical to pressed Enter at the byte level — both are `\n` or `\r`. Naive input watchers count any newline as a submit event.

## Root cause

Terminals with bracketed paste enabled wrap pasted text with `\x1b[200~` ... `\x1b[201~` markers. The markers tell you "everything between is a paste, not typed input". Ignoring them makes newlines inside the paste indistinguishable from actual submission.

## Lesson

Any detector that interprets user input must bracket-paste-aware: track `inBracketedPaste` state, enter on `\x1b[200~`, exit on `\x1b[201~`, and ignore Enter/newlines while inside. Two gotchas: (1) escape sequences can arrive split across multiple chunks, so buffer a small partial-escape state between calls; (2) if the end marker is somehow lost, add a timeout (5s is reasonable) to force-exit paste mode so you don't get stuck. Clean the state up in `dispose()`. The same logic is needed in InputTracker, ActivityMonitor, clear-command detection, and any future "did the user just submit something?" heuristic. Not every agent CLI emits bracketed paste (Gemini historically did not — see lesson 1283), so for non-bracketed agents you also need an output-settle fallback.

## Applies when

Writing or touching any service that classifies terminal input as "submitted" vs "typed", detects clear commands, or drives agent state transitions from input bytes. Also when users report "agent goes busy as soon as I paste" or "clear command doesn't fire when pasted".
