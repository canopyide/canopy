## Context

Canopy uses `CANOPY_E2E_SKIP_FIRST_RUN_DIALOGS` to suppress the `AgentSetupWizard` during Playwright E2E runs. The renderer reads this via `isCanopyEnvEnabled()` in `src/utils/env.ts`.

## Problem

The E2E flag was silently ignored in CI. The `AgentSetupWizard` kept appearing and blocking Playwright clicks on the welcome screen, even though the flag was set when launching Electron.

## Root cause

The renderer read the flag via `import.meta.env.CANOPY_E2E_SKIP_FIRST_RUN_DIALOGS`. Vite bakes `import.meta.env` values at **build time**, not at Electron launch time. CI builds don't set the env var during `npm run build` — only when launching Electron for the test run. So the baked value was always `undefined`, and the flag was always `false` at runtime.

The existing `CANOPY_E2E_FAULT_MODE` flag (a different E2E control) already used the correct pattern: the preload reads the env var at launch time and exposes it via `contextBridge.exposeInMainWorld`. The fix mirrors that pattern: the preload conditionally sets `window.__CANOPY_E2E_SKIP_FIRST_RUN_DIALOGS__`, and `isCanopyEnvEnabled()` checks the window bridge first before falling back to `import.meta.env`.

## Lesson

`import.meta.env` in Vite is a build-time snapshot. Any env var that needs to be controlled at Electron launch time (e.g. E2E flags, feature flags passed to the test runner) must travel through the preload via `contextBridge`. Never rely on `import.meta.env` for values that won't be known until after the build step. Use a `getRuntimeBridgeValue()` helper that checks `window.__VAR__` first so the bridge path takes precedence.

## Applies when

- Adding new E2E environment flags that control renderer behaviour
- Any `import.meta.env` usage that needs to be overridable at launch time vs build time
- Debugging why an env var works locally (where dev server sets it) but not in CI (where it's only set at run time)
