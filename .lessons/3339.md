## Context

The PR list GraphQL query declared `$orderBy: PullRequestOrder`, following the obvious naming convention for sorting pull requests.

## Problem

GitHub's GraphQL endpoint rejected the query with a type-mismatch error. PR listing in the worktree sidebar failed entirely.

## Root cause

`PullRequestOrder` does not exist in GitHub's public GraphQL schema. The `Repository.pullRequests` connection — despite being PR-specific — takes the same `IssueOrder` input type used by `Repository.issues`. The same is true for several other "issue-adjacent" connections in the GitHub schema. Local typechecking can't catch this because the input type is passed as a string inside the query literal.

## Lesson

When writing GitHub GraphQL against `Repository.pullRequests` (or any issue-related connection), use `IssueOrder` as the `$orderBy` input type. Add a unit test that matches the query string against `/\\$orderBy:\\s*IssueOrder/` so a regression surfaces without needing a live API call. Don't assume GitHub's type naming mirrors the connection name.

## Applies when

Any new GraphQL query against GitHub's `Repository.pullRequests`, `Repository.issues`, or similar order-able connections. Double-check the schema at https://docs.github.com/en/graphql/reference/input-objects before assuming an input type exists.
