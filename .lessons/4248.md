## Context

Canopy's setup wizard checks for prerequisite tools (Node.js, git, GitHub CLI) using `which` / `where`. It also has a "Re-check" button users click after installing a missing tool.

## Problem

The Re-check button consistently reported tools as "not found" even after the user had successfully installed them. A fresh terminal outside Canopy showed the tools working correctly.

## Root cause

Electron's main process inherits its `PATH` from the environment at launch time and never updates it. On macOS, apps launched from the Dock or Finder receive a minimal `launchd` PATH (`/usr/bin:/bin:/usr/sbin:/sbin`) that does not include Homebrew (`/usr/local/bin`, `/opt/homebrew/bin`), nvm, fnm, or custom installer paths.

Both `SystemHealthCheck.ts` and `CliAvailabilityService.ts` ran `execFileSync('which', [command])` against this stale `process.env.PATH`. Calling "Re-check" ran the same stale check again.

## Lesson

**Electron processes on macOS never receive the user's interactive shell PATH.** Apps launched from Dock/Finder get the launchd environment, not the shell's. To get the real PATH, you must spawn a login shell and read its output:

```bash
# macOS / Linux: spawn the user's login shell
$SHELL --login -c 'echo $PATH'
```

On Windows, read from the registry (`HKLM\SYSTEM\...\Environment` and `HKCU\...\Environment`).

This must happen on every "Re-check" — not once at startup — because the user may install tools between checks using a version manager (nvm, fnm) that modifies shell profiles, not system-wide paths.

Cache the result with a short TTL (a few seconds) to avoid spawning a shell on every tool check within a batch, but invalidate the cache before each user-triggered recheck.

This pattern is required for any Electron app that needs to detect user-installed CLI tools on macOS.

## Applies when

- `which`/`where` based tool detection in Electron main process
- Setup wizards or health checks that re-run after user installs tools
- macOS apps detecting Homebrew, nvm, fnm, pyenv, rbenv, or similar tools
- Any `execFileSync` call that depends on the user's interactive shell PATH
