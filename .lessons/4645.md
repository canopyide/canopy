## Context

`createWindow()` had an early-exit guard: if a window already existed, it focused the existing window instead of creating a new one. This was the explicit single-window assumption baked into the core lifecycle.

## Problem

Three collisions when the guard was removed without additional scoping:

1. **Terminal ID collisions**: two windows creating terminals with the same ID prefix would corrupt PTY routing.
2. **Global service re-initialization**: the second window triggered re-initialization of global services (WorkspaceClient, PtyClient) that should only be set up once.
3. **`second-instance` CLI paths**: when the app was already running and the user ran `canopy /path/to/repo`, the handler focused the existing window instead of opening a new one.

## Root cause

The app was designed with implicit single-window assumptions embedded in several places: the `second-instance` handler, `createWindow()`, `windowServices.ts`, and terminal ID generation.

## Lesson

- **Generate unique terminal ID prefixes per window** using the window's sequential index or a UUID. This prevents PTY routing collisions between windows sharing the same host.
- **Distinguish global vs. per-window service initialization**: `WindowServices.ts` should skip global service setup (WorkspaceClient connect, PtyClient connect) when a second window is created — these are already running.
- **`second-instance` must call `createWindow()` with the path arg**, not `win.focus()`. The existing behavior was correct for single-window (focus existing), but breaks the multi-window contract.
- **The `createWindow()` guard was load-bearing for single-window testing**: all unit tests assumed one window. After removing the guard, tests need explicit scoping of window references.
- **`activate` on macOS (dock click) should still focus an existing window** if all windows are closed, but not when windows are already open — this is distinct from `second-instance`.

## Applies when

Removing single-window guards from any Electron app being upgraded to multi-window. Audit terminal/session ID generation, service initialization ordering, and OS-level single-instance handlers.
