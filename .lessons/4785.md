## Context

The demo video capture pipeline was using `toPNG()` to capture frames from the Electron `WebContentsView` and writing them to disk before encoding with ffmpeg.

## Problem

`toPNG()` compresses each frame through zlib before returning, taking 20-50ms per frame. This capped real-world capture throughput at ~10-15 FPS — far below the 25 FPS target for smooth demo videos.

Additionally, `yuv420p` (4:2:0 chroma subsampling) was blurring syntax highlighting and UI text in the encoded output because chroma subsampling halves color resolution horizontally and vertically.

## Root cause

`toPNG()` was the obvious choice for "get frame as image" but its compression overhead is inappropriate for a video capture pipeline where frames are immediately consumed by an encoder. The codec choice was also wrong for text-heavy content.

## Lesson

**For Electron screen capture pipelines, use `toBitmap()` + raw BGRA piping, not `toPNG()`.**

`NativeImage.toBitmap()` returns raw BGRA pixel data with zero compression overhead. Pipe it directly to ffmpeg stdin:

```
ffmpeg -f rawvideo -pix_fmt bgra -video_size WxH -framerate 25 -i pipe:0 ...
```

This reduces per-frame overhead from 20-50ms to ~1-2ms, pushing throughput from 10-15 FPS to 25-35 FPS.

**For text/code content, use `libx264rgb` with 4:4:4 instead of `yuv420p`:**

- `yuv420p` (4:2:0): blurs text due to chroma subsampling. Standard for video playback compatibility.
- `libx264rgb` 4:4:4: preserves full color per pixel. Essential for syntax highlighting and UI text legibility.

Additional considerations:

- Handle stdin backpressure via the `drain` event — don't write the next frame until the previous write has flushed.
- Release `NativeImage` references immediately after extracting the bitmap to reduce GC pressure.
- Use `--force-device-scale-factor=1 --force-color-profile=srgb` launch flags for HiDPI/Retina consistency.
- Fill duplicate frames when capture lags to maintain constant frame rate in the output.

## Applies when

Any Electron app capturing screen content for video output. The toPNG vs toBitmap tradeoff applies whenever you need high-throughput frame capture without intermediate disk I/O.
