## Context

When a terminal panel was closed or the app quit, `TerminalProcess.kill()` called `ptyProcess.kill()` which sends SIGHUP to the immediate shell only. Interactive shells place foreground commands (`vite`, `next dev`, `webpack`) into separate process groups, so SIGHUP to the shell left all child processes alive as orphans adopted by init/launchd.

## Problem

User-visible symptoms included `EADDRINUSE` when restarting dev servers after closing their terminal, Activity Monitor filling with orphaned `node` processes, and macOS PTY device exhaustion after extended use (`posix_spawnp failed`). All three kill paths (user close, project close, app quit) ultimately called the same broken `ptyProcess.kill()`.

## Root Cause

`node-pty`'s `ptyProcess.kill()` only sends SIGHUP to the session leader. Interactive shells create new process groups for foreground commands via `setpgid()`, so even `process.kill(-pid)` (process-group signal) is insufficient — it misses processes in different groups. A recursive process tree traversal is required.

The codebase already had correct tree-killing in two other locations (`WorktreeLifecycleService.ts` using `process.kill(-pid, "SIGTERM")` and `cleanupOrphanedPtys()` with tree-kill fallback), confirming the issue was an oversight rather than a design choice.

## Lesson

Killing a PTY shell is not sufficient to clean up processes it spawned. Always kill the entire process tree:

1. Use `pgrep -P <pid>` recursively to discover all descendant PIDs
2. Send SIGTERM to the shell and all descendants
3. After 500ms, escalate to SIGKILL for any survivors

A `ProcessTreeCache` with a short TTL avoids redundant `pgrep` lookups during the SIGTERM/SIGKILL escalation window.

`ptyProcess.pid` may be undefined if the shell already exited — guard against this. On macOS, processes calling `setpgid()` (common in dev servers) will be in different process groups, so a process-group kill is not enough; recursive tree traversal is necessary.

Windows uses ConPTY with a different process model — `taskkill /T /F` is the equivalent.

## Applies when

- Any terminal close path in `TerminalProcess.kill()` and `onUnexpectedExit()`
- App quit sequences that kill terminal processes
- Diagnosing `EADDRINUSE` after terminal close in development workflows
