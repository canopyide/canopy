## Context

Canopy configures xterm.js with a scrollback limit (1000 lines). When output exceeds this limit, the oldest lines should be evicted. The test validates the buffer behaves correctly after outputting 5000+ lines (5× the limit).

## What the test verifies

- Output 5000 numbered lines via a shell loop
- Assert buffer holds approximately `scrollbackLimit` lines (±tolerance)
- Assert oldest retained line number is approximately `5000 - scrollbackLimit` (±100, tolerating shell overhead)
- Assert terminal is still interactive after the flood

## Design details that matter

**±100 tolerance**: Shell overhead (prompt lines, echo of the loop command itself) means the exact oldest line number is not `4000`. The test uses a tolerance band rather than exact equality.

**Configurable scrollback limit via `getScrollbackLimit()` IPC**: Rather than hardcoding `1000` in the test, `TerminalInstanceService` exposes `getScrollbackLimit()` through the `__canopyReadTerminalBuffer` debug API. If the configured limit changes, the test adapts automatically.

**ANSI-colored output**: The test deliberately uses ANSI color codes in the output lines to verify the buffer handles escape sequences correctly alongside line counting.

## Lesson

Terminal scrollback tests must use the configured scrollback limit constant (not a hardcoded number) and must include a tolerance band for shell overhead. The "is the terminal still interactive" assertion after a flood is as important as the buffer bounds check — it detects cases where xterm.js enters a degraded state after processing a large volume of lines.

Export `scrollbackLimit` from the service (or via the debug API) for tests to reference. Don't hardcode it in two places.

## Applies when

- Changes to xterm.js scrollback configuration
- Changes to how PTY data is chunked and sent to the renderer
- Testing buffer memory behavior under sustained high-volume output
