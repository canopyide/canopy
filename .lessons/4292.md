## Context

Canopy writes many config and state files to disk (settings, project state, notes, MCP server config, etc.). Several services had their own bespoke temp-file-then-rename patterns, while others used a plain `writeFile` without atomicity.

## Problem

Non-atomic writes leave a window where a crash (power loss, forced kill, OOM) can corrupt the target file — the write is partially complete when the process dies, leaving truncated or zero-byte data.

## Root cause

The existing `resilientWriteFile` utility retried on transient errors but was not atomic — it wrote directly to the target path. Six services duplicated their own temp+rename logic. Others used bare `writeFile`.

## Lesson

**Atomic write pattern for Node.js/Electron**: write to `${targetPath}.tmp`, call `fsync` on the file descriptor to flush OS buffers to disk, close the handle, then `rename` to the target path. The `rename` syscall is atomic on the same filesystem — the target is either the old file or the new file, never a partial write.

```
write → tmp file → fsync(fd) → close → rename(tmp, target)
```

**`fsync` is required before rename** on Linux (and recommended on macOS). Without it, the rename can complete while the data is still in the OS page cache; a crash after the rename but before the cache flush leaves an empty target.

**Consolidate into a shared utility** (`resilientAtomicWriteFile` and `resilientAtomicWriteFileSync`) rather than each service reimplementing the pattern. Services that only needed retries on transient errors should be upgraded to the atomic version for important data paths.

## Applies when

- Any file that, if corrupted, requires user recovery or causes app failure (settings, state, DB, config)
- Migrating from `writeFile` to crash-safe writes
- Cross-platform Electron file I/O (the temp+rename pattern works on all three OSes)
