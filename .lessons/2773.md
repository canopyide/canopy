## Context

`WorkspaceService` manages per-worktree `MonitorState` objects that run polling and emit PR/issue events. Users can switch projects, which changes the active scope.

## Problem

Previous fixes guarded the IPC boundary and the renderer store layer, yet stale monitors from a previous project were still emitting updates and mutating snapshots after a project switch, leaking into the new project's UI.

## Root cause

Async work in flight at the moment of a project switch cannot be reliably cancelled. Filtering at the boundary is insufficient — the monitors themselves keep running and calling `emitUpdate`, `deleteWorktree`, and event handlers (`sys:pr:detected`, etc.) without knowing they are stale.

## Lesson

For long-lived async producers that outlive a scope change, tag each producer with the scope id it was created under, then check `producer.projectScopeId === this.projectScopeId` at every emission point. Guard all pathways: update emissions, state getters, deletion handlers, and every event callback — missing one is enough to leak. Boundary filtering alone is a leaky abstraction for any cross-cutting scope concern (project, window, session).

## Applies when

Adding new emission pathways to `WorkspaceService` / `WorktreeMonitor`, introducing a new long-lived service that survives project switches, or debugging "stale data from the previous project appeared briefly" bugs.
