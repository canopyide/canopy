## Context

Canopy exposes CDP (Chrome DevTools Protocol) operations over IPC: debugger attach, Runtime enable, and console capture for browser panel webviews. These handlers accept a `webContentsId` parameter from the renderer.

## Problem

Without ownership validation, the renderer could pass any `webContentsId` — including IDs belonging to other project windows, internal Electron processes, or unrelated WebContents. This would allow the renderer to attach a debugger to or capture console output from arbitrary WebContents instances.

Additionally, the `register-panel` IPC call to register a webview was firing _after_ `start-console-capture` in `BrowserPane.tsx`, creating a race condition where CDP operations were attempted on unregistered WebContents.

## Root cause

The `WebviewDialogService.getPanelId()` registry already tracked which `webContentsId` values belonged to known panels — it just wasn't being consulted before handling CDP requests. The ownership check was a missing step, not a missing data structure.

## Lesson

Any IPC handler that accepts an ID for a resource (WebContents, PTY process, etc.) should validate that the ID belongs to a resource created by the calling context. An existing ownership registry (`getPanelId()` in this case) is the right check — no new infrastructure needed.

Also: in `BrowserPane.tsx`, IPC registration calls must fire before any operations that depend on that registration. `register-panel` must precede `start-console-capture` to avoid a race condition where CDP handlers reject the unregistered ID.

## Applies when

Adding new IPC handlers that accept resource IDs; reviewing CDP or debugger-attach operations in Electron; any handler where the renderer provides an ID that resolves to a native resource.
