## Context

Terminal output data flowed through a 4-hop path: pty-host → parentPort IPC → main process → ipcRenderer → renderer. A MessagePort between the renderer and PTY host already existed and handled renderer→pty input (write, resize), but the reverse direction (PTY output) still went through the main process relay.

## Problem

For high-throughput output (large builds, log streams, `cat` on large files), the main process acted as a CPU bottleneck — serializing and deserializing data at every hop. The `rendererPort` variable existed in `pty-host.ts` but was only used for input messages.

## Root cause

The MessagePort setup was asymmetric by design oversight. The `connect-port` handler wired the port for write/resize input but never sent pty data output over it. All output was routed through `sendEvent()` to parentPort regardless of whether a direct renderer port was available.

## Lesson

When a MessagePort connection already exists between two processes, both directions of high-frequency data should be evaluated for direct routing. An asymmetric MessagePort (one direction only) is often a sign that the optimization was partially implemented. After routing output directly, the IPC path is kept as a fallback for when the port is not connected or disconnects (renderer reload), making the transition seamless.

The IPC suppression flag (`messagePortConnected`) in `terminalClient.ts` prevents duplicate delivery — the IPC `onData` handler must check this flag and skip delivery when a port is active.

## Applies when

- Any high-frequency data channel (terminal output, audio, large file reads) that goes through the main process as relay
- Checking whether an existing MessagePort is being used asymmetrically
- Any situation where PTY data volume causes CPU spikes in the main process
