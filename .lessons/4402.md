## Context

On smaller screens (1200×763, common on Linux CI runners), Radix UI dropdown and context menus rendered beyond the visible viewport. The "Delete Worktree..." item was outside the viewport and unclickable, causing an E2E core test failure.

## Problem

`DropdownMenuContent` and `ContextMenuContent` had no height constraint, so menus could overflow the viewport bottom on short displays. The items were rendered but not reachable by click because they were off-screen.

## Root cause

Radix UI positions menus using its collision detection, but does not automatically add `overflow-y-auto` or a `max-height`. Without those, items that exceed the available space simply overflow. Radix does expose `--radix-context-menu-content-available-height` and `--radix-dropdown-menu-content-available-height` CSS variables that reflect the actual available space, but they have no effect unless the component explicitly uses them.

## Lesson

When using Radix UI menu primitives, always add `max-h-[var(--radix-*-content-available-height)] overflow-y-auto` to menu content components. This uses Radix's own computed available height (accounts for viewport position) and enables scrolling. Without it, menus overflow on small screens or when many items are present. This is especially important for CI environments that run at lower resolutions than developer machines.

## Applies when

- Adding or modifying `ContextMenuContent` or `DropdownMenuContent` components
- E2E tests fail because a menu item is "not clickable" or outside viewport
- Targeting Linux CI environments that use smaller screen resolutions
