## Context

Canopy's `App.tsx` previously rendered `WelcomeScreen` as a full replacement for `ContentGrid` when no project was open. This meant panels created before opening a project had no container to render in.

## Problem

Keyboard shortcuts, toolbar buttons, and the panel palette could all create terminals/browsers before a project was open, but those panels wouldn't appear because `ContentGrid` wasn't mounted. Additionally, `useActiveWorktreeSync.ts` resolved `defaultTerminalCwd` to `""` (empty string) when no project was active — which node-pty would interpret as the process working directory, not `~/`.

## Root cause

The `currentProject === null` state was treated as a special mode that replaced the panel grid entirely, rather than as an empty state _within_ the grid. The `WelcomeScreen` acted as a gate rather than as content inside the grid.

## Lesson

- Treat "no project open" as an empty state within the normal UI shell, not a separate rendering mode. `ContentGrid` should always be mounted; `WelcomeScreen` renders as the zero-panel placeholder inside it.
- When `currentProject` is null, always provide a sensible fallback cwd: `os.homedir()`. An empty string is not a valid cwd for node-pty.
- Review all panel creation paths (keyboard shortcuts, toolbar, palette) against the no-project state. They should all work without a project.

## Applies when

- Adding new panel types that need to work without a project
- Any hook that reads `currentProject` to derive a file path or cwd
- Onboarding flows where users might create panels before selecting a project
