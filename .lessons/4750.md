## Context

Canopy's multi-project architecture gives each project its own `WebContentsView` managed by `ProjectViewManager`, with an LRU cache to keep recently-used views warm for fast switching. The default cache size was 2.

## Problem

The "keep 2 views warm" default meant every user always had at least one hidden Chromium renderer sitting idle in memory, even when they rarely switched between projects.

## Root cause

The memory cost of a `WebContentsView` wasn't accounted for when setting the default. Each view is a full Chromium renderer process — 400–500 MB. A cache of 2 means up to 1 GB of renderer memory even if the user never opens a second project.

## Lesson

In Electron, `WebContentsView` / `BrowserView` / `BrowserWindow` instances are not lightweight objects. Each one is a full V8 + Blink renderer process. When building LRU caches or view pools:

- Default conservatively (1, not 2 or more)
- Document the per-view cost prominently near the default constant
- Test actual memory impact against typical user scenarios (single project vs. multi-project switchers)
- Project switching being "fast enough cold" (< 1 second) means caching more than 1 view rarely pays off

The 1–5 configurable range is still available for power users who do heavy project switching and have the RAM to spare.

## Applies when

- Setting defaults for any Electron view/window pool or LRU cache
- Evaluating whether a prewarming strategy is worth its memory cost
