## Context

`WorkspaceService.refreshAll()` triggers `updateGitStatus(true)` on all worktree monitors concurrently at startup. Each monitor runs a git status command against its own worktree path.

## Problem

After restart, multiple worktrees showed identical change stats — all mirroring the main worktree's numbers. Git's `fsmonitor` and `untracked-cache` daemon-side caches were producing stale/shared data across concurrent status calls hitting the same repo.

## Root cause

Git's `core.fsmonitor` and `core.untrackedCache` are designed for single-process sequential access. When multiple git processes query status concurrently against linked worktrees, the shared daemon cache can return stale results or mix up counts across worktrees. The issue is exacerbated on startup when all monitors fire simultaneously.

Two mitigations applied:

1. `hardenedGit.ts` now disables `core.fsmonitor` and `core.untrackedCache` for all git invocations in the app.
2. `WorkspaceService.ts` staggers the startup refresh so monitors don't all hit git at the same time.

## Lesson

Git's filesystem monitor and untracked-cache optimisations are unsafe for concurrent multi-worktree access patterns. Always disable them in the hardened git wrapper when the app may run multiple git status commands in parallel against linked worktrees.

Staggering concurrent git invocations at startup is a useful secondary defence even after disabling the cache, since git can still have lock contention on `.git` index files.

## Applies when

- Any code path that runs `git status` (or similar) on multiple worktrees in tight concurrency.
- New git wrapper utilities: check that `hardenedGit.ts` config flags are applied.
