## Context

Multiple UI components (AppDialog, AppPaletteDialog, SearchablePalette, FixedDropdown) each registered independent `window.addEventListener('keydown')` handlers for Escape. This worked by accident in most cases due to `stopImmediatePropagation`, but was fragile and could close multiple layers on a single keypress.

## Problem

With no coordination between overlay layers, pressing Escape could close both a palette AND a dialog simultaneously. `uiStore.overlayCount` tracked how many overlays existed but not which was topmost. If two `AppDialog` instances were open, both Escape handlers fired on the same keypress.

`SearchablePalette` had a two-step Escape pattern (clear search first, then close) implemented via `stopImmediatePropagation` — this was brittle and not composable.

## Root Cause

No central dispatch; each component assumed it was the only overlay. The overlay counter provided quantity but not ordering.

## Lesson

Use a LIFO stack (not a counter) for layered UI dismissal. The stack API: `pushHandler(id, fn)` / `popHandler()` / `removeHandler(id)`. A single global `window` keydown listener (mounted at app root) pops and executes the topmost handler. Components use `useEscapeStack(active, handler)` which pushes when active and removes on deactivation/unmount. Key integration points:

- `useGlobalKeybindings` chord-clearing still runs at capture phase, taking priority over the stack
- Radix UI's `onEscapeKeyDown` must be disabled/replaced to avoid double-firing
- `SearchablePalette`'s two-step pattern works naturally: first press calls the clear-search handler, second press fires the close handler
- When the stack is empty, Escape falls through to default terminal behavior

## Applies when

- Adding any new overlay, dialog, dropdown, or popover that responds to Escape
- Any new component that would have used `window.addEventListener('keydown')` for Escape handling
