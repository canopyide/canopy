## Context

When `ProjectViewManager` deactivates a project view (switching to another project), the view is detached from the window but kept alive in an LRU cache. React unmounts components, but V8's lazy GC doesn't immediately reclaim the freed memory from orphaned closures, detached DOM refs, and stale Zustand subscriptions.

## Problem

10-30MB of memory per deactivated view stayed on the heap indefinitely because V8's incremental GC deferred collection of the now-unreachable objects from the unmounted React tree.

## Solution

After detaching the view, defer a `window.gc()` call by 500ms via `webContents.executeJavaScript('window.gc && window.gc()')`. The 500ms delay ensures the GC runs after the view has fully transitioned off-screen, making any GC pause imperceptible to the user.

## Lesson

V8's lazy GC is not designed for tight memory budgets. After a predictable "large deallocation event" — React tree unmount, route change, tab switch — you can trigger a GC cycle explicitly via `window.gc()` (requires `--js-flags="--expose-gc"` or Electron's `--expose-gc` flag).

Guard the call: check that the view exists, its webContents are alive and not destroyed, and it is actually inactive. Use `webContents.executeJavaScript()` — not `webContents.send()` — because `gc()` is a synchronous V8 call that doesn't require a renderer IPC handler.

500ms deferral is the right tradeoff: enough time for the view to go off-screen, short enough to reclaim memory before the user switches back.

## Applies when

- `WebContentsView` LRU caches in Electron
- Tab-switching architectures where large React trees unmount
- Any scenario where predictable large allocations are freed at a known point
