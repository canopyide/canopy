## Context

Confetti particles used hardcoded Tailwind color classes, so they clashed with custom themes. The fix reads colors dynamically from CSS custom properties at render time.

## Problem

Static `PARTICLE_COLORS` arrays defined at module load time capture no theme state. When a user switches themes, the hardcoded colors remain.

## Root cause

Colors were baked in as Tailwind class-derived hex values instead of reading from the live theme token system.

## Lesson

When generating dynamic visual elements (particles, charts, canvas drawing) that should respect the active theme, read from CSS custom properties at the moment of rendering rather than at module initialisation:

```ts
function getThemeColors(): string[] {
  const style = getComputedStyle(document.documentElement);
  return [
    style.getPropertyValue("--color-accent-primary").trim(),
    style.getPropertyValue("--color-status-success").trim(),
    // ...
  ];
}
```

This pattern works because Canopy's theme system sets CSS custom properties on `documentElement` when themes are applied, so `getComputedStyle` always reflects the current theme.

**Fallback:** always provide a fallback array in case properties are undefined (e.g., in test environments where CSS is not applied).

**Test coverage note:** mock `getComputedStyle` to return controlled values per test case; this avoids brittle hardcoded hex assertions.

## Applies when

- Canvas drawing, particle systems, or any runtime-generated visuals that should match the theme
- Any component that builds a color palette at module load time
- Writing tests for components that read CSS custom properties
