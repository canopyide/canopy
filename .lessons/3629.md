## Context

Electron apps can crash the GPU process on machines with faulty or outdated graphics drivers. When the GPU process crashes, Electron attempts to restart it. Without a fallback, this loops indefinitely and can cause the entire app to become unusable.

## Problem

GPU process crashes are silent from the user's perspective: the app may freeze or display rendering artifacts without any mechanism to diagnose or recover.

## Root Cause

No crash counting, no persistence of crash state across restarts, and no user-accessible control to disable hardware acceleration.

## Lesson

Electron exposes GPU crashes via `child-process-gone` events with `type === "GPU"`. The recovery pattern:

1. Count crashes in-session via `child-process-gone`
2. After N crashes (3 is a reasonable threshold), write a flag file to userData and call `app.relaunch()` + `app.exit()`
3. On next launch, read the flag **before** `app.whenReady()` in the setup/environment phase and call `app.disableHardwareAcceleration()` early — this call must happen before the app is ready, so it must be in the earliest possible synchronous path
4. Expose a settings toggle so users can re-enable after driver updates

The flag file approach is required because `app.disableHardwareAcceleration()` must be called before app ready — it cannot be passed via argv or env in a way that works reliably across platforms.

Also include `gpuAccelerationDisabled` in the DiagnosticsCollector so support can see whether the fallback is active.

## Applies when

- GPU-related rendering issues are reported by users
- Building any crash recovery that requires changing pre-ready Electron configuration
- Adding troubleshooting toggles that affect Electron's process initialization
