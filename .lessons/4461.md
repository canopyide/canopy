## Context

The macOS app menu (the menu under the app name in the menubar) showed "About canopy-app" — the internal npm package name — instead of "About Canopy".

## Problem

Electron derives `app.name` from the `name` field in `package.json` if no `productName` is set. `name` must follow npm naming rules (lowercase, hyphens) so it was `"canopy-app"`. That internal identifier leaked directly into the macOS "About" menu item label.

## Root cause

`package.json` had `"name": "canopy-app"` but no `"productName"` field. Electron reads `productName` first and falls back to `name` if absent. Without `productName`, the display name was the npm-safe package name.

## Lesson

- Always set `"productName"` at the top level of `package.json` for Electron apps. This field controls `app.name`, which drives the macOS menu bar app name, the "About" dialog, and other OS-level UI strings.
- The `"name"` field must conform to npm naming conventions (lowercase, no spaces) — it's not suitable as a user-facing display name.
- **Watch out for data directory side effect**: changing `app.name` changes the default userData path (Electron appends `app.name` to the OS user data folder). After adding `productName`, hardcode the dev userData path in `environment.ts` to preserve existing dev environments (`app.getPath('userData')` would otherwise silently move to a new folder named after the new product name).

## Applies when

Setting up or renaming an Electron app. Always pair `productName` addition with an audit of any code that relies on `app.name` for filesystem paths.
