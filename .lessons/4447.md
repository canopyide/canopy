## Context

`CliInstallService` installs a `canopy` CLI symlink that points into the app's resource directory so users can invoke Canopy from the terminal. On Linux, this works by symlinking into `process.resourcesPath`.

## Problem

On AppImage distributions, `process.resourcesPath` points into an ephemeral FUSE mount path (e.g. `/tmp/.mount_Canopy<random>/usr/share/canopy/resources`). This path is different on every launch. Any symlink created against it becomes stale after the AppImage is closed and relaunched.

## Root cause

AppImage mounts itself at a temporary FUSE path at runtime via `$APPIMAGE`. `process.resourcesPath` reflects that ephemeral mount, not the AppImage file itself. The AppImage runtime exposes the actual file path via the `$APPIMAGE` environment variable (e.g. `/home/user/Canopy.AppImage`).

## Lesson

Never symlink directly into `process.resourcesPath` on AppImage. Instead, generate a small wrapper shell script at install time that resolves the AppImage path at runtime via `$APPIMAGE` env var:

```sh
#!/bin/sh
exec "${APPIMAGE:-$(command -v canopy-app)}" "$@"
```

Gate this behind an `isAppImage()` check (`process.env.APPIMAGE !== undefined`) so `.deb` and other non-AppImage installs are unaffected.

## Applies when

Any CLI tool or launcher that needs a stable, persistent reference to an AppImage executable's entry point — particularly on Linux with user-installed AppImages.
