## Context

Canopy's webview CSP included `'unsafe-eval'` in `script-src`, originally added because older Vite versions used eval-based HMR. The permission handler used a flat list without explicit categorization, with unknown permissions falling through without an explicit deny.

## Problem

Vite 6 no longer requires `'unsafe-eval'` for HMR — it switched to a non-eval approach. Keeping `'unsafe-eval'` in the CSP unnecessarily broadens the attack surface for webview content.

Also, the permission handler lacked a default-deny clause for unknown permission types. New Electron permission types added in future versions would be implicitly allowed rather than denied.

## Root cause

CSP entries added for a transient requirement (eval-based HMR) were never removed when the requirement went away. Permission handlers without explicit default-deny are open by default rather than closed.

## Lesson

**CSP**: Periodically audit `'unsafe-eval'` in webview CSP. Vite 6 HMR no longer requires it. Removing it is a safe, high-value hardening step with no functional regression.

**Permission handlers**: Structure Electron `setPermissionRequestHandler` and `setPermissionCheckHandler` with categorized permission sets (trusted, prompt-worthy, always-denied) and an explicit `default: return false` for unknown types. New Electron permission types should require explicit opt-in, not implicit allow.

Log all permission denials with partition context — this is critical for debugging CSP-related issues where silent failures are hard to trace.

## Applies when

Upgrading Vite (check if `'unsafe-eval'` was added for HMR and can be removed); reviewing Electron permission handlers; adding new webview features that request new permission types.
