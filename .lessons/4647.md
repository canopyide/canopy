## Context

Cross-project worktree contamination had been reported and "fixed" seven times before this PR (#2315, #2346, #2358, #2703, #2741, #2770, #4532). The singleton `WorkspaceService` utility process accumulated **16 distinct guard mechanisms** across 6 files to compensate for inherently shared mutable state. Each fix added another downstream layer; none addressed the architectural root cause.

## Problem

A singleton workspace host serves all windows and projects. It holds a single `projectScopeId`, a single `git` instance, and a single monitors map. When a user switches projects or opens a second window, shared mutable state causes cross-project contamination regardless of how many application-level guards are added.

## Root cause

Singleton processes with shared state cannot safely serve multiple projects. Client-side filtering (`windowScopes` map, generation counters, `isSwitching` flags, scope-checking, stale-closure detection) is fundamentally fragile — it requires every code path to be correct simultaneously under async race conditions.

## Lesson

- **The correct fix for shared-state contamination is isolation at the OS level, not more guards.** A separate utility process per project means contamination is impossible by construction — no amount of race conditions can leak state between separate processes.
- **Use the PTY host as the template**: `PtyClient` already manages a shared `utilityProcess` with per-window `MessagePort` distribution. The workspace host followed the same blueprint.
- **`WorkspaceClient` transforms from message router to process lifecycle manager**: spawning, health-checking (heartbeat), restart with backoff, and teardown. The renderer IPC contract stays unchanged.
- **Ref-counting for shared projects**: if two windows show the same project, they share one host via ref-counting rather than spawning two. Refcount on `attachWindow` / `detachWindow`.
- **Worker threads are not a substitute**: they share the parent's file descriptor limit (macOS soft 256), the libuv thread pool (4 threads), and provide no crash isolation. `utilityProcess` matches the existing PTY host pattern.
- Resource cost: ~30-50MB per active project. For 3-5 projects, 150-250MB — standard for a desktop IDE (VS Code Extension Host uses the same approach).

## Applies when

Any service that must serve multiple independent projects, workspaces, or user sessions — particularly when previous "guard" approaches have failed multiple times.
