## Context

`getTerminalSnapshot(id)` and `getAllTerminalSnapshots()` used a callback map keyed by terminal ID (or by the literal constant "all"). When the renderer made two concurrent snapshot calls for the same terminal — e.g. a hydration pass and a user-triggered refresh — the second call overwrote the first's callback, the first promise never resolved, and the timeout cleanup then deleted the second call's entry too.

## Problem

Snapshots intermittently hung or resolved with the wrong data. The bug was load-dependent so it was invisible in development and surfaced only when real users stacked requests.

## Root cause

Keying pending callbacks by a non-unique identifier (terminal ID) assumes at most one in-flight request per key. That assumption silently broke the moment a second caller arrived. Same-millisecond collisions made timestamp-based IDs unreliable too.

## Lesson

Every request-response IPC handshake needs a monotonically unique `requestId` generated by a broker (counter + PID, or UUID) on both sides. Store pending callbacks keyed by `requestId`, not by the resource they concern. Also track per-request timeouts in a parallel map so cleanup removes exactly the right entry. If you're tempted to key by resource ID because "we only ever have one request in flight", write the concurrent test first — you don't, and you will.

## Applies when

Building any main↔renderer or host↔client request/response layer, including PTY host, workspace host, plugin hosts, and MCP. Also applies to any async dedupe/cache layer where a second caller might arrive before the first completes.
