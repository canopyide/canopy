## Context

The Agent Settings "Re-check" button ran an async CLI availability probe and wrote the result into an in-memory cache. Rapid clicks, or a click that raced an in-flight probe, could cause a stale result to land after a newer one, so the UI showed outdated availability.

## Problem

Two async probes in flight simultaneously: A (older) and B (newer). B returns first and updates the cache. A returns later and overwrites the cache with its stale value. The user sees the older result permanently until they click again.

## Root cause

No ordering discipline between the issued request and the cache write. "Last write wins" is the wrong policy when writes arrive out of order.

## Lesson

For any async refresh that writes into a shared cache, issue a monotonically-increasing `checkId` when the request starts, capture it in closure, and only commit the result if that id still matches the current one at write time. Pattern:

```
const id = ++this.checkId;
const result = await probe();
if (id !== this.checkId) return; // stale, drop it
this.cache = result;
```

This is cheaper and simpler than AbortControllers when the work can't actually be cancelled (e.g. an external process probe). Also clear any `inFlightCheck` promise on forced refresh so a new request isn't dedup'd against a stale one.

## Applies when

Writing any service with a refreshable cache (CLI probes, GitHub rate-limit checks, project metadata, agent capabilities). Also when debugging "refresh button doesn't update UI" or "state reverts after a moment".
