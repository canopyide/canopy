## Context

Agent terminals didn't reliably turn green (working) when the user pressed Enter in the hybrid input bar. They looked idle even though the command had been sent.

## Problem

ActivityMonitor inferred busy transitions by watching `onInput()` character-by-character and matching heuristics (e.g. "a newline arrived, now we're working"). HybridInputBar submits a prompt in TWO async `write()` calls: one for the body, then one for Enter. The heuristic expected them to be contiguous but an output chunk from the agent could arrive between them, resetting the state machine. The submission was effectively invisible to the monitor.

## Root cause

Using input data streams as a proxy for "the user just submitted something" is fragile. PTY writes are not atomic with respect to the state machine; they race against the agent's own output. The actual semantic event ("user submitted a prompt") happens at a higher layer and should be signalled directly, not reverse-engineered from byte patterns.

## Lesson

When you need to detect a user-level action inside a byte-oriented stream, add a direct notification from the layer that knows — do not add more heuristics. `notifySubmission()` must be called both at enqueue time (for immediate UX feedback) AND at dequeue time (so queued submissions still fire once the previous one finishes, even if the state machine drifted to idle in between). Also sync any main-process state machine logic to the renderer worker equivalent (`AgentStateMachine` + `WorkerAgentStateService`) so the two do not disagree. Allow `idle→working` on input events as a safety net.

## Applies when

Adding or reviewing anything in `ActivityMonitor` / `AgentStateMachine`. Wiring a new input surface (command palette, context menu action, quick switcher) that submits to a terminal. Debugging "agent state is wrong" or "spinner did not appear" bugs. Adding new agent output heuristics — prefer direct notification calls where possible.
