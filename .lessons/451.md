## Context

Intermittently, `DEFAULT_AGENT_SETTINGS` and other exports from `shared/types/*` would be missing from the built `electron/` and `preload/` artifacts. The same clean build sometimes worked, sometimes didn't, with no code changes in between.

## Problem

Both `electron/tsconfig.json` and `electron/tsconfig.preload.json` had `shared/**/*.ts` in their `include` globs. When `tsc` ran them concurrently (or with stale incremental caches), two compilation processes raced to write `shared/...` output files into overlapping directories, producing torn or partial `.js` files at runtime.

## Root cause

`tsc` has no cross-process file locking. Two independent tsconfigs that emit to the same file paths will clobber each other under concurrent invocation. Incremental build caches (`.tsbuildinfo`) compound the problem by skipping rebuilds when they believe the output is current, even if another process just overwrote it.

## Lesson

When multiple tsconfigs share source files, convert the shared code to a **composite TypeScript project** with its own `outDir`, and use `references` from the consuming tsconfigs:

- `shared/tsconfig.json` becomes composite, builds first, writes to its own outDir.
- `electron/tsconfig.json` and `electron/tsconfig.preload.json` list `{ "path": "../shared" }` in `references` and drop `shared/**` from their own includes.
- Build scripts use `tsc -b` so dependency order and caching are handled by TypeScript itself.

Never let two tsconfigs emit into the same output location, even if they "happen to produce the same output" — incremental caches will lie to you eventually.

## Applies when

Any multi-target TypeScript build (main + preload + renderer + shared, or lib + tests + demo). If you see intermittent "export is undefined" or "module has no exports" errors that go away after `rm -rf dist && tsc`, suspect overlapping emit paths before blaming module resolution.
