## Context

Rapid agent state changes (multiple completions within 2 seconds) triggered sound storms: multiple `afplay`/`paplay` subprocesses spawning simultaneously, overlapping audio, and potential resource exhaustion.

## Problem

`SoundService` spawned a subprocess for each `play()` call with no concurrency control. With 10 agents completing simultaneously, 10 subprocesses launched at once.

## Root cause

No dampening layer existed. Each call was fully independent.

## Lesson

Four independent dampening mechanisms, layered in order:

**1. Debounce (150ms):** drop duplicate triggers of the same sound type within 150ms. Key is the base sound name, not the variant — this prevents bypass via cycling through variants of the same sound.

**2. Exponential volume decay:** track consecutive sounds within a 2s window. Each play at 70% of the previous volume (floor 10%). Resets after 2s silence. Implementation: `afplay --volume <0.0-1.0>` on macOS, `paplay --volume <0-65536>` on Linux (no-op on Windows). Volume is injected into the subprocess args, not a separate API.

**3. Max 3 concurrent voices with priority-based eviction:** maintain a pool of active voices. When full, evict the lowest-priority voice for a higher-priority incoming sound. Equal or lower priority sounds are dropped (not queued). Priority order: error > waiting > chime/complete > ping.

**4. Batch completion chording:** 2+ completion sounds within 2s triggers a single "chord" at full volume (overriding decay) using the canonical base sound. Gives a satisfying multi-agent completion cue without a pile-up.

**Preview bypass:** all dampening must be bypassed for preview playback in settings UI — users testing a sound expect it to play at full volume every time.

## Applies when

- Any feature that triggers audio as a side effect of rapid state changes
- Implementing subprocess-based audio on macOS/Linux where concurrent process limits matter
- Designing notification or feedback systems where user perception degrades with repetition
